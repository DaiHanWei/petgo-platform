# 保真验收标准（Fidelity Acceptance Standard）

> 2026-06-18 制定。背景：此前「截图验收通过」但实际与原型差异巨大（典型 #7 成长档案，自评 ~90% 实际 ~45%）。
> 根因：① 无真值图可比；② 验的是「我的改动渲染了吗」而非「成品像原型吗」；③ 盲信计划的错误基线；④ 确认偏误；⑤ 无 FAIL 判据。
> 本标准是所有 `pages/*.html` 保真工作的**唯一验收口径**。原型 HTML 是**绝对真值**。

---

## 0. 铁律

1. **先有真值图，再谈验收。** 没有 `reference/<x>.png` 不允许判「通过」。
2. **验成品，不验 diff。** 验收对象是「整屏 vs 原型」，不是「这次改了什么」。
3. **不信任何历史保真分。** 「已 ~90%」「小改即可」一律作废，逐屏重新基线。
4. **没有 FAIL 条件的验收 = 没验收。** 每屏必须能被判「驳回」。
5. **实现者不自证。** 终判由独立 reviewer（以原型为真值、默认 NEEDS WORK）出具。

---

## 1. 真值图产出（步骤 0）

工具（已验证可用）：`/tmp/fidtool/shot.mjs`（playwright-core + 本机 Google Chrome，截 `.phone` 元素）。

```bash
node /tmp/fidtool/shot.mjs \
  /Users/dai/work/petgo-platform/_bmad-output/pages/<x>.html \
  /Users/dai/work/petgo-platform/_bmad-output/fidelity/reference/<x>.png
```

- 输出 **1170×2532**（390×844 @3x），含状态栏/内容/底栏，与模拟器截图同构。
- `.phone` 不存在的页 → 退化整页截图（极少数）。
- 真值图全部存 `_bmad-output/fidelity/reference/`，随仓库留档。

## 2. 条件对齐（截 App 图前）

App 截图必须与原型**同条件**，否则差异不可比：
- **语言 = 印尼语**（原型语言）。验收用 locale=id。
- **视口** = iPhone 逻辑 390×844（iPhone 17 Pro 近似可用，记录实际机型）。
- **数据语义贴近原型**：原型有的条目类型（如健康事件红条、首个里程碑星标）在 mock 中都应出现，否则标注「数据态不可达」。
- 直达：`--dart-define=DEV_USER=true`（登录态）/ `DEV_VET=true`（兽医态）+ `DEV_ROUTE=/<route>`；弹窗/状态屏见覆盖矩阵的「可达方式」。

## 3. 逐元素 Rubric（步骤 2）

把原型按 DOM 拆成**区域**，逐区按 6 维判定。任一维不达标记一条 delta。

| 维度 | 判据 |
|---|---|
| ① 存在性 | 该区块在 App 中存在；**且 App 没有多出原型没有的区块** |
| ② 布局/对齐 | 位置、对齐方式（左/居中）、堆叠顺序一致 |
| ③ 组件类型 | 卡片 vs 内联、分段按钮 vs 图标、行式 vs 整图、网格列数…必须同型 |
| ④ 颜色 | 背景/文字/描边 hex 或 token 与原型一致（紫体系 #845EC9 等） |
| ⑤ 文案 | 逐字一致（印尼语）；占位文案也算 |
| ⑥ 尺寸/间距 | 字号、圆角、padding、间距量级一致（±20% 内算过） |

**评分**：勾选率 = 达标维度 / 总维度。每屏产出 delta 清单。

## 4. FAIL 门（步骤 3，硬性）

命中任一即**驳回**（无论勾选率多高）：
- 缺任一原型区块，或多出原型没有的区块（例：成长档案多了 "Days together" 卡）。
- **组件类型不符**（例：统计 2 内联 vs 原型 3 列块；时间线整图卡 vs 原型行式；切换小图标 vs 原型分段按钮）。
- 主结构/对齐不符（例：名片居中 vs 原型左对齐）。
- 颜色用错体系 / 文案语言或内容不符。

**通过 = 所有 FAIL 项清零 且 勾选率 ≥ 95%。** 否则 → NEEDS WORK，进重做清单。

## 5. 独立 reviewer（步骤 4，去确认偏误）

- 不由实现者放行。派独立子代理（建议 Evidence-Collector / Reality-Checker 风格），提示词固定：
  > 「你是保真审查员。原型图 `reference/<x>.png` 是唯一真值，App 图 `actual/<x>.png` 是被审对象。默认判 NEEDS WORK，逐区域列出**每一处**差异（按上面 6 维），只认有图证据的事实，不要替实现找借口。输出 delta 清单 + FAIL 命中项 + 勾选率 + 通过/驳回。」
- reviewer 只看图、不看代码，避免被实现意图带偏。

## 6. 留档（步骤 5）

每屏一份 `_bmad-output/fidelity/<x>.md`：

```
# <x>.html 保真验收
- reference: ./reference/<x>.png
- actual:    ./actual/<x>.png   （机型/locale/路由/日期）
- rubric 表（逐区 6 维 √/✗）
- delta 清单
- FAIL 命中：[...]
- 勾选率：NN%　判定：通过 / 驳回
```

## 7. 流程钩子

- 任何屏声明「完成」前，必须附 `fidelity/<x>.md` 且判定=通过。
- 计划/审计里的保真分若无对应 `fidelity/<x>.md` 支撑，视为**未验证**，不得作为「小改即可」的依据。
