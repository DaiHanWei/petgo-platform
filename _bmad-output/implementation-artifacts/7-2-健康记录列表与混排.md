# Story 7.2: 健康记录列表与混排

Status: review

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

> **所属**：V1.1 Epic 7 第二个 Story（**全栈**）。承接 7-1（health_records CRUD）。交付：① **后端**——扩展 `GET /pet-profiles/me/health-records` 为**混排**（结构化记录 `editable=true` + 问诊存档 `health_events` ARCHIVED `editable=false` 只读，按 event_date 倒序）+ **里程碑第四触发路径**（健康记录创建自动完成对应里程碑，FR-45C）；② **前端**——健康记录列表页 `p-health-list`（可编辑结构化 + 🏥问诊只读混排，`editable` 区分可点，UX-DR10）。
> **里程碑映射（本 Story 设计决策，规格未细化取合理默认）**：`VACCINE`→里程碑 **M3**（完成第一次疫苗接种）、`DEWORM`→**M4**（完成第一次驱虫）；`MENSTRUATION`/`NEUTER`/`CUSTOM` 无对应目录节点 → 不触发。既有 M3/M4 亦可经打卡完成（USER_CHECKIN），本路径为**第四条**（健康记录创建）——幂等：首次完成即可，重复创建不重复完成。

## Story

As a **用户**,
I want **健康记录与问诊存档只读混排**,
so that **我在一处看全宠物健康时间线（FR-45B/45C，UX-DR10）**。

## Acceptance Criteria

> **验证层**：**L0**（混排 DTO/映射单测、`flutter analyze`/widget）· **L1**（混排端点真 pg：两源合并倒序 + editable 区分 + 里程碑第四路径完成）· **L2**（模拟器健康列表视觉 + 编辑/只读区分）。

### AC1 — 混排列表：结构化可编辑 + 问诊只读

**Given** 结构化记录（`editable=true`）+ 🏥问诊存档（`health_events` ARCHIVED，`editable=false` 只读，来自 FR-16）
**When** `GET /api/v1/pet-profiles/me/health-records`
**Then** 返回**两源合并**、按 `event_date` 倒序（同日按稳定次序）；每项带 `editable` 标志（结构化 true、问诊 false）；**问诊条目不入 `health_records` 表**（仅只读混排，运行时合并）
> 验证层：**L1**（造结构化记录 + ARCHIVED 问诊事件 → 合并倒序、editable 正确、问诊不落 health_records 表）+ **L0**（HealthEvent→只读 item 映射 + 排序单测）。

### AC2 — 健康记录触发里程碑第四路径（FR-45C）

**Given** 里程碑 M3（疫苗）/M4（驱虫）+ 健康记录创建
**When** 创建 `VACCINE`/`DEWORM` 记录
**Then** 自动完成对应里程碑（VACCINE→M3、DEWORM→M4，`SYSTEM_AUTO`）——**第四触发路径**（补既有打卡路径）；**幂等**（首次完成，重复创建不重复完成）；其它类型不触发
> 验证层：**L1**（创建 VACCINE → M3 完成；重复创建不重复完成；DEWORM → M4；CUSTOM 不触发）+ **L0**（listener purpose/type 分派单测）。

### AC3 — 前端健康列表页（editable 区分可点）

**Given** 健康列表页 `p-health-list`（UX-DR10）
**When** 渲染
**Then** 结构化记录可点编辑/删除、问诊条目只读（不可编辑，🏥 标识）；新增/编辑结构化记录（type/date/name/note）；空态引导
> 验证层：**L0**（editable 分支 widget：结构化有编辑入口、问诊无；新增表单调 CRUD）+ **L2**（模拟器视觉 + 编辑/只读区分）。

## Dev Notes

### 混排数据源（同 profile 域，直用 repo）

- 结构化：`health_records`（7-1，`HealthRecordRepository`）editable=true。
- 问诊存档：`health_events`（V5，`HealthEventRepository.findByPetId` → 过滤 `archiveDecision=ARCHIVED`）editable=false 只读；字段 `symptomSummary`/`aiLevel`/`eventDate`。**症状/建议属健康数据，只读展示，不外泄明文到日志**。
- 两源均在 profile 域，`HealthRecordService` 可直接注入两 repo 合并（不跨模块，无需 timeline 的 source 接口）。按 `eventDate` 倒序合并。

### 里程碑第四路径（照既有事件订阅范式）

- 既有 `MilestoneAutoCompleteListener` 用 `@Async @TransactionalEventListener`（AFTER_COMMIT）订阅领域事件 → `MilestoneCompletionService.completeForOwner(ownerId, suffix, source)`（suffix 如 "M3"，按宠物类型解析 C-/D-；唯一约束保幂等）。本 Story 加 `HealthRecordCreatedEvent` 一路，**禁 MQ/新中间件**。
- 映射：VACCINE→M3、DEWORM→M4（合理默认，规格未细化）。M3/M4 亦可打卡完成——第四路径与打卡**互不冲突**（唯一约束幂等，先到先完成）。
- **AFTER_COMMIT**：健康记录 create 事务提交后才发（里程碑完成是非关键、可最终一致；与资金入账的同步 MANDATORY 相反——里程碑照既有 8.3 用 AFTER_COMMIT+@Async）。

### 强制护栏

- 健康数据 PII：日志不落 symptom/advice/note 明文。
- 里程碑完成幂等（唯一约束）；禁 MQ/Quartz/缓存/新中间件（8.3 护栏延续）。
- 问诊条目**不入 health_records 表**（运行时合并，只读）。
- App 不渲染后端显示串：type 前端按 code 本地化（memory `petgo-i18n-model-and-debt`）。
