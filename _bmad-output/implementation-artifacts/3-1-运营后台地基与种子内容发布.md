# Story 3.1: 运营后台地基与种子内容发布

Status: ready-for-dev

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

## Story

As a **运营人员**,
I want **一个受保护的后台用于平台上线前发布种子内容**,
so that **用户首次打开时 Feed 不为空、有内容氛围**。

> 本 Story 是 Epic 3 的**地基故事，后端重**。它首建架构 Gap G-1 补齐的 **`com.petgo.admin/` slice**（`role=ADMIN` 受保护端点 + 极简 Thymeleaf 后台 shell），并在其上实现**种子内容发布**（FR-18）。这套 Admin shell 是横切设施：后续 **Story 3.7（举报审核队列）、Epic 5（兽医账号 CRUD/评分查看/封禁）** 都复用同一 slice，不再各自造后台。
>
> **做**：① ADMIN 角色门控（JWT `role=ADMIN`，与 user/vet 隔离）；② Thymeleaf 登录页 + 后台框架页（导航壳，预留举报/兽医/评分入口位）；③ 种子内容发布表单（三类内容类型）→ 写入 `content_posts` 全平台内容池；④ 种子内容与用户内容**完全混排、不打标记**。
> **不做**：举报审核队列处理界面（Story 3.7 在本 shell 上叠加）；兽医账号 CRUD / 评分查看 / 封禁（Epic 5）；Feed 读取与硬过滤（Story 3.2）；点赞/评论/详情（Story 3.3~3.5）。
> **依赖前序**：Story 1.1（脚手架 + `shared/security` + ProblemDetail + Flyway）、Story 1.3（JWT 签发与角色 claim 机制 + `users` 表）、Story 2.1（媒体 STS/公开桶——种子内容图走公开桶）、Story 2.3（`content_posts` 表与内容写入路径——本 Story **复用** content service 的写入能力，**不重复建表**；若 Story 2.3 已建表，本 Story 仅追加 ADMIN 写入路径）。

## Acceptance Criteria

> **验证层标注**：每条 AC 末尾标注验证层级与所需环境——
> **L0 静态**（编译/lint/MockMvc 切片，无需真实 DB） · **L1 集成/运行时**（需 Docker daemon + postgres + redis，验真实写库/门控/迁移） · **L2 端到端/外部凭证**（需真实第三方或真机）。
> 本 Story 全部 AC 落在 **L0 / L1**——Admin 是纯后端 + Thymeleaf 服务端渲染，无 Flutter 侧，无外部第三方依赖（种子图可用已上传的公开桶 URL 占位，不强制真跑 OSS）。

### AC1 — Admin slice 与 ADMIN 角色门控首建

**Given** 后端
**When** 建立 `admin` slice（`role=ADMIN` 门控 + 极简 Thymeleaf 后台）
**Then** 仅 ADMIN 账号可访问后台端点，复用各模块 service，不引入独立系统（G-1 落点）
**And** ADMIN 后台地基在此首建，后续举报审核/兽医 CRUD/评分查看挂同一 slice
> 验证层：**L1**（须真实 Spring Security 上下文 + JWT 校验链才能验「非 ADMIN→403、ADMIN→200」；非 ADMIN 含 user/vet/游客三态都要拦）。门控配置的存在性、URL 匹配规则可在 **L0** 用 `@WebMvcTest` + mock JWT 切片验证。

### AC2 — 种子内容发布并与用户内容混排不标记

**Given** 运营在后台
**When** 发布种子内容（三类内容类型）
**Then** 种子内容进入全平台内容池，与用户内容在 Feed 中混排、不做特殊标记区分（FR-18）
**And** 种子内容支持三类内容类型（日常分享 / 专业知识科普 / 成长日历快乐时刻），写入与用户内容**同一张 `content_posts` 表、同一套字段**，仅 `author_id` 指向运营账号；Feed 读取侧无任何「种子/官方」标识字段
> 验证层：**L1**（须真实写 `content_posts` 并断言落库行与用户帖结构一致、无区分标记列；混排效果在 Story 3.2 Feed 读取处验，本 Story 只验「写入即进同一池、无标记」）。表单字段校验（1000 字计数、图片 ≤9 张）可在 **L0** MockMvc 验。

## Tasks / Subtasks

> **按"后端子任务 / 前端子任务 / 联调验收"三段组织**。本 Story **前后端分布严重偏后端**：前端（Flutter）**无任何工作**（Admin 是 Thymeleaf 服务端后台，不在 App 内）；"前端"段记录的是 Thymeleaf 模板侧。建议执行顺序：后端门控 → Thymeleaf shell → 种子发布 → 联调。

### 🟦 后端子任务（petgo-backend / Spring Boot — admin slice）

- [ ] **B1. 建 `com.petgo.admin/` slice 包骨架** (AC: 1)
  - [ ] `com.petgo.admin/{web,service,domain,repository,dto}`（admin 是横切 slice，**复用各模块 service，自身不重复建业务表**）。
  - [ ] `AdminWebController`（Thymeleaf 页面控制器，返回视图名）与（如需）`AdminApiController`（表单 POST 处理）分离；区别于 `/api/v1/**` 的 JSON 控制器，Admin 走 `/admin/**` 服务端渲染。
- [ ] **B2. ADMIN 角色门控** (AC: 1)
  - [ ] 在 `shared/security/SecurityConfig` 增加 `/admin/**` 规则：要求 `role=ADMIN`（JWT claim，复用 Story 1.3 的 `JwtAuthFilter`/`RoleGuard`）；user/vet/游客访问 `/admin/**` 一律 **403 越权**（非 401，已登录但角色不符）；未登录访问后台页面跳 Admin 登录页。
  - [ ] ADMIN 账号来源：复用 `users` 表 + role 字段，或独立最小化方式由运营在库内置（本 Story 不做 ADMIN 注册 UI，账号由 Flyway 种子/手工入库，密码 BCrypt；在 Dev Notes 记录开户方式）。
  - [ ] **护栏**：ADMIN 端点绝不暴露在公开 `/api/v1`；`/admin/**` 与用户/兽医路由完全隔离（单 App 角色门控延伸到后台）。
- [ ] **B3. 极简 Thymeleaf 后台 shell** (AC: 1)
  - [ ] `src/main/resources/templates/admin/login.html`（账密登录）+ `admin/layout.html`（侧边导航壳）+ `admin/dashboard.html`（首页）。
  - [ ] 导航壳预留入口位（**占位不实现**）：「种子内容发布」（本 Story 实现）、「举报队列」（Story 3.7）、「兽医账号」「评分查看」（Epic 5）——预留可见但后三者链接指向占位页或 disabled。
  - [ ] 极简即可：无需设计 token / 无障碍门槛（那是 App 侧要求），后台只求可用。
- [ ] **B4. 种子内容发布表单与写入** (AC: 2)
  - [ ] `templates/admin/seed-post.html`：内容类型下拉（DAILY / KNOWLEDGE / GROWTH_MOMENT 三类，落库 UPPER_SNAKE）+ 文字输入（≤1000 字符，服务端权威校验）+ 图片 URL 列表（≤9 张；种子图为已传公开桶的 URL，本 Story 不做后台直传 UI，URL 由运营粘贴或 Story 2.1 STS 预上传）。
  - [ ] `AdminContentService.publishSeed(...)` **复用 `content` 模块的 `ContentService` 写入接口**（经 service 调用，**禁止 admin 直接访问 content repository**——跨模块只经 service，架构 §模块边界）。
  - [ ] 写入 `content_posts`：`author_id` = 运营 ADMIN 账号；`type` ∈ {DAILY,KNOWLEDGE,GROWTH_MOMENT}；`visibility` 默认全平台公开；**无任何「seed/official」区分列**——与用户帖结构完全一致（AC2 关键）。
  - [ ] 若 `content_posts` 表尚不存在（Story 2.3 未先落地），本 Story 负责经 content 模块建表；若已存在则仅追加写入路径，**不改表结构**。
- [ ] **B5. 错误与日志规范** (AC: 1, 2)
  - [ ] Admin API/表单错误统一走 ProblemDetail（JSON 端）或后台页面内联错误（Thymeleaf 表单回显）；403 越权语义正确。
  - [ ] 日志 MDC 带 `adminUserId`；**严禁记录 PII/令牌**；种子发布记 INFO 业务事件（含 postId、type，不含图 URL 明文若涉私密——种子均公开桶可记）。

### 🟩 前端子任务（Thymeleaf 模板侧；Flutter App 无工作）

- [ ] **F1. Thymeleaf 模板实现** (AC: 1, 2)
  - [ ] 实现 B3/B4 列出的 `login.html` / `layout.html` / `dashboard.html` / `seed-post.html`；文字计数可用纯服务端校验 + 表单回显，无需前端 JS 框架（极简）。
  - [ ] **明确**：本 Story **不触碰 `petgo_app/`（Flutter）任何文件**——Admin 后台不在移动端内。Epic 3 的 Flutter 工作从 Story 3.2 开始。

### 🟨 联调验收子任务（端到端门控 + 写入 + 混排前置）

- [ ] **J1. 角色门控验收** (AC: 1 / **L1**)

## Dev Notes

### 关键架构约定（本 Story 必须落实）

**Admin slice 定位**（架构 §Gap G-1 / §Project Structure）：
- `com.petgo.admin/` 是**横切 slice 非独立系统**：`role=ADMIN` 受保护端点 + 极简 Thymeleaf 后台，**复用各模块 service**承载运营能力。本 Story 首建 shell + 种子发布；后续 FR-25 举报审核（3.7）、FR-29 兽医 CRUD、FR-33 评分查看、FR-19 封禁（Epic 5）全部挂此 slice，**不做第二个后台**。
- 路由分区：用户/兽医走 `/api/v1/**`（JSON）；Admin 走 `/admin/**`（Thymeleaf 服务端渲染）；H5 名片走 `/p/{cardToken}`（Story 2.6）。三者互不重叠。

**命名映射链**（架构 §Naming，全程沿用）：DB `snake_case` ↔ Java/Dart `camelCase` ↔ JSON `camelCase`。
- `content_posts` 表：复数 snake_case；主键 `id`(bigint)；`author_id`、`created_at`/`updated_at`(timestamptz UTC)、软删 `deleted_at`；`type` 枚举落库 `varchar` UPPER_SNAKE ∈ {DAILY, KNOWLEDGE, GROWTH_MOMENT}。
- 内容类型术语对齐 FR-12：日常分享=DAILY、专业知识科普=KNOWLEDGE、成长日历快乐时刻=GROWTH_MOMENT。

**ProblemDetail（RFC 9457）**：Admin 的 JSON 端点错误统一信封（type/title/status/detail/instance/traceId）；403 越权、401 未认证语义不混用。

**游标分页**（Feed 契约，本 Story 不直接用但需知晓）：列表统一 `{items, nextCursor, hasMore}`，20 条/批——Story 3.2 用，Admin 后台列表可用简单分页。

### 强制护栏（架构 §Enforcement —— 违反即返工）

- **禁止擅自引入 MQ / 通用缓存层 / 新中间件**；种子发布是同步写库，不需异步。
- `ddl-auto=validate`，schema 归 Flyway（若需建/改 `content_posts` 用 `V<序号>__*.sql`）。
- 模块间**只经 service 接口或领域事件通信，禁止 admin 直接访问 content repository**——`AdminContentService` 调 `ContentService`。
- ADMIN 端点绝不混入公开 `/api/v1`；对外暴露标识用不可枚举 token（本 Story 后台内部用 id 即可，不对外）。
- 凭证 env 注入绝不入库；ADMIN 密码 BCrypt。
- 注销内容匿名化（NFR-8）：种子内容 author 为运营账号、不涉注销；但写入路径需与用户内容一致以便 Feed 统一处理匿名化（在 Story 3.2/7.3 体现）。

### 范围边界（防 scope creep —— 本 Story 明确不做）

- ❌ 不做 Feed 读取/硬过滤/瀑布流（Story 3.2）。
- ❌ 不做举报队列处理 UI（Story 3.7，仅预留入口位）。
- ❌ 不做兽医账号 CRUD / 评分查看 / 封禁（Epic 5，仅预留入口位）。
- ❌ 不做点赞/评论/详情/删除（Story 3.3~3.6）。
- ❌ 不碰 Flutter App 任何文件。
- ❌ 不做后台直传图 UI（种子图用公开桶已传 URL；如需上传走 Story 2.1 STS）。
- ✅ 只做：admin slice + ADMIN 门控 + Thymeleaf shell（含后续入口位预留）+ 种子内容三类发布写入同一内容池不打标记。

### Project Structure Notes

- 后端：`com.petgo.admin/{web,service,domain,repository,dto}`（横切 slice）；门控在 `com.petgo.shared/security/`（SecurityConfig 增 `/admin/**` 规则）；Thymeleaf 模板在 `src/main/resources/templates/admin/`；内容写入复用 `com.petgo.content/service/ContentService`。
- `content_posts` 表归属 `content` 模块；Admin 经 service 写入，不拥有该表。
- 前端：**无** `lib/features/content/` 或任何 Flutter 改动（与黄金模板 Story 1.1「双产物」不同，本 Story 仅后端）。
