# Story 1.6: 新用户引导——昵称确认与宠物状态选择

Status: review

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

## Story

As a **新用户**,
I want **登录后确认昵称并选择我的宠物状态**,
so that **App 能据此定制我的首页内容方向**。

> 本 Story 实现 **新用户引导流的前两步**（昵称确认 + 宠物状态选择），消费 Story 1.3 的新用户分流信号。范围：① 昵称确认页（默认读取 Google Display Name 可改，≤20 字，超出禁止继续）；② 宠物状态选择页（A 我有宠物 / B 暂无但计划养 / C 宠物爱好者，必选不可跳过，写 `users.pet_status`）；③ 分叉——选 A → 进档案创建引导（Story 1.7）；选 B/C → 直接进首页且不显示档案提示条；④ 已完成引导用户后续在「我的」/「成长档案」改状态时，状态更新成功且首页内容方向据新状态即时调整（与 FR-17 联动）。
>
> **依赖**：Story 1.3（`users` 表含 `nickname`/`pet_status`/`onboarding_completed`、新用户分流信号、JWT）、Story 1.2（外壳 + i18n + token）、Story 1.4（引导完成后回跳 pendingAction）。
> **被依赖**：Story 1.7（选 A 进档案创建引导 + 提示条逻辑依赖 `pet_status==A && !onboarding/档案完成`）、Epic 3 FR-17（按 `pet_status` 硬过滤 Feed）、Epic 7 FR-20/21（「我的」/「成长档案」改状态复用本 Story 状态选择界面）。
>
> **不做**：档案创建引导页与首页提示条（Story 1.7）、档案表单本体（FR-11，Epic 2）、Feed 硬过滤实现（FR-17，Epic 3，本 Story 只保证状态变更触发「首页应刷新」信号）。

## Acceptance Criteria

> **验证层标注**：每条 AC 末尾标注验证层级与所需本地环境——
> **L0 静态**（编译/lint/widget/MockMvc） · **L1 集成/运行时**（需 Docker+postgres+redis，验 `users` 字段持久化） · **L2 端到端/外部凭证**（需真实 Google + 真机/模拟器走新用户全流程）。
> 本 Story 前端表单/路由分叉为 **L0**；昵称/状态写库为 **L1**；新用户首登完整引导为 **L2**。

### AC1 — 昵称确认（默认 Display Name 可改，≤20 字）

**Given** 新用户首次 Google 授权成功（由 Story 1.3 分流至此）
**When** 进入昵称确认页
**Then** 昵称默认读取 Google Display Name 且可修改，限制 ≤20 字（超出禁止继续），确认后进入宠物状态选择（FR-0E）
> 验证层：**L0**（前端 widget：默认填充 displayName、实时计数、>20 字禁「继续」；后端 MockMvc：`PATCH` 昵称 >20 → 422 ProblemDetail、≤20 → 200）+ **L1**（昵称写入 `users.nickname` 持久化，需 postgres）。

### AC2 — 宠物状态必选（A/B/C 写 pet_status）+ 分叉

**Given** 昵称确认完成
**When** 进入宠物状态选择页
**Then** 必选一项（A 我有宠物 / B 暂无但计划养 / C 宠物爱好者），不可跳过；选择写入用户 `pet_status`（FR-0F）
**And** 选 A → 进入档案创建引导（Story 1.7）；选 B/C → 直接进入首页且不显示档案提示条
> 验证层：**L0**（前端 widget：三选一必选、未选禁继续、不可跳过；分叉路由 A→档案创建引导占位(Story 1.7)、B/C→首页且无提示条；后端 MockMvc：写 `pet_status ∈ {A,B,C}`、非法值 422、写状态同时置 `onboarding_completed=true`）+ **L1**（`pet_status` 持久化）。

### AC3 — 后续改状态即时联动首页

**Given** 已完成引导的用户
**When** 后续在「我的」或「成长档案」修改宠物状态
**Then** 状态更新成功，首页内容方向据新状态即时调整（与 FR-17 联动）
**And** 若本次为 **B/C → A** 的状态变更，则**同样触发 FR-0G 宠物档案创建引导**（逻辑与注册时选 A 完全复用，Story 1.7），而非仅刷新首页信号
> 验证层：**L0**（前端：改状态成功后发出「首页应按新状态刷新」信号——本 Story 提供状态选择界面 + 复用写状态端点 + 刷新信号；**B/C→A 时复用 Story 1.7 建档引导入口/提示条逻辑**；FR-17 实际硬过滤在 Epic 3）+ **L1**（`PATCH pet_status` 更新持久化）。⚠️「我的」/「成长档案」入口本体分别在 Epic 7/Epic 2，本 Story 产出**可复用的状态选择界面 + 写状态端点 + 刷新信号**，供其复用（FR-21/FR-0F 共享界面）。
> 🔄 **PRD V1.0.0 修订（FR-0F · 2026-06-08）：** B/C→A 切换语义已转正——运行期改状态选 A 与注册时选 A **等价**，均进 FR-0G 建档引导（复用 Story 1.7 引导路由/跳过/提示条逻辑），不只是刷新首页信号。提示条计数沿用 A 态既有持久计数（不重置），见 Story 1.7 AC2。

## Dev Notes

### 关键架构约定（本 Story 相关部分）

**数据（架构 §Data / Story 1.3 `users` 表）**：`nickname varchar`（≤20）、`pet_status varchar NULL`（`A|B|C`）、`onboarding_completed boolean`。本 Story **不新建表**，只 UPDATE 这些字段（Story 1.3 已建 `users`）。枚举落库 `varchar` + UPPER（`pet_status` 用 `A/B/C` 字面，符合 PRD 语义；如需可记为 `HAS_PET/PLANNING/ENTHUSIAST`，**本 Story 统一用 `A/B/C` 与 PRD 一致**，Completion Notes 注明）。

**API（架构 §API/Naming）**：`PATCH /api/v1/me`（**当前用户主体统一端点**——全平台「当前用户」一律用 `/api/v1/me`，不用 `/users/me`，与 1.7/2-4/7-1/7-3 对齐）；body camelCase（`nickname`/`petStatus`）；422 校验 ProblemDetail；`Accept-Language` 驱动错误文案双语。仅作用当前 JWT 用户。

**校验时机（架构 §Process Patterns）**：服务端 Bean Validation 权威（`@Size(max=20)`、枚举校验）；客户端实时预校验仅体验（字数计数）。

**前端结构**：引导页 `features/auth/presentation/{nickname_page, pet_status_page}`；`PetStatusSelector` 做成可复用组件（FR-0F/FR-21/FR-20 共享，架构 FR-21「复用 FR-0F 状态选择界面」）；`pet_status` 入 `shared_preferences`。

**FR-17 联动**：改状态即时刷新首页内容方向——本 Story 只**发信号**，硬过滤（A/C 全显、B 不显成长日历、未登录全显）在 Epic 3 实现。

### 强制护栏（架构 §Enforcement —— 违反即返工）

- **`pet_status` 复用同一选择界面**（FR-0F/FR-21/FR-20）——做成单一可复用组件，禁止三处各写一份（防不一致）。
- 服务端校验权威：昵称 ≤20、状态枚举合法在后端强制（客户端计数仅体验）。
- 端点仅作用当前 JWT 用户，**禁止接受任意 userId**（防越权改他人资料）。
- 无写死 UI 文案 / 无硬编码视觉常量（沿用 1.2 token + i18n）。
- 凭证/PII 不入日志；不引入新中间件。

### 范围边界（防 scope creep —— 本 Story 明确不做）

- ❌ 不实现档案创建引导页 / 首页档案提示条（Story 1.7）——选 A 仅路由到其占位。
- ❌ 不实现宠物档案表单本体（FR-11，Epic 2）。
- ❌ 不实现 Feed 硬过滤（FR-17，Epic 3）——仅发「首页应刷新」信号。
- ❌ 不实现「我的」/「成长档案」页本体（Epic 7/Epic 2）——仅产出可复用 `PetStatusSelector` + 写状态端点供其复用。
- ❌ 不实现 Google 登录/分流本体（Story 1.3）——仅消费分流信号。
- ✅ 只做：昵称确认页 + 宠物状态选择（可复用组件）+ 分叉路由（A/B/C）+ onboarding 完成 + 改状态刷新信号 + 对应后端 PATCH/GET 端点。

### 页面多态完整性（本 Story 相关态）

- 昵称为空/全空格 → 禁继续（或回退 displayName，按实现，禁止空昵称写库）。
- 昵称 >20 字 → 客户端禁继续 + 服务端 422 双保险。
- 状态未选 → 禁「完成」（不可跳过）。
- 写昵称/状态网络失败 → 顶部 inline banner 重试（UX-DR10），引导停留不丢已填，不崩。
- 已 onboarding 用户被错误重新路由到引导 → 由分流逻辑（Story 1.3 `onboardingCompleted`）拦截，不重复引导。
