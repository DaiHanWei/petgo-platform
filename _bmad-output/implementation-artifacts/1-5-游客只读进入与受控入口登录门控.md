# Story 1.5: 游客只读进入与受控入口登录门控

Status: review

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

## Story

As a **未登录访客**,
I want **不登录就能进入 App 浏览首页**,
so that **我在被要求注册前先体验内容、降低使用门槛**。

> 本 Story **接线游客态门控**，把 Story 1.4 的登录引导组件挂到真实触发源上。范围：① 游客可进入并滚动首页 Tab（FR-0A；Feed 实际内容由 Epic 3 填充，本 Story 为可滚动容器 + 空状态占位）；② 未登录点击受控 Tab（成长档案/[+]/问诊/我的）或受控入口（问诊/发布/创建档案）→ 检测未登录 → 调 Story 1.4 **强弹窗**，不进入目标（FR-0C 触发侧 + FR-19 门控）；③ 客户端 dio 拦截器对受控 API 的 401 统一进入 Story 1.4 登录引导流；④ 后端确保未登录写操作被拒（401 ProblemDetail），游客只读端点 permitAll。登录成功后回到触发点（复用 Story 1.4 回跳）。
>
> **依赖**：Story 1.2（Tab 外壳——本 Story 把「五 Tab 全可点」收紧为「仅首页游客可访问」）、Story 1.3（登录态/JWT/SecurityConfig/dio 拦截器）、Story 1.4（强弹窗 + 回跳 + pendingAction）。
> **被依赖**：Epic 2/3/4/5 各受控入口复用本 Story 的门控 helper；Epic 3 Feed 在首页可滚动容器内填充真实内容并接软浮层滚动触发。
>
> **不做**：软浮层滚动触发（Epic 3 提供首页滚动深度）、Feed 真实内容、任何受控功能本体——只做「拦截 + 弹强弹窗 + 401 接线 + 游客只读容器」。

## Acceptance Criteria

> **验证层标注**：每条 AC 末尾标注验证层级与所需本地环境——
> **L0 静态**（编译/lint/widget/MockMvc） · **L1 集成/运行时**（需 Docker+postgres+redis，验后端 401/permitAll 真实生效） · **L2 端到端/外部凭证**（需真实 Google + 真机/模拟器走完整门控→登录→回跳）。
> 本 Story 前端门控/路由为 **L0**；后端「未登录写操作被拒/游客只读放行」需 **L1**；完整门控→登录→回跳为 **L2**。

### AC1 — 游客只读进入首页 + 后端拒绝未登录写

**Given** 未登录用户打开 App
**When** 落在首页 Tab
**Then** 首页内容区对游客可见、可滚动（FR-0A；Feed 实际内容由 Epic 3 填充，此处为可滚动容器 + 空状态占位）
**And** 未登录状态下任何写操作请求被后端拒绝（401/ProblemDetail）
> 验证层：**L0**（前端：未登录可进首页、可滚动容器 + 空状态占位渲染，widget test）+ **L1**（后端：无 JWT 访问任意写端点 → 401 ProblemDetail；游客只读端点放行——需 postgres/SecurityConfig 真实生效；本 Story 用一个最小受保护写端点 + 一个游客只读占位端点验证门控对称性）。

### AC2 — 受控 Tab/入口门控 + dio 401 → 登录引导 → 回跳

**Given** 未登录用户
**When** 点击成长档案 / [+] / 问诊 / 我的 任一受控 Tab，或点击问诊/发布/创建档案等受控入口
**Then** 系统检测到未登录态并拦截，调用 Story 1.4 的强登录引导弹窗，不进入目标功能（FR-0C 触发侧、FR-19 门控）
**And** 客户端 dio 拦截器对受控 API 的 401 统一进入 Story 1.4 登录引导流；登录成功后回到触发点（复用 Story 1.4 回跳）
> 验证层：**L0**（前端 widget/单元：未登录点 4 个受控 Tab + 受控入口 → 弹强弹窗 + 不进入目标 + 注入 pendingAction；dio 拦截器 401（refresh 也失败后）→ 进登录引导流而非静默吞掉）+ **L2**（真机：门控→真实 Google 登录→回到触发点）。

## Dev Notes

### 关键架构约定（本 Story 相关部分）

**鉴权与门控（架构 §Authentication & Security / §Architectural Boundaries）**：
- `/api/v1/**` 默认需 JWT；**例外 = Feed/详情只读对游客可见**（FR-0A/17）。本 Story 落实这条「读放行/写拒绝」边界。
- `/p/{cardToken}` 公开 H5 名片放行（Epic 2，本 Story 不涉及但 SecurityConfig 已含）。
- 授权按 `role` claim 门控；游客态贯穿 Feed/详情只读。
- **鉴权流（架构 §Process Patterns）**：401 → refresh 续期一次 → 重放；失败则落游客态/登录引导（FR-0C）；**游客触发受控功能即弹 FR-0C**。

**FR-19 门控**：底部 Tab Bar 仅首页未登录可访问，其余触发 FR-0C；一级页常显。本 Story 把 Story 1.2 的「全可点」收紧为此规则。

**前端结构**：门控 helper 在 `features/auth/domain`；首页占位容器在 `features/content`（Epic 3 落点）或 Story 1.2 占位；强弹窗复用 `shared/widgets/login_hard_dialog`（Story 1.4）；拦截器在 `core/network/auth_interceptor`（Story 1.3）。

**错误规范**：401 未认证（→FR-0C 登录引导）与 403 越权（→友好提示+返回，UX-DR18 ⑤）**严格区分不混用**。

### 强制护栏（架构 §Enforcement —— 违反即返工）

- **安全规则只升不降**：本 Story 在 1.3 基础上**进一步收紧**前端 Tab 门控，不得放松任何 1.3 已设的后端门控。
- **单一门控入口**：所有受控点经 `requireLogin` helper / go_router guard，禁止在各页散落 `if (!loggedIn) ...` 重复判断（防 Epic 间不一致）。
- **401 vs 403 不混用**：登录拦截走 FR-0C；越权走友好提示。
- 无写死 UI 文案 / 无硬编码视觉常量（沿用 1.2 token + i18n）。
- 不擅自引入新中间件。

### 范围边界（防 scope creep —— 本 Story 明确不做）

- ❌ 不做软浮层滚动触发（Epic 3 提供首页滚动深度后接 FR-0B）——本 Story 只接强弹窗门控（FR-0C）。
- ❌ 不实现 Feed 真实内容（Epic 3）——首页仅可滚动容器 + 空状态占位。
- ❌ 不实现任何受控功能本体（问诊/发布/创建档案/成长档案/我的）——仅拦截 + 弹窗 + 注入 pendingAction。
- ❌ 不实现强弹窗/回跳本体（Story 1.4）——仅接线触发与 pendingAction 注入。
- ❌ 不实现 Google 登录/JWT/dio 拦截器本体（Story 1.3）——仅消费与接线。
- ✅ 只做：游客只读首页容器 + 受控 Tab/入口门控接线 + dio 401→引导接线 + 后端门控对称性收口（读放行/写拒绝）。

### 页面多态完整性（本 Story 相关态，UX-DR18）

- 游客首页空状态（②空）：Feed 未填充时显示 `empty_state` 占位（UX-DR8），不白屏。
- 未登录写请求 401（登录类拦截）→ 走 FR-0C 登录引导（**不与 403 越权混用**）。
- refresh 彻底失效（会话过期）→ 落游客态 + 后续受控操作弹引导，不死循环。
- 并发多 401 → 单例引导，不叠多窗。
- 已登录但越权（如普通用户访兽医端，Epic 5）→ 403 友好提示（UX-DR18 ⑤），结构在本 Story 备好但不实现具体角色端点。
