# Story 1.7: 状态 A 档案创建引导与首页提示条

Status: review

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

## Story

As a **选择"我有宠物"的新用户**,
I want **被引导去创建宠物档案，且即使跳过也能被适时提醒**,
so that **我能顺畅建立宠物档案这一核心留存资产**。

> 本 Story 收口 Epic 1 引导流的最后一步。范围：① 状态 A 用户选完状态后进入**档案创建引导页**（入口 + 「跳过，稍后创建」；创建表单本体复用 FR-11，由 Epic 2 实现，本 Story 只负责引导路由与跳过逻辑）；② **首页档案提示条**——仅「状态 A 且未完成档案」用户首页顶部显示（含「立即创建」CTA + 关闭 X）；**前 3 次重启均显示，第 3 次关闭后或档案完成上传后永不再显示**；③ 状态 B/C 用户首页**不触发任何提示条**。
>
> **依赖**：Story 1.6（状态 A 分流 + `pet_status`）、Story 1.2（首页外壳/token/i18n）、Story 1.3（`users` + JWT；判断「档案是否已创建」需后端 `hasPetProfile` 信号）。
> **被依赖**：Epic 2 Story 2.2（档案创建表单本体；创建成功后回填本 Story 引导/提示条的「完成」语义）。
>
> **不做**：宠物档案表单/`pet_profiles` 表（FR-11，Epic 2 Story 2.2）——本 Story 仅引导路由 + 跳过 + 提示条计数逻辑，「立即创建」CTA 跳转到 Epic 2 创建页（本 Story 期可为占位）。

## Acceptance Criteria

> **验证层标注**：每条 AC 末尾标注验证层级与所需本地环境——
> **L0 静态**（编译/lint/widget/单元，含重启计数纯逻辑） · **L1 集成/运行时**（需 Docker+postgres+redis，验 `hasPetProfile` 后端信号） · **L2 端到端/外部凭证**（需真实 Google + 真机/模拟器跨重启走查）。
> 本 Story 提示条 3 次重启计数为可纯单元测试的 **L0**；「档案是否已创建」依赖后端为 **L1**；跨真机重启走查为 **L2**。

### AC1 — 状态 A 档案创建引导（入口 + 跳过）

**Given** 新用户在状态选择中选了 A
**When** 完成选择
**Then** 进入档案创建引导页（入口 + 「跳过，稍后创建」），创建表单本体复用 FR-11（Epic 2 实现，此处负责引导路由与跳过逻辑）（FR-0G）
**And** 完成创建 → 进入首页且不显示提示条；跳过 → 进入首页并进入提示条逻辑
> 验证层：**L0**（前端 widget：选 A 后进引导页 + 「跳过，稍后创建」存在；「立即创建」→ 跳 Epic 2 创建页占位；跳过 → 进首页并激活提示条逻辑；mock「完成创建」→ 进首页且不显示提示条）。文案 .arb 双语。

### AC2 — 首页档案提示条（仅状态 A 未完成档案，3 次重启逻辑）

**Given** 状态 A 且尚未完成档案上传的用户
**When** 进入首页
**Then** 首页顶部显示档案提示条（含「立即创建」CTA + 关闭 X）（FR-0H）
**And** 关闭后当次 session 不再显示、下次重启继续显示；前 3 次重启均显示，第 3 次关闭后或档案完成上传后永不再显示
> 验证层：**L0**（前端单元/widget：计数状态机——重启计数 ≤3 显示；当次关闭→本 session 不再显示；第 3 次关闭后→永不显示；档案完成→永不显示。计数持久化于 `shared_preferences`，纯逻辑可单测）+ **L1**（「档案是否完成」依赖后端 `GET /api/v1/me`.hasPetProfile，需 postgres——本 Story 该信号可先由 Epic 2 回填，1.7 期默认 false=未创建）。

### AC3 — 状态 B/C 不触发提示条

**Given** 状态 B 或 C 的用户
**When** 进入首页
**Then** 不触发任何档案提示条
> 验证层：**L0**（前端 widget：`pet_status ∈ {B,C}` → 首页无提示条、计数逻辑完全不激活）。

## Dev Notes

### 关键架构约定（本 Story 相关部分）

**FR-0H 精确语义（验收硬门槛）**：仅「状态 A 且未完成档案」用户首页顶部显示档案提示条（含「立即创建」CTA + 关闭）；**前 3 次重启显示，第 3 次关闭后或完成上传后永不再显示**。提示条是**纯客户端状态**（`shared_preferences` 计数），后端只提供 `hasPetProfile`。

**前端结构（架构 §Frontend）**：
- 提示条/引导属 profile 域 → `features/profile/{domain,presentation}`；通用条组件 `shared/widgets/profile_prompt_bar.dart`。
- 计数/标志入 `core/storage/prefs`（`shared_preferences`——非敏感偏好：语言/petStatus/**引导计数**，架构 §令牌存储明列「引导计数」走 prefs）。
- 显示判定写成可单测纯函数；Riverpod provider 管状态，副作用进 provider。

**FR-11 边界**：档案表单本体在 Epic 2 Story 2.2（建 `pet_profiles` 表）；本 Story「立即创建」仅路由到其入口（占位），`hasPetProfile` 由 Epic 2 回填。

**i18n / token**：引导页 + 提示条全部文案 .arb 双语；视觉引用 Story 1.2 token；无障碍 ≥44pt 触摸目标（NFR-13）。

### 强制护栏（架构 §Enforcement —— 违反即返工）

- **引导计数/标志走 `shared_preferences`（非敏感偏好）**，不入 secure_storage、不入后端（后端不存「提示条已关几次」）。
- **提示条状态机写成可单测纯函数**——FR-0H 的「3 次重启 + 第 3 次关闭/完成永不显示」语义复杂，必须有单测覆盖边界。
- **B/C 完全不激活提示条逻辑**（不只是隐藏，是不进计数/不持久化），防 B→A 切换时计数污染（如有切换语义，Completion Notes 记录处理）。
- 无写死 UI 文案 / 无硬编码视觉常量（沿用 1.2 token + i18n）。
- 不引入新中间件；不在本 Story 建 `pet_profiles` 表（Epic 2）。

### 范围边界（防 scope creep —— 本 Story 明确不做）

- ❌ 不实现宠物档案创建表单 / `pet_profiles` 表（FR-11，Epic 2 Story 2.2）——「立即创建」仅跳占位。
- ❌ 不实现成长档案 Tab 时间线（FR-37，Epic 2）。
- ❌ 后端不存提示条计数/关闭状态（纯客户端）。
- ❌ 不实现 Feed 内容（Epic 3）——提示条挂在首页容器顶部即可。
- ✅ 只做：状态 A 档案创建引导页（入口+跳过）+ 首页提示条组件 + 3 次重启显示状态机（纯函数可单测）+ B/C 不触发 + 档案完成/永久关闭回填锚点 + 后端 `hasPetProfile` 信号占位。

### 页面多态完整性（本 Story 相关态）

- A 用户跳过后首页：提示条显示（符合计数）；提示条不应遮挡 Feed 首条（顶部条，可关闭）。
- 提示条「立即创建」跳转目标（Epic 2 创建页）本 Story 期未实现 → 占位页（不崩、UX-DR18 友好占位），Epic 2 替换。
- A 用户已在别处创建了档案（`hasPetProfile==true`，Epic 2 回填）→ 即使计数未满也不显示提示条。
- B/C 用户 → 无引导页、无提示条（AC3）。
- 计数 prefs 损坏/缺失 → 默认从 0 开始（首启显示），不崩。
