# Story 2.2: 宠物档案创建（单账号单宠物）

Status: ready-for-dev

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

## Story

As a **有宠用户**,
I want **为我的宠物创建一张专属档案**,
so that **我拥有承载宠物记忆的家，并获得可对外分享的名片**。

> 本 Story 在 Epic 2 中**首次建立 `pet_profiles` 表**，是 2.3（成长日历绑定宠物）、2.4（档案 Tab 时间线）、2.6（名片 H5）、2.7（分享 FAB）、2.8（编辑）的数据根。本 Story 只做"创建 + 单账号单宠物约束 + 生成不可枚举名片 token + 已有档案直达"，**不做时间线、不做名片页、不做编辑**。
>
> **依赖前序**：Story 1.1（脚手架/profile 模块空壳/ProblemDetail）、Story 1.3（JWT，创建需登录用户）、Story 1.6/1.7（状态 A 用户 + 创建引导入口，本 Story 承接其「立即创建」按钮跳到本表单）、Story 2.1（头像上传复用其压缩/直传/权限工具）。

## Acceptance Criteria

> **验证层标注**：**L0 静态**（编译/lint/MockMvc/widget 测试）· **L1 集成/运行时**（需 Docker daemon + postgres + redis）· **L2 端到端/外部凭证**（真实阿里 OSS/STS、真机相册）。
> 本 Story 核心落 **L0/L1**；头像直传依赖 2.1 的能力，其真实 OSS 验证已在 2.1 的 L2 完成，本 Story 只需用 2.1 工具产出的 URL 落库（L1）。

### AC1 — 创建档案 + 单账号单宠物 + 生成名片 token

**Given** 状态 A 用户（含 Story 1.7 引导入口或档案 Tab 入口）
**When** 填写头像（相册/拍照）、名字、品种、生日、一句话介绍（≤30 字）并提交
**Then** 创建宠物档案成功（写 `pet_profiles` 表，本故事创建该表），V1 限制单账号单宠物（已存在档案时不允许再建）（FR-11）
**And** 创建后自动生成不可枚举的对外名片 token（供 FR-14 使用），不暴露顺序 id
> 验证层：**L1**（Flyway 建表迁移成功 + 创建端点写库 + 重复创建被 409 拦 + token 列唯一非顺序，需 postgres）。Bean Validation（名字 ≤20、介绍 ≤30、必填项、生日合法）与 token 不可枚举生成可在 **L0**（MockMvc/单测）验证。头像 URL 经 2.1 直传得到后随表单提交（**L0** 验 DTO 接收 URL；真实直传 **L2** 已在 2.1 覆盖）。

### AC2 — 已有档案直达，不重复创建

**Given** 已有档案的用户
**When** 再次进入创建入口
**Then** 直接进入该宠物档案，不出现重复创建表单
> 验证层：**L1**（已有档案时 `GET /pet-profiles/me` 返回既有档案；前端据此路由到档案 Tab 而非创建表单）+ **L0**（前端路由守卫/provider 分支 widget 测试）。

## Tasks / Subtasks

> 三段组织。本 Story **前后端均衡**：后端建表 + 创建/查询端点 + 单宠物约束 + token 生成；前端创建表单 + 头像上传（复用 2.1）+ 已有档案直达。建议：后端 → 前端 → 联调。

### 🟦 后端子任务（petgo-backend / `com.petgo.profile`）

- [ ] **B1. Flyway 建 `pet_profiles` 表** (AC: 1)
  - [ ] `db/migration/V<序号>__init_profile.sql`：`pet_profiles(id bigserial pk, owner_id bigint not null, avatar_url varchar, name varchar(20) not null, breed varchar, birthday date, intro varchar(30), card_token varchar not null, created_at timestamptz default now(), updated_at timestamptz)`。
  - [ ] 约束/索引：`uq_pet_profiles_owner_id`(**单账号单宠物：owner_id 唯一**)、`uq_pet_profiles_card_token`、`fk_pet_profiles_owner`→users(id)。命名严格 `idx_/uq_/fk_` 前缀。
  - [ ] **删除级联设计前置**（架构不可砍地基#4）：表与 owner 的 FK 设计需支持注销级联（FR-20），注释标注「注销时此表 + 头像对象需级联删除」。
- [ ] **B2. 实体 / DTO / Repository** (AC: 1, 2)
  - [ ] `profile/domain/PetProfile`(@Entity)；`profile/repository/PetProfileRepository`（`findByOwnerId`、`existsByOwnerId`、`findByCardToken`）。
  - [ ] DTO record：`PetProfileCreateRequest{avatarUrl, name(≤20,@NotBlank), breed, birthday, intro(≤30)}`、`PetProfileResponse{id, avatarUrl, name, breed, birthday, intro, cardToken, createdAt}`。Bean Validation 注解齐全。
- [ ] **B3. 名片 token 生成（不可枚举）** (AC: 1)
  - [ ] `profile/service`：用 `SecureRandom`/UUIDv4 base62（≥128bit 熵）生成 `card_token`，落库唯一；**绝不用顺序 id 派生**。供 2.6 `/p/{cardToken}` 使用。
- [ ] **B4. 创建 + 查询服务/控制器 + 单宠物约束** (AC: 1, 2)
  - [ ] `ProfileService.create(ownerId, req)`：先 `existsByOwnerId` → 已存在抛 `AppException`(409 冲突, type `https://petgo/errors/profile-exists`)；否则生成 token 写库。
  - [ ] `ProfileApiController`：`POST /api/v1/pet-profiles`（JWT，201 + `PetProfileResponse`，带 `Idempotency-Key` 去重）；`GET /api/v1/pet-profiles/me`（返回当前用户档案或 404）。
  - [ ] 套写端点限流（`RedisRateLimiter`）。**owner_id 取自 JWT，绝不信任客户端传入**。

### 🟩 前端子任务（petgo_app / `lib/features/profile`）

- [ ] **F1. 创建表单页** (AC: 1)
  - [ ] `features/profile/presentation/pet_profile_create_page.dart`：头像（点击→2.1 的选图/权限/压缩/直传，得 avatarUrl）、名字（≤20 实时计数）、品种、生日（date picker）、一句话介绍（≤30 实时计数）；前端实时预校验（体验），服务端为权威。
  - [ ] 提交调 `ProfileRepository.create(...)`，带 `Idempotency-Key`；成功后路由到档案 Tab（2.4）。
- [ ] **F2. data/domain 层** (AC: 1, 2)
  - [ ] `features/profile/data/profile_repository.dart`（dio + ProblemDetail 映射 + DTO）；`domain/pet_profile.dart`(不可变 + copyWith)；`presentation` provider `petProfileProvider`(AsyncValue)。
- [ ] **F3. 已有档案直达守卫** (AC: 2)
  - [ ] 进入创建入口前先读 `petProfileProvider`/`GET /pet-profiles/me`：已存在→直接 go_router 跳档案 Tab，不渲染创建表单；不存在→渲染表单。
- [ ] **F4. 错误/重复创建处理** (AC: 1, 2)
  - [ ] 409 profile-exists → 本地化提示并跳档案（防并发双开窗重复提交）；头像未传/必填缺失 → 422 映射到字段级提示。i18n 双套，无写死串。

### 🟨 联调验收子任务

- [ ] **J1. 创建闭环（L1）** (AC: 1) — 起栈→登录态→填表+头像→201→DB 查得记录、card_token 非顺序且唯一。
- [ ] **J2. 单宠物约束（L1）** (AC: 1) — 同用户再次 POST → 409；`GET /pet-profiles/me` 返回既有。
- [ ] **J3. 已有档案直达（L0/L1）** (AC: 2) — 已有档案用户进创建入口→直接落档案 Tab，无重复表单。
- [ ] **J4. 校验（L0）** (AC: 1) — 名字 >20 / 介绍 >30 / 缺必填 → 422 字段级 ProblemDetail。

## Dev Notes

### 架构约定

- **命名映射链**：表 `pet_profiles`(复数 snake_case)、列 `owner_id`/`card_token`/`created_at`(timestamptz UTC)；实体 `PetProfile`、DTO `PetProfileCreateRequest/Response`(record)、provider `petProfileProvider`。JSON camelCase、时间 ISO-8601 UTC、null 省略。
- **对外标识**：`card_token` 不可枚举（SecureRandom，≥128bit），**绝不暴露顺序 id 于对外名片路径**（2.6 用）。已授权资源响应可带数字 `id`。
- **错误**：重复创建 409、校验 422、未登录 401、ProblemDetail 统一信封。
- **头像媒体**：复用 Story 2.1——公开桶（头像对外可见于名片/档案）+ CDN URL；EXIF 已在 2.1 客户端剥离。

### 强制护栏

- **禁 MQ/缓存层/新中间件**；Redis 仅限流/幂等。
- `ddl-auto=validate`——schema 只由本 Story 的 Flyway 脚本管，**禁 Hibernate 自动建表**。
- 凭证 env 注入；owner_id 来自 JWT 不信任客户端。
- 删除级联路径设计前置（注销时连带 OSS 头像）。
- 日志不落 PII（名字/介绍属用户内容，按需 INFO 可记业务事件 id，**不记健康数据**——本表无健康字段）。

### 范围边界（不做）

- ❌ 不做时间线/记录展示（2.4）、名片 H5（2.6）、分享 FAB（2.7）、编辑（2.8）、多宠物。
- ❌ 不做 content_posts（2.3）。
- ✅ 只做：`pet_profiles` 表 + 创建/`/me` 查询端点 + 单宠物 409 + 不可枚举 token + 创建表单 + 已有档案直达 + 头像复用 2.1。

### Project Structure Notes

- 后端 `com.petgo.profile.{web/ProfileApiController, service/ProfileService, domain/PetProfile, repository/PetProfileRepository, dto/...}`；迁移 `resources/db/migration/V<n>__init_profile.sql`（与架构示例 `V2__init_profile.sql` 对应）。
- 前端 `lib/features/profile/{data,domain,presentation}`，头像复用 `lib/shared/utils`(2.1) 与 `core/media`。
