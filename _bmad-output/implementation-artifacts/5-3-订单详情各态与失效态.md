# Story 5.3: 订单详情各态与失效态

Status: review

> V1.1 **Epic 5 收官 story**，**全栈**（订单详情 `GET /orders/{token}` + 前端详情页 `p-order-detail`，替换 5-2 占位路由，L0/L1/L2）。点进任一订单看到**按类型分支的详情**（兽医多态含退款子阶段 / AI / 充值 / HD 预留）+ **宠物已删→失效占位态**（FR-54D，非裸失败）。**无 Flyway 迁移**（复用 5-1 三源 + refund_requests；宠物硬删靠 findById 空判定）。
> 源：`epics-v1.1.md` Story 5.3（兽医 6 态/AI/充值/HD 详情；退款态两子阶段/子变体 L0/L1；宠物已删→订单失效占位 FR-54D L1；模拟器 L2）· FR-54A~D · UX-DR7（详情 6 态）· 架构 `architecture-v1.1-delta.md` §6 `/orders/{token}`（line 230）· §缺失态（line 291「宠物已删订单失效态」）。
> **⚠️ 承接现状（5-1/5-2 已建，照契约扩）**：5-1 `com.tailtopia.order`（`OrderCenterService` 聚合 3 源 + `OrderSummaryView`/`OrderType`/`OrderStatusColor` + `OrderController` GET /orders）；5-2 前端 `features/order/`（列表 + 卡片 + `/me/orders/:token` **占位详情路由**待本 story 替换）。三源：`consult_orders`(ConsultOrder，pet_profile_id 无 FK)/`ai_consult_orders`(AiConsultOrder)/`payment_intents`(PaymentIntent PAWCOIN_TOPUP)。退款：`refund_requests`(RefundRequest：need_decision/approval_status/net_amount，绑 order_id)。宠物：`PetProfile`(name/petType/avatarUrl；**硬删**——`ProfileDeletionService.petProfiles.delete(pet)` 级联，无软删标志) + `PetProfileQueryService`/`PetProfileRepository.findById`。
> **决策（承接 5-1/5-2 + 架构，autonomously 定，无新增待澄清）**：① 详情按 token 跨 3 源查（token 全局唯一不可枚举，无前缀 → 依次试 consult_orders.order_token / ai_consult_orders.order_token / payment_intents.public_token，命中即 owner 校验）。② **退款子阶段派生 refund_requests**（REFUNDING 订单 join：approval_status null→待选方式/PENDING_APPROVAL→待审批/APPROVED→待打款/PROCESSING→打款中；COMPLETED+refund_rejected→退款未通过），即「6 态」的退款子变体。③ **宠物已删失效**：vet/AI 订单 `pet_profile_id`→`findById` 空 → `petDeleted=true` 占位（订单本身持久存活，非 500，FR-54D）。④ 详情**不下发 title/subtitle 显示串**（前端按 orderType+statusCode+refundStage 本地化）。
> **范围边界**：本 story = 详情端点（3 源分支 + 退款子阶段 + 宠物失效）+ 前端详情页（替换占位）。**不做**：列表/卡片（5-2 已建）；聚合列表（5-1）；HD 订单（Epic 6，详情类型预留）；退款操作（4-x）；深链跳分诊/会话结果（OPEN 留后续）。**无迁移**。

## Story

As a 用户,
I want 点进订单看到对应类型的详情、状态与退款进度,
so that 我能查支付凭证、看退款到哪一步（FR-54A~D），宠物删了也看到失效说明而非报错（FR-54D）。

## Acceptance Criteria

1. **后端：`GET /orders/{orderToken}` 详情（3 源分支）（L1）**
   **Given** 用户有某订单（兽医/AI/充值）
   **When** `GET /api/v1/orders/{orderToken}`（JWT USER，仅 owner）
   **Then** 按 token 跨 3 源定位 → 返回 `OrderDetailView`（orderType/statusCode/statusColor/amount(可空)/payChannel/createdAt/paidAt + 类型专属字段）；非 owner / 不存在 → 404（防枚举）（**L1**）
   **And** 兽医：pet 信息（name/type/avatar）+ session 起止 + 退款子阶段；AI：triageTaskId（预留深链）；充值：coins(=amount)（**L1**）

2. **后端：退款态两子阶段/子变体（L1）**
   **Given** 兽医订单 `status=REFUNDING` 或 `COMPLETED+refund_rejected`
   **When** 取详情
   **Then** join `refund_requests`(by order_id) 派生 `refundStage`：`AWAITING_METHOD`(approval_status null，待用户选方式 4-5)/`AWAITING_APPROVAL`(PENDING_APPROVAL，待主管 4-6)/`AWAITING_PAYOUT`(APPROVED，待财务 4-6)/`PROCESSING`；`COMPLETED_REFUND_REJECTED`→ `refundStage=REJECTED`（退款未通过）+ 可带 `refundNetAmount`（**L1**）
   **And** 非退款态订单 `refundStage=null`（**L1**）

3. **后端：宠物已删失效占位态（FR-54D）（L1）**
   **Given** 兽医/AI 订单的 `pet_profile_id` 对应宠物**已被硬删**（`ProfileDeletionService` 级联删除，findById 空）
   **When** 取详情
   **Then** 订单**仍返 200**（持久存活）+ `petDeleted=true` + pet 字段占位 null（前端显「宠物已删除」失效态，**非 500/非裸失败**，FR-54D）（**L1**）
   **And** 订单金额/状态/凭证字段照常（失效仅作用 pet 区块）（**L1**）

4. **前端：订单详情页 `p-order-detail`（各态 + 失效态）（L0/L2）**
   **Given** 从 5-2 列表卡进入
   **When** 渲染 `/me/orders/{token}`（替换 5-2 占位路由）
   **Then** 按 `orderType` 分支：**兽医**（pet 卡 / 已删占位 + 状态 + session + **退款进度条/子阶段文案** + 金额）/ **AI**（状态 + 金额 + triage 预留）/ **充值**（coins + 渠道 + 状态）；退款中→info 蓝、退款未通过→提示样式（**L0/L2**）
   **And** 加载/错误/404 态（订单不存在友好占位）；宠物已删区块占位文案；**离线不留白不转圈**（**L0/L2**）

5. **契约 + 验收（L0/L1/L2）**
   **Given** 全部实现
   **When** 验证
   **Then** `DB_NAME=petgo_l1 mvn -B test` 绿（详情 3 源分支 + 退款子阶段 + 宠物已删 200 占位 + owner 404）+ `flutter analyze` 零 + `flutter test` 绿（详情模型 fromJson + 分支渲染）（**L0/L1**）
   **And** 不破坏 5-1 列表聚合 / 5-2 列表页 / consult/ai/pay/refund 既有（回归绿）；模拟器：兽医多态 + 退款子阶段 + 宠物已删失效 + 充值/AI 详情（**L2 留本地**）

### 承接现状（已核实，照此复用/勿改）

- **三源定位**：token 全局唯一不可枚举（CardTokenGenerator 22/32 位 base62），无源前缀 → `getDetail` 依次 `consultOrders.findByOrderToken` / `aiOrders.findByOrderToken`(需加) / `intents.findByPublicToken`，命中即停 + owner 校验（userId 不符→404 防枚举）。**AiConsultOrderRepository 需加 `findByOrderToken`**（现仅 findByPaymentIntentToken）。
- **退款子阶段（RefundRequest）**：`refund_requests` 绑 order_id（`RefundRequestRepository` 加 `findByOrderId`）。兽医订单 REFUNDING → 取 refund_request 派生 `refundStage`：`approval_status==null`→AWAITING_METHOD（4-5 待用户选方式/填收款）；`PENDING_APPROVAL`→AWAITING_APPROVAL（4-6 待主管）；`APPROVED`→AWAITING_PAYOUT（4-6 待财务）；`PROCESSING`→PROCESSING。订单 `COMPLETED+refund_rejected`→statusCode `COMPLETED_REFUND_REJECTED`(5-1 已映) + refundStage=REJECTED。可带 `refundNetAmount`(refund_request.net_amount，QRIS 填账户后有值)。
- **宠物硬删（FR-54D）**：`ProfileDeletionService` 物理 `delete(pet)` 级联（无软删标志）。`consult_orders.pet_profile_id` **无 FK**（订单存活）。详情 `petProfiles.findById(petProfileId)` 空 → `petDeleted=true` + pet 字段 null（前端占位「宠物已删除」，非 500）。`PetProfile`：getName/getPetType/getAvatarUrl。
- **门控**：`/api/v1/orders/{token}` — SecurityConfig 现仅 `GET /api/v1/orders`(exact)。扩为 `GET /api/v1/orders/**` hasRole('USER')（或加第二 matcher）。owner 校验在 service（jwt.sub）。
- **前端 order 模块（5-2）**：`features/order/` domain/repo/list/l10n/widgets；`/me/orders/:token` 现为占位 Scaffold → 本 story 替换为 `OrderDetailPage`。statusColor→色映射（INFO 紫柔底非红）+ orderType 图标/label 复用 5-2 `order_l10n`。

### 关键设计决策（dev 照此）

- **D-1 3 源 token 依次试 + owner 404**：无源提示则依次查；首命中 owner 校验。若三源皆无或非 owner→404（不暴露存在性）。
- **D-2 退款子阶段=「6 态」子变体**：兽医 6 态 = IN_PROGRESS / COMPLETED / COMPLETED(refund_rejected) / REFUNDING(AWAITING_METHOD/APPROVAL/PAYOUT/PROCESSING 子阶段) / REFUNDED。前端退款进度按 refundStage 展示步骤条/文案。**退款中恒 info 蓝非红**（承接 5-1/5-2）。
- **D-3 宠物失效占位（FR-54D）**：仅作用 pet 区块（petDeleted=true + null 字段）；订单金额/状态/凭证照常。**绝不因宠物缺失 500**（防御 findById 空）。前端显「宠物已删除」灰占位。
- **D-4 不下发显示串**：详情返 orderType/statusCode/refundStage/字段码，前端本地化（[[petgo-i18n-model-and-debt]]）。
- **D-5 detail 富化边界**：兽医含 pet + session + refundStage；AI 含 triageTaskId（深链跳分诊结果留后续 OPEN）；充值含 coins。**兽医名/头像、分诊/会话结果深链留 OPEN**（本 story 详情聚焦支付凭证 + 退款进度 + 宠物；不做跨模块深链）。
- **D-6 无迁移**：加 repo 派生方法（findByOrderToken/findByOrderId）+ DTO + service/controller + 前端页；无 schema 改。
