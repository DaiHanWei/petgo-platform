# Story 1.4: 登录引导浮层与登录后回跳

Status: review

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

## Story

As a **未登录访客**,
I want **在恰当时机看到清晰的登录引导并在登录后回到原操作**,
so that **我不会因登录中断而迷失，转化更顺滑**。

> 本 Story 是 **纯前端登录引导组件库 + 回跳机制**。范围：① 实现两个自包含、可被任意触发源调用的组件——软性登录推荐浮层（FR-0B，底部半屏、主 CTA 显著、关闭弱化、**每 session 最多触发一次**）与强登录引导弹窗（FR-0C，视觉强度更高、含「登录后继续使用该功能」说明）；② 登录后回跳机制（登录成功自动回到触发点继续原操作；新用户先经 Story 1.6 引导再回触发点）。两个组件**本身不自带触发条件**，由调用方注入。
>
> **依赖**：Story 1.2（外壳 + i18n + token，浮层/弹窗样式与文案）、Story 1.3（组件内「Google 登录」按钮调用其登录流程 + 新老分流信号 + 回跳锚点 `pendingAction` 结构）。
> **被依赖**：Story 1.5（受控入口/Tab 门控注入强弹窗触发 + dio 401→引导）、Epic 3 Feed（首页浏览至第 3 页滚动深度→注入软浮层触发）、各 Epic 受控入口（问诊/发布/创建档案触发强弹窗）。
>
> **不做**：任何触发条件的接线（门控触发=Story 1.5；滚动深度触发=Epic 3 提供滚动信号后接），本 Story 只建组件 + 每 session 一次去重逻辑 + 回跳机制，并用一个开发自测按钮手动触发以验证。**无后端改动**——登录态/JWT 由 Story 1.3 后端提供，本 Story 复用其前端登录流程。

### AC1 — 两个自包含登录引导组件（软浮层 FR-0B / 强弹窗 FR-0C）

**Given** 前端需要一套可被任意触发源调用的登录引导组件
**When** 实现软性登录推荐浮层（FR-0B）与强登录引导弹窗（FR-0C）
**Then** 两者均为自包含组件：软浮层=底部半屏、主 CTA「Google 登录」（调用 Story 1.3 登录）显著、关闭按钮小号浅色、每 session 最多触发一次；强弹窗=视觉强度更高、含说明「登录后继续使用该功能」+ Google 登录按钮 + 关闭
**And** 组件本身不自带触发条件，由调用方注入触发——受控入口门控触发见 Story 1.5；软浮层「首页浏览至第 3 页」滚动触发在 Epic 3 Feed 提供滚动深度后接线（此处先建组件 + 每 session 一次的去重逻辑）
> 验证层：**L0**（widget test：软浮层=底部半屏 + 主 CTA 显著 + 关闭弱化 + 同一 session 第 2 次调用不再弹；强弹窗=更高强度 + 「登录后继续使用该功能」说明 + Google 按钮 + 关闭；组件接受注入式触发而非自带条件）。文案双语走 .arb。

### AC2 — 登录后回到触发点（含新用户先经引导）

**Given** 用户在登录引导（软浮层或强弹窗）中完成登录
**When** 登录成功
**Then** 自动回到触发点继续原操作（如继续进入问诊/发布/创建档案）；新用户先经 Story 1.6 引导再回到触发点
> 验证层：**L0**（widget/单元测试：注入 `pendingAction`（如「打开问诊」）→ 触发引导 → mock 登录成功(老用户)→ 自动执行 pendingAction；mock 新用户→先路由引导占位(Story 1.6)→引导完成回调后再执行 pendingAction）。+ **L2(模拟器)** 真实链路联动 Story 1.3。

### AC3 — [R2] 授权失败回到触发前页面 + 重试（FR-0C/FR-0D · 决策 F13）

**Given** 用户从某触发点（AI问诊 / 兽医咨询 / 发布 / 建档 / 首页滚动浮层）经登录引导（软浮层或强弹窗）发起 Google 登录
**When** 授权失败（取消 / 网络超时 / Google 异常 / 后端校验失败，复用 Story 1.3 AC5 失败信号）
**Then** 回到**触发登录前的原页面**——**保留回跳目标 `pendingAction` 不清空**，展示「登录失败，请重试」+ 重试入口（决策 F13 输入类失败口径）；**不前进到注册引导**（不路由到 Story 1.6 占位）、不执行 pendingAction
**And** 用户点重试可重新发起授权（成功后仍按 AC2 用保留的 pendingAction 回跳触发点）；用户主动关闭则取消 pendingAction、停留原页（与 FR-0B/0C 可关闭一致）
> 验证层：**L0**（widget/单元测试：注入 `pendingAction`「打开问诊」→ 触发引导 → mock 登录回调失败（null / 异常）→ 断言 `pendingAction` **仍保留未清空**、**未路由到引导占位**、引导显示失败态 + 重试入口；mock 重试成功(老用户) → 用保留的 pendingAction 回跳；主动关闭 → pendingAction 清空 + 停留）。纯前端，**无 L1**；真实失败链路属 Story 1.3 L2。

## Dev Notes

### 关键架构约定（本 Story 相关部分）

**前端结构（架构 §Project Structure / Frontend）**：
- 软浮层/强弹窗是**跨特性通用 UI** → 放 `lib/shared/widgets/`（与 `bottom_tab_bar`/`empty_state` 同级）。
- 触发协调器/回跳逻辑属 auth 域 → `lib/features/auth/{domain,presentation}`；登录调用复用 `features/auth/data`（Story 1.3）。
- 路由用 go_router（Story 1.2 接入）；回跳通过 `pendingAction`/`RouteIntent` 表达，落在 `core/router`。
- Riverpod 状态不可变；`AsyncValue` 表达登录中/成功/失败；副作用进 provider 不写在 build 内。

**转场原语（UX-DR11/12）**：modal/bottom sheet 自底上滑 300ms spring + 可拖拽/点背景关闭；软浮层用 bottom sheet、强弹窗用居中 dialog（视觉强度区分）。

**Voice & Tone（UX-DR14）**：双语 microcopy，每条最多 1 emoji；软性浮层语气邀请性、强弹窗简短明确「登录后继续使用该功能」。

**登录态来源（架构 §Authentication）**：登录动作 = Story 1.3 的 Google→JWT 流程；本 Story 不碰 JWT/secure_storage 本身，只消费 `authStateProvider` + `LoginResponse{isNewUser, onboardingCompleted}`。

### 强制护栏（架构 §Enforcement —— 违反即返工）

- **无写死 UI 字符串**：浮层/弹窗全部文案走 .arb（NFR-11）。
- **无硬编码视觉常量**：颜色/字号/圆角/elevation/动效时长引用 Story 1.2 token。
- **组件零触发条件**：软浮层/强弹窗**只暴露调用 API**，触发条件由调用方注入——这是「可被任意触发源复用」的硬约束，防止把首页滚动/门控逻辑耦合进组件。
- **软浮层 session 级去重用内存标志，非持久 prefs**（每 session 一次≠每用户一次）。
- 不擅自引入新中间件/状态库（坚持 Riverpod + go_router）。

### 范围边界（防 scope creep —— 本 Story 明确不做）

- ❌ 不接任何真实触发源：受控入口/Tab 门控触发=Story 1.5；首页滚动至第 3 页触发=Epic 3 Feed 提供滚动深度后接线。本 Story 仅 `@dev` 自测按钮手动触发。
- ❌ 不实现 Google 登录链路本体 / JWT / secure_storage（Story 1.3）——仅调用。
- ❌ 不实现新用户引导页本体（Story 1.6）——回跳分叉路由到其占位。
- ❌ 不做 dio 401→引导的接线（Story 1.5 复用本 Story 组件 + 回跳）。
- ❌ 无后端改动。
- ✅ 只做：软浮层 + 强弹窗两组件 + 每 session 去重 + pendingAction 回跳机制（含新用户先引导后回跳）+ 自测触发。

### 页面多态完整性（本 Story 相关态）

- 引导中登录失败/取消 → 引导保持/可重试或关闭，不崩；停留原页。
- 软浮层已在本 session 触发过 → 再次请求 no-op（静默，不闪现）。
- pendingAction 目标在登录后已失效（如内容被删）→ 回跳到目标页时由目标页自身的「资源不存在」态（UX-DR18 ④）兜底，本 Story 不重复处理但回跳不应崩溃。

### Project Structure Notes

- 复用：`features/auth/data`（Story 1.3 登录）、`features/auth/domain/auth_state`、Story 1.2 token + i18n + go_router shell。

#### R2 增量（AC3 / F5 · 决策 F13 · 2026-06-08）

- **断档定位**：原 `_attemptLogin` 只处理「成功 / 取消(null)」两态，runner 抛出的**非取消异常**（网络/Google/后端校验失败）未捕获 → 失败时无失败态、pendingAction 行为未定义。R2 补齐授权失败回触发前页面 + 重试。
- **三态化**：`_attemptLogin` 改返回 `LoginGuideOutcome{success, cancelled, failed}`（新增 `lib/shared/widgets/login_guide_outcome.dart`，置 shared 层避免 widget 反向依赖 features）。runner 抛异常 → `try/catch` 归一为 `failed`；返回 null → `cancelled`；成功 → `success`。**失败/取消均不清 `_pending`**（仅「关闭」清），失败**不路由 `/onboarding`**、不执行 pendingAction。
- **失败态 UI**：`LoginSoftSheet`/`LoginHardDialog` 由 Stateless 改 Stateful，`onLogin` 返回 `Future<LoginGuideOutcome>`；`failed` 时内联展示「登录失败，请重试」（复用既有 `loginFailed` .arb 双语 + `AppColors.danger` token，零新文案）。**主 CTA「Google 登录」即重试入口**（再点重新发起，复用同一 runner），登录中禁用 CTA/关闭防双击。成功由协调器关闭引导，`if(!mounted) return` 防 setState-after-dispose。
- **取消差异化保留**：取消（`LoginCancelled`→null）静默保持引导、不显失败态——与 Story 1.3「已取消 vs 登录失败」差异化口径一致（PRD FR-0D 字面「取消也统一提示失败」的归一仍待产品确认，未在本轮改）。
- **L0 全绿**：`flutter analyze` 零警告；`flutter test` 全套 **240 通过**（login_guide 新增 4 例：强弹窗失败态+pending 保留+不路由 / 失败后重试成功回跳 / 失败后关闭清 pending / 软浮层失败态）。widget 签名改动对 1-5 门控、content/triage 等既有引用零回归。
- **待本地（L2）**：真实失败链路（断网/取消/后端 4xx）观感留模拟器（headless 不可验）。
