# Story 5.2: 兽医工作台框架与在线状态管理

Status: review

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

## Story

As a **兽医**,
I want **一个专属工作台并能切换在线/离线**,
so that **我能管理我的接单可用性与会话，在线态即时影响用户侧能否找到我**。

> 本 Story 在 5.1（兽医账号 + `role=vet` 登录 + role 分流壳）之上，交付**兽医工作台的导航框架**（独立 Tab Bar：待接单/进行中/历史/我的）与**在线/离线状态管理**（FR-32，Redis 在线态，即时联动 FR-4B 可用性与接单队列）。
>
> 做：兽医独立 Tab Bar 骨架（与用户侧 5-Tab 完全不同）、四个 Tab 的壳页与导航、「我的」Tab 在线/离线开关、Redis 在线态读写、用户侧问诊入口对在线态的读取（「当前暂无兽医在线」）。
> 不做：待接单队列里**真实的请求**（请求由 5.3 写入 `consult_sessions` WAITING）、接单（5.5）、进行中会话内容（5.5）、历史记录内容（5.8/5.6）、评分（5.6）。本 Story 四个 Tab 是**框架 + 在线态**，列表内容多为占位/空态，后续 Story 填充。
>
> 依赖前序：5.1（vet 登录与 `GET /api/v1/vet/me`、role 守卫）。本 Story 首次引入 Redis 兽医在线态——架构明确 Redis 收窄用于「兽医在线态(在线/忙/离线) + 问诊队列态」，本 Story 落「在线态」，队列态在 5.3 落。

## Acceptance Criteria

> **验证层标注**：**L0**（壳页/Tab 切换/lint，纯静态或前端单测） · **L1**（需 Docker + redis：在线态读写、用户侧入口读在线态需真实 Redis 跑） · **L2**（真机/模拟器跑兽医 App 工作台 + 用户 App 入口联动）。
> 本 Story 在线态读写主体在 **L1**（Redis），双端联动（兽医切在线 → 用户侧入口文案变）需 **L2**。无腾讯 IM。

### AC1 — 兽医工作台独立 Tab Bar（与用户侧完全不同）

**Given** 兽医登录后
**When** 进入工作台
**Then** 展示与用户侧完全不同的 Tab Bar（待接单/进行中/历史记录/我的）（FR-30 框架）
> 验证层：Tab Bar 渲染四 Tab + 切换 = **L0/L2**（前端 widget 测 + 真机）。**严禁**复用用户侧 5-Tab（凸起「+」发布 FAB 等）——兽医壳是另一套底部导航，无发布入口、无 Feed/档案。role 守卫保证只有 `role=vet` 能进（承接 5.1）。

### AC2 — 「我的」Tab 切换在线/离线，状态即时生效（Redis）

**Given** 兽医在「我的」Tab
**When** 切换在线/离线
**Then** 状态即时生效（Redis 在线态）：在线可接新请求并出现在可接单队列、离线不接新请求且用户侧问诊入口显示「当前暂无兽医在线」（FR-32，联动 FR-4B）
> 验证层：在线/离线开关写 Redis（`PUT /api/v1/vet/online-status`）+ 读回 = **L1**（真实 Redis）。「出现在可接单队列」= 在线态被 5.3 的匹配逻辑读取（本 Story 落「在线态可被查询」的能力，队列匹配在 5.3）。双端即时联动 = **L2**。

### AC3 — 用户侧问诊入口读取在线态（无兽医在线时的离线态文案）

**Given** 用户进入兽医咨询入口
**When** 系统检测当前在线兽医数
**Then** 有 ≥1 兽医在线 → 入口可发起；全部离线 → 入口显示「当前暂无兽医在线」（FR-32 联动 FR-4B 离线态）
> 验证层：聚合查询「当前是否有在线兽医」(读 Redis 在线态集合) = **L1**；用户侧入口据此渲染在线/离线两态 = **L2**。**注意**：架构要求在线态「概率性展示、不暴露实时人数」——本 Story 后端只回「是否有兽医在线(bool)」，**不回精确人数**（精确人数即使内部算出也不透传前端）；离线态完整软引导（恢复时段 + 「先用 AI 分诊」）在 5.3 完成，本 Story 落「有/无兽医在线」这一开关位。

### AC4 — 在线态的鲁棒性（断线/退出/会话占用）

**Given** 兽医退出登录 / App 被杀 / token 失效
**When** 在线态需失效
**Then** 兽医离线（在线态过期/清除），不再被计为可接单；在线态用带 TTL 的 Redis 键 + 心跳/登录态续期，避免「幽灵在线」
> 验证层：**L1**（Redis 键 TTL 到期自动离线；显式登出清键）。**注意多态**：本 Story 显式覆盖「兽医并非显式点离线、而是退出/掉线」的非 happy-path——用 TTL 兜底，不依赖前端一定发离线请求。

### AC5 — 待接单抢单列表（前端）[R2·回改]

> **代码已实现，本轮回改补齐**（F11 抢单模式前端，与 5.3 后端原子接单配套；本 AC 只落待接单列表/请求详情/预览计时的前端，后端先到先得原子写入归 5.3）。

**Given** 兽医在线 + 进入「待接单」Tab
**When** 系统以**抢单模式**展示待接单请求并兽医点开某请求
**Then** 按以下流程：
- 待接单 Tab 列表展示**请求卡片**（症状摘要 + 图片预览缩略图），多在线兽医并发可见同一批 WAITING 请求（先到先得，接单原子写在 5.3）；
- 点卡片进**请求详情页**查看完整描述 / 媒体，进入即**开始 3 分钟预览计时**；
- 预览期内三种返回列表态须显式覆盖（决策 F11 / spec-page-state-completeness 多态完整性）：
  1. 用户在预览期取消请求（WAITING→CANCELLED，5.3）→ 提示「此请求已关闭」→ 返回列表；
  2. 其他兽医率先接单（5.3 原子写命中，本兽医接单影响 0 行）→ 提示「此请求已被其他兽医接单」→ 返回列表；
  3. 3 分钟预览未操作 → 自动返回列表，该请求继续对其他兽医可见（不被本兽医独占）。
> 验证层：**L0**（待接单抢单列表 + 请求卡片渲染 + 3min 预览计时器 + 三种返回态文案/路由 widget 测试，纯前端）。后端先到先得原子接单/退单/预览超时回队列在 **5.3**——本 Story 不写 `consult_sessions`、不做原子接单。**禁 MQ / 分布式锁中间件**：并发互斥语义由 5.3 的 DB 原子条件更新承担，本 AC 前端只消费「接单成功 / 已被抢 / 已关闭」三种后端响应并渲染对应返回态。状态机沿用 `WAITING/IN_PROGRESS/.../CANCELLED`（决策 E1），本 AC 不引入新状态。任务：前端（`features/vet/presentation` 待接单列表 + 请求详情 + 预览计时 + 三态返回）。

## Dev Notes

### 关键架构约定（本 Story 必须落实）

- **兽医工作台是独立导航壳**：与用户侧 5-Tab 物理隔离（架构 §Frontend Architecture / Boundaries，`features/vet` 独立 Tab）。用户侧凸起「+」发布、Feed、成长档案在兽医壳中**完全不存在**。
- **Redis 在线态（收窄用途之一）**：架构明确 Redis 仅用于「auth 限流 / 幂等 / **兽医在线态(在线/忙/离线)** / 问诊队列态 / 角标」。本 Story 落在线态，**不**把它扩成通用缓存（§Data Architecture / Enforcement）。
- **在线态概率性展示、不暴露实时人数**：FR-4B 要求用户侧入口概率性展示在线、不显示实时在线人数。后端对用户侧只回 `vetOnline: bool`，精确人数即便内部用于匹配也不透传（5.3 的「工作日 8–23 通常有兽医在线」文案是前端静态/配置文案，非实时人数）。

### 会话状态机专项（本 Story 的定位）

> Epic 5 状态机 `WAITING → IN_PROGRESS → PENDING_CLOSE → CLOSED / INTERRUPTED`，超时(1min)/评分门(30min) 为迁移触发器，全在 Java 侧编排腾讯 IM；兽医在线态/队列在 Redis。
> **本 Story 落「兽医在线态」这一半**——它是 5.3 接单队列匹配的前置（只有在线兽医出现在可接单队列、收到新请求推送）。**队列态（WAITING 请求集合）在 5.3 落**，本 Story 不写 `consult_sessions`、不写任何状态迁移。在线态的 `BUSY` 子态（进行中会话占用）由 5.5 在接单时置位，本 Story 仅留枚举位。

### 强制护栏（架构 §Enforcement —— 违反即返工）

- **禁 MQ / 通用缓存 / 新中间件**：在线态用 Redis 原生键/集合 + TTL，**不**引入 Redisson 分布式锁框架等额外中间件（V1 单机够用）；用 Spring Data Redis 原生操作。
- **Redis 收窄**：只存在线态键/集合，不存兽医档案、不存会话内容。
- **模块间经 service**：consult 的 availability 经 vet `VetPresenceService` 接口拿，不直访 vet 的 Redis 键名/repository。
- **角色门控**：在线态端点 `hasRole('VET')`；availability 对用户侧可见（含游客？按 5.3——发起需登录，availability 查询可放开或要 USER，依 5.3 入口约定，本 Story 与 5.3 对齐，先要 `role=user`）。

### 范围边界（防 scope creep —— 本 Story 明确不做）

- ❌ 不写 `consult_sessions` 表 / WAITING 状态 / 接单队列里的真实请求（Story 5.3）。
- ❌ 不做接单、不建 IM 会话、不做进行中会话内容（Story 5.5）。
- ❌ 不做历史记录列表内容、评分（Story 5.6 / 5.8）。
- ❌ 不做离线态完整软引导（恢复时段 + AI 分诊跳转）——本 Story 只落「暂无兽医在线」开关位，完整软引导在 Story 5.3。
- ❌ 不做封禁→离线钩子的封禁逻辑本身（Story 5.7，本 Story 仅留 `goOffline` 可被调用）。
- ✅ 只做：兽医独立四 Tab 框架（壳 + 导航 + 空态占位）+ 在线/离线开关 + Redis 在线态读写(TTL 兜底)+ 用户侧入口读「是否有兽医在线」bool。

**Redis 在线态数据结构（实际落地）**：
- 单一 sorted-set `vet:online`：`member=vetId`，`score=lastSeen(epochMillis)`。
- 「在线」判定 = 成员 lastSeen 在 TTL 窗口内（`TTL=3min`，> 心跳间隔 60s，留掉线宽限）。
- 惰性清理 `pruneStale()`（`ZREMRANGEBYSCORE` 移除过期成员）在 `anyOnline()`/`onlineVetIds()` 调用时触发 → **TTL 兜底离线，不依赖前端一定发离线请求**（防幽灵在线）。
- 选 sorted-set 而非「TTL 键 + 集合」：集合成员不会随 TTL 键自动失效会留幽灵，sorted-set 用 score 窗口 + 惰性清理一把解决。**未引入 Redisson**，纯 Spring Data Redis `opsForZSet`。
- `BUSY` 子态：仅留枚举位 `VetPresenceStatus.BUSY`，置位由 5.5 接单时做（本故事只读写 ONLINE/OFFLINE）。

**心跳实现**：前端 `VetMePage` 在线时 `Timer.periodic(60s)` 打 `POST /vet/heartbeat` 续期；App 退后台（`didChangeAppLifecycleState`）停止心跳 → TTL 兜底离线；登出调 `POST /vet/logout` 即时 `goOffline`。5.5 IM 在线可后续复用此续期点。

**只回 bool 不回人数**：`GET /consult/availability` 只 `{vetOnline:bool}`，经 `VetPresenceService.anyOnline()`，consult 不直访 vet 的 Redis 键名（模块间经 service）。
