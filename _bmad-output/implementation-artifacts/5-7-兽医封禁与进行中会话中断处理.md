# Story 5.7: 兽医封禁与进行中会话中断处理

Status: ready-for-dev

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

## Story

As a **运营**,
I want **封禁违规兽医并妥善处理其进行中的会话**,
so that **平台能及时止损且用户不被悬空**。

> 本 Story 在 5.1（Admin slice + `vet_accounts.status` ACTIVE/BANNED + 「BANNED 不可登录」）、5.5（IN_PROGRESS 会话 + IM）、5.6（PENDING_CLOSE）之上，落地**封禁的两件事**：① Admin 封禁兽医 → 立即无法登录（5.1 已落「BANNED 不可登录」，本 Story 补「正在登录的会话失效/被踢」）；② 封禁时该兽医有进行中会话 → 会话迁移 `INTERRUPTED`、用户侧系统消息 + 软引导、中断会话入历史标「已中断」、用户可立即重新发起。
>
> 做：Admin 封禁端点的「封禁即生效」（踢下线 + 在线态清除 + 不可再操作）、进行中会话（IN_PROGRESS 含 PENDING_CLOSE）批量迁移 INTERRUPTED、用户侧 IM 系统消息「兽医已临时下线…请重新发起」、中断会话历史标记、用户重新发起入口。
> 不做：封禁 UI 的完整 Admin 后台（5.1 已建 status 切换端点 + Thymeleaf，本 Story 复用并补会话中断副作用）、解封后会话恢复（中断不可恢复，用户须重新发起）、问诊历史列表渲染（5.8，本 Story 产 INTERRUPTED 数据 + 终态标签）、推送本身（Epic 6，本 Story 发事件 + IM 系统消息）。
>
> 依赖前序：5.1（封禁 status + Admin）、5.5（IN_PROGRESS + IM 系统消息能力 + 在线态 BUSY）、5.6（PENDING_CLOSE 也可能被封禁波及）。本 Story 是状态机的 `IN_PROGRESS/PENDING_CLOSE → INTERRUPTED` 旁路。

## Acceptance Criteria

> **验证层标注**：**L0**（状态迁移 / lint） · **L1**（需 Docker + pg + redis：封禁 status 落库、在线态清除、进行中会话批量迁 INTERRUPTED、历史标记、重新发起） · **L2**（需真实腾讯 IM + 真机：用户侧 IM 系统消息「兽医已临时下线」、被封兽医对话被中断、用户重新发起入会话）。
> 本 Story 状态机旁路与封禁副作用主体在 **L1**；用户侧中断系统消息与重新发起在 **L2**（IM）。

### AC1 — 封禁即生效（立即无法登录 + 踢下线）

**Given** 运营在 Admin 后台封禁某兽医账号
**When** 封禁生效
**Then** 该账号立即无法登录（FR-19、G-1）
> 验证层：Admin 封禁 `PUT /api/v1/admin/vets/{id}/status`(BANNED)（5.1 端点）→ `vet_accounts.status=BANNED`、已签发 JWT 失效/被拒（在线态清除 + 后续请求被 RoleGuard/状态校验拒）= **L1**；被封兽医正在用的 App 被踢回登录页 = **L2**。**注意**：5.1 已保证「BANNED 不能新登录」；本 Story 补「已登录会话立即失效」（不依赖 token 自然过期）。

### AC2 — 进行中会话迁移 INTERRUPTED + 用户侧处理

**Given** 封禁时该兽医有进行中会话
**When** 会话被中断
**Then** 会话迁移 INTERRUPTED，用户侧收到系统消息「兽医已临时下线，本次问诊已中断，请重新发起咨询」，中断会话进入用户问诊历史标记「已中断」，用户可立即重新发起
> 验证层：封禁副作用——查该 vet 所有 IN_PROGRESS（及 PENDING_CLOSE）会话 → 批量 CAS 迁 `INTERRUPTED` + 出在线态/队列 = **L1**；用户侧 IM 系统消息「兽医已临时下线…请重新发起」= **L2**（IM）；中断会话入历史标「已中断」(5.8 渲染) = **L1**（数据）；用户重新发起（解除「同时仅 1 个」占用，因原会话已 INTERRUPTED 非进行中）= **L1+L2**。**多态**：被封兽医的 WAITING 待接单请求（未接的）不受影响（那是用户的请求，非该兽医的会话）——只中断该 vet 已接的进行中会话。

## Tasks / Subtasks

> 三段组织。后端重（封禁副作用编排：踢下线 + 会话批量中断 + IM 系统消息 + 历史标记）。前端：用户侧中断处理 + 重新发起；被封兽医踢回登录。建议顺序：后端封禁编排 → 前端中断处理 → 联调（含 IM L2）。

### 🟦 后端子任务（petgo-backend / Spring Boot）

- [ ] **B1. 封禁即生效（踢下线）** (AC: 1)
  - [ ] `com.petgo.admin` / `vet.service.VetAccountService.ban(vetId)`（复用 5.1 status 切换）：置 `status=BANNED` → 清在线态（5.2 `VetPresenceService.goOffline`）→ 移出待接单队列匹配。
  - [ ] 已登录会话失效：JWT 校验链增加「vet status=BANNED → 拒」（`shared/security` 在 vet 端点校验账号 status，或维护失效标记），保证不等 token 自然过期即被拒。记实现方式（每次请求查 status vs 失效黑名单）。
- [ ] **B2. 进行中会话批量中断** (AC: 2)
  - [ ] `ConsultSessionService.interruptByVetBan(vetId)`：查该 vet 所有 `IN_PROGRESS`（及 `PENDING_CLOSE`）会话 → 逐个 CAS 迁 `INTERRUPTED`、记 `interrupted_reason=VET_BANNED`、记中断时间。
  - [ ] 每个中断会话经 IM 发系统消息给用户「兽医已临时下线，本次问诊已中断，请重新发起咨询」（`shared/im/TencentImClient`）。
  - [ ] 发 `ConsultInterruptedEvent`（过去式，供 notify 推送 Epic 6 + 历史更新）；**INTERRUPTED 不触发 FR-16 存档**（中断非正常结束，按 PRD 标「已中断」入历史，不强制存档；若需保留对话记录入历史由 5.8 处理，但不走 `ConsultClosedEvent` 存档桥接）。记 Dev Notes。
  - [ ] 封禁事务：封禁 status 与会话中断在同一业务流（status 落库 + 会话迁移 + 事件），失败回滚保证一致。
- [ ] **B3. 中断会话历史与重新发起** (AC: 2)
  - [ ] INTERRUPTED 会话进用户问诊历史（5.8 渲染），终态标签「已中断」。
  - [ ] 用户重新发起：原会话已 INTERRUPTED（非 WAITING/IN_PROGRESS/PENDING_CLOSE），不再占用「同时仅 1 个进行中」（5.3 约束）→ 用户可立即新发起。
  - [ ] 被中断会话的 IM 会话设为只读/不可继续发（用户侧打开是只读终态，5.8 只读态）。

### 🟩 前端子任务（petgo_app / Flutter）

- [ ] **F1. 用户侧中断处理** (AC: 2)
  - [ ] `features/consult/presentation`：进行中会话收到 INTERRUPTED（IM 系统消息/轮询）→ 对话转只读终态 + 展示「兽医已临时下线，本次问诊已中断」+ 软引导「重新发起咨询」按钮（复用 5.3 发起入口）。
  - [ ] 中断会话在问诊历史(5.8)显示「已中断」终态标签。
- [ ] **F2. 被封兽医踢回登录** (AC: 1)
  - [ ] `features/vet`：兽医端请求被拒（status=BANNED）→ 清登录态、踢回登录页（提示「账号已被停用，请联系运营」）。不可再进工作台/操作会话。
- [ ] **F3. Admin 封禁入口** (AC: 1)
  - [ ] 复用 5.1 Admin Thymeleaf 兽医列表：封禁按钮（ACTIVE→BANNED）+ 二次确认；封禁后列表显示 BANNED 态 + 解封入口（解封不恢复已中断会话）。

### 🟨 联调验收子任务

- [ ] **J1. 封禁即生效** (AC: 1 / **L1+L2**)
  - [ ] Admin 封禁 → status=BANNED、在线态清除；被封兽医正在用的 App 下次请求被拒、踢回登录；不可再登录（5.1）。
- [ ] **J2. 进行中会话中断** (AC: 2 / **L1+L2**)
  - [ ] 封禁有进行中会话的兽医 → 该会话迁 INTERRUPTED、用户收 IM 系统消息「兽医已临时下线…请重新发起」、会话转只读。PENDING_CLOSE 会话亦中断。
- [ ] **J3. 历史标记 + 重新发起** (AC: 2 / **L1+L2**)
  - [ ] 中断会话入历史标「已中断」；用户「同时仅 1 个」占用解除 → 立即重新发起成功进新会话。
- [ ] **J4. 多态边界** (AC: 2 / **L1**)
  - [ ] 被封兽医的 WAITING 待接单(未接)请求不受影响（仍是用户的请求，可被其他在线兽医接）；中断只针对该 vet 已接会话。

## Dev Notes

### 关键架构约定（本 Story 必须落实）

- **封禁中断在 Java 侧编排 IM**：架构 §API & Communication「后端通过腾讯 IM REST/回调建会话、发系统消息、**封禁中断**」。封禁副作用（踢下线 + 会话迁 INTERRUPTED + IM 系统消息）全在 Java 侧。
- **会话状态机 INTERRUPTED 旁路**：`IN_PROGRESS（/PENDING_CLOSE）→ INTERRUPTED`（架构状态机含 INTERRUPTED 终态）。这是与正常 `CLOSED` 并列的异常终态，不经评分门、不经正常存档。
- **Admin 封禁挂 admin slice（G-1）**：FR-19 封禁挂 `com.petgo.admin`（5.1 已建 status 端点 + Thymeleaf）；本 Story 复用并补会话中断副作用。
- **跨模块经事件**：`ConsultInterruptedEvent` 供 notify（推送 Epic 6）；admin/vet status 变更触发 consult 中断经 service 调用或事件——记选型（同步 service 调用更简单，保证封禁原子；事件用于推送/历史）。

### 会话状态机专项

> 本 Story 的旁路：`IN_PROGRESS →(封禁,本Story) INTERRUPTED`、`PENDING_CLOSE →(封禁,本Story) INTERRUPTED`。INTERRUPTED 是终态，**不经 PENDING_CLOSE、不评分、不走 `ConsultClosedEvent` 存档桥接**（中断非正常结束）。中断会话入历史标「已中断」（5.8 渲染只读终态），用户「同时仅 1 个进行中」占用随之解除可重新发起。
> 与 5.6 对比：5.6 是兽医主动正常结束 → PENDING_CLOSE → CLOSED → 存档；本 Story 是运营封禁强制中断 → INTERRUPTED → 不存档、提示重发。两条出口都让用户不被悬空。

### 强制护栏（架构 §Enforcement —— 违反即返工）

- **禁 MQ / 通用缓存 / 新中间件**：批量中断用 DB 事务 + CAS；踢下线用 5.2 Redis 在线态清除。
- **封禁即生效不依赖 token 过期**：JWT 校验链查 vet status（或失效标记），保证立即生效。
- **跨模块经 service / 事件**：封禁原子性优先（status + 会话中断同事务/同业务流）；推送/历史经 `ConsultInterruptedEvent`，不直调 notify repository。
- **角色门控**：封禁 `role=ADMIN`；中断后用户重新发起 `role=user`。
- **INTERRUPTED 不存档**：不误触发 `ConsultClosedEvent` / `ImToOssArchiver`（避免给中断会话错误存档）。

### 范围边界（防 scope creep —— 本 Story 明确不做）

- ❌ 不做 FR-19 封禁在其他场景的应用（如用户封禁——本 Story 仅兽医封禁 + 会话中断）。
- ❌ 不做解封后会话恢复（中断不可恢复，用户重新发起）。
- ❌ 不做问诊历史列表渲染 / 只读会话页结构（Story 5.8，本 Story 产 INTERRUPTED 数据 + 终态标签 + 只读语义）。
- ❌ 不做推送本身（Epic 6，本 Story 发 `ConsultInterruptedEvent` + IM 系统消息）。
- ❌ 不做 INTERRUPTED 会话存档（按 PRD「已中断」入历史，不走存档桥接）。
- ✅ 只做：封禁即生效(踢下线 + 在线态清除 + 已登录失效)+ 进行中(含 PENDING_CLOSE)会话批量迁 INTERRUPTED + 用户侧 IM 系统消息 + 软引导重新发起 + 历史「已中断」标记 + Admin 封禁入口(复用 5.1)。
