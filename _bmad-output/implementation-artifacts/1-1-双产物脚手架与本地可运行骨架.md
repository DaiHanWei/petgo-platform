# Story 1.1: 双产物脚手架与本地可运行骨架

Status: ready-for-dev

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

## Story

As a **开发者**,
I want **一套可运行的 Flutter 移动端 + Spring Boot 后端脚手架与本地编排**,
so that **团队能在统一约定（命名/分层/错误规范）上"改一行就跑起来"，为后续所有功能提供地基**。

> 这是整个实现阶段的**第一个 Story**（架构文档 §Implementation Handoff 明确指定）。本 Story 不实现任何业务功能（无登录、无 Feed、无分诊），只交付"双产物能本地跑起来 + 统一约定就位 + CI 绿灯"的地基。auth/triage 等业务从 Story 1.3 起在此骨架上叠加。

## Acceptance Criteria

> **验证层标注**：每条 AC 末尾标注验证层级与所需本地环境——
> **L0 静态**（编译/lint，无需 DB） · **L1 集成/运行时**（需 Docker daemon + postgres + redis） · **L2 端到端/外部凭证**（需真实第三方或真机模拟器）。
> 本 Story 全部 AC 落在 **L0 / L1**，无 L2——这是它适合作为首个自动化验收节点的原因（不依赖任何外部凭证）。

### AC1 — 双产物脚手架与目录骨架

**Given** 干净的开发环境
**When** 执行架构文档指定的两条 init 命令（`flutter create --org com.petgo --project-name petgo petgo_app` + Riverpod/go_router/dio/intl/flutter_lints；`spring init` Maven/SB4.0.x/Java25 含 web,validation,data-jpa,postgresql,data-redis,oauth2-resource-server,security,thymeleaf,actuator,flyway,lombok）
**Then** 生成的两套工程均能本地启动：`flutter run` 出空白首页、后端 `mvn spring-boot:run` 启动成功
**And** 后端按 8 模块 + shared 的 package-by-feature→layer 目录骨架建立，前端按 core/shared/features 骨架建立
> 验证层：**L0**（目录结构、`flutter analyze`、`mvn compile` 可纯静态验证）+ **L1**（`mvn spring-boot:run` 实际起进程需 postgres/redis 在跑，见 AC2；`flutter run` 需模拟器但出空白页不依赖后端）。

### AC2 — 本地编排、迁移基线、健康检查与错误/文档规范

**Given** 后端工程
**When** 启动 `docker-compose up`（app + PostgreSQL + Redis，德国单机配置占位）
**Then** Flyway 执行基线迁移成功、`GET /actuator/health` 返回 `UP`、Redis 连接就绪
**And** `@RestControllerAdvice` 全局异常处理统一输出 RFC 9457 ProblemDetail，springdoc 暴露 `/v3/api-docs`（OpenAPI 3.1）
> 验证层：**L1**（必须 Docker daemon + postgres + redis 真实在跑才能验 health=UP / Flyway 迁移 / Redis 就绪）。ProblemDetail 输出可用一个临时 `/api/v1/_ping-error` 端点或 MockMvc 测试在 **L0** 验证信封结构。

### AC3 — CI 骨架双绿

**Given** 代码仓库
**When** 推送触发 GitHub Actions CI 骨架
**Then** 后端 `mvn package` 出 jar、前端 `flutter build` 通过，CI 绿灯（部署步骤可占位）
> 验证层：**L0**（CI 里 `mvn package` 与 `flutter build` 均为编译/打包级，CI runner 用 service container 起 postgres 供测试阶段；部署步骤本 Story 仅占位 `echo`，不接德国机凭证 = 不引入 L2）。

### 关键架构约定（本 Story 必须落实，后续 Story 全程沿用）

**目录/分层**（架构 §Project Structure）：
- 后端 **package-by-feature → layer**：`com.petgo.<module>/{web,service,domain,repository,dto,event}` + `com.petgo.shared/{config,security,error,async,media,im,ai,ratelimit}`。
- 前端 **feature-first**：`lib/core/{network,router,theme,l10n,storage}` + `lib/shared/{widgets,utils}` + `lib/features/<feature>/{data,domain,presentation}`。
- 模块间只经 service 接口或领域事件通信，**禁跨模块直接访问对方 repository**；`shared/` 只放跨模块基础设施，不放业务。

**命名映射链**（核心，架构 §Naming）：DB `snake_case` ↔ Java/Dart `camelCase` ↔ JSON `camelCase`，由 JPA + Jackson 自动桥接。
- DB 表复数 snake_case；主键 `id`(bigint)；外键 `<单数>_id`；时间戳 `created_at`/`updated_at`(`timestamptz`, **一律 UTC**)；枚举落库 `varchar` + UPPER_SNAKE；Flyway 脚本 `V<序号>__<描述>.sql`。
- API：`/api/v1`，资源路径小写复数连字符（`/api/v1/pet-profiles`），查询参 camelCase，自定义头 `Idempotency-Key`/`Accept-Language`。
- Java：`XxxController/Service/Repository`、实体 `Xxx`、DTO `XxxRequest`/`XxxResponse`(record)、事件 `XxxEvent`(过去式)。
- Dart：文件 snake_case、类 PascalCase、provider `xxxProvider`、状态 `XxxState`(不可变 copyWith/freezed)。

**错误响应（统一 RFC 9457 ProblemDetail）** 示例：
```json
{ "type":"https://petgo/errors/validation", "title":"Validation Failed",
  "status":422, "detail":"...", "instance":"/api/v1/...", "traceId":"...",
  "errors":[{"field":"nickname","message":"≤20"}] }
```
**HTTP 状态码语义表**：200 读/改 · 201 创建 · 202 异步受理 · 400/422 校验 · 401 未认证 · 403 越权 · 404 不存在 · 409 冲突/重复 · 429 限流 · 5xx 服务端。

**成功响应格式**：不包通用 wrapper，直接返回资源；列表用游标分页 `{items,nextCursor,hasMore}`；camelCase；时间 ISO-8601 UTC；null 字段省略（Jackson NON_NULL）。

**日志**：SLF4J + logback **JSON**；MDC 带 traceId/userId；**严禁记录 PII/健康数据/令牌/签名 URL**。

### 强制护栏（架构 §Enforcement —— 违反即返工）

- 异步只用 `@Async` + DB 状态机，**禁止擅自引入 MQ / 通用缓存层 / 新中间件**（违背 V1 轻量姿态）。本 Story 不写异步，但**勿顺手加 Kafka/Caffeine/RabbitMQ**。
- `ddl-auto` 必须 `validate`（schema 归 Flyway），**禁用 `update`/`create`**。
- 新增对外暴露标识一律用不可枚举 token（本 Story 无对外标识，记住即可）。
- 凭证全部 env 注入，**绝不入库**。
- 红色态零变现 / 安全规则层"只升不降" —— 本 Story 不涉及，但写代码时勿埋违反点。

### 范围边界（防 scope creep —— 本 Story 明确不做）

- ❌ 不实现登录/OAuth/JWT（Story 1.3）——虽然 `oauth2-resource-server`/`security` 依赖已引入，本 Story 只让它们存在且不拦截，**SecurityConfig 先放行全部**（或 permitAll），避免脚手架阶段被 401 挡住 health/docs。
- ❌ 不做设计 token / Tab Bar 视觉 / 凸起+弧形缺口（Story 1.2）。
- ❌ 不做任何业务表（users 在 1.3 建、pet_profiles 在 2.2 建……）。
- ❌ 不接 Cloudflare/德国机真实部署（CI 部署步骤仅占位）。
- ✅ 只做：两套工程生成 + 目录骨架 + 本地编排起栈 + Flyway 基线 + ProblemDetail + OpenAPI + Actuator + i18n 空壳 + CI 双绿。
