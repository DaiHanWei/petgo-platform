# Story 6.3: 内容互动推送

Status: review

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

## Story

As a **内容作者**,
I want **我的内容被点赞或评论时收到提醒**,
so that **我能及时感知社区互动**。

> 本 Story 在 6.1 已建的统一推送出口 + 深链路由表上，接入**内容互动两个推送场景**（FR-22B）：被点赞、被评论。
> **依赖**：推送下发/写库/角标/深链复用 **Story 6.1**；互动事件源来自 **Epic 3（content 模块的点赞/评论）**——通过 content 发布的领域事件（如架构示例 `ContentLikedEvent → notify`）触发推送，**notify 不直访 content repository**。被评论深链复用 6.1 的评论区锚点能力。

## Acceptance Criteria

> **验证层标注**：**L0** 静态/单测 · **L1** 集成/运行时（Docker + postgres + redis） · **L2** 端到端/外部（真实 APNs/FCM 经腾讯 IM、真机、三态）。
> 推送送达与点击直达（含定位评论区）为 L2（真机）；自互动不推、逐条不合并、深链组装可在 L0/L1 验证。

### AC1 — 被点赞/被评论推送，点击跳详情（被评论定位评论区，FR-22B）

**Given** 他人对用户内容点赞/评论（来自 Epic 3 互动事件）
**When** 触发推送
**Then** 被点赞推「有人赞了你的 [内容标题/宠物名]」、被评论推「有人评论了你的内容，点击查看」，点击跳对应详情页（被评论定位评论区）（FR-22B）
> 验证层：**L1**（点赞/评论事件 → notify 写库 + 角标 +1 + IM 离线投递，需 pg+redis；标题/宠物名占位由事件携带或经 content service 取，不在 notify join content 表）+ **L2**（真机收推送、点击落详情页、被评论滚到评论区为端到端）。**L0**：深链组装（CONTENT_LIKED→详情 / CONTENT_COMMENTED→详情+`focus=comments`）单测。

### AC2 — 逐条不合并 + 自互动不推送

**Given** 触发互动推送
**When** 同一内容短时间多次被互动 / 用户对自己内容点赞评论
**Then** V1 逐条推送不合并；用户对自己内容互动不推送
> 验证层：**L1**（自互动判定：互动发起人 == 内容作者 ⇒ 不发，需造作者本人互动数据）+ **L0**（逐条不合并 = 每个互动事件独立 send，无聚合/去抖窗口；自互动过滤判定单测）。

## Tasks / Subtasks

> 三段组织。本 Story 形态 = **后端中（事件订阅 + 自互动过滤 + 推送触发）+ 前端轻（复用 6.1 详情/评论区深链落点）**。执行顺序：后端 → 前端 → 联调。

### 🟦 后端子任务（petgo-backend / Spring Boot）

- [ ] **B1. 订阅点赞事件触发推送** (AC: 1, 2)
  - [ ] notify 订阅 content 的 `ContentLikedEvent{contentToken, authorUserId, actorUserId, titleOrPetName}`（字段以 Epic 3 实际为准；标题/宠物名由事件携带，避免 notify join content 表）→ `NotificationService.send(type=CONTENT_LIKED, body=「有人赞了你的 [标题/宠物名]」, deepLink=详情 token)`。
- [ ] **B2. 订阅评论事件触发推送** (AC: 1, 2)
  - [ ] notify 订阅 `ContentCommentedEvent{contentToken, authorUserId, actorUserId}` → `send(type=CONTENT_COMMENTED, body=「有人评论了你的内容，点击查看」, deepLink=详情 token + 评论区锚点)`。
- [ ] **B3. 自互动过滤** (AC: 2)
  - [ ] send 前判定 `actorUserId == authorUserId ⇒ 跳过`（用户对自己内容点赞/评论不推送）。
- [ ] **B4. 逐条不合并** (AC: 2)
  - [ ] 每个互动事件独立调用一次 send，**V1 不做聚合/去抖/合并窗口**（不引入定时聚合任务、不引 MQ）。

### 🟩 前端子任务（petgo_app / Flutter）

- [ ] **F1. 互动推送深链落点** (AC: 1)
  - [ ] 复用 6.1 `pushPayloadToLocation`：`CONTENT_LIKED` → content 详情页（token）；`CONTENT_COMMENTED` → 详情页 + 评论区锚点（`?focus=comments`，落地后滚动并聚焦评论区）。
- [ ] **F2. 失效内容兜底** (AC: 1)
  - [ ] 目标内容已删除（content 软删 `deleted_at`）→ 走 6.1 not-found 兜底，不白屏、不回首页。

### 🟨 联调验收子任务

- [ ] **J1. 点赞/评论推送端到端** (AC: 1 / L2)
  - [ ] 他人点赞作者内容 → 作者真机收「有人赞了你的 [标题/宠物名]」→ 点击落详情；他人评论 → 收推送 → 点击落详情并滚到评论区。
- [ ] **J2. 自互动 + 不合并** (AC: 2 / L1)
  - [ ] 作者对自己内容点赞/评论 → 无推送；同内容连续 3 次他人互动 → 收 3 条独立推送（不合并）。
- [ ] **J3. 深链 + 过滤单测** (AC: 1, 2 / L0)
  - [ ] CONTENT_LIKED/CONTENT_COMMENTED 映射 + 评论区锚点 + 自互动过滤单测覆盖。

## Dev Notes

### 关键架构约定（沿用全局）

- 后端 `com.tailtopia.notify.{web,service,domain,repository,dto,event}`；事件源 content（Epic 3）经领域事件，**禁 notify 直访 content repository**。
- 前端复用 6.1 `core/router/deep_link_routes`（详情 + 评论区锚点）+ `features/content` 详情页。
- 标题/宠物名由事件携带（事件富化），避免跨模块 join。

### 推送专项约定

- 下发统一经 6.1 `NotificationService` → `shared/im` 离线推送，**不开新通道、不引 TPNS**。
- 角标计数复用 6.1 `notify:unread:{userId}`。
- 架构明示 `ContentLikedEvent → notify` 为模块间事件通信范式——本 Story 落地该范式。

### 强制护栏（违反即返工）

- **禁 MQ / 独立推送服务 / 聚合定时任务**；逐条 send，V1 不合并。
- **禁** notify 跨模块直访 content repository——经领域事件（事件携带标题/宠物名）。
- **禁** 深链带顺序 id（用内容 token）。
- 日志严禁记录内容正文/PII。

### 范围边界（本 Story 明确不做）

- ❌ 不做兽医回复/新请求推送（Story 6.2）。
- ❌ 不申请推送权限（Story 6.4）。
- ❌ 不实现点赞/评论功能本体（Epic 3——本 Story 仅订阅其事件触发推送）。
- ❌ V1 不做互动推送合并/聚合（明确逐条）。
- ✅ 只做：被点赞/被评论推送（含自互动过滤、逐条不合并）的事件订阅、推送触发与深链落点。

### Project Structure Notes

- notify 订阅 content 发布的 `ContentLikedEvent` / `ContentCommentedEvent`（名以 Epic 3 实际为准，过去式）。
- 评论区定位复用 6.1 已实现的 `focus=comments` 锚点参数。
