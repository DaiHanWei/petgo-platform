# Story 6.6: 全局通知中心（铃铛）

Status: ready-for-dev

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

## Story

As a **用户**,
I want **一个铃铛入口汇聚我的所有通知**,
so that **我能集中查看与管理未读消息**。

> 本 Story 实现**首页顶部铃铛入口 + 未读红色角标（Redis 计数）+ 通知中心列表（倒序 + 点击跳目标 + 标记已读）**（FR-34）。
> **依赖**：通知数据来自 **Story 6.1** 已建的 `notifications` 表（6.2/6.3 写入的四类通知：兽医已回复 / 问诊已结束 / 被点赞 / 被评论）；角标读 **6.1** 已建的 Redis 计数 `notify:unread:{userId}`；点击跳目标复用 **6.1** 深链路由表（FR-38）。本 Story 是 Epic 6 的**消费端/收口 Story**。

## Acceptance Criteria

> **验证层标注**：**L0** 静态/单测 · **L1** 集成/运行时（Docker + postgres + redis：列表查询、角标计数、标记已读）· **L2** 端到端/外部（深链点击直达三态、真机）。
> 列表/角标/标记已读为 L1（pg+redis）；点击条目深链直达复用 6.1 三态（L2 真机）。

### AC1 — 首页铃铛 + 未读红色角标（FR-34、与问诊 Tab 红点并存）

**Given** 首页顶部导航栏
**When** 渲染
**Then** 右侧显示铃铛图标，有未读时显示红色数字角标（与问诊 Tab 红点并存不互斥）（FR-34、FR-19 角标规则）
> 验证层：**L1**（角标数字 = Redis `notify:unread:{userId}` 计数，需 redis 在跑；有未读→显示数字、零→隐藏）+ **L0**（角标渲染：0 隐藏 / >0 显示 / 与问诊 Tab 红点共存不互斥的 widget test）。

### AC2 — 点击铃铛进通知列表，倒序展示四类，点击跳目标并标记已读

**Given** 用户点击铃铛
**When** 进入通知列表
**Then** 按时间倒序展示所有通知（兽医已回复/问诊已结束/被点赞/被评论），点击条目跳对应目标并标记为已读（FR-34、FR-38）
> 验证层：**L1**（列表按 `created_at desc` 拉取四类、点击条目 `is_read=true`+`read_at` 写入、Redis 角标递减/重算，需 pg+redis）+ **L2**（点击条目经 6.1 深链路由直达目标页，三态真机）。**L0**：列表项 → 深链 location 映射（复用 6.1 `pushPayloadToLocation`）+ 空列表态渲染单测。

## Tasks / Subtasks

> 三段组织。本 Story 形态 = **前端重（铃铛 + 角标 + 通知列表 UI + 标记已读交互 + 深链跳转）+ 后端中（列表查询端点 + 标记已读端点 + 角标计数维护）**。执行顺序：后端 → 前端 → 联调。

### 🟦 后端子任务（petgo-backend / Spring Boot）

- [ ] **B1. 通知列表查询端点** (AC: 2)
  - [ ] `GET /api/v1/notifications?cursor=&limit=`（需 JWT，仅返回当前用户 `recipient_user_id` 的通知）→ 游标分页 `{items,nextCursor,hasMore}`，按 `created_at desc`。item 含 `type, title, body, deepLinkType, deepLinkToken, isRead, createdAt`（**不返回顺序 id / target_ref 内部字段**，对外暴露 token）。
- [ ] **B2. 未读角标计数端点** (AC: 1)
  - [ ] `GET /api/v1/notifications/unread-count` → 读 Redis `notify:unread:{userId}` 返回 `{count}`。Redis 缺值时按 `notifications.is_read=false` 回算并回填（容错，不依赖角标键永不丢）。
- [ ] **B3. 标记已读端点** (AC: 2)
  - [ ] `POST /api/v1/notifications/{token}/read`（token 定位单条）→ `is_read=true` + `read_at` + Redis 角标递减（不低于 0）。可选 `POST /api/v1/notifications/read-all` 全标已读、角标清零。
  - [ ] 越权防护：只能标记自己的通知（`recipient_user_id == 当前用户`），否则 403。
- [ ] **B4. 角标计数一致性** (AC: 1, 2)
  - [ ] 与 6.1 send（INCR）协同：标记已读 DECR、读取容错回算。**Redis 仅角标用途**，不当通用缓存。

### 🟩 前端子任务（petgo_app / Flutter）

- [ ] **F1. 首页顶部铃铛 + 角标** (AC: 1)
  - [ ] 首页顶栏右侧铃铛图标（`features/notify/presentation` + `shared/widgets`）；角标用 `unread-count`：0 隐藏、>0 显示红色数字。**与问诊 Tab 红点并存不互斥**（遵守 FR-19 角标规则）。
  - [ ] 角标随推送到达 / 列表标记已读刷新（进入前台/收到推送时拉一次 unread-count）。
- [ ] **F2. 通知中心列表** (AC: 2)
  - [ ] 点击铃铛进 `features/notify` 列表页：游标分页倒序展示四类通知；每类图标/文案区分（兽医已回复/问诊已结束/被点赞/被评论）；未读项视觉高亮。
  - [ ] **空态**：无通知时展示空态（复用 `shared/widgets/empty_state`），不白屏。
- [ ] **F3. 点击条目跳目标并标记已读** (AC: 2)
  - [ ] 点击条目：调 `POST /notifications/{token}/read` → 经 6.1 `pushPayloadToLocation(deepLinkType, deepLinkToken)` → `router.go` 目标页；目标失效走 6.1 多态兜底（not-found/已关闭/已中断）。
  - [ ] 已读后该项更新为已读态、角标递减。
- [ ] **F4. 双语文案** (AC: 1, 2)
  - [ ] 铃铛/列表/空态/分类标签走 i18n（id/en），无写死字符串。

### 🟨 联调验收子任务

- [ ] **J1. 角标计数验收** (AC: 1 / L1)
  - [ ] 6.2/6.3 触发若干推送 → 角标显示正确未读数；与问诊 Tab 红点同时存在不互斥；标记已读后角标递减、清空后隐藏。
- [ ] **J2. 列表倒序 + 四类验收** (AC: 2 / L1)
  - [ ] 造四类通知 → 列表 `created_at desc` 正确排序、四类齐全、未读高亮；空账号显示空态。
- [ ] **J3. 点击跳目标 + 标记已读** (AC: 2 / L1+L2)
  - [ ] 点击「兽医已回复」→ 标记已读 + 深链直达会话；点击「被评论」→ 直达详情评论区；指向已删内容的条目 → 标记已读 + 落 not-found 兜底。
- [ ] **J4. 越权 + 容错** (AC: 1, 2 / L1)
  - [ ] 标记他人通知 → 403；Redis 角标键缺失 → unread-count 按库回算正确。
- [ ] **J5. 映射/空态单测** (AC: 2 / L0)
  - [ ] 列表项 → 深链 location 映射（复用 6.1）+ 空列表态 + 角标 0/>0 渲染单测。

## Dev Notes

### 关键架构约定（沿用全局）

- 后端 `com.petgo.notify.{web,service,domain,repository,dto,event}`；读自己 6.1 建的 `notifications` 表与 Redis 角标键，**不跨模块 join**。
- 前端 `lib/features/notify/{data,domain,presentation}`（铃铛/角标/列表）+ 复用 `lib/core/router/deep_link_routes`（6.1）。
- API：游标分页 `{items,nextCursor,hasMore}`、camelCase、ISO-8601 UTC、null 省略；对外只暴露 token。

### 推送/通知中心专项约定

- **未读角标 = Redis 计数** `notify:unread:{userId}`（6.1 建，send INCR / 标记已读 DECR / 读取容错回算）——[architecture.md §缓存 Redis 收窄：未读角标计数]。
- 点击条目跳目标复用 **6.1 深链路由表（FR-38）**，目标失效复用 6.1 多态兜底。
- 角标与问诊 Tab 红点**并存不互斥**（FR-19 角标规则）。

### 强制护栏（违反即返工）

- **禁** Redis 当通用缓存（角标键仅未读计数用途）。
- **禁** 跨模块直访 consult/content repository——通知内容已在 `notifications` 表持久化（6.1/6.2/6.3 写入），列表直接读自己的表。
- **禁** 对外返回顺序 id / `target_ref` 内部字段——只暴露 token + deepLinkType。
- 越权：只能读/标记自己的通知（403 防护）。
- 日志严禁记录通知正文中的 PII/健康相关文本。

### 范围边界（本 Story 明确不做）

- ❌ 不建 `notifications` 表 / 不建角标计数键（Story 6.1 已建，本 Story 消费）。
- ❌ 不实现推送下发（6.2/6.3 写入通知，本 Story 只读展示）。
- ❌ 不建深链路由表本体（6.1，本 Story 复用）。
- ❌ V1 不做通知分类筛选/删除单条/免打扰设置（除非 PRD 另有要求）；不做服务端推送合并。
- ✅ 只做：首页铃铛 + 未读角标 + 通知中心列表（倒序四类 + 空态）+ 点击跳目标 + 标记已读 + 对应后端读取/已读/计数端点。

### Project Structure Notes

- 铃铛入口在首页顶栏（与 Tab Bar/凸起+FAB 视觉体系协同，FR-19 角标规则）。
- 标记已读用 token 定位单条；列表 item 的 deepLink 字段由 6.1 写入时落库，本 Story 直读，保证前后端深链口径一致。
