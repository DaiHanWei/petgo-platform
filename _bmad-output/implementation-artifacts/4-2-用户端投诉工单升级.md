# Story 4.2: 用户端投诉工单升级

Status: review

> V1.1 **Epic 4 第 2 story**，**纯前端**（消费 4-1 已建后端 API，本 story 不建后端端点）。把 V1.0「帮助反馈 = WhatsApp/邮箱直连抽屉」升级为**在线提交投诉工单**：投诉表单（主题/正文 + 联系方式类型+值 + 标签多选 + ≤5 图附件 + 可选关联订单）→ `POST /api/v1/support-tickets` → 我的工单列表 + 工单详情。附件复用**既有 media 预签名直传**（`mediaUploadUseCaseProvider`），前端拿 objectKey 回传。
> 源：`epics-v1.1.md` Story 4.2（提投诉能附图 + 留联系方式，客服更快联系；`POST /support-tickets` 提交带附件与联系方式 → 上传成功、工单可查 L1；`flutter analyze` 绿 L0；模拟器验收 L2）· FR-52 · AB-5A~5H · 架构 `architecture-v1.1-delta.md` §3.6 + 端点表 line 235（`/support-tickets` POST user）。
> **⚠️ 承接 4-1（后端已就位，勿重建）**：4-1 已建 `com.tailtopia.support` 后端全套并提交（`8b60333`）——`POST /api/v1/support-tickets`（建单，返回用户视图）、`GET /api/v1/support-tickets`（我的工单列表）、`GET /api/v1/support-tickets/{ticketToken}`（详情，owner 404）。门控 `role=USER` 已配。**本 story 只做前端**（Flutter），不碰后端。
> **范围边界**：本 story = 投诉入口升级（`customer_service_sheet` 加「在线提交投诉」CTA + 「我的工单」入口）+ 投诉表单页（含 ≤5 图上传）+ 我的工单列表页 + 工单详情页 + repository/domain/route/i18n。**不做**：后端（4-1 已完）、退款申请（4-3）、客服判定/解锁回落（4-4）、CSAT 问卷填写 UI（4-7，详情页 `csat*` 字段本 story 只读展示占位/不做）、admin 处理（4-4/4-7）、**详情页历史附件缩略图现签**（见 Dev Notes D-4，V1 只显数量）。

## Story

As a 用户,
I want 提投诉时能附图并留联系方式、并能查看我提过的工单,
so that 客服更快联系我、我也能追踪工单状态（FR-52，AB-5A~5H）。

## Acceptance Criteria

1. **前端：投诉入口升级（L0/L2）**
   **Given** V1.0「我的」页顶栏「帮助反馈」图标 → `showCustomerServiceSheet`（WhatsApp/邮箱直连抽屉）
   **When** 升级入口
   **Then** 客服抽屉**保留** WhatsApp/邮箱直连行（勿删，紧急直连仍要），**顶部新增主 CTA**「在线提交投诉」→ push 投诉表单页 `/me/support-tickets/new`；**新增**「我的工单」入口 → push 列表页 `/me/support-tickets`
   **And** `flutter analyze` 零 issue（**L0**）；模拟器点入口跳转正确（**L2**）

2. **前端：投诉表单页（≤5 图 + 联系方式 + 标签）（L0/L1/L2）**
   **Given** 投诉表单页
   **When** 用户填写并提交
   **Then** 表单含：主题（可空）、正文（**必填**，空则禁用提交/提示）、联系方式类型单选（`EMAIL`/`WHATSAPP`，本地化标签）、联系方式值（**必填**）、标签多选（8 个 `TicketLabelType` 本地化 chip，可 0 选）、≤5 图附件（复用 `mediaUploadUseCaseProvider.pickAndUploadOne(scope: MediaScope.private)` 直传拿 objectKey，**满 5 张禁再加**，上传中禁提交）、可选关联订单（见 Dev Notes D-3，本 story 默认不做手动选单，`relatedOrderToken` 传 null）
   **And** 提交调 `POST /api/v1/support-tickets`（body `{subject?, body, contactType, contactValue, needContact, relatedOrderToken?, labels[], attachmentObjectKeys[]}`）；成功（201）→ `pushReplacement` 到工单详情页 + 成功 toast；`DioException` → 失败 toast（不清空表单，可重试）（**L1** 真机/模拟器提交落库可查，**L2** 视觉）
   **And** `flutter analyze` 绿（**L0**）

3. **前端：我的工单列表页（L0/L2）**
   **Given** `GET /api/v1/support-tickets`（`List<SupportTicketView>`，created_at 倒序）
   **When** 进列表
   **Then** 卡片列表（主题或正文首行摘要 + status 本地化徽章 + 标签 + 相对时间）；空态友好文案「暂无工单」；点卡片 → 详情 `/me/support-tickets/{ticketToken}`；下拉刷新
   **And** 无数据/网络失败按空态兜底不崩（**L0/L2**）

4. **前端：工单详情页（L0/L2）**
   **Given** `GET /api/v1/support-tickets/{ticketToken}`（`SupportTicketView`，非本人后端 404）
   **When** 查看详情
   **Then** 展示：主题、正文、联系方式（类型+值）、标签、status 本地化、创建时间、附件**数量**（「已附 N 张图片」，**不渲染缩略图**——见 D-4）、`needContactCustomer`/`contactedCustomer` 状态提示（可选）；404 → 友好「工单不存在」页不崩
   **And** **绝不展示** `handledBy`/内部备注等（4-1 用户视图 DTO 本就不含，前端亦不构造内部字段）；CSAT 字段（`csatScore`/`csatComment`）本 story 只读占位或不展示（填写 UI 留 4-7）（**L0/L2**）

5. **前端：静态与视觉验收（L0/L2）**
   **Given** 全部实现
   **When** 验证
   **Then** `flutter analyze` 零 issue + `flutter test`（新增 model/widget 测试）绿（**L0**）；紫色主题（`AppColors`，勿硬编码色值）+ 双语 arb（en/id，无缺键）
   **And** 模拟器（Android）走通：入口 → 表单填 3 图+联系方式+标签 → 提交 → 详情 → 返回列表可见（**L2**，连真后端 `api.tailtopia.id` 或本地）

### 关键设计决策（dev 照此）

- **D-1 新 `features/support/` 前端模块**（domain/data/presentation），照 `consult`/`vet` 分层。Riverpod provider + go_router route + dio repository。
- **D-2 联系方式**：`contactType` 单选（EMAIL/WHATSAPP）+ `contactValue` 自由文本（必填）。前端传枚举名字符串（`"EMAIL"`/`"WHATSAPP"`），后端已按 String 解析（非法 422，但前端单选不会非法）。`needContact` 默认 true（可加开关，默认勾选）。
- **D-3 关联订单 `relatedOrderToken`（本 story 传 null）**：后端支持传 orderToken 关联咨询订单（归属不符忽略存 null）。本 story **默认不做手动选单 UI**，`relatedOrderToken=null`。未来「从订单详情/咨询结果发起投诉」带 token 预填留后续（4-3 退款工单才强绑单）。dev 若行有余力可加「从订单跳投诉预填」，非必需。
- **D-4 附件展示 = 只显数量（V1）**：⚠️ 4-1 `SupportTicketView.attachmentObjectKeys` 返 **objectKey 非签名 URL**（4-1 因 dev 无 OSS 凭证 `signAll→buildClient` 抛 500，走 OPEN-2 返 objectKey）。项目既有私密图展示范式是**后端 DTO 返签名 URL**（`consult` `caseContext`/`aiContext`），但 consult L1 靠**空图规避** signAll（现签留 L2/生产）。故本 story **详情页不渲染历史附件缩略图**，只显「已附 N 张图片」。**表单页**可展示**刚 pick 的本地 bytes 预览**（`_PickedPhoto.bytes`，同病例表单，无需签名）。若产品要求详情页缩略图 → 需后端补 `SupportTicketView` 附件现签 URL（照 consult 范式 + L1 mock `SignedUrlService` 或空图 + L2 验证），属独立增量，标 [OPEN-1]。
- **D-5 隐私红线（前端亦守）**：详情页**绝不**展示/请求内部字段（后端 DTO 本就无 `handledBy`/内部备注）。前端不臆造内部字段。
- **D-6 status 展示**：OPEN/IN_PROGRESS/RESOLVED/CLOSED 本地化徽章（本 story 用户端只读；流转由 admin 4-4/4-7）。CSAT（`csatScore`/`csatComment`）本 story 只读占位或不展示，填写 UI 留 4-7。
- **无迁移 / 无后端改动**：本 story 纯前端。`api_paths` 加常量、`app_router` 加路由属前端。
