# Story 3.3: 兽医接单与限时支付窗

Status: ready-for-dev

> V1.1 **Epic 3 第三个** story，**纯后端**。承接 3-1（`tryAccept` CAS 已建）+ 3-2（QUEUEING 入队 + 广播 + 超时删）。兽医接单 → CAS `QUEUEING→ACCEPTED_AWAIT_PAY` + 开 1.5 分钟限时支付窗（`pay_deadline_at=+1.5min`）、兽医侧看到「等待支付」倒计时中间态（FR-53A）；服务端权威计时 + `@Scheduled` 扫支付窗过期 → **作废本次接单、回 QUEUEING 重播**（vet 释放）。**无迁移**（3-1 已建表）。
> 源：`epics-v1.1.md` Story 3.3（`POST /consultations/{token}/accept` CAS；接单成功置 ACCEPTED_AWAIT_PAY、pay_deadline=+1.5min、兽医侧同一倒计时；服务端权威计时、@Scheduled 扫过期）· 架构 `architecture-v1.1-delta.md` §3.2/§5.1（两表流、服务端权威计时、CAS 单列 H-4）· FR-53A（兽医「等待支付」中间态）。
> **⚠️ brownfield 关系**：V1.0 兽医接单 `POST /api/v1/vet/consult-sessions/{id}/accept`（`ConsultAcceptService`，`consult_sessions` WAITING→IN_PROGRESS **即建 IM 会话免费流**）不动。**V1.1 计费流接单是新端点**（接 `consult_requests`，接单**不建会话**——会话在支付成功 3-4 才建）。复用 `VetPresenceService.goBusy/goAvailable/isBusy`、`currentVetId(jwt)`、3-2 广播事件、3-2 `@Scheduled` scanner 范式。
> **范围边界（与 3-4 切分）**：本 story = 接单 CAS + 开支付窗（pay_deadline）+ 兽医占用（goBusy）+ 兽医侧待支付视图 + **支付窗超时作废接单回 QUEUEING 重播（vet 释放 + rebroadcast_count++）**。**不做**：`POST /consultations/{token}/pay` 支付建单/建会话（3-4）、重播**上限**（5 次/30min → `failed_consult_requests`，3-4）、跳充值**暂停顺延**（A-4，3-4）、支付成功 IM 建会话失败退款（3-4）、前端（3-5/3-6）、既有 V1.0 accept 改动。

## Story

As a 兽医,
I want 接单后系统给用户开 1.5 分钟限时支付窗、我侧看到「等待支付」倒计时,
so that 我知道这单是否会成、不空等（FR-53A）。

## Acceptance Criteria

1. **后端：兽医接单（CAS，接单不建会话）（L1）**
   **Given** 3-1 `ConsultRequestRepository.tryAccept`（单列 state CAS）+ 3-2 QUEUEING 请求
   **When** 兽医 `POST /api/v1/vet/consultations/{requestToken}/accept`（`hasRole('VET')`，`vetId` 取自 JWT）
   **Then** 按 token 解析请求 → `tryAccept(id, vetId, payDeadline=now+90s)`（CAS `WHERE state=QUEUEING`）：**1 行=接单成功**（置 `ACCEPTED_AWAIT_PAY`、`vet_id`、`pay_deadline_at`）+ `presence.goBusy(vetId)`；**0 行=已被抢/已过期删** → **409**「该请求已被接单或已过期」（先到先得，H-4）；**接单绝不建 `consult_sessions`/`consult_orders`**（会话/订单在 3-4 支付成功才建）（**L1**）
   **And** **兽医占用互斥**：`presence.isBusy(vetId)` 时拒绝接新单（**409**「您有进行中的接单」），避免一兽医占多单（**L1**）

2. **后端：限时支付窗 + 兽医侧待支付倒计时（服务端权威）（L0/L1）**
   **Given** 接单成功
   **When** 返回 / 兽医侧查看
   **Then** accept 响应含 `{requestToken, state=ACCEPTED_AWAIT_PAY, payDeadlineAt}`（服务端权威 timestamptz，兽医侧据此渲染「等待支付」倒计时中间态，FR-53A）；`pay_deadline_at` 为 DB 时间 `now + 90s`（1.5min），**不信客户端计时**；提供兽医侧查询（accept 响应即含，或 `GET` 待支付态——工作台轮询在 3-6）（**L0** 常量/响应 / **L1** 落库时间）

3. **后端：支付窗超时作废接单、回 QUEUEING 重播（@Scheduled，无 MQ）（L1）**
   **Given** `pay_deadline_at`（服务端权威）
   **When** 到期仍 `ACCEPTED_AWAIT_PAY`（用户未支付）
   **Then** `@Scheduled`（照 3-2/`ConsultCloseScanner` 范式，**禁 MQ**）扫过期接单 → 逐条 **CAS 回退**（`WHERE state=ACCEPTED_AWAIT_PAY AND pay_deadline_at < now`）：置回 `QUEUEING`、清 `vet_id`/`pay_deadline_at`、`rebroadcast_count++`、`queue_deadline_at=now+1min`；回退成功则 `presence.goAvailable(vetId)`（释放兽医）+ **重播**（发 3-2 广播事件，再次通知在线兽医）（**L1**）
   **And** 回退用**单列 state CAS**（非时间戳比较，H-4）；已支付（3-4 转订单删 request）/ 已被其它路径处理的行不受影响；回退幂等可重扫（**L1**）
   **And** **重播上限**（5 次/30min → 彻底失败写 `failed_consult_requests`）**属 3-4**，本 story 只做「作废回队 + 重播通知」不设上限（3-4 加封顶，防无限重播）（**L1**，本 story 标注上限缺口由 3-4 补）

4. **后端：契约 + 归属/鉴权（L0/L1）**
   **Given** 兽医端点
   **When** 调用
   **Then** 非 VET 角色 403、缺 JWT 401、不存在 token → 409（与「已被接单」同码防枚举）；accept 响应 DTO 契约（`requestToken`/`state`/`payDeadlineAt`）；`consult_orders` 全程零新增（接单阶段不建单）（**L0** DTO / **L1** 鉴权/契约）

## Dev Notes

> 纯后端。**核心复用 3-1 CAS（tryAccept）+ 3-2 广播事件/scanner 范式 + VetPresenceService**。新增只有：接单 service（CAS + goBusy + 占用互斥）、兽医端点、支付窗超时**回退 CAS**（作废接单回队重播）。**接单不建会话/订单**（3-4 才建）是本 story 红线。

### 承接现状（已 review，照此复用）
- **3-1**：`ConsultRequestRepository.tryAccept(id, vetId, payDeadline)`（`@Modifying UPDATE ... WHERE id AND state=QUEUEING` 返回行数，FQN 枚举字面量）；`findByRequestToken`；`deleteIfState`。`ConsultRequest`（`state`/`vetId`/`payDeadlineAt`/`rebroadcastCount`/`queueDeadlineAt`，工厂 `queue`）。**取消/超时物理删、无 CANCELLED。**
- **3-2**：`ConsultRequestService`（`createRequest` 占用+入队+广播、`purgeExpiredQueue`）；`ConsultRequestTimeoutScanner`（`@Scheduled(:30000)` 调 `purgeExpiredQueue`，try/catch）；`ConsultRequestQueuedForBillingEvent(requestId)`（→ `ConsultNotifyListener` 广播在线兽医）。**本 story 接单成功不发广播（已被接走）；超时回退时重发此事件（re-broadcast）。**
- **既有 vet 端（不动）**：`VetConsultController`（`@RequestMapping("/api/v1/vet/consult-sessions")`，`hasRole('VET')` 由 `SecurityConfig`，`currentVetId(jwt)` 私有静态）；`ConsultAcceptService.accept(vetId, sessionId)`（**V1.0 consult_sessions WAITING→IN_PROGRESS + IM 建会话**，与本 story 无关）。本 story 新端点 `/api/v1/vet/consultations/{token}/accept` 接 `consult_requests`。
- **VetPresenceService**：`goBusy(vetId)`（ONLINE→BUSY 不接新单）、`goAvailable(vetId)`（BUSY→ONLINE）、`isBusy(vetId)`。**接单 goBusy、支付窗超时回退 goAvailable。**（在线态显式、无 TTL——记忆库 vet-presence-explicit-only。）

### 关键设计（dev 照此）
- **接单不建会话/订单（红线）**：CAS 只把 `consult_requests` 从 QUEUEING 推到 ACCEPTED_AWAIT_PAY + 开支付窗。**IM 会话、`consult_orders` 都在 3-4 支付成功才建**。L1 断言接单阶段 `consult_orders` 零新增、无 `consult_sessions` 新建。
- **H-4 单列 CAS**：接单 `tryAccept`（`WHERE state=QUEUEING`）、回退 `revertExpiredAcceptance`（`WHERE state=ACCEPTED_AWAIT_PAY AND pay_deadline_at<now`）——均单列 state 谓词（非时间戳比大小判先后；pay_deadline 只作过期筛选谓词，判定归属仍看 state）。并发接单恰一兽医胜。
- **goBusy/goAvailable 时序**：CAS 写在 `@Transactional` 内；`goBusy`（Redis）建议放事务**提交后**（避免 CAS 回滚但 Redis 已置 BUSY 的不一致）——可用 `TransactionSynchronization.afterCommit` 或 service 内 CAS 成功后置（本 story 单机、Redis 与 DB 非同事务，取「CAS 成功即置 BUSY，回退时 goAvailable」；极端崩溃残留 BUSY 由兽医重上线/9-x 兜底，非资金不苛求强一致）。dev 择稳妥实现并注释。
- **支付窗超时回退 = 作废接单回队重播**：架构 §5.1「支付超时 → 作废接单 → rebroadcast」。本 story 做「回退 CAS（QUEUEING + 清 vet + rebroadcast_count++ + 新 queue_deadline）+ goAvailable + 重播事件」。**重播上限（5 次/30min → `failed_consult_requests`）留 3-4**——本 story 不设上限，Dev Notes 显式标注，3-4 必须补封顶防无限重播（否则一直没人付会永久循环）。
- **服务端权威计时（A-4/A-5）**：`pay_deadline_at`/`queue_deadline_at` 皆 DB timestamptz，扫描/判定以 DB 为准。兽医侧倒计时（FR-53A）用 accept 响应的 `payDeadlineAt` 渲染（前端 3-6）。**跳充值暂停顺延（A-4，`paused_at`）属 3-4**，本 story 不处理暂停。
- **端点归属**：兽医动作走 `/api/v1/vet/` 前缀（既有惯例），token 寻址（不可枚举）。不存在 token 与「已被接单」返同一 409 防枚举。
