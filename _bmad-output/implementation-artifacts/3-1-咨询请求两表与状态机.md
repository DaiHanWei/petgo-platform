# Story 3.1: 咨询请求两表与状态机

Status: review

> V1.1 **Epic 3（兽医咨询计费流）开篇** story，**纯后端数据层地基**。建 A-5 两表拆分模型：`consult_requests`（付费前临时 CAS 状态机，取消/超时即删、不进订单中心）+ `consult_orders`（支付成功才建、持久、进订单中心）+ 子表 `consult_order_stage_events`（append-only 节点时间戳）。**本 story 只建表 + 实体 + repo（含单列 CAS 原语）+ 并发 CAS 测试**——入队/接单/支付端点在 3-2/3-3/3-4。
> 源：`epics-v1.1.md` Story 3.1（A-5 两表、未扣费不建单、订单中心无「已取消」态）· 架构 `architecture-v1.1-delta.md` §3.2（consult 两表字段全集）· 决策 A-5（两表拆分根因）/ H-4（accept/cancel 竞态走 `state` 单列 compare-and-set，非时间戳）· 决策 E2（Flyway 单调顺延）。
> **⚠️ brownfield 关系（务必理解，勿重建）**：V1.0 已有 `consult_sessions`（V14，`SessionStatus{WAITING,IN_PROGRESS,PENDING_CLOSE,CLOSED,INTERRUPTED,CANCELLED}`，免费 queue→accept→session）、`ConsultQueueService`（Redis ZSET 待接单）、`failed_consult_requests`（V37）、`consult_anomalies`（V44）。**V1.1 两表是新增的「计费前置层」**：`consult_requests`=付费前临时态（新），`consult_orders`=计费订单（新），IM 会话仍用既有 `consult_sessions`（在 3-4 支付后建）。**本 story 不改 `consult_sessions`/`ConsultQueueService`/既有 consult 代码**，只加新表新实体新 repo。
> **范围边界**：3 张新表 + 实体/枚举 + repo（CAS accept、条件 delete、建单、stage event 追加）。**不做**：`POST /consultations` 入队（3-2）、accept 端点（3-3）、pay 建单/建会话（3-4）、前端（3-5/3-6）、兽医分成计算（3-7）、封禁挂起（3-8）、既有 consult_sessions 逻辑改动。

## Story

As a 平台,
I want `consult_requests`（付费前临时 CAS）+ `consult_orders`（支付成功才建）两表,
so that 未扣费不建单、取消/超时不留痕、订单中心无「已取消」态（A-5）。

## Acceptance Criteria

1. **后端：三张表（V66 迁移）（L0/L1）**
   **Given** A-5 两表拆分 + 架构 §3.2 字段全集
   **When** 建计费数据层
   **Then** 新建 Flyway **`V66__init_consult_billing.sql`**（接 2-3 的 V65 顺延，决策 E2）：
   - **`consult_requests`**（付费前临时态）：`id` PK；`request_token` VARCHAR 不可枚举唯一；`user_id` bigint NOT NULL；`pet_profile_id` bigint NOT NULL；`vet_id` bigint 可空（接单前 null）；`state` VARCHAR(24) NOT NULL CHECK `IN('QUEUEING','ACCEPTED_AWAIT_PAY')`（**仅两活态；取消/超时即删行、无 CANCELLED 态**，H-4/A-5）；`queue_deadline_at` timestamptz（入队 1min）；`pay_deadline_at` timestamptz 可空（接单后 +1.5min）；`paused_at` timestamptz 可空（跳充值暂停锚 A-4）；`rebroadcast_count` int NOT NULL DEFAULT 0；`created_at`/`updated_at`
   - **`consult_orders`**（支付成功才建，持久）：`id` PK；`order_token` VARCHAR 不可枚举唯一；`user_id`/`vet_id`/`pet_profile_id` bigint NOT NULL；`status` VARCHAR(16) NOT NULL CHECK `IN('IN_PROGRESS','COMPLETED','REFUNDING','REFUNDED')`（**无 CANCELLED**——未扣费不建单）；`amount` bigint NOT NULL；`pay_channel` VARCHAR(16) CHECK `IN('QRIS','PAWCOIN')`；`payment_intent_id` bigint 可空（关联 payment_intents）；`vet_payout` bigint（分成快照，如 Rp30,000）；`vet_share_rate_snapshot` int；`unit_price_snapshot` bigint（成交时快照，后台改价不影响历史）；`refund_rejected` boolean NOT NULL DEFAULT false（A-2 驳回回落 COMPLETED 置真）；`session_started_at`/`session_ended_at`/`paid_at` timestamptz 可空；`created_at`/`updated_at`
   - **`consult_order_stage_events`**（子表，append-only 历史）：`id` PK；`consult_order_id` bigint NOT NULL；`event_type` VARCHAR(24) NOT NULL CHECK（`ACCEPTED`/`PAID`/`SESSION_STARTED`/`SESSION_ENDED`/`REFUND_REQUESTED`/`REFUND_COMPLETED`/`REFUND_REJECTED`）；`occurred_at` timestamptz NOT NULL；`note` VARCHAR 可空；`created_at`（**保留历史不覆盖**——只 INSERT，无 UPDATE/DELETE）
   **And** 唯一索引 `uq_consult_requests_token`/`uq_consult_orders_token`；索引 `idx_consult_requests_state`（供 @Scheduled 扫超时）、`idx_consult_order_stage_events_order`；VARCHAR 列长度≥16 无 CHAR(1) 坑（**L0/L1**）

2. **后端：实体 + 枚举（UPPER_SNAKE，@Enumerated STRING）（L0）**
   **Given** 三表
   **When** 映射
   **Then** `ConsultRequest`（+`ConsultRequestState{QUEUEING,ACCEPTED_AWAIT_PAY}`）、`ConsultOrder`（+`ConsultOrderStatus{IN_PROGRESS,COMPLETED,REFUNDING,REFUNDED}`）、`ConsultOrderStageEvent`（+`ConsultStageEvent{ACCEPTED,PAID,SESSION_STARTED,SESSION_ENDED,REFUND_REQUESTED,REFUND_COMPLETED,REFUND_REJECTED}`）；枚举落库 varchar UPPER_SNAKE + `@Enumerated(STRING)`，照既有 `SessionStatus`/`PawCoinTxnType` 范式；money/timestamp 遵命名映射链（bigint IDR、timestamptz UTC）（**L0**）

3. **后端【H-4 核心】：单列 state compare-and-set 原语（L1）**
   **Given** accept 与 cancel/timeout 高并发竞争同一 `QUEUEING` 请求
   **When** 走 CAS repo
   **Then** `ConsultRequestRepository`：
   - `tryAccept(id, vetId, payDeadline)` = `@Modifying UPDATE consult_requests SET state='ACCEPTED_AWAIT_PAY', vet_id=:vetId, pay_deadline_at=:payDeadline WHERE id=:id AND state='QUEUEING'`（返回受影响行数：1=接单成功 / 0=已被抢或已删）
   - `deleteIfQueueing(id)` = `@Modifying DELETE ... WHERE id=:id AND state='QUEUEING'`（取消/超时删行，返回行数）
   - `deleteIfState(id, state)`（通用条件删，供支付成功转单后删 request）
   **And** **竞态先到先得、杜绝双写**：N 线程并发对同一 QUEUEING 请求 `tryAccept` 恰 1 次返回 1、其余 0（行锁串行化 `WHERE ... AND state=?`）；accept 与 delete 并发时二者互斥（一方成一方 0 行）；**判定基于 `state` 单列非时间戳比较**（H-4，防时钟偏移误判）（**L1**）

4. **后端：建单 + stage event 追加原语（L0/L1）**
   **Given** 支付成功（3-4）需原子建 `consult_orders`(IN_PROGRESS) + 首条 stage event
   **When** 提供建单能力
   **Then** `ConsultOrderRepository`/`ConsultOrderStageEventRepository` 常规 save；`ConsultBillingService`（或本 story 仅 repo + 轻封装）暴露 `createOrder(...)`（建 IN_PROGRESS 单）+ `appendStageEvent(orderId, type, occurredAt, note)`（**只追加、绝不改历史**）；`order_token` 用 `CardTokenGenerator`（不可枚举）；**本 story 不接支付/会话**（3-4 调），仅建原语 + 单测（**L0** 单测 / **L1** 落库）

5. **后端：schema validate 一致 + 契约（L0/L1）**
   **Given** `ddl-auto=validate`（护栏）
   **When** 上下文启动
   **Then** 三实体 ↔ 三表**字段/类型/枚举一致**，Flyway V66 后 `validate` 通过（启动即验）；枚举线格式 UPPER_SNAKE；无 CHAR(1) 坑；`consult_orders.status` 无 `CANCELLED`、`consult_requests` 无 CANCELLED 态（A-5 落表层）（**L0** 编译 / **L1** 启动 validate）

## Dev Notes

> 纯数据层地基。**核心难点只有一个：H-4 单列 state CAS**（用 `@Modifying UPDATE ... WHERE state=?` 返回行数，非应用层读改写、非时间戳比较）。其余是标准建表 + 实体。**全新表，不碰既有 consult_sessions/队列服务**——V1.1 两表是既有免费会话之上的「计费前置层」。

### 承接现状（brownfield，已读，勿混用/勿重建）
- **既有 consult（V1.0，5.x）**：`consult_sessions`（V14，`SessionStatus{WAITING,IN_PROGRESS,PENDING_CLOSE,CLOSED,INTERRUPTED,CANCELLED}`）= 免费咨询会话（IM 会话根）；`ConsultQueueService`（Redis ZSET `consult:waiting` 待接单）；`ConsultAcceptService`/`ConsultSessionService`/`ConsultCloseService` 等；`failed_consult_requests`（V37）；`consult_anomalies`（V44）；`ConsultSource{DIRECT,AI_UPGRADE}`。**本 story 全部不动**。
- **V1.1 两表定位**：`consult_requests`（新）= 付费前 QUEUEING→ACCEPTED_AWAIT_PAY 临时态（3-2 入队 / 3-3 接单）；`consult_orders`（新）= 支付成功建的计费订单（3-4）；**IM 会话仍复用既有 `consult_sessions`**（3-4 支付后建会话、`consult_orders.session_started_at` 记时）。三者关系：request（付费前）→ 支付成功 → 建 order + 建 consult_session + 删 request。本 story 只建 request/order/stage 三表与原语。
- **CAS 范式（直接照抄）**：`PawCoinWalletRepository.applyDelta(userId, delta)` = `@Modifying @Query("update ... where ... and balance+delta>=0")` 返回行数（0=失败）。本 story `tryAccept` 同构：`WHERE id=? AND state='QUEUEING'` 返回 1/0。行锁串行化，天然先到先得。**禁应用层「先查 state 再改」**（TOCTOU 竞态）。
- **token 生成**：`CardTokenGenerator.generate()`（profile 模块，22 位 base62 不可枚举，已跨模块复用）→ `request_token`/`order_token`。
- **pay 关联**：`PaymentPurpose.VET_CONSULT` 已存在（1.1）；`PayChannel{QRIS,PAWCOIN}`；`consult_orders.payment_intent_id` 关联 `payment_intents`（3-4 支付时落）。本 story 仅列，不接支付。
- **既有 `failed_consult_requests`（V37）**：3-4 重播上限（5 次/30min）彻底失败时写它；本 story 不动、不重建。

### 关键设计（dev 照此）
- **无 CANCELLED（A-5 落表层）**：`consult_requests.state` 只有 QUEUEING/ACCEPTED_AWAIT_PAY 两活态——**取消/超时 = 物理删行**（`deleteIfQueueing`/`deleteIfState`），不留「已取消」态；`consult_orders.status` 只有 IN_PROGRESS/COMPLETED/REFUNDING/REFUNDED——**无 CANCELLED**（未扣费根本不建单）。这是「订单中心无已取消态」的表层保证，CHECK 约束钉死。
- **H-4 单列 CAS（非时间戳）**：accept/cancel 竞态一律 `WHERE state=?` 单列比较，**绝不用 `updated_at`/deadline 时间戳比大小**判先后（时钟偏移会误判）。accept=`UPDATE state WHERE state=QUEUEING`，cancel=`DELETE WHERE state=QUEUEING`；两者对同一行的 QUEUEING 谓词经行锁串行，恰一方生效。
- **stage events append-only**：`consult_order_stage_events` 每个节点一行 INSERT，**绝不 UPDATE/DELETE 旧行**（对账/审计留痕，比照 `ledger_entries` append-only 铁律）。`ConsultBillingService.appendStageEvent` 只 `save` 新行。
- **快照字段**：`vet_payout`/`vet_share_rate_snapshot`/`unit_price_snapshot` 成交时落快照（后台 9-2 改价不影响历史订单/退款）。本 story 建列；填值在 3-4/3-7（价来自 config，渐进 env→9-2 DB，同 2-1/2-3）。
- **pay_deadline/queue_deadline 服务端权威**：deadline 是 DB timestamptz，服务端 `@Scheduled` 扫（3-3/3-4）；本 story 只建列 + `idx_consult_requests_state` 供扫描，不建 scheduler。
