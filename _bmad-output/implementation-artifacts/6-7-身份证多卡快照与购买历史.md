## 1. 背景：当前实现为何是错的

现状（Story 6-1/6-3）把身份证做成了**"一账号一张卡 + 一次性永久解锁"**：

- 身份证数据**实时从当前 `PetProfile` 渲染**（`IdCardDataResponse.from(PetProfile, hdUnlocked)`），改档案卡面就变——**没有"信息锁定/卡快照"概念**。
- `PetProfile.serialId` **一个宠物只分配一次、永不换号**（`IdCardService.generateSerial` 幂等、已有号不换）。
- HD 解锁 = `id_card_hd_purchases` 表 **`UNIQUE(user_id)`**，`existsByUserId` 短路，**付一次终身解锁**。

**实际业务**（用户 2026-07-22 澄清）：解锁=**锁定这次的信息成一张卡**；用户可用**新信息新建卡**，**需重新支付**才能解锁新卡的 HD。→ 天然有多张卡、多次购买 → 才有"历史列表"。

## 2. 已拍板的设计决策（用户 2026-07-22 选定）

| # | 决策 | 选定 |
|---|---|---|
| ① | 卡身份/解锁绑定 | **新增卡快照表**：每次建卡冻结当时信息成一条快照，购买/解锁绑到 `card_id`（非 user_id） |
| ② | serial 编号规则 | **每张卡新号**：每次新建卡从号池 `allocate()` 取一个新 serial，历史各卡编号独立 |
| ③ | 建新卡后旧卡去留 | **保留可看可下载**：历史列表里旧卡仍可查看；已付费旧卡仍可下载 HD |
| ④ | "新建卡"入口 | **独立建卡器**：KTP 页独立"新建身份证"流程，卡信息与宠物档案**解耦**（可填与档案不同的信息，默认预填档案值） |

## 3. 数据模型

### 3.1 新表 `id_cards`（卡快照）— Flyway **V90**（当前最新 V89，顺延）

```sql
-- Story 6-7：身份证卡快照（决策①/②）。每张卡=一次锁定的信息快照 + 独立 serial。
-- 解锁绑 card_id（去掉旧 UNIQUE(user_id) 的永久解锁语义）。ddl-auto=validate，schema 归 Flyway。
CREATE TABLE id_cards (
    id            BIGINT       GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    user_id       BIGINT       NOT NULL,
    serial_id     BIGINT       NOT NULL,                 -- 每张卡独立 serial（决策②），从 pet_serial_seq 号池取
    -- 冻结快照字段（建卡时定格，之后改档案不影响本卡）：
    name          VARCHAR(60)  NOT NULL,
    pet_type      VARCHAR(16),                            -- 枚举 UPPER_SNAKE（与 PetProfile.petType 一致，可空）
    breed         VARCHAR(80),
    birthday      DATE,
    avatar_key    VARCHAR(255),                           -- 头像对象 key（快照当时）；渲染时签名，勿存签名 URL
    intro         VARCHAR(200),
    hd_unlocked   BOOLEAN      NOT NULL DEFAULT false,    -- 本卡 HD 是否已解锁（购买到账后置 true）
    created_at    TIMESTAMPTZ  NOT NULL DEFAULT now(),    -- UTC
    CONSTRAINT uq_id_cards_serial UNIQUE (serial_id),     -- serial 全局唯一（与 pet_profiles.serial_id 同池，不冲突：见注）
    CONSTRAINT fk_id_cards_user FOREIGN KEY (user_id) REFERENCES users (id) ON DELETE CASCADE
);
CREATE INDEX ix_id_cards_user_created ON id_cards (user_id, created_at DESC);  -- 历史列表倒序
```

> serial 号池说明：现 `pet_serial_seq` 高水位序列 + `pet_serial_pool` 回收表由 `SerialAllocationService.allocate()` 统一发号。本卡也走同一 `allocate()`，与 `pet_profiles.serial_id` 共享同一号空间保证全局不撞。**注意**：`pet_profiles.serial_id` 与 `id_cards.serial_id` 两处 UNIQUE 各自独立，但同源号池不会发重号；旧 `PetProfile.serialId` 逻辑保持不动（老用户既有卡不受影响，见 §6 迁移策略）。

### 3.2 改表 `id_card_hd_purchases` — Flyway **V91**

购买记录从"按 user 一行"改成"按 card 一行"：

```sql
-- Story 6-7：购买绑到具体卡快照（决策①）。去掉一账号一行的永久解锁语义。
ALTER TABLE id_card_hd_purchases ADD COLUMN card_id BIGINT;
ALTER TABLE id_card_hd_purchases DROP CONSTRAINT uq_id_card_hd_purchases_user;   -- 去永久解锁
ALTER TABLE id_card_hd_purchases ADD CONSTRAINT uq_id_card_hd_purchases_card UNIQUE (card_id);  -- 一卡至多一次成功购买（幂等兜底）
ALTER TABLE id_card_hd_purchases ADD CONSTRAINT fk_id_card_hd_purchases_card
    FOREIGN KEY (card_id) REFERENCES id_cards (id) ON DELETE CASCADE;
-- pet_profile_id 列保留（历史/溯源），不再作幂等键。
```

> **决策 E2/回溯**：已提交迁移 V75 冻结，加列/改约束一律新起 ALTER（本 V91）。存量 `id_card_hd_purchases` 行 `card_id` 为 NULL（老"永久解锁"购买）——迁移策略见 §6。

## 4. 后端改动

### 4.1 领域
- 新增 `IdCard` 实体（`profile/domain/IdCard.java`）：映射 §3.1，工厂 `IdCard.snapshot(userId, serial, name, petType, breed, birthday, avatarKey, intro)`。

### 4.2 仓库
- 新增 `IdCardRepository`：`List<IdCard> findByUserIdOrderByCreatedAtDesc(long userId)`（历史列表）、`Optional<IdCard> findByIdAndUserId(long id, long userId)`（归属校验取单卡）。
- `IdCardHdPurchaseRepository`：`existsByUserId` → **改为** `boolean existsByCardId(long cardId)`；保留 user 维度查询仅作后台统计（可留）。

### 4.3 Service
- 新增 `IdCardService.createCard(ownerId, CreateIdCardRequest)`（**独立建卡器**，决策④）：
  - 入参为用户填写的卡信息（默认前端预填档案值，允许改）；服务端校验（name 必填、长度、petType 枚举、birthday 合法）。
  - `serial = serialAllocation.allocate()`（决策②，同一事务内）。
  - `idCards.save(IdCard.snapshot(...))` → 返回新卡 DTO。
  - avatar：接受对象 key（复用现有上传预签名链路，勿存签名 URL）。
- 新增 `IdCardService.listMyCards(ownerId)` → 历史列表 DTO（倒序）。
- 新增 `IdCardService.getMyCard(ownerId, cardId)` → 单卡详情（归属校验，非本人 404 防枚举）。
- `IdCardHdService` 全面改为**按 card**：
  - `isUnlocked(userId)` → **`isUnlocked(cardId)`** = `card.hdUnlocked`（或 `purchases.existsByCardId`）。
  - `purchase(userId, cardId, channel)`：入口短路 `card.hdUnlocked`；幂等键 `id-hd:<cardId>`；成交价仍读 `platformConfig.pricing().getIdHdDownloadPrice()`（决策：每张卡各付一次）；PawCoin 即时扣费+`card.markHdUnlocked()`+建购买行(cardId)；QRIS 建意图。
  - `completePurchase(userId, cardId, channel, intentId)`：`MANDATORY` 同事务；`existsByCardId` 幂等；置 `card.hdUnlocked=true` + 建购买行。
  - **`IdHdPaidHandler`**：回调需能拿到 `cardId`——把 cardId 编入 payment_intent 的业务引用（现幂等键 `id-hd:<petProfileId>` 改 `id-hd:<cardId>`，回调据此解析 cardId）。

### 4.4 Controller（`ProfileApiController`，路径 `/api/v1/pet-profiles/me/...` 或新资源 `/api/v1/me/id-cards`）
> 命名建议用 `/api/v1/me/id-cards`（资源小写复数连字符，当前用户 `/me`，决策 C1）。旧 `/pet-profiles/me/id-card` 单卡端点**保留**兼容老前端一段时间或前端一并切换。

- `POST /me/id-cards`（建卡，body=`CreateIdCardRequest`）→ 201 新卡 DTO。
- `GET /me/id-cards`（历史列表，倒序）→ `List<IdCardResponse>`。
- `GET /me/id-cards/{cardId}`（单卡详情，归属校验）→ `IdCardResponse`（含 `hdUnlocked`）。
- `POST /me/id-cards/{cardId}/hd-download`（购买/解锁本卡 HD）→ `HdPurchaseResponse`（复用现结构）。
- DTO：新增 `CreateIdCardRequest`（name/petType/breed/birthday/avatarKey/intro）+ `IdCardResponse`（id/serialId/快照字段/hdUnlocked/createdAt）。

## 6. 存量迁移策略（老"永久解锁"用户）

- 老用户在 `pet_profiles.serialId` 上有一个号 + `id_card_hd_purchases` 一行（`card_id=NULL`）。
- 迁移选项（V91 内或一次性回溯 SQL，**发通知否？否**）：
  - **推荐**：为每个"已永久解锁"的老用户，用其当前 `pet_profiles`（serial/信息）**回填生成一张 `id_cards` 快照**（`hd_unlocked=true`），并把其购买行 `card_id` 指向该卡。→ 老用户重进即见"1 张已解锁的历史卡"，无缝。
  - 回溯 SQL 幂等、只读老数据、不发 App 通知（参考 V77 回溯不发通知的先例）。
- 老端点 `GET/POST /pet-profiles/me/id-card` 与 `hd-download`：前端切换后可下线；过渡期保留只读兼容。
