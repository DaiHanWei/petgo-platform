# Story 3.6: 内容删除（无编辑）

Status: ready-for-dev

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

## Story

As a **内容作者**,
I want **删除我发布的内容**,
so that **我能撤回不想保留的帖子**。

> 本 Story 实现内容删除（FR-36）——作者二次确认删除、从 Feed 与成长档案时间线同步移除、点赞评论一并删；**V1 无内容编辑**。**后端中、前端轻**：后端软删 `deleted_at` + 级联清点赞评论 + 同步移出 Feed/时间线；前端「···」菜单删除项（仅作者可见）+ 二次确认。
>
> **做**：① 作者「···」菜单「删除」（仅作者可见）；② 二次确认弹窗；③ 软删 `content_posts.deleted_at`；④ 从 Feed 与成长档案时间线同步移除；⑤ 该内容的点赞与评论数据一并删除（级联）；⑥ 删除后他人访问详情走 404 占位页（Story 3.3 多态 ④）。
> **不做**：内容编辑（V1 明确无编辑）；运营下架（Story 3.7 复用本 Story 软删能力，但触发方在 Admin）；详情页多态 404 渲染（Story 3.3 已做，本 Story 触发软删使其生效）。
> **依赖前序**：Story 2.3（`content_posts` 表 + 成长日历帖入时间线）、Story 3.3（详情页「···」菜单容器 + 404 占位页）、Story 3.4（点赞表，删除时级联清）、Story 3.5（comments 表，删除时级联清）、Story 2.4（成长档案时间线——删除须同步移除快乐时刻条目）。

## Acceptance Criteria

> **验证层标注**：**L0**（编译/lint/widget test/Service 单元 + MockMvc） · **L1**（需 postgres+redis，验软删 + 级联清 + Feed/时间线移除） · **L2**（真机，二次确认交互）。
> 软删 + 级联 + 同步移除为 **L1**（DB 状态 + 多处读取一致性是核心）；「···」删除可见性/二次确认为前端 **L0 + L2**。无外部第三方依赖。

### AC1 — 作者删除 + 二次确认 + 软删 + 同步移除 + 级联清

**Given** 内容作者在详情页
**When** 通过「···」菜单点击「删除」（仅作者可见）
**Then** 弹确认「删除后内容将永久移除，无法恢复」，确认后内容从 Feed 与成长档案时间线同步移除（FR-36）
**And** 该内容的点赞与评论数据一并删除；V1 不提供内容编辑（采用软删 `deleted_at`）
> 验证层：**L1**（软删后：Feed 读取(Story 3.2)不返回该帖、成长档案时间线(Story 2.4)不显该条、详情(Story 3.3)走 404、点赞(Story 3.4)与评论(Story 3.5)级联清；删除权限=仅作者，他人删→403）。「···」删除项仅作者可见 + 二次确认文案为前端 **L0 widget test** + **L2 真机**。删除写端点带 `Idempotency-Key`、需 JWT（未登录/非作者拦截）。

## Tasks / Subtasks

> **三段组织**，后端中前端轻。建议执行顺序：后端软删+级联+移除 → 前端删除菜单+二次确认 → 联调（多处一致性）。

### 🟦 后端子任务（petgo-backend / Spring Boot — content 模块）

- [ ] **B1. 内容删除端点（软删 + 权限）** (AC: 1)
  - [ ] `DELETE /api/v1/content-posts/{id}`：权限 = **仅内容作者本人**；非作者→**403**；未登录→401。带 `Idempotency-Key`。
  - [ ] 软删：置 `content_posts.deleted_at = now()`（**不物理删**，架构 §Naming 软删约定 + 注销匿名化需保留历史结构）；已删再删→幂等。
- [ ] **B2. 级联清点赞与评论** (AC: 1)
  - [ ] 事务内：内容软删时**级联软删其全部评论（comments，含一/二级）+ 删除其点赞行（content_likes）**；或评论级联软删、点赞物理删（点赞无保留价值）——在 Dev Notes 定策略并一致。
  - [ ] commentCount/likeCount 随之归零/失效（详情已 404，无须再显）。
- [ ] **B3. Feed 与时间线同步移除** (AC: 1)
  - [ ] Feed 读取（Story 3.2 `WHERE deleted_at IS NULL`）天然排除软删帖——确认 Story 3.2 查询已含此条件（若无则补）。
  - [ ] 成长档案时间线（Story 2.4，GROWTH_MOMENT 帖入时间线）：时间线读取须 `deleted_at IS NULL` 过滤——确认/补该条件，使删除即从时间线消失。
  - [ ] **无缓存**：因 Feed/时间线直读 PostgreSQL（无 Redis 缓存），软删即时生效，无需失效缓存。
- [ ] **B4. 详情 404 触发** (AC: 1)
  - [ ] 详情端点（Story 3.3）对 `deleted_at` 非空返回 **404 统一防枚举文案**——确认 3.3 已实现，本 Story 软删使其生效。
- [ ] **B5. （为 Story 3.7 预留）下架复用** (AC: 1)
  - [ ] 软删能力封装为 `ContentService.softDelete(postId, reason)`，**Story 3.7 运营下架复用同一 service 方法**（reason 区分 author_delete / admin_takedown）——本 Story 实现作者删，下架触发方在 Admin（3.7）。

### 🟩 前端子任务（petgo_app / Flutter — features/content）

- [ ] **F1. 「···」删除菜单（仅作者可见）** (AC: 1)
  - [ ] 详情页顶部「···」菜单（Story 3.3 容器）：当 `isAuthor==true` 显示「删除」项（非作者不显）；游客/他人无此项。
  - [ ] 「我的发布」（Epic 7 Story 7.1）的单条目删除复用同一删除调用——本 Story 提供 deleteProvider，7.1 接入。
- [ ] **F2. 二次确认** (AC: 1)
  - [ ] 点删除 → 二次确认 dialog「删除后内容将永久移除，无法恢复」（双语 .arb）；确认调 B1 端点；成功后 pop 详情回 Feed（该帖已不在列表）。
  - [ ] 删除中加载态 + 失败 toast（UX-DR10）。

### 🟨 联调验收子任务（多处一致性 + 权限）

- [ ] **J1. 软删 + 权限** (AC: 1 / **L1**)
  - [ ] 作者删自己帖→deleted_at 置位；非作者删→403；未登录→401；重复删→幂等。
- [ ] **J2. 级联 + 多处移除一致性** (AC: 1 / **L1**)

- [ ] **J3. 前端删除菜单 + 确认** (AC: 1 / L0+**L2**)

## Dev Notes

### 关键架构约定（本 Story 必须落实）

**内容删除模型**（FR-36）：作者删二次确认；从 Feed 与成长档案时间线同步移除；点赞评论一并删；**V1 无编辑**；采用**软删 `deleted_at`**。

**软删而非物理删**（架构 §Naming：需软删处 `deleted_at`；NFR-8 注销匿名化需保留内容结构）：内容软删保留行，Feed/时间线/详情读取一律 `deleted_at IS NULL` 过滤。这也使注销内容匿名化（Story 7.3）与删除两条路径不冲突——注销是改 author 投影、删除是 deleted_at 置位。

**级联策略**：内容软删 → 评论级联软删（comments）+ 点赞清除（content_likes）。在事务内完成，保证一致性。无 Redis 缓存故无失效问题。

**与 Story 3.7 复用**：`ContentService.softDelete(postId, reason)` 封装软删，作者删（本 Story）与运营下架（3.7）共用；reason 区分来源，便于审计但不暴露给前端。

**多态联动**（架构 §UX-DR18 / Story 3.3）：删除后他人/陈旧入口/推送深链访问详情 → 404 统一防枚举文案 + 返回（Story 3.3 已实现占位页，本 Story 触发软删使其生效）。

**命名映射链**：`content_posts.deleted_at`(timestamptz UTC, 可空)；DELETE 端点；HTTP 403 非作者、401 未登录、200/204 删除成功、幂等重复 200。

### 强制护栏（架构 §Enforcement —— 违反即返工）

- **禁止擅自引入 MQ / 通用缓存层 / 新中间件**；删除同步走 DB 软删 + 事务级联，无缓存失效。
- `ddl-auto=validate`；`deleted_at` 列若 Story 2.3 未建则本 Story Flyway 补（与 2.3 协调，避免重复）。
- 软删（`deleted_at`）**不物理删**——满足注销匿名化保留与审计。
- 删除权限服务端权威（仅作者）；前端隐藏入口仅体验。
- 模块间只经 service——时间线移除是 profile 读取侧加 `deleted_at` 过滤（profile 经 content service 或共享 deleted_at 语义），**不在 profile repository 改 content 表**。
- 写端点 Idempotency-Key。

### 范围边界（防 scope creep —— 本 Story 明确不做）

- ❌ 内容编辑（V1 无编辑）。
- ❌ 运营下架触发（Story 3.7 复用本 Story `softDelete`，触发方在 Admin）。
- ❌ 详情 404 占位页渲染（Story 3.3 已做）——本 Story 触发使其生效。
- ❌ 评论删除（Story 3.5 已做评论级别删除；本 Story 是内容级删除连带级联评论）。
- ✅ 只做：作者删除端点（软删 + 权限）+ 二次确认 + 级联清点赞评论 + Feed/时间线同步移除 + 复用封装供 3.7。

### Project Structure Notes

- 后端：`com.petgo.content/{web,service,repository}`——`ContentController` 增 DELETE、`ContentService.softDelete`（含级联，供 3.7 复用）；时间线移除过滤在 `com.petgo.profile`（读取侧加 deleted_at，经 content service 或共享语义）。
- 前端：`lib/features/content/{data,domain,presentation}`——详情「···」删除项（isAuthor 可见）+ deleteProvider + 二次确认 dialog（供 Epic 7「我的发布」复用）。
