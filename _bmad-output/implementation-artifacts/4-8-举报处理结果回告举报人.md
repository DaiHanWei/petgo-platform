# Story 4.8: 举报处理结果回告举报人

Status: review

> V1.1 **Epic 4 第 8 story**，**纯后端验证型**（举报处置→举报人模糊闭环通知，L1）。**核心发现：本 story 的行为已被 V1.0 admin epic（Story 4.1「举报处理」）完整实现**——每条下架/驳回（含批量逐条）经 `AdminModerationService.notifyReporter` 发 `ReportResolvedEvent` → `ModerationNotifyListener`（`@TransactionalEventListener`）发 `REPORT_REVIEWED` 模糊通知（下架/驳回**文案完全一致、不透露结果**），`NotificationService.send` 自带 **REQUIRES_NEW**（避 AFTER_COMMIT 吞写，[[notify-after-commit-tx-swallow-bug]]）。**无新增生产代码 / 无迁移**。
> 源：`epics-v1.1.md` Story 4.8（举报处理完成→通知举报人「已处理」不透露结果 L1；通知 INSERT 走 REQUIRES_NEW L1）· FR-51 · AB-3A。
> **⚠️ 承接现状（已就位，勿重建）**：`AdminModerationService.takedown/dismiss/batch`（每条处置 publishEvent `ReportResolvedEvent(reportId, reporterId, now)`，仅 reporterId 非空时发）+ `ModerationNotifyListener.onReportResolved`（`@TransactionalEventListener` → `send(reporterId, REPORT_REVIEWED, "举报已处理", "感谢你的举报，我们已完成审核。", …, targetRef=null)`，护栏：**下架/驳回文案一致、不透露内容/作者/结果、无申诉入口、deepLink 不导向内容**）+ `NotificationService.send`（REQUIRES_NEW）+ `NotificationType.REPORT_REVIEWED`（枚举 + V40 CHECK 已就位）。既有测试：`AdminModerationServiceTest`（L0：takedown/dismiss 两路径均 verify publishEvent）+ `ModerationNotifyListenerTest`（L0：listener 发 REPORT_REVIEWED 给 reporter 泛化文案）。
> **本 story 的唯一增量**：既有测试是 **L0（mock）**——`AdminModerationServiceTest` 只验「发了事件」、`ModerationNotifyListenerTest` 只验「listener 调了 send」，**无 L1 端到端确认通知真落库**（经 event→AFTER_COMMIT listener→REQUIRES_NEW send 的真实 Spring 路径）。而 4-8 两条 AC 均标 **L1**，且这条路径正是 [[notify-after-commit-tx-swallow-bug]] 曾静默吞写的高危路径。故本 story = **补一个 L1 集成测试锁定闭环**（举报处置后举报人 REPORT_REVIEWED 通知真落库 + 下架/驳回文案一致），无生产改动。
> **范围边界**：本 story = L1 闭环验证测试。**不做**：任何生产代码改动（已实现）；申诉入口 / 举报结果查询（AB-3A 明确不做）；迁移。

## Story

As a 举报人,
I want 举报被处理后收到「已处理」通知,
so that 我知道平台已跟进（FR-51），而处置结果/内容/作者对我保持模糊（AB-3A）。

## Acceptance Criteria

1. **后端（验证）：举报处置→举报人模糊通知真落库（L1）**
   **Given** 举报人对某内容举报（`content_reports` PENDING）
   **When** 运营下架（`takedown`）**或**保留/驳回（`dismiss`）
   **Then** 举报人收到 1 条 `REPORT_REVIEWED` 通知落库（`notifications.recipient_user_id=reporterId`）；**下架与驳回的 title/body 完全一致**（不透露具体结果，AB-3A）；通知 `targetRef` 为空（deepLink 不导向内容）（**L1**）
   **And** 通知**不含**被举报内容/作者信息（护栏）；reporterId 为空的历史举报不发（防御）（**L1**）

2. **后端（验证）：通知 INSERT 走 REQUIRES_NEW（避 AFTER_COMMIT 吞写）（L1）**
   **Given** 处置在 `@Transactional` 内，`@TransactionalEventListener` 默认 AFTER_COMMIT 阶段消费事件
   **When** 事件触发 `NotificationService.send`
   **Then** 通知**真正提交落库**（REQUIRES_NEW 挂起已提交事务、新开独立事务）——L1 端到端断言可见该行（若退回默认 REQUIRED 会静默丢写，此测试即回归闸）（**L1**）

3. **契约 + 验收（L0/L1）**
   **Given** 全部就位
   **When** 验证
   **Then** `DB_NAME=petgo_l1 mvn -B test` 绿：新增 L1 `ReporterNotificationClosedLoopIntegrationTest`（下架/驳回各发 1 条 REPORT_REVIEWED、文案一致、targetRef 空）+ 既有 `AdminModerationServiceTest`/`ModerationNotifyListenerTest`/`AdminModerationIntegrationTest` 回归绿（**L0/L1**）
   **And** 无生产代码改动、无迁移（**L0**）

## Dev Notes

> 本 story 是 **verification story**：4-8 的产品行为在 V1.0 admin epic 已交付，本 story 补 L1 回归闸锁定「举报处置→举报人模糊通知真落库 + REQUIRES_NEW」这条高危闭环（曾被 AFTER_COMMIT 静默吞写）。

### 承接现状（已核实，勿改）

- **举报处置发事件**：`AdminModerationService.notifyReporter(ContentReport r)` → `if (r.getReporterId() != null) events.publishEvent(new ReportResolvedEvent(r.getId(), r.getReporterId(), Instant.now()))`。`takedown`/`dismiss`/`batch`（逐条）均调。
- **监听器发通知**：`ModerationNotifyListener.onReportResolved`（`@TransactionalEventListener`，默认 AFTER_COMMIT）→ `notificationService.send(reporterId, REPORT_REVIEWED, "举报已处理", "感谢你的举报，我们已完成审核。", REPORT_REVIEWED.name(), null)`。**文案对下架/驳回完全一致**（AB-3A 模糊）；`targetRef=null`（deepLink 不导向内容）。
- **REQUIRES_NEW**：`NotificationService.send` `@Transactional(propagation = REQUIRES_NEW)`——AFTER_COMMIT 阶段触发事务已提交但同步资源仍绑定，REQUIRES_NEW 强制新开独立事务保证 INSERT 真提交（[[notify-after-commit-tx-swallow-bug]]）。**本 story L1 测试即验此**。
- **既有测试为何不够**：`AdminModerationServiceTest`（L0 mock events）只验 publishEvent 被调；`ModerationNotifyListenerTest`（L0 mock notify）只验 send 被调；`AdminModerationIntegrationTest`（L1）验状态/软删/审计但**未查 notifications 表**。→ 4-8 补 L1 直查 notifications 表的闭环测试。

### 关键设计决策（dev 照此）

- **D-1 零生产改动**：4-8 行为已实现，本 story 只加测试。若实现中发现闭环有缺口（如某处置路径漏发事件），才补生产代码并记录；预期无缺口。
- **D-2 L1 直查 notifications 表**：测试后置断言 `SELECT count(*) FROM notifications WHERE recipient_user_id=? AND type='REPORT_REVIEWED'`（reporterId 本人）。下架、驳回各造独立举报人，各断言 count=1。
- **D-3 文案一致断言**：取下架产生的通知 title/body 与驳回产生的通知 title/body 比对相等（AB-3A：不透露处置结果）。
- **D-4 事务时序**：`moderation.takedown/dismiss` 是 `@Transactional`；返回即提交 → AFTER_COMMIT listener 同步执行 → REQUIRES_NEW send 提交 → 测试方法（非 @Transactional，ApiIntegrationTest 真写库）后续 jdbc 查询可见。
