# Story 6.1: 流水号与身份证数据

Status: review

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

> **所属**：V1.1 Epic 6「宠物身份证与分享拉新」的**第一个 Story（后端建模/基建型）**。为 6.2（三风格证件卡渲染）、6.3（HD 付费下载）、6.4（分享落地页拉新）铺数据地基。本 Story **纯后端**，无前端子任务：只交付 ① `pet_profiles.serial_id` 全平台自增流水号 + 号池回收机制；② 身份证数据接口（含老用户「尚未生成」标志）；③ 注销/删除时流水号释放回收布线。三风格渲染（6.2）、HD 下载（6.3）、分享落地（6.4）均不在本 Story。

## Story

As a **平台**,
I want **全平台自增流水号 + 身份证数据接口 + 老用户未生成引导态**,
so that **身份证有唯一编号、老用户有引导（FR-49A）**。

## Acceptance Criteria

> **验证层标注**：**L0 静态**（`mvn -B compile\|test-compile\|package`，无需 DB）· **L1 集成/运行时**（需 Docker daemon + postgres + redis 真跑：并发分配、号池回收、幂等、落库）· **L2 端到端**（本 Story 无 L2——纯后端，无真机/外部凭证依赖）。

### AC1 — `serial_id` 唯一 + 号池回收机制（Flyway 迁移）

**Given** `pet_profiles.serial_id` bigint **唯一**（允许 NULL：老用户/未生成为 NULL）+ 号池/序列回收机制（**Flyway V74**，非架构旧稿的 V55/V70——见 Dev Notes「Flyway 撞号」）
**When** 分配流水号
**Then** 从 **1** 起自增、删除释放回收（被删档案的号回到池中可再分配）、**并发不撞号**（`serial_id` UNIQUE 约束为终极兜底 + 分配串行化保证无重复）
> 验证层：**L1**（真 pg：并发多线程分配全部互异、删除后号回池、下次分配优先复用回收号；`serial_id` 唯一约束 + 分配锁生效）+ **L0**（`SerialAllocationService`/迁移 DDL 编译通过、迁移文件命名与内容静态可核）。

### AC2 — 身份证数据接口 + 老用户「尚未生成」标志

**Given** 当前用户档案（`/api/v1/pet-profiles/me` 既有单宠物范式）
**When** 请求身份证数据 `GET /api/v1/pet-profiles/me/id-card`
**Then** 返回身份证渲染所需数据（宠物名/类型/品种/生日/头像 + `serialId` + `generated` 标志）：
- 已生成（`serial_id != null`）→ `generated=true` + `serialId`
- **老用户/未生成**（`serial_id == null`）→ `generated=false`、`serialId=null`（前端据此渲染「尚未生成」引导态，FR-49A）
- 无档案 → **404**（与 `GET /pet-profiles/me` 一致，不 500、不空对象）
**And** 生成动作 `POST /api/v1/pet-profiles/me/id-card`（分配流水号）**幂等**：已有 `serial_id` 则原样返回既有号（不重复分配、不换号），未有则分配后落库并返回
> 验证层：**L1**（老用户档案无 serial → GET 返 `generated=false`；POST 分配后 GET 返 `generated=true`+号；POST 二次调用返同号不换号；无档案 404 需 pg 造数据）+ **L0**（`generated` 派生逻辑 = `serialId != null` 的映射单测、DTO 结构 MockMvc 可验）。

### AC3 — 分享/快照绑内部 id，不绑 serial_id

**Given** 分享名片（`card_token` / 内部 `id`）与只读快照（`PetProfileSnapshot`）
**When** 生成分享链接 / 定时扫描 / 跨模块只读取档案
**Then** 对外定位与回查一律用 **`card_token`（不可枚举）或内部 `id`**，**`serial_id` 仅作展示编号**（如「编号 #123」），**绝不**作为分享链接、深链、快照回查的定位键
**And** `PetProfileSnapshot`（6.7 定时扫描用）不新增 `serialId` 字段；身份证数据 DTO 中 `serialId` 为纯展示值，非资源定位符
> 验证层：**L0**（快照 DTO 不含 serial_id 的静态核查 + 身份证 DTO 中 serial 仅展示用的断言；分享/落地页定位键仍是 card_token 的代码走查）。

## Dev Notes

### 关键架构约定（沿用全局）

- **目录/分层**：后端 `com.tailtopia.profile.{web,service,domain,repository,dto}`；号池分配归 `profile.service`（`SerialAllocationService`），身份证数据接口归 `profile.web`。**不跨模块直访对方 repository**。
- **命名映射链**：DB `snake_case`(`serial_id`/`pet_serial_pool`/`pet_serial_seq`) ↔ Java/Dart `camelCase`(`serialId`) ↔ JSON `camelCase`；时间戳 UTC；Flyway `V<序号>__<描述>.sql`。
- **对外标识**：`serial_id` 是**展示用编号**，**不是**对外资源标识——分享/深链/快照定位一律 `card_token`（不可枚举）或内部 `id`（AC3 红线）。
- **错误**：无档案 → 404 语义（ProblemDetail），不裸抛、不 500、不空对象。

### 号池回收设计（本 Story 核心）

- **为什么不用纯 SEQUENCE**：AC 要求「删除释放回收」——纯序列不回收会留永久空洞。设计 = **高水位序列 `pet_serial_seq`（新号）+ free-list 表 `pet_serial_pool`（回收号）**，分配优先出池、否则 `nextval`；释放入池。这样编号紧凑、删除的号可复用。
- **并发正确性**：`pg_advisory_xact_lock(常量)` 串行化分配临界区（分配罕见、量小，最简且可证无撞号）；`serial_id` UNIQUE 约束为终极兜底。**advisory 锁是 Postgres 原生能力，非新中间件/缓存层，不违「禁 MQ/缓存/中间件」护栏**。不用 Redis（角标/幂等键场景才用 Redis，此处 DB 内解决）。
- **分配时机（惰性）**：GET 数据接口**不**分配号（老用户永远能看到「尚未生成」引导态直到主动生成）；仅 POST 生成动作分配。6.2 前端在用户「制作身份证」时调 POST。→ 保证 FR-49A「老用户有引导」。
- **幂等**：POST 生成对已有号的档案直接返回既有号、绝不 re-alloc（防换号 / 防序列空耗）。

### 强制护栏（违反即返工）

- **禁** 引入 MQ / 通用缓存层 / 新中间件——号池分配用 pg advisory 锁 + free-list 表 + sequence（全 DB 内）。
- **禁** 把 `serial_id` 当对外资源标识（分享/深链/快照定位）——一律 token/内部 id（AC3）。
- `ddl-auto=validate`——serial_id/序列/池表全归 Flyway，**禁** `update`/`create`。
- 日志严禁记录 PII/健康数据/令牌；身份证 DTO 不带 owner PII / 健康明文。

**身份证数据接口（AC2）**：`GET/POST /api/v1/pet-profiles/me/id-card`（owner 取自 JWT）。GET 只读、**不分配号**（老用户 `generated=false` → 前端「尚未生成」引导态持续可见）；POST 惰性分配且**幂等**（已有号原样返回、不重复分配、不换号，避免序列空耗）。无档案 → 404。`IdCardDataResponse` 不含 owner PII / 健康数据。

**删除释放布线（AC1）**：`ProfileDeletionService.deleteByUserId` 在 `petProfiles.delete(pet)` 前 `serialAllocation.release(serialId)`（同 `@Transactional` 原子）。主动删档（F18）与注销 7.3 两条路径共用此 service → 均自动回收号。L1 端到端验证：删档 → 号入池 → 下次分配复用。

**AC3（分享/快照绑内部 id）**：`PetProfileSnapshot`（6.7）未新增 serialId，仍绑内部 id；`serial_id` 仅作展示编号，分享/深链/落地页定位键仍为 `card_token`/内部 id。未新增任何以 serial_id 为对外定位符的路径。

**范围**：纯后端，无前端（三风格渲染/引导态 UI 留 6.2）。`id_card_hd_purchases`（6.3）另起迁移，本 story 只做 serial。

**云端注意**：云端 headless 无 Docker → 只能到 L0；L1 需 scratch 库（共享 dev 库有跨分支迁移污染）。本地已全部 L1 验证通过。
