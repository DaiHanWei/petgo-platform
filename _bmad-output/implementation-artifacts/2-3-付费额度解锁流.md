# Story 2.3: 付费/额度解锁流

Status: review

> V1.1 **Epic 2 第三个（集成枢纽）** story，把 2-1（免费额度 `FreeQuotaService.tryConsume`）+ 2-2（`TriageTask.unlock` 不可覆盖跃迁 / 分字段下发）+ 1.x（`PaymentIntentService` / `PawCoinWalletService` / `PaymentIntentPaidEvent`）**串成解锁流**。**纯后端**：新表 `ai_consult_orders`（已结算订单，独立命名空间）+ `POST /triage/{id}/unlock` + 三条解锁路径（免费额度 / PawCoin 站内扣 / QRIS 现金）+ 现金到账处理器。前端 paywall 在 2-4。
> 源：`epics-v1.1.md` Story 2.3 · 架构 `architecture-v1.1-delta.md` L92（`payment_intents` purpose=`AI_UNLOCK`）/L107（`ai_consult_orders` 独立命名空间）/L126（`pricing_config.ai_unlock_price`）/L160-161（全链路幂等 NFR-2 + 「生成结果+额度扣减+解锁来源写入」同一事务 NFR-3）/L339（最简收费场景先跑通幂等/解锁）· 决策 C1（`/api/v1/me`、不可枚举 token）。
> **⚠️ 承接纠偏（2-2 已确立）**：解锁态在真实表 **`triage_tasks`**（非 `triage_records`）；解锁来源枚举 `UnlockSource{LOCKED,FREE_QUOTA,PAID}` + `UnlockChannel{QRIS,PAWCOIN}` 已由 2-2 建。**triage 用数字 `{id}` 寻址**（既有 `GET /triage/{id}`，越权/不存在统一 403 防枚举）——架构 `/triage/{token}/unlock` 是宽泛措辞，本 story 用 `POST /triage/{id}/unlock` 与既有一致。
> **范围边界**：解锁编排 + 三路径 + `ai_consult_orders` + 现金到账处理器。**不做**：paywall/额度展示前端（2-4）、退款（Epic 4）、后台改价 UI（9-2，价走 env）、兽医咨询订单（Epic 3 的 `consult_orders`，与本 `ai_consult_orders` 独立命名空间隔离）。

## Story

As a 用户,
I want 用免费额度或 PawCoin/现金解锁 AI 详细建议,
so that 我能看到完整养护建议。

## Acceptance Criteria

1. **后端：已结算订单表（V65 ai_consult_orders，独立命名空间）（L0/L1）**
   **Given** 架构 L107「`ai_consult_orders`（AB-8G，独立 ID 命名空间，与兽医订单隔离）」
   **When** 建付费解锁订单存储
   **Then** 新建 Flyway **`V65__init_ai_consult_orders.sql`**（接 2-2 的 V64 顺延，决策 E2）：表 `ai_consult_orders`（`id` PK；`order_token` VARCHAR 不可枚举唯一；`user_id`；`triage_task_id`（关联被解锁分诊）；`amount` bigint；`pay_channel` VARCHAR(16) CHECK `IN('QRIS','PAWCOIN')`；`payment_intent_token` VARCHAR 可空（现金路径关联 `payment_intents`，站内 PawCoin 为 null）；`status` VARCHAR(16) CHECK `IN('PENDING_PAYMENT','COMPLETED','ABNORMAL')`；`paid_at` TIMESTAMPTZ 可空；`created_at`/`updated_at`）+ 唯一索引 `uq_ai_consult_orders_order_token`、`uq_ai_consult_orders_intent`(`payment_intent_token` where not null 或普通唯一)（**L0/L1**）
   **And** **仅付费路径（PAWCOIN/QRIS）建订单**；免费额度解锁不建订单（`triage_tasks.unlock_source=FREE_QUOTA` 即记录）（**L0/L1**）
   **And** status 相对架构「COMPLETED/ABNORMAL」**增 `PENDING_PAYMENT`**（现金异步到账前的中间态，供回调关联 intent→triage；本 story 决策，已记 Dev Notes）（**L0**）

2. **后端：解锁编排 service + 端点（L0/L1）**
   **Given** `FreeQuotaService`（2-1）/`TriageTask.unlock`（2-2）/`PawCoinWalletService.debit`（1.2）/`PaymentIntentService.createIntent`（1.1）
   **When** `POST /api/v1/triage/{id}/unlock`（JWT，body `{method: FREE_QUOTA|PAWCOIN|QRIS}`）
   **Then** `AiUnlockService` 按 method 分派；`userId` 取自 JWT；仅解锁**自己的** triage（非 owner/不存在统一 **403** 防枚举，照既有 `TriageService.getResult`）；仅 `status=DONE` 可解锁（非 DONE→409/422）；price 取 `TriageProperties.aiUnlockPrice`（env `TRIAGE_AI_UNLOCK_PRICE` 默认 10000，`ai_unlock_price` 9-2 后台后换 DB）（**L0/L1**）

3. **后端【同一事务 NFR-3】：免费额度 & PawCoin 同步解锁（L1）**
   **Given** method=FREE_QUOTA 或 PAWCOIN
   **When** 解锁
   **Then** **单一 `@Transactional`** 内原子完成：
   - **FREE_QUOTA**：`freeQuotaService.tryConsume(userId)` → true 则 `task.unlock(FREE_QUOTA, null)`；false（额度耗尽）→ 抛「额度不足需付费」（**402/409**，不改任何状态、回滚）（**L1**）
   - **PAWCOIN**：`walletService.debit(userId, price, SPEND, "AI_UNLOCK", triageId, "ai-unlock:"+triageId)` → `task.unlock(PAID, PAWCOIN)` → 建 `ai_consult_orders(COMPLETED, amount=price, PAWCOIN, paid_at=now)`；余额不足 → `debit` 抛冲突（**402/409**）、整事务回滚（**L1**）
   **And** 三写（额度扣减/PawCoin 扣 + 解锁来源写入 + 订单）**要么全提交要么全回滚**（NFR-3「同一事务」）（**L1**）

4. **后端：QRIS 现金解锁（异步，复用 payment_intents 双通道到账）（L1/L2）**
   **Given** method=QRIS（Midtrans 收款，异步）
   **When** 发起解锁
   **Then** `AiUnlockService` `createIntent(userId, AI_UNLOCK, channel, price, "IDR", "ai-unlock:"+triageId)`（1.1 幂等建单）+ 建 `ai_consult_orders(PENDING_PAYMENT, triageId, channel, payment_intent_token=intent.publicToken)`；返回 `PaymentIntentResponse`（QR/deeplink）——**此刻不解锁**（**L1**）
   **And** **到账处理器** `AiUnlockPaidHandler`（`@EventListener` + `@Transactional(MANDATORY)`，仅处理 `PaymentIntentPaidEvent.purpose==AI_UNLOCK`，照 `TopupPaidHandler` 范式）：按 `intent.publicToken` 查 `ai_consult_orders` → `task.unlock(PAID, channel)` + 订单 `→COMPLETED`+`paid_at`；**与 `applyCallback` 的 `markPaid` 同一事务**（禁 AFTER_COMMIT 异步，血泪护栏）（**L1** 处理器单测 / **L2** sandbox 真付一次）

5. **后端【幂等 NFR-2】：解锁后可复看、二次解锁不重复扣费（L1）**
   **Given** 已解锁的 triage（`unlock_source∈{FREE_QUOTA,PAID}`）
   **When** 再次 `POST /unlock`（任意 method）
   **Then** **短路成功**：直接返回当前已解锁结果（`locked=false`），**不 tryConsume、不 debit、不建新 intent/order、不再扣费**（服务入口先判 `alreadyUnlocked` → 早返）；`task.unlock` 的不可覆盖守卫（2-2）+ PawCoin `debit` 幂等键 `ai-unlock:{triageId}` + intent 幂等键 + `ai_consult_orders` 唯一约束 = **多层兜底不双扣**（**L1**）
   **And** 现金回调重放/双通道（回调+轮询）：1.1 `applyCallback` 对已 PAID intent 不再发事件 + 订单已 COMPLETED 短路 → **绝不重复解锁/重复入订单**（**L1**）

6. **后端【安全攸关】：红色永不锁不走扣费（L0/L1）**
   **Given** `dangerLevel==RED`（详建响应层恒放行，2-2）
   **When** 对 RED triage `POST /unlock`
   **Then** **短路成功不扣费**：不 tryConsume、不 debit、不建 intent/order，返回结果（`locked=false`）——红色本就免费（FR-43C，即使额度耗尽也不该因「解锁」被扣）；红色永不进付费分支（**L0/L1**）

7. **后端：可复看下发（复用 2-2 分字段）（L0/L1）**
   **Given** 解锁成功后
   **When** `GET /triage/{id}`（既有）或 unlock 端点返回体
   **Then** 复用 2-2 `TriageResultResponse.from`：解锁后 `advice`/`medicationRef` 下发、`locked=false`、`unlockSource` 反映来源（FREE_QUOTA/PAID）；unlock 端点成功可直接返回 `TriageResultResponse`（同步路径）或支付信息（现金路径未支付）（**L0/L1**）

## Dev Notes

> 集成枢纽：**不重复造轮子**——额度（2-1）、解锁跃迁（2-2）、PawCoin 扣（1.2）、支付意图+回调（1.1）、到账事件（1.1）全已就绪，本 story 是**编排 + 一张订单表 + 一个到账处理器**。同步两路径是核心（先跑通幂等/解锁）；现金路径纯复用 payment_intents 双通道，加个 `AiUnlockPaidHandler` 照 `TopupPaidHandler` 抄。

### 承接（已 done / 已 review，直接调用勿重写）
- **2-1 `FreeQuotaService`**（`triage/service`）：`tryConsume(userId)`（`@Transactional` REQUIRED，`insertIfAbsent`+原子条件 UPDATE，返回 boolean；`limit<=0` 短路 false）；`status(userId)`；WIB period 惰性重置。**REQUIRED 便于并入本 story 外层解锁事务**（2-1 已按此设计）。
- **2-2 `TriageTask`**：`unlock(UnlockSource, UnlockChannel)` —— 仅 `LOCKED/NULL→FREE_QUOTA/PAID` 一次性跃迁，已解锁抛 `IllegalStateException`（不可覆盖）；`PAID` 必带 channel、`FREE_QUOTA` channel 必 null；`markDone` 已置 `LOCKED`。`getUnlockSource()`/`getDangerLevel()`。`TriageResultResponse.from(task)` 分字段 + 红色永不锁（`isUnlocked` 单点）。
- **1.2 `PawCoinWalletService.debit(userId, coins, PawCoinTxnType, refType, refId, idempotencyKey)`**：`requirePositive`→幂等短路（Redis+DB）→`applyDelta(-coins)`（`WHERE balance-coins>=0`，0 行=余额不足抛冲突）→ `LedgerService.post` 平衡分录（贷方 `PLATFORM_REVENUE`）+ 写 `pawcoin_transactions`。**幂等键=`ai-unlock:{triageId}`**（稳定可重放，勿用随机/时间戳）。`PawCoinTxnType` 有 `SPEND`。
- **1.1 `PaymentIntentService.createIntent(userId, PaymentPurpose, PayChannel, amount, currency, idempotencyKey)`**：幂等建单（`IdempotencyService` 重放取回）+ 限流；返回 `PaymentIntentResponse`（含 QR/deeplink）。`applyCallback` 双通道单一收口，PENDING→PAID 后发 `PaymentIntentPaidEvent`（同 `@Transactional` 内同步内联）；已 PAID intent 不再发事件（回调/轮询去重）。
- **1.1 `PaymentIntentPaidEvent(intentId, publicToken, userId, purpose, channel, amount, currency)`** + **`TopupPaidHandler`（范式模板）**：`@EventListener @Transactional(propagation=MANDATORY)`，`if(purpose!=X) return;`，同事务入账。**照抄给 `AiUnlockPaidHandler`。**
- **`PaymentPurpose.AI_UNLOCK` 已存在**（1.1 建枚举时预留）；`PayChannel{QRIS,PAWCOIN}`。
- **既有 `TriageController`**：`GET /{id}`（数字 id + JWT）；`TriageService.getResult(userId, id)` 越权/不存在统一 403 防枚举。**本 story 加 `POST /{id}/unlock` 复用同鉴权/寻址范式。**

### 关键设计决策（dev 照此）
- **intent→triage 关联（现金路径核心缺口的解法）**：`PaymentIntent`/`PaymentIntentPaidEvent` **不带解锁目标**（它们对充值等用途通用）。故现金发起时**建 `ai_consult_orders(PENDING_PAYMENT)` 存 `payment_intent_token` 与 `triage_task_id`** 作为关联锚；到账处理器按 `intent.publicToken` 反查订单拿 `triage_task_id` 去 unlock。**不改 pay 模块**（不给 intent 加 ref、不改事件）——关联落在本 story 自己的订单表（owning 模块自持，照跨模块解耦惯例）。
- **`ai_consult_orders` 三态**：架构只列 `COMPLETED/ABNORMAL`，本 story **增 `PENDING_PAYMENT`**（现金异步到账前的锚态）。`ABNORMAL` = 到账但订单缺失/triage 已被其它路径解锁/金额不符等对账异常（处理器记录不 crash，供 9-x 对账）。同步路径（PawCoin）直接建 `COMPLETED`；FREE_QUOTA **不建订单**。
- **红色永不锁不扣费（AC6，安全攸关）**：service 入口 `if(dangerLevel==RED) return 已解锁结果;` **在任何扣费分支之前**短路。红色详建响应层本就放行（2-2），此处保证「解锁」动作也不触发扣费（即使客户端误传 method）。与 2-2 的响应层红色单点呼应——**双保险**：展示侧红色恒放行 + 收费侧红色恒不扣。
- **已解锁短路（AC5）**：入口 `if(task.getUnlockSource() 已是 FREE_QUOTA/PAID) return 结果;`（**在扣费前**）。防二次扣费第一道闸；`task.unlock` 不可覆盖 + debit/intent 幂等键 + 订单唯一约束是后续兜底。
- **同一事务（AC3，NFR-3）**：同步路径用一个 `@Transactional`：额度/扣币 + `task.unlock`（JPA 脏检查 flush）+ 建订单。任何一步抛 → 全回滚。`FreeQuotaService.tryConsume`（REQUIRED）与本事务合并。**现金路径的「同一事务」在到账处理器侧**：`AiUnlockPaidHandler`（MANDATORY）与 `applyCallback.markPaid` 同事务，unlock+订单 COMPLETED 与 markPaid 原子。
- **归属**：`ai_consult_orders` 是「AI 解锁订单」——建议放 `triage` 模块（`triage/domain/AiConsultOrder`），因它绑 triage 且价来自 triage 配置；`AiUnlockPaidHandler` 放 `pay` 模块（与 `TopupPaidHandler` 同处，消费 pay 事件），注入 triage 侧 service（照 account 编排调 owning service 的解耦向：pay 处理器调 `AiUnlockService.completeCashUnlock(intentToken)` 而非直 join triage 表）。dev 择一并保持解耦（pay 不直 join triage 表 / triage 不直 join pay 表）。

### 幂等多层兜底（AC5，血泪）
1. service 入口 `alreadyUnlocked` 短路（第一闸，防二次扣费）。
2. `task.unlock` 不可覆盖抛 `IllegalStateException`（第二闸，DB 脏检查前拦）。
3. PawCoin `debit` 幂等键 `ai-unlock:{triageId}`（Redis+DB+总账唯一约束）。
4. `createIntent` 幂等键 `ai-unlock:{triageId}`（同 key 重放取回既有 intent，不双建单）。
5. `ai_consult_orders` 唯一约束（order_token + intent token）。
6. 现金回调/轮询双通道：`applyCallback` 已 PAID 不再发事件 + 处理器订单 COMPLETED 短路。
> **禁 `AFTER_COMMIT` 异步入账/解锁**（记忆库血泪：notify AFTER_COMMIT+默认 REQUIRED 静默吞写）。到账处理器用同步 `@EventListener` + `MANDATORY`。
