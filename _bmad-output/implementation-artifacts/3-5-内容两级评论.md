# Story 3.5: 内容两级评论

Status: review

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

## Story

As a **用户**,
I want **对内容发表评论并回复他人评论**,
so that **我能与社区展开讨论**。

> 本 Story 实现两级评论的**写入与删除**（FR-24）——Story 3.3 已做评论**只读**展示，本 Story 补齐发表/回复/删除及权限。**后端中、前端中**：后端建 `comments` 表（自引用二级）+ 发表/回复/删除端点 + 删除权限 + 级联删二级 + 产出互动事件；前端接底部评论框发表、回复入口、删除菜单。
>
> **做**：① 一级评论（≤200 字，时间正序）写入；② 二级回复（≤200 字，挂一级下，**不再嵌套**，最多两级）；③ 删除权限（作者删自己任意级评论 / 内容主删其内容下任意评论，删一级**级联删二级**）；④ 需登录，未登录触发 FR-0C；⑤ 评论产出互动事件供 FR-22B 推送消费。
> **不做**：评论**读取/分页展示**（Story 3.3 已做，本 Story 复用其展示）；推送下发（Epic 6）；@提及/评论点赞/评论举报（V1 不做，FR-25 明确 V1 不做评论举报）。
> **依赖前序**：Story 3.3（详情页评论区展示 + 底部评论框门控 + 评论读取端点；若 `comments` 表读路径已在 3.3 建，本 Story 复用并补写/删路径）、Story 1.4/1.5（FR-0C）、Story 1.3（用户身份）、Story 2.3（`content_posts` + 内容作者身份用于删除权限判定）。

## Acceptance Criteria

> **验证层标注**：**L0**（编译/lint/widget test/Service 单元 + MockMvc） · **L1**（需 postgres+redis，验评论写入 + 二级挂载 + 删除权限 + 级联删 + 事件） · **L2**（真机，评论框/回复交互）。
> 评论写入/删除权限/级联/事件为 **L1**（DB 读写 + 权限矩阵是核心）；评论框/回复 UI 为前端 **L0 + L2**。无外部第三方依赖。

### AC1 — 一级评论 + 二级回复（最多两级，正序）

**Given** 登录用户在详情页
**When** 发表一级评论（≤200 字）
**Then** 即时显示、按时间正序排列（写 `comments` 表，本故事创建该表）（FR-24）
**And** 发表二级评论（回复一级，≤200 字）显示在该一级评论下方，二级下不再嵌套（最多两级）
> 验证层：评论表创建 + 一级/二级写入 + 时间正序 + 二级挂载到正确 parent + **回复二级时强制归并到其一级（不产生三级）** 为 **L1**；≤200 字服务端权威校验为 **L1**（客户端实时计数为体验 L0）；即时显示（乐观/重拉）为前端 **L0 + L2**。

### AC2 — 删除权限 + 级联删二级 + 需登录 + 互动事件

**Given** 评论删除权限
**When** 用户删除评论
**Then** 用户可删自己任意级别评论、内容作者可删其内容下任意评论；删一级评论时其下二级一并删除
**And** 评论需登录、未登录触发 FR-0C；评论产生互动事件供 FR-22B 推送消费
> 验证层：**L1**（删除权限矩阵：评论作者删自己✓、内容主删其内容下任意✓、无关用户删→403、删一级**级联删全部二级**、删二级仅删该条；需登录→401→FR-0C；发表评论发布 `ContentCommentedEvent`，自评不通知）。权限矩阵是核心，须穷举验证。

### AC3 — 评论发送失败保留输入 `[R2·回改]`

> **代码已实现，本轮回改补齐**（review→ready-for-dev）。统一 F13 输入类失败口径——发送失败不丢用户已输入内容。

**Given** 登录用户在底部评论框已输入一级评论或二级回复
**When** 发送失败（网络/服务器错误）
**Then** 提示「发送失败，请重试」（F13、双语 .arb，每条 ≤1 emoji），**输入框内容保留不清空**，回复态（`replyTarget`）也保留，用户可**直接重试无需重新输入**
**And** 仅在发送成功后才清空输入框并退出回复态、刷新评论区
> 验证层：**L0**（widget/provider 测试：模拟提交抛错 → 输入框 controller 文本保留、回复态保留、失败提示出现；模拟成功 → 输入框清空 + `commentsRefreshProvider` bump。区别于服务端 ≤200 字 422 校验——422 同样保留输入可改后重发）。任务：前端。

## Dev Notes

### 关键架构约定（本 Story 必须落实）

**两级评论模型**（FR-24）：一级 ≤200 字时间**正序** + 二级回复 ≤200 字**不再嵌套**（最多两级）；删除权限（作者删自己/内容主删任意，删一级**级联删二级**）；触发 FR-22B；需登录。**回复二级评论时归并到其一级父**（绝不产生三级）——这是 dev 易错点，service 层强制。

**与 Story 3.3 分工**：3.3 做评论**只读展示 + 分页**（首 10 一级、二级默认 3、查看全部）；本 Story 做**写入/回复/删除/权限/事件**。`comments` 表归属本 Story（若 3.3 为读路径先建表，本 Story 复用补写删——在 File List 标注实际归属）。commentCount 口径需与 3.3 详情计数一致。

**互动事件**（架构 §Communication）：`ContentCommentedEvent` 经进程内事件 → notify（Epic 6）消费；自评不通知；二级携带内容主 + 一级作者供推送去重。**禁跨模块调 notify repository**。

**软删与级联**（架构 §Naming：软删处 `deleted_at`）：评论软删；删一级事务内级联软删二级；内容被删（Story 3.6）时其评论一并删（3.6 负责，本 Story 表结构支持级联）。

**命名映射链**：`comments`(post_id/author_id/parent_id/body/created_at/deleted_at)、索引 `idx_comments_post`/`idx_comments_parent`；DB snake_case ↔ JSON camelCase；事件 `ContentCommentedEvent`。

**校验时机**（架构 §Process）：服务端 Bean Validation 权威（≤200→422）；客户端实时计数仅体验。

### 强制护栏（架构 §Enforcement —— 违反即返工）

- **禁止擅自引入 MQ / 通用缓存层 / 新中间件**；评论直读写 PostgreSQL。
- `ddl-auto=validate`；comments 表/索引/外键走 Flyway。
- 模块间只经 service/事件——评论事件给 notify，**禁跨 repository**。
- **两级硬约束**：回复二级归并一级，绝不三级嵌套。
- 删除权限服务端权威（前端隐藏入口仅体验，后端必校验作者/内容主）。
- 自评不通知；写端点 Idempotency-Key。
- 注销作者匿名化（NFR-8）。

### 范围边界（防 scope creep —— 本 Story 明确不做）

- ❌ 评论读取/分页展示（Story 3.3 已做）——本 Story 复用。
- ❌ 推送下发（Epic 6）——只产出 `ContentCommentedEvent`。
- ❌ @提及 / 评论点赞 / **评论举报**（V1 不做，FR-25 明确仅内容举报）。
- ❌ 内容删除级联清评论（Story 3.6 触发，本 Story 表结构支持）。
- ✅ 只做：comments 表 + 一级/二级写入（两级约束）+ 时间正序 + 删除权限矩阵 + 级联删二级 + FR-0C 门控 + 互动事件。
