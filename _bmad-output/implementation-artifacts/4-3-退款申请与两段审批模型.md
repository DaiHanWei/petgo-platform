# Story 4.3: 退款申请与两段审批模型

Status: review

> V1.1 **Epic 4 第 3 story**，**纯后端**（模型 + 服务原语 + 职责分离 + 加密 + 审计；L1）。**全系统最高危节点**（架构 §12 H-8，A-1）：退款三级职责分离建模。本 story 建 `refund_requests` 两段审批数据模型 + `net_amount` 后端权威计算 + payout PII 字段级加密 + **同单职责互斥服务层强制（SUPER_ADMIN 不豁免）** + 接入既有 AdminAuditService 哈希链。
> 源：`epics-v1.1.md` Story 4.3（`refund_requests` 两段字段 + `net_amount` + 加密 payout PII，Flyway；`net=order−fee` 后端权威 L1；同 admin 兼任两职→服务层拒绝 SUPER_ADMIN 不豁免 + 记审计 L1）· A-1/A-2 · FR-NFR-4/5 · 架构 `architecture-v1.1-delta.md` §3.4 refund_requests（line 111-122）· §5.2 退款两段审批 · 决策表 A-1/A-2/A-9。
> **⚠️ 范围边界（4-3 = 模型 + 受控状态迁移原语，行为流留后续）**：本 story 建**表 + 实体 + 三段角色分配服务原语（submit 需求 / approve 审批 / payout 打款）+ 职责分离守卫 + 净额计算 + payout 加密 + 审计**。**不做**：客服判定的订单解锁/回落 COMPLETED+refund_rejected 行为与 App 通知（4-4）、用户端选退款方式与填收款 UI（4-5）、主管审批 UI + 财务实际 Midtrans Iris/Disbursement 打款 + PawCoin 转币（4-6）、结案通知/CSAT（4-7）、`REFUND_REJECTED` 等通知类型扩枚举（4-4，`extend_notification_types_v11`）、admin 前端 Thymeleaf 页面（各自 story）。本 story 只保证：模型正确、净额权威、职责互斥不可绕过、PII 加密不入日志、写操作留审计——均 L1 可测。

## Story

As a 平台,
I want `refund_requests` 明确分离客服判定与主管审批两段、算净额、加密收款账号、职责分离,
so that 退款可审计、UI 不撒谎、内控不越界（A-1/A-2/FR-NFR-4）。

## Acceptance Criteria

1. **后端：`refund_requests` 表（Flyway V71/顺延）（L0/L1）**
   **Given** 无既有退款表（V1.0 零资金）
   **When** 建表
   **Then** `refund_requests`：`id`、`refund_token`(VARCHAR(32) 不可枚举 `CardTokenGenerator`)、`order_id`(bigint FK `consult_orders`)、`related_ticket_id`(bigint FK `feedback_tickets`，可空)、`user_id`(bigint)；**两段字段**：`need_decision`(varchar `PENDING`/`APPROVED`/`REJECTED` 默认 PENDING，CHECK)、`submitter_admin_id`(bigint 可空)、`approval_status`(varchar `PENDING_APPROVAL`/`APPROVED`/`REJECTED`/`PROCESSING`/`DONE` 可空，CHECK)、`approver_admin_id`(bigint 可空)、`payer_admin_id`(bigint 可空)；**金额**：`order_amount`(bigint)、`channel_fee`(bigint)、`net_amount`(bigint)；**收款 PII（加密列）**：`payout_channel`(varchar 可空 `BCA`/`OVO`/`GOPAY`，CHECK)、`payout_account`(varchar 加密密文，可空)、`account_holder_name`(varchar 加密密文，可空)；`created_at`/`updated_at`(timestamptz)；唯一 `refund_token`；唯一 `order_id`（一订单一退款请求，防重复退款）
   **And** Flyway **V71**（当前 max V70=4-1）——移动靶：merge 时按当时全局 max 顺延（决策 E2）；VARCHAR≥16 无 CHAR(1) 坑（**L0**）

2. **后端：净额后端权威计算（L1）**
   **Given** 订单金额 + 渠道费表（`BCA=0` / `OVO=2500` / `GoPay=2500`）
   **When** `RefundService` 计算净额
   **Then** `net_amount = order_amount − channel_fee`，渠道费按 `payout_channel` 查权威费率表（后端常量/枚举，非前端传入）；前端展示须与后端一致（FR-NFR-5 跨文档校验）；`net_amount` 落库快照（**L1**）
   **And** 渠道费率集中定义（`PayoutChannel` 枚举带 fee 或 `RefundProperties`），9-2 后台可调则外部化预留（本 story 常量即可）（**L0/L1**）

3. **后端：三段角色分配原语 + 职责分离守卫（A-1，最高危）（L1）**
   **Given** 三权限点 `refund.submit`/`refund.approve`/`refund.payout`（`AdminPermissions` 新增 + `ALL` 白名单）
   **When** `RefundService` 分配 `submitter_admin_id`（提交需求）/ `approver_admin_id`（主管审批）/ `payer_admin_id`（财务打款）
   **Then** **服务层强制** `submitter_admin_id`/`approver_admin_id`/`payer_admin_id` **两两不可相等**——任一后置角色的 `admin_id` 等于同一 `refund_request` 上已占的另一角色 → 拒绝（`AppException.forbidden`/409）+ `AdminAuditService.record` 记违规尝试；**含 SUPER_ADMIN 不豁免**（守卫是 admin_id 相等判定，非权限判定——SUPER_ADMIN 虽持全部权限点仍被拦）（**L1**）
   **And** 每次角色分配（submit/approve/payout）为成功写操作时接入 `AdminAuditService.record(actorAccountId, action, "refund_request", refundToken, summary)`；**summary 严禁含 payout PII / 账号**（护栏）（**L1**）

4. **后端：payout PII 字段级加密 + 日志红线（FR-NFR-4）（L1）**
   **Given** `payout_account`/`account_holder_name` 为敏感 PII
   **When** 持久化
   **Then** 落库**加密密文**（JPA `AttributeConverter` + AES-GCM，密钥 env 注入 `REFUND_PII_KEY`，绝不入库/入日志——凭证 env 红线）；读取自动解密；**日志绝不出现明文账号/持有人名**（**L1** 断言密文列非明文 + 往返解密一致）
   **And** 缺密钥（env 未配）→ 启动期明确失败或加密调用抛可控异常，不静默存明文（**L1**）

5. **后端：契约 + 静态验收（L0/L1）**
   **Given** 全部实现
   **When** 验证
   **Then** `DB_NAME=petgo_l1 mvn -B test` 绿：L0（V71 建表、净额计算单测[各渠道费]、职责分离守卫单测、加密往返单测）+ L1（建 refund_request 落库、净额权威、职责互斥三组合[submit=approve / approve=payer / submit=payer 均拒]含 SUPER_ADMIN、审计留痕、payout 密文非明文、唯一 order_id 防重）；schema `validate` 一致（`ddl-auto=validate`）（**L0/L1**）
   **And** 无前端（用户端选退款方式在 4-5、admin UI 在 4-6）；本 story 纯后端模型 + 原语（**L1**）

## Dev Notes

> 纯后端建模，**照 `support`(4-1) / `consult`(3-x) 分层范式**（domain/repository/service）+ 对接**既有 admin 权限/审计基建**。**全系统最高危**：职责分离守卫不可绕过、payout PII 加密不入日志、写操作全审计。实际打款（Midtrans Iris）+ 订单回落 + 通知 + App UI 均**不在本 story**。

### 承接现状（已核实，照此复用/勿改）

- **admin 权限基建（已存在）**：`admin/account/domain/AdminPermissions.java` = 全小写点分 `<模块>.<动作>` **String 常量** + `ALL` 白名单（UI 勾选 + 合法校验取此）。SUPER_ADMIN 经 `hasRole('SUPER_ADMIN')` **隐式全权**（不注入全集）。本 story **加 `refund.submit`/`refund.approve`/`refund.payout` 三常量 + 追加 ALL**。⚠️ **职责分离守卫与权限点正交**：权限点管「能否做该类操作」，职责分离管「同一人不能在同一退款单上兼任两角色」——后者是**服务层 admin_id 相等判定**，SUPER_ADMIN 持全部权限点仍被此守卫拦（A-1「不豁免」正是指这个）。
- **审计基建（已存在，必接入）**：`admin/audit/service/AdminAuditService.record(Long actorAccountId, String actionType, String targetType, String targetId, String summary)`（`@Transactional` REQUIRED，被业务事务调用即加入同事务；哈希链 append-only）。`targetType="refund_request"`，`targetId=refundToken`。**summary 严禁含密码/令牌/签名 URL/健康数据/payout PII**（既有护栏）。action 常量照 `admin/audit/service/AuditActions.java` 范式新增（如 `REFUND_SUBMIT`/`REFUND_APPROVE`/`REFUND_PAYOUT_ASSIGN`/`REFUND_DUTY_VIOLATION`）。
- **admin 主体**：admin 控制器 `@AuthenticationPrincipal AdminUserDetails admin`（`admin.getAccountId()`）。本 story 是**服务原语**（4-6 admin 控制器才接线），service 方法**入参传 adminId**（由 4-6 控制器从 principal 取），4-3 用测试直接传 adminId 验守卫。
- **订单关联（勿改订单态）**：`consult/repository/ConsultOrderRepository.findByOrderToken` 拿 order；`refund_requests.order_id` 关联。ConsultOrder 已有 `status`(含 REFUNDING/REFUNDED) + `refund_rejected`（V66）。**本 story 不改订单状态**——客服驳回→订单回落 COMPLETED+refund_rejected 是 **4-4** 行为；本 story 只建退款请求模型与角色分配守卫。
- **工单关联**：`support/domain/FeedbackTicket`（4-1）→ `related_ticket_id`（退款需求经 FR-52 工单发起）。可空（本 story 允许无 ticket 直建，绑定校验从简）。
- **token**：`profile/service/CardTokenGenerator.generate()` → `refund_token`。
- **无加密工具（本 story 引入）**：全仓 `grep Encrypt/Cipher/AES` 无既有工具。本 story 新建 `EncryptedStringConverter`（JPA `AttributeConverter<String,String>`，AES-GCM，密钥 `REFUND_PII_KEY` env base64，随机 IV 前置密文）。**密钥缺失即启动/调用失败，绝不静默存明文**（凭证 env 红线 + PII 加密红线）。
- **资金分录不在本 story**：`ledger_entries`(V61) `REFUND_OUT` 分录 + PawCoin `wallet.credit(REFUND)` + 实际 Midtrans Iris 打款是 **4-6**（`payer_admin_id` 打款那一步才落分录/发起 disbursement）。本 story `net_amount` 仅落库快照，不动账本。

### 关键设计决策（dev 照此）

- **D-1 新 `com.tailtopia.pay.refund` 模块**（domain/repository/service/crypto，归 `pay` 下——退款属资金域）。admin 控制器（4-6）在 `admin/refund` 复用本 service（照 moderation：domain 在业务域、admin 控制器在 admin 下）。
- **D-2 职责分离守卫（最高危，A-1）**：`submitter_admin_id`/`approver_admin_id`/`payer_admin_id` 两两不等。守卫在**每次后置角色分配时**校验：`approve` 时 `approverId != submitterId`；`assignPayout` 时 `payerId != submitterId && payerId != approverId`。相等 → `throw AppException.forbidden(...)`（或 409 conflict）+ 审计 `REFUND_DUTY_VIOLATION`（**注意：违规也要记审计**，但 record 在抛异常前调用需保证事务——record @Transactional REQUIRED 会随抛出回滚！故违规审计用 `REQUIRES_NEW` 或先 record 再 throw 于同事务外——dev 择：违规记审计可用独立 `REQUIRES_NEW` 事务确保留痕，参考通知 AFTER_COMMIT 事务吞写坑教训 [[notify-after-commit-tx-swallow-bug]]）。**含 SUPER_ADMIN 不豁免**（守卫不看角色只看 id）。
- **D-3 净额后端权威**：`net = order_amount − channelFee(payout_channel)`。渠道费 `PayoutChannel.fee`（BCA=0/OVO=2500/GOPAY=2500）常量。`order_amount` 从 `consult_orders.amount` 快照。net<0 兜底 0 或校验（正常 order≥fee）。9-2 后台调费预留（本 story 常量）。
- **D-4 payout PII 加密**：`payout_account`/`account_holder_name` `@Convert(EncryptedStringConverter)`。密文列 VARCHAR 足够长（AES-GCM base64，如 VARCHAR(512)）。`payout_channel` 非 PII（枚举）不加密。**日志绝不记这三个字段**。
- **D-5 唯一 order_id 防重复退款**：一订单至多一 refund_request（DB 唯一 + service `existsByOrderId` 预检 409）。已交付退款反套利（C-1，仅当天+退真钱）语义在 4-5/4-6 落，本 story 建模不阻。
- **D-6 状态迁移极简（本 story）**：`need_decision` PENDING→(4-4 客服)APPROVED/REJECTED；`approval_status` null→PENDING_APPROVAL(选完方式)→(4-6 主管)APPROVED→PROCESSING→DONE。本 story 建枚举 + 三段分配原语 + 守卫；完整流转编排（含订单回落/通知/打款）留 4-4/4-5/4-6。
- **Flyway V71**：当前 max V70（4-1 占）。移动靶顺延（决策 E2）。加密密文列非长度=1，无 CHAR(1) 坑。
