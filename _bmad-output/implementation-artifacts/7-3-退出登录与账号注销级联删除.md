# Story 7.3: 退出登录与账号注销级联删除

Status: review

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

## Story

As a **用户**,
I want **安全退出，并能按法规要求彻底注销账号**,
so that **我对自己的数据拥有可达的控制权（PDP 数据主体权利）**。

> 本 Story 是 Epic 7 的**合规核心、后端重** story（FR-20 注销部分 / NFR-8）。两部分：①退出登录（轻，清本地态回游客）；②**账号注销级联删除**（重，PDP 强制）——双重确认 → 跨**全部业务模块各表 + OSS 私密/公开桶图片**级联删除个人数据，同时 UGC（帖子/评论）**匿名化保留**于 Feed。这是架构 §不可协商地基第 4 条（"数据模型设计之初即考虑各表 + 对象存储照片的级联删除路径"）的兑现 story。删除**不可遗漏任何一张表或任一 OSS 对象**，否则即合规缺口 + PII 残留。注销作为可靠异步作业（DB 状态机驱动）执行，保证可重试、可审计、不半途残留。

## Acceptance Criteria

> **验证层标注**：每条 AC 末尾标注验证层级与所需本地环境——
> **L0 静态**（编译/lint，无需 DB） · **L1 集成/运行时**（需 Docker daemon + postgres + redis） · **L2 端到端/外部凭证**（需真实 OSS 删除 / 真机 / 真实 Google 账号）。
> 注销级联删除的**真实 OSS 图片删除**与**真实 Google 账号全链路**为 **L2**；各表删除/匿名化/状态机可在 **L1** 用 Docker+pg+redis 验。

### AC1 — 退出登录回游客态

**Given** 用户点击退出登录
**When** 确认
**Then** 清除本地登录状态，App 停留在首页保持游客浏览态（FR-20、FR-0A）
> 验证层：**L0**（清 `flutter_secure_storage` access/refresh + 重置 auth provider 为游客态、留首页 widget test）+ **L1**（后端令牌/refresh 失效语义；下次写操作触发 FR-0C 登录引导，需起栈）。退出**不删任何数据**，与注销严格区分。

### AC2 — 账号注销双重确认 + 各表 + OSS 级联删除

**Given** 用户发起账号注销
**When** 双重确认（第一次弹窗说明「删除后数据不可恢复」、第二次要求输入「确认注销」或点击高危按钮）
**Then** 执行级联删除：用户账号、宠物档案、AI 分诊记录（`users`/`pet_profiles`/`triage_tasks` 等纯个人数据表）+ 全部私密/个人 OSS 图片**删除**（数据模型设计之初即纳入路径）；**兽医问诊会话与评分（`consult_sessions`/`consult_ratings`）改为匿名化保留**——剥离 user PII（解关联 `user_id`），保留症状/AI 评级/评分供运营分析与未来 FR-5 历史库；其私密桶图片随个人图片一并删除（FR-20、NFR-8）
> 验证层：**L1**（双重确认门控；触发后**逐表删除** `users`/`pet_profiles`/`triage_tasks` 等纯个人数据表 + 关联私密桶图片，**`consult_sessions`/`consult_ratings` 走匿名化（保留行、剥 PII）**；用 pg 验删除表行消失、匿名化表行仍在但 `user_id` 解关联、删除调用清单完整）+ **L2**（真实阿里 OSS 私密桶/公开桶对象**确实被删除**、真实 Google 账号注销全链路、个人数据各表+OSS 图片确实清空——合规可测核心断言）。
>
> 📌 **决策（D1）**：consult 会话/评分匿名化而非删除，因 ① 评分 FR-33「仅运营可见」供运营分析；② 兽医侧历史记录(5.8)与未来 FR-5 历史库需问诊语料。与 UGC 匿名化策略一致——剥离 PII 即满足 PDP，不必物理删除。triage（纯个人 AI 健康记录、无跨方价值）仍物理删除。

### AC3 — UGC 匿名化保留（不删，剥离 PII）

**Given** 注销后用户发布的帖子/评论
**When** 匿名化保留
**Then** 内容保留在 Feed 与他人评论区不删除，作者头像替换默认占位、昵称显示「已注销用户」，仅保留文字与图片不保留任何个人身份信息，点击「已注销用户」头像不触发 FR-26（NFR-8）
> 验证层：**L1**（注销后 `content_posts`/`comments` 行**保留**但 `author_id` 解关联/置匿名标志、头像→默认占位、昵称→「已注销用户」i18n 文案；Feed 与评论区仍可见这些内容；点匿名头像**不触发** FR-26 迷你主页卡——需 pg + 起栈验）+ **L2**（真机 Feed 视觉验证）。

### AC4 — [R2·回改] 注销时 H5 宠物名片链接立即失效（F14，承接 2-6 / D2 邻接）

> **[PRD1.0.0✏️·R2]** 代码已实现，本轮仅**追加名片 token 失效保证**。上游 FR-20 明确「宠物档案（含 H5 名片链接）删除，名片链接注销后立即失效」。**既有 D1（consult 匿名化 / triage 删除）/ D2（IM 媒体）级联删除实现一律不回退**——本 AC 不改 AC2/AC3/B3/B5/B5b/B5c 任何删除/匿名化逻辑，仅补名片 token 失效一项。

**Given** 用户注销、其宠物档案被删除（AC2 已删 `pet_profiles`）
**When** 任何人访问该宠物的 H5 名片链接（FR-14 不可枚举 token）
**Then** 名片 token 立即失效，H5 返回失效态「这只宠物的档案已不存在」（承接 2-6 FR-14 链接失效态），**不展示任何宠物数据、不暴露资源曾存在**（防枚举 404 语义）
**And** 名片失效随 `pet_profiles` 删除即时生效（token 解析依赖档案存在性），无需额外 TTL；本 AC **仅新增名片 token 失效保证**，D1/D2 既有合规实现不动
> 验证层：**L1**（注销后名片 token 解析 → 失效态，不返回宠物数据 / 不 200 泄露存在性）+ **L0**（失效态文案 + 防枚举 404 语义断言）。

## Tasks / Subtasks

> **按"后端子任务 / 前端子任务 / 联调验收"三段组织**。本 Story **后端重**（级联删除作业是核心）。建议顺序：后端 → 前端 → 联调。

### 🟦 后端子任务（petgo-backend / auth + profile + 级联删除作业）

- [ ] **B1. 退出登录** (AC: 1)
  - [ ] `POST /api/v1/auth/logout`：使当前 refresh 失效（access 短时自然过期）；不触碰任何业务数据。
- [ ] **B2. 注销发起端点 + 双重确认契约** (AC: 2)
  - [ ] `DELETE /api/v1/me`（或 `POST /api/v1/me/deletion`）：要求二次确认载荷（如确认短语/高危确认标志），缺失→422 ProblemDetail。
  - [ ] 写入 `account_deletions`（或复用 `users.status=DELETING` + DB 状态机：`PENDING→PROCESSING→DONE/FAILED` + `retry_count`），返回 202 受理（异步作业，见架构 §异步）。
- [ ] **B3. 级联删除作业（核心，跨全部模块各表 + OSS）** (AC: 2)
  - [ ] `@Async` + DB 状态机驱动的 `AccountDeletionJob`：**穷举**当前用户在全部业务模块的个人数据表，逐表删除——
    - `auth`: `users`（账号/OAuth 绑定/refresh token）
    - `profile`: `pet_profiles`、成长时间线（快乐时刻/健康事件）等
    - `triage`: `triage_tasks` 及关联结构化结果/JSONB（纯个人 AI 健康记录，**删除**）
    - ⚠️ `consult`: `consult_sessions`/`consult_ratings` **不在此删除列表**——走 B5 匿名化保留（剥 user PII，保留症状/评级/评分供运营与未来 FR-5 库）。仅删其私密桶图片。
    - `notify`: 该用户 `notifications`、推送绑定/角标态
    - 任何以 `user_id`/`author_id`/`pet_id` 关联的个人数据表
  - [ ] **跨模块经各 owning service 删除**（禁直接 join 他模块表，架构 §模块边界）；profile 订阅/编排，各模块提供 `deleteByUserId` service 方法。
  - [ ] **OSS 图片删除路径**（不可遗漏）：删私密桶②（AI 分诊图/健康历史图）+ 公开桶①中**纯个人**图片（头像；档案图随档案删——注意 Feed 帖子图属匿名化保留，见 B5 不删）；经 `shared/media` 收集该用户全部对象 key 逐一删除。
  - [ ] 删 Redis 中该用户痕迹（在线态/队列态/角标计数/幂等键残留）。
  - [ ] 作业**幂等 + 可重试**（状态机重扫，部分失败重跑剩余）；全部完成才置 `DONE`，半途失败 `FAILED`+`retry_count` 不丢。
- [ ] **B4. 删除完整性守护（强制护栏）** (AC: 2)
  - [ ] 维护一份"个人数据表 + OSS 桶/前缀"清单作为删除范围单一事实源；新增个人数据表时必须登记（设计之初纳入语义）。
  - [ ] 作业完成校验：按清单逐项确认该 user 残留为零（各表 `count=0`、OSS 列举无残留 key），否则不置 `DONE`。
- [ ] **B5. UGC 匿名化（保留，剥 PII）** (AC: 3)
  - [ ] content 模块 `anonymizeByUserId`：`content_posts`/`comments` **保留行**，解除/置空作者 PII 关联——`author_id` → 匿名占位、作者头像 → 默认占位、昵称 → 「已注销用户」(i18n key)；保留正文与图片，**不保留任何个人身份信息**。
  - [ ] 匿名作者**不可点开 FR-26 迷你主页卡**（标志位让前端禁用触发）。
  - [ ] 帖子图（公开桶①）随匿名化**保留**，与 B3 的个人图删除区分清楚。
- [ ] **B5b. consult 会话/评分匿名化（决策 D1）** (AC: 2)
  - [ ] consult 模块 `anonymizeByUserId`：`consult_sessions`/`consult_ratings` **保留行**，解关联/置空 `user_id` 与任何用户 PII（昵称快照等），**保留** `symptom`/`danger_level`/`rating`/`vet_id` 等供运营分析（FR-33 仅运营可见）与未来 FR-5 历史库。
  - [ ] consult 私密桶②图片（分诊升级图/聊天存档图）随 B3 个人图**删除**（图片属个人 PII，不随会话保留）。
  - [ ] 兽医侧历史记录(5.8)在匿名化后仍可见会话（作者显示「已注销用户」），不空屏。

## Dev Notes

### 关键架构约定（本 Story 必须落实）

- **目录**：后端 `com.petgo.auth/`（账号/令牌）+ `com.petgo.profile/`（注销编排 + 级联删除作业）+ 各模块提供 `deleteByUserId`/`anonymizeByUserId` service；前端 `lib/features/me/{data,domain,presentation}` + `core/storage/secure_storage`。
- **异步可靠性**：注销走 `@Async` + DB 状态机（`PENDING→PROCESSING→DONE/FAILED` + `retry_count` + 启动重扫），**不引入 MQ**（架构 §异步）。半途失败可重跑剩余，绝不留半删状态当成功。
- **跨模块边界**：删除/匿名化经各 owning service 接口，**禁直接 join 他模块表**（架构 §模块边界）。profile 作为编排者。
- **媒体三层**：私密桶②（分诊/健康图 + consult 存档图，**删除**）+ 公开桶①（头像随删 / 帖子图匿名保留）+ IM 托管（聊天媒体）：**注销时调腾讯 IM 删除该用户会话媒体，或确认 IM 侧 TTL 自动清理（二选一，dev 落实并记录）**；经 `shared/media` 删除。
- **匿名化 vs 删除分类（决策 D1/D2 后的权威清单）**：**删除**=`users`/`pet_profiles`/`triage_tasks`/`notifications` + 所有个人 OSS 图片（头像/分诊/健康/consult 存档图）+ IM 聊天媒体 + Redis 痕迹；**匿名化保留**=`content_posts`/`comments`（UGC）+ `consult_sessions`/`consult_ratings`（运营/历史库价值）。
- **错误**：双重确认缺失 → 422 ProblemDetail；越权注销他人 → 403。

### ⚖️ 合规攸关约束

- **账号注销级联删除（NFR-8 / 架构不可协商地基 #4）**：删除路径在**数据模型设计之初即纳入**——本 Story 兑现。删除**穷举各表 + OSS 图片**，由 B4 的删除范围清单（单一事实源）守护；任一表或对象遗漏 = PII 残留 = 合规失败。**新增个人数据表必须登记到清单**（前向约束）。
- **注销内容匿名化保留（NFR-8）**：UGC 不删但**彻底剥离个人身份信息**——`author_id` 解关联、头像默认占位、昵称「已注销用户」、不可触发 FR-26。仅保留文字与图片，确保无任何 PII 可回溯到已注销主体。
- **PDP 基础（NFR-8）**：注销是「数据主体权利可达路径」的核心兑现——收集前同意（FR-0D）+ 目的告知 + 真实隐私政策 + **可达的删除路径**四者构成 PDP 地基。注销入口须从「我的」（7.1）可达。**跨境出境为已挂账风险**（暂缓≠豁免）。
- **日志不落 PII**：删除作业日志只记代理 id + 状态机进度 + 计数，严禁昵称/头像 URL/令牌/签名 URL/健康数据。保留不含 PII 的注销受理时间作合规留证。

### 强制护栏（违反即返工）

- **级联删除不可遗漏任何表 / OSS 对象**：以删除范围清单 + 完成校验（各表 count=0 + OSS 列举零残留）守门，未达不置 `DONE`。
- **退出 ≠ 注销**：退出绝不删数据；注销绝不残留 PII。二者代码路径严格分离。
- **匿名化 vs 删除边界清晰**：UGC 帖子图保留、个人图（头像/分诊/健康图）删除，**不可混淆**。
- 异步只用 `@Async`+DB 状态机；删除作业幂等可重试；env 注入凭证。
- 红色态/安全语义不在本 Story 范围但勿埋违反点。

### 范围边界（本 Story 明确不做）
