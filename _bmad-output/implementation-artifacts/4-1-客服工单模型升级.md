# Story 4.1: 客服工单模型升级

Status: ready-for-dev

> V1.1 **Epic 4 开篇** story，**纯后端建模**（L1，无前端——用户端投诉 UI 在 4-2）。Epic 4（退款与客服工单）依赖 Epic 3（有订单才退款）。本 story 建**客服工单数据模型**：`feedback_tickets` 主表 + 三子表（`ticket_attachments` OSS 附件 / `ticket_labels` 标签 / `ticket_internal_notes` 内部备注用户不可见）+ 建工单原语（≤5 图 OSS 预签名）+ CSAT/`cs_rating` 预留列（避免二次迁移）。
> 源：`epics-v1.1.md` Story 4.1（工单支持图片附件、自填联系方式、CSAT、内部备注、标签；工单表 ALTER + 子表 + `cs_rating` 预留 Flyway V53→顺延；建工单带 ≤5 图 OSS 预签名 → 附件/联系方式/标签落库、内部备注用户不可见 L1；schema validate L0/L1）· FR-52（客服工单）· AB-5A~5H · 架构 `architecture-v1.1-delta.md` §3.6（工单扩展列 + 子表 line 133-136）· 端点表 line 235（`/support-tickets`·`/{id}/csat`）。
> **⚠️ 现状（已核实）**：**无既有客服工单表**——V1.0「帮助反馈」仅「简单表单或跳转 WhatsApp」（PRD line 588），无落库工单系统。故架构 §3.6「ALTER 既有客服工单表（或新 `feedback_tickets`）」在本仓 = **新建 `feedback_tickets`**（不是 ALTER）。**标题「模型升级」= 从 V1.0 无工单升级到真工单系统**。最近的同构范式 = `moderation.ContentReport`（FR-51 举报：用户建、admin 处理、status 流转、`handled_by`/`handled_at`）——本 story 照其结构（domain/dto/repository/service/web）。OSS 预签名上传**已具备**（`shared/media/PresignedUploadService` + `MediaController` + `UploadUrlRequest/Response` + `AliyunOssClient`）——附件**复用既有预签名上传**，前端拿 objectKey 回传，本 story 只**存 objectKey**，不新建上传端点。
> **范围边界**：本 story = `feedback_tickets`(V70) + 三子表 + 实体/repo + 建工单 service（用户建：主题/正文 + 自填联系方式 + 标签 + ≤5 附件 objectKey）+ **用户视图 DTO 排除内部备注** + 加内部备注原语（admin 用，用户不可见契约）。**不做**：前端投诉 UI（4-2）、退款申请两段审批（4-3）、客服判定/订单解锁回落（4-4）、CSAT 问卷发放/结案通知（4-7，`csat_*`/`cs_rating` 列本 story 只**预留**不填不发）、通知类型扩枚举（`extend_notification_types_v11` 属 4-4）、admin 处理 UI/流转（4-4/4-7）、工单 7 天自动关闭 scanner（§调度⑦，4-7）。

## Story

As a 平台,
I want 客服工单支持图片附件、自填联系方式、CSAT、内部备注、标签的完整数据模型,
so that 客服能完整处理投诉（FR-52，AB-5A~5H）。

## Acceptance Criteria

1. **后端：`feedback_tickets` 主表 + 三子表（Flyway V70/顺延）（L0/L1）**
   **Given** 无既有工单表
   **When** 建表
   **Then** `feedback_tickets`：`id`、`ticket_token`(VARCHAR(32) 不可枚举 `CardTokenGenerator`)、`user_id`(bigint)、`subject`(varchar 可空)、`body`(text 投诉正文)、`contact_type`(varchar `EMAIL`/`WHATSAPP`，CHECK)、`contact_value`(varchar)、`need_contact_customer`(bool 默认 true)、`contacted_customer`(bool 默认 false)、`related_order_id`(bigint 可空，关联 `consult_orders`)、`status`(varchar `OPEN`/`IN_PROGRESS`/`RESOLVED`/`CLOSED`，CHECK，默认 OPEN)、**预留列**（本 story 不填）`csat_score`(smallint 可空 1-5)、`csat_comment`(varchar(100) 可空)、`csat_deadline`(timestamptz 可空)、`cs_rating`(smallint 可空，AB-5G 后续)、`handled_by`(bigint 可空 admin)、`resolved_at`(timestamptz 可空)、`created_at`/`updated_at`(timestamptz)；唯一 `ticket_token`
   **And** 子表：`ticket_attachments`(`id`/`ticket_id` FK/`object_key` VARCHAR OSS/`created_at`)；`ticket_labels`(`id`/`ticket_id` FK/`label` varchar 枚举 `BUG`/`FEATURE`/`CONSULT_COMPLAINT`/`REFUND`/`CONTENT`/`ACCOUNT`/`PRAISE`/`OTHER` CHECK；唯一 `(ticket_id,label)`)；`ticket_internal_notes`(`id`/`ticket_id` FK/`admin_id`/`note` text/`created_at`——**用户不可见**)（**L0** 迁移文件 / **L1** validate 一致）
   **And** Flyway **V70**（当前文件 max V69）——移动靶：merge 时按当时全局 max 顺延（决策 E2）；VARCHAR≥16 无 CHAR(1) 坑；smallint 长度非 1 列注意（**L0**）

2. **后端：实体 + 仓储（L0/L1）**
   **Given** 表
   **When** 建实体
   **Then** `FeedbackTicket`（工厂 `create(userId, subject, body, contactType, contactValue, needContact, relatedOrderId)` 建 OPEN + token）+ `TicketAttachment`/`TicketLabel`/`TicketInternalNote`（工厂 `of(...)`）；`FeedbackTicketRepository`（`findByTicketToken`、`findByUserIdOrderByCreatedAtDesc`）+ 子表 repos（`findByTicketId...`）；命名映射 snake↔camel、时间戳 UTC、枚举落库 UPPER_SNAKE varchar（**L0** 编译 / **L1** 持久化）

3. **后端：建工单原语（用户建，≤5 图 OSS 预签名）（L1）**
   **Given** 用户投诉 + 已用既有预签名上传拿到 ≤5 个 `object_key`
   **When** `SupportTicketService.createTicket(userId, subject, body, contactType, contactValue, needContact, relatedOrderToken?, labels, attachmentObjectKeys)`
   **Then** 原子建 `feedback_tickets`(OPEN) + 逐 `object_key` 建 `ticket_attachments`（**校验 ≤5，超限拒绝**）+ 逐 label 建 `ticket_labels`（去重）；`relatedOrderToken` 有则解析为 `related_order_id`（校验属本人订单，不符则忽略/拒绝——见 Dev Notes）；`contact_type` 非法 422（**L1**）
   **And** 附件仅存 `object_key`（**不新建上传端点**，复用 `shared/media` 预签名）；`ticket_token` 不可枚举；**绝不记录 PII/联系方式到日志**（`contact_value` 脱敏，护栏）（**L1**）

4. **后端：内部备注用户不可见 + 用户视图 DTO（L1）**
   **Given** `ticket_internal_notes`（admin 加）
   **When** 用户查看工单
   **Then** 用户视图 DTO（`SupportTicketView`）**只含**用户可见字段（主题/正文/联系方式/标签/附件/status/csat 相关）、**绝不含** `ticket_internal_notes`/`handled_by` 等内部字段；加内部备注原语 `addInternalNote(ticketId, adminId, note)`（admin 用，L1 断言用户视图不含之）（**L1**）
   **And** 用户只能查自己的工单（owner 校验，非本人 404 防枚举）（**L1**）

5. **后端：契约 + 静态验收（L0/L1）**
   **Given** 全部实现
   **When** 验证
   **Then** `mvn -B test` 绿：L0（迁移文件建表、DTO 契约、实体工厂、≤5 附件校验单测）+ L1（建工单落库[主表+附件+标签]、内部备注不入用户视图、owner 404、contact_type 非法 422、`consult_orders` 无污染）；schema `validate` 一致（`ddl-auto=validate`）（**L0/L1**）
   **And** 无前端（用户端投诉 UI 在 4-2）；本 story 纯后端建模 + 建工单原语（**L1**）

## Dev Notes

> 纯后端建模，**照 `moderation.ContentReport` 同构范式**（用户建、admin 处理、status、`handled_by`）。附件复用 `shared/media` 预签名（只存 objectKey）；token 用 `CardTokenGenerator`。CSAT/`cs_rating` 只**预留列**（4-7 填），避免二次迁移。**无既有工单表 → 全新建**（非 ALTER）。

### 承接现状（已核实，照此复用/勿改）

- **无既有客服工单表**：V1.0「帮助反馈」= 表单/WhatsApp 跳转（PRD line 588），无落库。故 `feedback_tickets` **新建**（架构 §3.6「ALTER 既有 或 新」在本仓落为新建）。**勿找不存在的旧表去 ALTER。**
- **同构范式 `moderation.ContentReport`（FR-51 举报，照结构）**：`moderation/domain/ContentReport`（`@Entity @Table` + `create(...)` 工厂 + `status`(ReportStatus PENDING) + `handled_by`/`handled_at` + `@PrePersist/@PreUpdate`）；`moderation/{repository,service,web}` 分层；user 建 + admin 处理。**本 story 的 `support` 模块照此分层**（domain/dto/repository/service/web）。
- **OSS 预签名上传（`shared/media`，复用只存 key）**：`PresignedUploadService`/`MediaController`(`POST /media/upload-url`)/`UploadUrlRequest`/`UploadUrlResponse`/`AliyunOssClient`/`SignedUrlService`（读签名）。**前端已用此上传拿 objectKey**（照 content/consult 图片流）；本 story 建工单端点**接收 objectKey 列表存 `ticket_attachments`**，**不新建上传端点**。附件读展示（签名 URL）在 admin/用户查看时用 `SignedUrlService`（4-2/4-4）。
- **token**：`profile/service/CardTokenGenerator.generate()` → `ticket_token`（不可枚举，勿用自增 id 外露）。
- **relatedOrder**：`consult/repository/ConsultOrderRepository.findByOrderToken(token)`（用户投诉关联订单——传 orderToken 解析成 `related_order_id`，校验属本人；退款工单 4-3 用 `related_order_id`）。
- **通知类型不在本 story**：`TICKET_RESOLVED`/`CSAT_SURVEY`/`REFUND_REJECTED`/`IDENTITY_REQUIRE_MODIFY` 扩枚举 = `extend_notification_types_v11`（4-4）。本 story **不碰 `NotificationType`**。

### 关键设计决策（dev 照此）

- **D-1 模块归属 = 新 `com.tailtopia.support`**（feature 模块，非埋 admin 下）：工单**用户建**（`support/web` 用户端点）+ **admin 处理**（4-4/4-7 在 `admin/support` 或复用 support service，照 moderation：domain 在 `moderation`、admin 控制器在 `admin/moderation`）。本 story 只建 `support` 模块 + 用户建单/查单；admin 处理 UI 后续。
- **D-2 CSAT/cs_rating 预留不填**：`csat_score`/`csat_comment`/`csat_deadline`/`cs_rating` 建列即 null，**本 story 不写不发**（4-7 结案后发 CSAT 问卷填 `csat_*`；`cs_rating` AB-5G 后续版本，预留避免二次迁移）。`handled_by`/`resolved_at` 同——admin 处理（4-4/4-7）才填。
- **D-3 内部备注用户不可见（契约红线）**：`ticket_internal_notes` **绝不进用户视图 DTO**（`SupportTicketView` 无此字段，L1 断言）。admin 视图（4-4）才含。这是 AB-5 的隐私契约。
- **D-4 附件 ≤5 校验 + 只存 key**：`attachmentObjectKeys.size() > 5` → 422；只存 `object_key`（不存签名 URL，签名 URL 会过期，展示时 `SignedUrlService` 现签）。**日志绝不记 `contact_value`/PII**（脱敏，护栏）。
- **D-5 status 极简**：`OPEN`（建）→ `IN_PROGRESS`（admin 接手 4-4）→ `RESOLVED`（客服解决 4-4）→ `CLOSED`（7 天自动 或 CSAT 后，4-7）。本 story 只建 OPEN + 枚举定义；流转在 4-4/4-7。
- **Flyway V70**：当前文件 max V69（3-8 占 V69）。移动靶顺延（决策 E2）。smallint/bool 列非长度=1 varchar，无 CHAR(1) 坑；`csat_comment` VARCHAR(100)。
