# Story 2.2: AI 结果分字段下发与锁定态

Status: review

> V1.1 **Epic 2 第二个** story，承接 2-1（免费额度模型，已 review）。**纯后端**：给既有 `triage_tasks` 加解锁态两列 + 把分诊结果**按「安全免费部分 / 详建锁定部分」分字段下发**。**安全攸关**：红色评级详建**永不锁**（即使额度耗尽），护栏在**响应层单点**，只升不降不可绕过（FR-43C）。
> 源：`epics-v1.1.md` Story 2.2（FR-43A/43C、C-7）· 架构 `architecture-v1.1-delta.md` L105（ALTER 加 `unlock_source`/`unlock_channel`、分字段下发、红色永不锁）+ L271（红色态零变现强化：付费墙永不拦红，分字段下发，安全规则层只升不降）· `CROSS-STORY-DECISIONS.md` 继承护栏。

> **范围边界**：加两列 + 实体字段 + `markDone` 时初始化 `unlock_source=LOCKED` + 结果 DTO 分字段（锁定 advice/medicationRef）+ 红色永不锁响应规则 + 解锁态查询字段 + 幂等不可覆盖的 `unlock()` 实体方法（供 2-3 调）。**不做**：`POST /triage/{token}/unlock` 解锁事务与额度扣减/支付（2-3）、前端 paywall UI（2-4）、triage 生成/安全层逻辑改动（4.1/4.2 不动）。

## Story

As a 用户,
I want 始终免费看到等级图标/颜色/黄色时效/红色强提醒，详细建议按解锁下发,
so that 安全信息永不被付费墙挡住（FR-43A/43C，C-7）。

   - `unlock_channel VARCHAR(16)`（可空；`CHECK(unlock_channel IS NULL OR unlock_channel IN ('QRIS','PAWCOIN'))`）
   **And** **对 `triage_tasks`（非 `triage_records`）**；两列 `VARCHAR(16)` 非长度=1（无 CHAR(1) 坑）；不加 NOT NULL 默认（历史行 + PENDING/FAILED 行 `unlock_source` 保持 NULL = 无解锁语义）（**L0/L1**）

2. **后端：实体解锁态字段 + 枚举（L0）**
   **Given** `TriageTask` 实体
   **When** 映射两列
   **Then** 新增枚举 `com.tailtopia.triage.domain.UnlockSource {LOCKED, FREE_QUOTA, PAID}` + `UnlockChannel {QRIS, PAWCOIN}`（落库 varchar UPPER_SNAKE，照 `DangerLevel`/`TriageStatus` 范式）；`TriageTask` 加 `@Enumerated(STRING)` 字段 `unlockSource`/`unlockChannel` + getter（**L0**）

3. **后端：生成成功即置 LOCKED，失败不置（「生成失败不建记录」）（L0/L1）**
   **Given** `TriageProcessor` 状态机：`markDone(level, raw, parsed)` 仅在**成功生成 + 后置安全裁决后**调用；失败走 `markFailed`（不经 markDone）
   **When** 分诊完成
   **Then** `TriageTask.markDone(...)` 内**初始化 `unlockSource=LOCKED`**（详建默认锁定，待 2-3 解锁）；`unlockChannel` 保持 NULL；**FAILED/PENDING 任务 `unlockSource` 恒 NULL**——天然满足架构「生成失败不建记录」（无 parsed_result 无可解锁详建）（**L0** 实体单测 / **L1** 走处理器落库）

4. **后端：结果分字段下发（安全免费 vs 详建锁定）（L0/L1）**
   **Given** `TriageResultResponse.from(task)` 现返回 DONE 全字段
   **When** 返回分诊结果
   **Then** **始终免费下发**（安全信息，永不受锁影响）：`status`/`dangerLevel`（图标/颜色）/`disclaimer`（免责）/`observation`（黄色三要素含 `timeWindow` 时效）/`emergencySteps`/`emergencyAvoid`（红色强提醒）
   **And** **锁定详建**（SARAN PERAWATAN）：`advice`/`medicationRef` **仅在已解锁时下发**，锁定时置 null（Jackson NON_NULL 省略）

5. **后端【安全攸关】：红色评级详建永不锁（响应层单点，只升不降不可绕过）（L0/L1）**
   **Given** FR-43C「红色态零变现，即使月度免费额度耗尽也永不加付费墙」（安全规则层只升不降）

   **Then** 响应层**无条件下发** `advice`/`medicationRef`、`locked=false`，**与 `unlock_source` 取值无关**（即使 `unlock_source=LOCKED`、即使额度耗尽）——这是红色永不锁的**唯一判定点**，写在响应组装处（`TriageResultResponse.from`）；**不在别处埋绕过/降级分支**（安全单点、不可旁路，比照 4.2 `SafetyRuleLayer` 只升不降精神）（**L0/L1**，红色锁态用例是本 story 头等断言）

6. **后端：解锁不可覆盖的实体方法（供 2-3，幂等/immutable）（L0）**

   **Then** `TriageTask.unlock(UnlockSource source, UnlockChannel channel)`：仅允许 `LOCKED/NULL → FREE_QUOTA/PAID` 一次性跃迁；已是 `FREE_QUOTA/PAID` 时**幂等拒绝**（抛 `IllegalStateException` 或返回 false，dev 择一并测）；`PAID` 必带 `channel`、`FREE_QUOTA` 的 `channel` 恒 NULL（校验）；**本 story 不接调用点**（2-3 在其解锁事务内调），仅建方法 + 单测覆盖不可覆盖性（**L0**）

## Dev Notes

> 安全攸关 story。**唯一红线**：红色详建永不锁，判定必须**单点**在响应层（`TriageResultResponse.from`），不得散落。其余是标准 ALTER + 实体扩字段 + DTO 分字段。**生成/安全层（4.1/4.2）逻辑零改动**，只在 `markDone` 尾部加一行置 LOCKED。

### 承接现状（已读，dev 照此，勿臆测）
- **表 `triage_tasks`（V12，Story 4.1）**：`id/user_id/pet_id/status(varchar16 CHECK PENDING|PROCESSING|DONE|FAILED)/danger_level(varchar8 CHECK GREEN|YELLOW|RED,可空)/symptom_text/image_object_keys(jsonb)/gemini_raw(jsonb)/parsed_result(jsonb)/retry_count/idempotency_key(uq)/response_locale/created_at/updated_at`。**没有 triage_records 表**——架构文档笔误，一律用 `triage_tasks`。
- **`TriageTask` 实体**（`@Table("triage_tasks")`）：`status`/`dangerLevel` `@Enumerated(STRING)`；`parsedResult` `@JdbcTypeCode(JSON) Map<String,Object>`。状态机方法 `submit`（静态工厂建 PENDING）/`markProcessing`/`markDone(level,raw,parsed)`/`markRetry`/`markFailed`。**本 story 在 `markDone` 尾加 `unlockSource=LOCKED`，新增 `unlock(...)` 方法，加两字段 getter。**
- **`TriageProcessor.process`**：领任务→PROCESSING→签图 URL→Gemini→`SafetyRuleLayer.enforce`（4.2 只升不降升红唯一挂载点）→`markDone(finalLevel, raw, toParsed)`→DONE；失败 `markRetry`（≤3）超限 `markFailed`。**本 story 零改此文件逻辑**（`markDone` 内部置 LOCKED 即自动生效）。
- **`TriageResultResponse.from(task)`**（record）：现字段 `status/dangerLevel/advice/medicationRef/disclaimer/observation(TriageObservation)/emergencySteps/emergencyAvoid`；非 DONE 仅回 status（其余 null，Jackson NON_NULL 省略）；DONE 从 `parsedResult` map 取 `advice/medicationRef/disclaimer/observation/emergencySteps/emergencyAvoid`。**本 story 在此加分字段逻辑 + 两个新字段 `unlockSource`/`locked`。**
- **`DangerLevel {GREEN,YELLOW,RED}`**（`atLeast` 只升不降）；`TriageStatus`。枚举落库 varchar UPPER_SNAKE + `@Enumerated(STRING)`——`UnlockSource`/`UnlockChannel` 照此。

### 分字段下发模型（AC4/AC5 核心）
- **安全免费部分（永不锁，与 unlock_source 无关）**：`dangerLevel`（等级图标/颜色）、`disclaimer`（免责前置 NFR-9）、`observation`（黄色三要素：`indicators`/`timeWindow` 时效/`escalationTriggers`）、`emergencySteps`/`emergencyAvoid`（红色对症院前应急强提醒）。这些是「安全信息永不被付费墙挡住」（C-7）。
- **锁定详建部分（SARAN PERAWATAN）**：`advice`（详细养护建议）、`medicationRef`（用药参考）。锁定时置 null。
- **解锁判定 `unlocked(task)`**（**单点**）：`dangerLevel==RED || unlockSource∈{FREE_QUOTA,PAID}`。红色永不锁就靠 `dangerLevel==RED` 这一支——**即使 unlock_source=LOCKED、即使额度耗尽，红色 advice 照发**。这一判定**只此一处**，是 FR-43C 安全单点。
- **响应新增**：`unlockSource`（前端知锁态来源；null 按 LOCKED 语义）、`locked`（=`!unlocked(task)`，前端渲染 paywall 直接依据）。

### 「生成失败不建记录」的落地（AC3）
- 架构语「生成失败不建记录」在真实 `triage_tasks`（submit 即建 PENDING 行、失败转 FAILED）语境下，落为：**`unlock_source` 只在 `markDone`（成功生成）时置 LOCKED**；PENDING/FAILED 行 `unlock_source` 恒 NULL = **无可解锁详建**（FAILED 无 parsed_result）。故「失败不建记录」= 失败不产生「可解锁的详建记录态」，天然成立，无需额外拦截。

### 红色永不锁——安全实现纪律（AC5，勿埋违反点）
- **只在响应层判 `dangerLevel==RED` 放行详建**，不在 service/controller/2-3 解锁流里为红色写特殊分支。原因同 4.2「安全规则层只升不降、单点、不可旁路」——散落判定会在某处被漏改而破防。
- 红色 `advice` 可能为空（模型 RED 时未产出详建 → `parsed_result` 无 advice）：此时 `advice=null` 是「无内容」非「锁定」，`locked` 仍为 false（前端红色态回退通用步骤，见 4.4）。区分「锁定 null」与「无内容 null」靠 `locked` 布尔。
- **2-3 依赖本 story 契约**：解锁流对红色**不应调 tryConsume 扣额度**（红色本就免费），或调了也不影响展示（响应层已放行）——本 story 只保证展示侧红色永不锁；扣费策略在 2-3。

### 回归风险（AC7）
- **行为变更**：本 story 后，GREEN/YELLOW 的 DONE 结果 `advice`/`medicationRef` **默认锁定不下发**（需 2-3 解锁）。这会让「submit→result 看到 advice」的既有 triage L1/L0 测试失败——**须按新语义更新断言**（GREEN/YELLOW 锁定态 advice=null+locked=true；新增 RED 仍有 advice）。这是有意的付费墙行为，非 bug。
- **版本级依赖**：2-2（后端锁）必须与 2-3（解锁流）+2-4（paywall UI）**同版本发布**，否则用户 GREEN/YELLOW 详建无从解锁。story 序内按 2-2→2-3→2-4 推进，发布时整体上线。
- **端点签名不变**：`GET /triage/{id}` 仅响应体扩 `unlockSource`/`locked` 字段（加字段兼容），不改路径/方法/鉴权。
