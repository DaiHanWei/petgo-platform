# Story 8.1: 里程碑数据基建与清单分配

Status: review

> 🆕 **决策 F16（2026-06-09 产品拍板，反转 F2②）：** FR-42 里程碑本体**全量纳入 V1.0.0**，Epic 8 转入实现。本 Story 是 Epic 8 的**数据地基（纯后端）**：建 `pet_milestones`（roster）/ `milestone_completions`（完成项）两表（归 **profile 域**）、后端固定清单常量 `MilestoneCatalog`（猫30/狗30/通用15 + C-Lx 组合依赖规则）、建档按 `pet_type` 自动分配 roster、`GET /api/v1/pet-profiles/me/milestones` 列表/进度端点（L/M/S 分区 + 完成状态）。
>
> **护栏**：清单为后端固定常量，V1 不做运营可编辑清单；`milestone_completions` 唯一约束保证自动完成幂等、不可撤销；对外标识用里程碑 **code**（如 `C-S1`，稳定非顺序）不外露自增 id；新表归 profile 域、随档案级联删除（D1/D2 家族）。**禁 MQ/Quartz/缓存/新中间件**。`pet_type` 创建后不可改（F6）。

## Story

As a **有宠物档案的用户**,
I want **系统按我的宠物类型自动分配一套里程碑清单**,
so that **我能看到为我的猫 / 狗 / 其他宠物量身定制的成长目标与进度**。

## Acceptance Criteria

> **验证层标注**：**L0 静态**（编译/纯单测/契约金标，无需 DB） · **L1 集成**（需 Docker pg+redis：真分配 roster + 进度查询落库） · **L2 端到端**（无，本 Story 纯数据/接口）。

### AC1 — 按 pet_type 分配清单并返回 L/M/S 分区进度（FR-42）
**Given** 宠物档案已创建（`pet_type` = CAT/DOG/OTHER）
**When** 请求 `GET /api/v1/pet-profiles/me/milestones`
**Then** 返回对应固定清单（猫 30 / 狗 30 / 其他 15），每项含 `code`、级别（S/M/L）、标题、触发方式（SYSTEM_AUTO/USER_CHECKIN/PUSH_PUBLISH）、完成状态、完成日期；按 **L/M/S 分区**，每级给出已完成/总数进度与顶部总进度。
> 验证层：**L0**（`MilestoneCatalog` 计数金标：CAT=30/DOG=30/OTHER=15、每套 S/M/L 数量正确；`MilestoneListResponse` 契约金标字段集）+ **L1**（真建档→roster 落库→查询返回分区进度）。

### AC2 — 不预插完成行；进度由「清单总数 vs 完成数」计算
**Given** 清单为后端固定常量 `MilestoneCatalog`（含三套清单与 C-Lx 组合依赖规则）
**When** 建档完成
**Then** 物化 `pet_milestones` roster（每只宠物一份清单），**不预插 `milestone_completions` 完成行**；完成态由「该 milestone 是否存在 completion 行」决定；进度 = roster 总数 vs completion 数。
> 验证层：**L0**（service 单测：分配 roster 后 completion 数=0、进度全 0）+ **L1**（落库验证）。

### AC3 — 注销级联 + 不可枚举对外标识
**Given** 数据生命周期
**When** 注销账号
**Then** 里程碑数据随宠物级联删除（归 profile 域，`ProfileDeletionService` 先删 completions→roster→档案）；对外标识用 milestone `code`（非顺序 id），响应不含任何自增 DB id。
> 验证层：**L0**（契约金标断言响应无 `id` 字段、`code` 在；deletion service 单测顺序）+ **L1**（注销真删）。

## Dev Agent Record

### 设计决策 / 偏差说明
- **两表 vs 单表**：依权威台账 CROSS-STORY-DECISIONS（F16 行 + 表归属总表两处明列 `pet_milestones`/`milestone_completions`）建**两表**。`pet_milestones` = 建档时按 `MilestoneCatalog` 物化的 per-pet roster（含 code/level/trigger/sort_order，自描述 + 唯一约束兜底幂等）；`milestone_completions` = 仅记完成项（FK→roster，`pet_milestone_id` 唯一保自动完成幂等不可撤销，`linked_content_id` partial-unique 保「一内容一里程碑」）。AC「不预插完成行」= 不预插 completion 行（roster 物化不算完成行），满足。`MilestoneCatalog` 仍是清单**唯一事实源**（标题/级别/触发/组合依赖），roster 仅承载结构位供查询自包含。
- **对外标识**：用 catalog `code`（C-S1 等，稳定非顺序），响应不含任何自增 DB id（契约金标守护）。
- **幂等物化**：`assignRoster` 先查 roster 是否已存在再插，且唯一约束 (pet_profile_id, code) 兜底并发/重复建档。存量档案（本功能上线前已建）经首次读端点 lazy ensure 兜底。
