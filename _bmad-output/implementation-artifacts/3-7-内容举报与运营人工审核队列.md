# Story 3.7: 内容举报与运营人工审核队列

Status: ready-for-dev

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

## Story

As a **用户与运营**,
I want **举报不当内容并由运营人工处理**,
so that **平台内容保持健康可信**。

> 本 Story 实现内容举报（FR-25，**含 Admin 审核队列**）——用户侧「···」举报单选类型提交，运营侧 Admin 后台审核队列人工处理（**无自动下架**）。**后端中（含 Admin）、前端轻**：后端建举报工单表 + 举报提交端点（moderation 模块）+ Admin 队列页（复用 Story 3.1 的 Admin shell）+ 人工下架（复用 Story 3.6 的 `softDelete`）；前端「···」举报菜单 + 类型单选 + 提交反馈。
>
> **做**：① 用户侧「···」菜单举报（类型单选：违法违规/虚假信息/不当内容/骚扰/其他）；② 提交反馈「已收到你的举报…」+ 写举报工单表；③ 进运营审核队列**无自动下架**；④ Admin 后台举报队列页（复用 3.1 shell 预留入口位）查看详情 + 人工下架（复用 content `softDelete`，所有决定人工做出）。
> **不做**：评论举报 / 用户举报（V1 明确不做，仅内容举报）；自动下架/机器审核（V1 人工）；Admin shell 框架（Story 3.1 已建，本 Story 填充举报队列入口）。
> **依赖前序**：Story 3.1（Admin slice shell + ADMIN 门控 + 举报队列入口位）、Story 3.6（`ContentService.softDelete` 复用做人工下架）、Story 3.3（详情页「···」菜单容器 + 举报入口位）、Story 3.2（Feed 卡片长按 context menu——举报/分享）、Story 1.4/1.5（FR-0C，举报需登录）。

## Acceptance Criteria

> **验证层标注**：**L0**（编译/lint/widget test/Service 单元 + MockMvc） · **L1**（需 postgres+redis，验举报写入 + 队列读取 + 人工下架 + ADMIN 门控） · **L2**（真机，举报菜单交互）。
> 举报提交/队列/下架/门控为 **L1**（DB + Admin slice 是核心）；「···」举报菜单/类型单选为前端 **L0 + L2**。无外部第三方依赖。

### AC1 — 用户举报单选类型 + 反馈 + 进队列无自动下架

**Given** 用户在内容卡片/详情页「···」菜单
**When** 发起举报并选择类型（违法违规/虚假信息/不当内容/骚扰/其他，单选）
**Then** 提交后显示「已收到你的举报，我们会尽快处理」，举报进入运营审核队列（写举报工单表，本故事创建该表）（FR-25）
**And** V1 不做自动下架、不做评论/用户举报
> 验证层：**L1**（举报工单表创建 + 提交写入含 type/reporterId/postId/status=PENDING；**提交后内容仍正常展示——无自动下架**；防重复举报策略：同用户对同内容重复举报幂等/合并）。「···」举报菜单 + 5 类型单选 + 提交反馈文案为前端 **L0 widget test** + **L2**；举报需登录→未登录触发 FR-0C。

### AC2 — Admin 审核队列人工下架

**Given** 运营在 Admin 后台举报队列
**When** 处理举报工单
**Then** 可查看举报详情并人工下架内容（复用 content service 软删），所有处理决定由人工做出（G-1 落点）
> 验证层：**L1**（ADMIN 门控：仅 ADMIN 可访问举报队列；队列按时间/状态列举工单；点工单看详情含被举报内容 + 举报类型/数；「下架」调 `ContentService.softDelete(postId, reason=admin_takedown)` → 内容走 404（Story 3.3）+ 从 Feed/时间线移除（Story 3.6 一致）；工单状态流转 PENDING→RESOLVED/DISMISSED）。**所有决定人工**——无规则自动下架。

## Tasks / Subtasks

> **三段组织**，后端中（含 Admin）前端轻。建议执行顺序：后端举报表+提交端点 → Admin 队列页 → 前端举报菜单 → 联调。

### 🟦 后端子任务（petgo-backend / Spring Boot — moderation 模块 + admin slice）

- [ ] **B1. 举报工单表** (AC: 1)
  - [ ] Flyway 建 `content_reports`（moderation 模块）：`id`(bigint)、`post_id`(fk content_posts)、`reporter_id`(fk users)、`reason_type`(varchar UPPER_SNAKE ∈ {ILLEGAL, MISINFO, INAPPROPRIATE, HARASSMENT, OTHER})、`status`(varchar UPPER_SNAKE ∈ {PENDING, RESOLVED, DISMISSED})、`created_at`/`updated_at`(timestamptz UTC)、`handled_by`(可空, ADMIN id)、`handled_at`(可空)；索引 `idx_content_reports_status`、`idx_content_reports_post`。
  - [ ] 防重复：同 reporter 对同 post 重复举报→幂等（唯一约束或合并；在 Dev Notes 定）。
- [ ] **B2. 举报提交端点** (AC: 1)
  - [ ] `POST /api/v1/content-posts/{postId}/reports`：body 含 `reasonType`（单选，必填，枚举校验）；**需 JWT**（未登录→401→前端 FR-0C）；带 `Idempotency-Key`。
  - [ ] 写工单 status=PENDING；**不触发任何自动下架**（AC1）；返回 200 + 反馈语义（前端显「已收到…」）。
  - [ ] **V1 仅内容举报**——无评论/用户举报端点。
- [ ] **B3. Admin 举报队列（复用 Story 3.1 shell）** (AC: 2)
  - [ ] `admin` slice 填充举报队列：`AdminWebController` 增 `/admin/reports`（Thymeleaf 页，复用 3.1 的 layout shell 与预留入口位）；**ADMIN 门控**（3.1 已建 `/admin/**` 规则，本 Story 复用）。
  - [ ] 队列列表：按 status=PENDING 倒序 + 简单分页；点工单看详情（被举报内容快照/链接 + reason_type + 举报次数）。
  - [ ] `AdminModerationService` 复用 moderation repository 读工单 + 调 `ContentService.softDelete(postId, "admin_takedown")` 下架（**经 content service，禁直接改 content 表**）；工单状态置 RESOLVED（下架）/ DISMISSED（驳回），记 `handled_by`/`handled_at`。
  - [ ] **所有决定人工**——无任何自动下架规则；下架后内容走 Story 3.6 软删的全套同步（Feed/时间线移除、详情 404）。

### 🟩 前端子任务（petgo_app / Flutter — features/content；Admin 为 Thymeleaf 无 Flutter）

- [ ] **F1. 用户侧「···」举报菜单** (AC: 1)
  - [ ] 详情页「···」菜单（Story 3.3 容器）+ Feed 卡片长按 context menu（Story 3.2/UX-DR12）增「举报」项。
  - [ ] 举报 sheet：5 类型**单选**（违法违规/虚假信息/不当内容/骚扰/其他，双语 .arb）→ 提交调 B2；未登录触发 FR-0C（复用 1.4）。
  - [ ] 提交成功 toast「已收到你的举报，我们会尽快处理」（双语，每条 ≤1 emoji）。
- [ ] **F2. Admin 队列页（Thymeleaf）** (AC: 2)
  - [ ] `templates/admin/reports.html`（队列列表 + 工单详情 + 下架/驳回按钮）——服务端渲染，**不在 Flutter App 内**。

### 🟨 联调验收子任务（举报写入 + 队列 + 人工下架 + 门控）

- [ ] **J1. 举报提交无自动下架** (AC: 1 / **L1**)
  - [ ] 用户举报某帖（选类型）→ 工单 PENDING 落库；**该帖仍正常在 Feed/详情展示**（无自动下架）；同用户重复举报→幂等；未登录举报→401→FR-0C。
- [ ] **J2. Admin 队列 + 人工下架** (AC: 2 / **L1**)
  - [ ] ADMIN 登录后台 `/admin/reports` 见 PENDING 工单；非 ADMIN→403（复用 3.1 门控）；点工单看详情；「下架」→内容软删→Feed/时间线移除、详情 404；工单→RESOLVED 记 handled_by；「驳回」→DISMISSED。
- [ ] **J3. 前端举报菜单** (AC: 1 / L0+**L2**)
  - [ ] widget test：5 类型单选、提交反馈文案。真机：详情/Feed 长按举报、未登录 FR-0C。

## Dev Notes

### 关键架构约定（本 Story 必须落实）

**举报模型**（FR-25）：卡片「···」菜单举报**单选类型**；进运营人工审核队列**无自动下架**；**V1 不做评论/用户举报**。这是「不可协商地基」之一（架构 §Non-Negotiables 第 5 条：内容举报入口 + 人工下架能力）。

**Admin slice 复用**（架构 §Gap G-1 / Story 3.1）：举报队列**挂 Story 3.1 首建的 `com.petgo.admin/` shell**，复用 `/admin/**` ADMIN 门控与 layout——**不另造后台**。本 Story 是 3.1 预留「举报队列」入口位的填充。下架经 `ContentService.softDelete`（Story 3.6 封装），所有决定人工。

**模块归属**：举报工单表/提交端点在 `com.petgo.moderation/`；Admin 队列页在 `com.petgo.admin/`（经 moderation service 读工单 + content service 下架）。跨模块只经 service，**禁 admin/moderation 直接访问 content repository**。

**命名映射链**：`content_reports`(post_id/reporter_id/reason_type/status/handled_by/handled_at/created_at)；枚举 UPPER_SNAKE（reason_type {ILLEGAL,MISINFO,INAPPROPRIATE,HARASSMENT,OTHER}、status {PENDING,RESOLVED,DISMISSED}）；索引 `idx_content_reports_status`；DB snake_case ↔ JSON camelCase。

**人工下架联动**：下架 = `softDelete(postId, "admin_takedown")` → 走 Story 3.6 全套软删同步（Feed 不返回、时间线移除、详情 404 统一防枚举文案 UX-DR18 ④）。

### 强制护栏（架构 §Enforcement —— 违反即返工）

- **禁止擅自引入 MQ / 通用缓存层 / 新中间件**；举报同步写库。
- `ddl-auto=validate`；`content_reports` 表/索引走 Flyway。
- **无自动下架**——无任何规则自动改内容状态，全部人工经 Admin。
- 模块边界：moderation 提交 + admin 处理，下架经 content service `softDelete`，**禁跨 repository**。
- ADMIN 端点复用 3.1 `/admin/**` 门控，绝不混入公开 `/api/v1`；非 ADMIN→403。
- 举报需登录（FR-0C）；写端点 Idempotency-Key + 防重复举报。
- V1 仅内容举报——**不加评论/用户举报端点**。

### 范围边界（防 scope creep —— 本 Story 明确不做）

- ❌ 评论举报 / 用户举报（V1 明确不做）。
- ❌ 自动下架 / 机器审核（V1 人工）。
- ❌ Admin shell 框架/登录/门控（Story 3.1 已建，本 Story 复用填充举报队列）。
- ❌ 内容软删机制本身（Story 3.6 已封装 `softDelete`，本 Story 调用）。
- ✅ 只做：举报工单表 + 用户举报提交（单选类型，无自动下架）+ Admin 举报队列页（复用 shell）+ 人工下架（复用 softDelete）。

### Project Structure Notes

- 后端：`com.petgo.moderation/{web,service,domain,repository,dto}`（举报提交 + 工单）；`com.petgo.admin/web`（`/admin/reports` 队列页，复用 3.1 shell）；下架经 `com.petgo.content.ContentService.softDelete`（Story 3.6）。
- 前端：`lib/features/content/{data,domain,presentation}`——「···」举报菜单 + 类型单选 sheet（详情 + Feed 长按）；Admin 队列为 `templates/admin/reports.html`（Thymeleaf，无 Flutter）。
