# Story 3.3: 删除用户账号（D1 注销 / D2 违规）

Status: ready-for-dev

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

> **范围域**：管理后台 V1.0.0 · Epic 3（用户账号治理）· Story 3.3（AB-UA-03）。不可逆删除：两类型（用户申请注销 D1 / 违规处置 D2），二次确认 + 必填备注 + 永久审计记录。**安全攸关节点（对齐 App 注销决策 D1/D2，PDP 合规）。**
> **产物归属**：仅后端 `petgo-backend`（**无 Flutter 侧**）。brownfield 扩 `com.tailtopia.admin.usermgmt`，**复用既有 Story 7.3 `AccountDeletionService`**。
> **前置**：Epic 1（admin_accounts + 会话 + 门控 + 审计 1.3）+ 既有 7.3 注销编排。**与 3.1/3.2 各自独立**。

## Story

As a **运营**,
I want **按「用户申请注销（D1）」或「违规处置（D2）」两种类型不可逆地删除一个用户账号，并匿名化处理其历史数据**,
so that **同时满足印尼 PDP 合规（匿名化而非硬删保留审计链）与平台内容处置诉求，且每次删除都有永久不可篡改的记录**。

## Acceptance Criteria

> 验证层：**L0** 编译/单测/MockMvc（mock service，无需 DB）· **L1** 集成（Docker postgres+redis，验级联删除/匿名化/下架/审计真跑）· **L2** 外部凭证（IM 媒体级联删除依赖腾讯 IM 凭证，留本地/真凭证手测）。

1. **AC1（不可逆 + 二次确认 + 类型 + 必填备注，L0/L1）**
   **Given** 删除操作
   **When** 执行前
   **Then** 二次确认弹层显示「此操作不可撤销」；**必须选删除类型**（`USER_REQUEST` 注销 / `VIOLATION` 违规）；**备注必填**（空则校验拒绝）。

2. **AC2（D1 用户申请注销，L1）**
   **Given** 类型 = 用户申请注销（D1）
   **When** 删除
   **Then** 账号移除不可登录；其 UGC **匿名化保留**（作者显示「已注销用户」+ 默认头像、不触发迷你预览卡 FR-26）；宠物档案 + H5 名片**立即删除失效**；历史问诊**匿名化保留**（剥 PII，保留 danger_level/vet_id/评分）。**= 复用既有 `AccountDeletionService` 现有级联（见 Dev Notes）。**

3. **AC3（D2 违规处置，L1）**
   **Given** 类型 = 违规处置（D2）
   **When** 删除
   **Then** 账号移除；其内容**同步下架**（Feed/我的发布/成长档案全移除，软删 `deleted_at` + `DeleteReason.ADMIN_TAKEDOWN`）；宠物档案 + H5 名片删除失效；历史问诊匿名化。**= 在 D1 级联之上，前置「下架该用户全部内容」一步（见 Dev Notes 对齐/差异）。**

4. **AC4（永久记录 = 审计，L1）**
   **Given** 删除完成
   **When** 记录
   **Then** 经 `AdminAuditService.record(...)` 写一条 append-only 审计（`action_type=USER_DELETED`、target=userId、summary 含**操作人 / 时间 / 删除类型 / 备注**）；永久保留不可删（A6）。**备注/类型落审计，不落 PII**。

5. **AC5（复用既有编排，不破坏 App 注销，L0/L1）**
   **Given** App 端注销（7.3）走同一 `AccountDeletionService`
   **When** 后台删除接入
   **Then** **不破坏 App 自助注销路径**；后台删除复用 `requestDeletion(userId)`（DB 状态机 PENDING→…→DONE + 失败重扫，禁 MQ）；幂等可重跑。

6. **AC6（权限门控，L0）**
   **Given** 逐端点授权（A5）
   **When** 无删除权限的后台账号请求删除端点
   **Then** 方法级 `@PreAuthorize` 拒绝 403；`SUPER_ADMIN` 隐式全权（权限码见 Dev Notes）。

7. **AC7（回归绿，L0/L1）**
   **Given** 接入既有删除编排
   **When** `mvn -B compile` + L0 + 本地 L1 全量回归
   **Then** 全绿；既有 7.3 注销与全部既有端点无回归。

## Dev Notes

### 架构约束（必须遵守）
- **删除走匿名化（PDP 合规），对齐 App 注销 D1/D2**；安全攸关节点（与 4.2/4.5/7.3 同级），写代码勿埋违反点。[Source: architecture.md#NFR4 / Cross-Cutting #6；CLAUDE.md 强制护栏]
- 异步只 `@Async` + DB 状态机（既有 7.3 即此范式），**禁引入 MQ/缓存/新中间件**。[Source: architecture.md#护栏；CLAUDE.md]
- 跨模块只经既有 service/事件，**禁 admin 直写 content/consult/profile/auth 表**。[Source: architecture.md#Architectural Boundaries]
- 写操作经 `AdminAuditService.record` 写审计（append-only 哈希链，A6），`action_type=USER_DELETED`；**审计 summary 落类型/备注/操作人，绝不落 PII**（备注由运营填写，约定不写入用户隐私）。[Source: architecture.md#Communication Patterns / Format Patterns 一次性敏感值]
- 授权方法级 `@PreAuthorize`（A5）；`SUPER_ADMIN` 隐式全权。[Source: architecture.md#Process Patterns 删除合规]

### 既有 7.3 注销编排（READ — 复用核心，现状 / 对齐 / 差异）
**现状（`account/service/AccountDeletionService`，Story 7.3，决策 D1/D2）：**
- `requestDeletion(userId)`：登记 `account_deletions` PENDING（幂等）+ 发 `AccountDeletionRequestedEvent`（AFTER_COMMIT）→ `@Async` 执行 `execute()`；失败置 FAILED，`ApplicationReadyEvent` 启动重扫续跑（不丢、不半残、可重跑）。
- `execute(deletionId)` 级联：`profileDeletion.deleteByUserId`（宠物档案 + 健康事件 + 里程碑 + **名片图/og 图**物理删）→ `triageDeletion` → `consultAnonymization.anonymizeByUserId`（会话/评分**保留行剥 PII**，收集 AI 私密图 key）→ `notificationDeletion` → `authDeletion.deleteByUserId`（**物理删 users 行 + refresh 句柄**，返回头像图）→ OSS 个人图删（私密 + 公开头像/名片）→ IM 会话媒体删。
- **关键事实**：`auth` 删用户行后，其 UGC（content_posts/comments）**经 `AccountQueryService` 解析为 `AuthorView.anonymized`（「已注销用户」+默认头像、不触发 FR-26）自动匿名化保留**——即**内容默认是「匿名化保留」，不下架**。
- `account/domain/AccountDeletion` 是 **PII-free 作业表**：仅 `userId/status/retryCount/requestedAt/completedAt`，**无 type/operator/note**。

**对齐 / 差异结论（D1 vs D2）：**
- **D1（用户申请注销）= 既有编排原样**：内容匿名化保留 + 档案/名片删 + 问诊匿名。后台 D1 **直接复用 `requestDeletion(userId)`，零改 7.3**。
- **D2（违规处置）= D1 + 前置「内容下架」一步**：既有编排**不含**内容下架分支（它只做匿名化保留）。故 D2 在调用 `requestDeletion` **之前**，先经 content service 把该用户全部内容软删下架（`DeleteReason.ADMIN_TAKEDOWN`）；之后 users 行删除时这些内容已软删、Feed 不可见，差异即「下架」而非「匿名保留」。**差异落在 admin 编排层，不改 7.3 的 `AccountDeletionService` 与 `account_deletions` schema。**
- **「删除类型」承载位置**：不在 `account_deletions` 加 `type` 列（保持 7.3 表冻结 + PII-free）。D1/D2 的分支逻辑在 `AdminUserService` 决策；类型 + 备注 + 操作人写入**审计日志**（AC4）作为永久记录。

### 既有内容下架基线（READ）
- `content/service/ContentService.softDelete(postId, DeleteReason)` + `content/domain/DeleteReason{AUTHOR_DELETE, ADMIN_TAKEDOWN}`：现成软删 + 关联清理（likes 等）。D2 逐条调之（或在 content service 加「按作者批量下架」只读编排方法），**经 service**。
- `content/domain/PostStatus` 当前仅 `PUBLISHED`，下架用 `deleted_at` 软删（非状态枚举）。

### 审计依赖（READ）
- `AdminAuditService`（Story 1.3）：撰写时 audit 子域尚未在代码库落地。横切依赖：按 1.3 约定的 `record(actionType, targetType, targetId, summary)` 与写操作同事务接线；若 1.3 未就绪，接口占位 + log 占位并在 Completion Notes 标「审计落库待 1.3」（**不得**跳过审计设计点 — 此为 AC4 硬要求）。

### 关键边界 / 防坑
- **不可逆 + 二次确认 + 必填备注 + 必选类型**是合规硬门槛：四者缺一即拒绝；确认弹层必须明示「此操作不可撤销」。
- **D2 顺序**：先下架内容、后触发 `requestDeletion`。反过来（先删用户行）会让 UGC 先被匿名化保留，再去下架时作者已不存在、易漏；按「先下架后注销」最稳。
- **幂等**：`requestDeletion` 本身幂等（按 userId 复用 PENDING 行）；D2 内容下架也须幂等（已软删跳过）。
- **PII 纪律**：审计 summary 写「类型/备注/操作人/时间」；备注由运营填，约定**不抄入用户隐私**；日志只记代理 id + 进度（沿用 7.3：「日志只记代理 id+进度+计数，绝不落 PII」）。
- 仅作用于 `role=USER`；不得删后台账号（admin_accounts 只停用不删，1.5）或兽医账号。
- 删除 ≠ 停用：删除不可逆走级联匿名化；停用可逆（3.2）。两者入口分开，勿混。

### Flyway
- **本故事无新迁移**：D1/D2 分支在 admin 编排层 + 审计日志承载，`account_deletions`（7.3）与 `users` 表均不加列。若 dev 评估确需在 `account_deletions` 记类型（**不推荐**，破坏 7.3 表冻结与 PII-free），须先与架构对齐再从顺延序号加 ALTER（决策 E2）；默认**不动 Flyway**。
