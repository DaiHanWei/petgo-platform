# Story 2.9: 问诊请求未成功记录

Status: ready-for-dev

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

> **范围域**：管理后台 V1.0.0 · Epic 2 · Story 2.9（AB-2G）。**独立队列**——记录从未建立会话的问诊请求（区别于 Epic 5 已建立会话的异常工单）。
> **产物归属**：仅后端 `petgo-backend`（**无 Flutter 侧**）。SSR + HTMX，`/admin/failed-requests`，不返 JSON；建 `failed_consult_requests`（Flyway **V35**）。
> **brownfield 关键**：用户取消/超时/系统故障发生在既有 consult 排队流程（Story 5.3 `ConsultSessionService`/`ConsultQueueService`）。本故事**经领域事件**从 consult 流程喂入记录（禁跨 repo）。

## Story

As a **运营成员**,
I want **查看并跟进所有未成功建立会话的问诊请求（取消/超时/系统故障）**,
so that **监控平台接诊能力、排查系统异常、确保故障类被跟进**。

## Acceptance Criteria

> 验证层：**L0** MockMvc 切片 + 单测 · **L1** 集成（Docker postgres+redis，验事件落库/筛选/归档）。

1. **AC1（建表，L0 编译 + L1 迁移）**：新增 Flyway `V35__init_failed_consult_requests.sql` 建 `failed_consult_requests`（`id` / `request_token`(不可枚举 token) / `user_id` / `submitted_at` / `cancelled_at` / `cancel_reason`(varchar UPPER_SNAKE：`USER_CANCEL`|`TIMEOUT`|`SYSTEM_FAILURE`) / `online_vet_count`(int) / `followed_up`(bool) / `note` / `archived_at`(nullable) / `created_at`/`updated_at`）；JPA 实体精确匹配（`ddl-auto=validate` 绿）。
2. **AC2（事件落记录，L1）**：请求未成功时（用户取消 / 超时无人接单 / 系统故障）经 consult 模块发布 `ConsultRequestFailedEvent(reason, userId, sessionId/requestToken, submittedAt, onlineVetCount)`，admin 侧监听落 `failed_consult_requests`；`online_vet_count` 取失败时刻 `VetPresenceService.onlineVetIds().size()`。
3. **AC3（列表 + 系统故障视觉预警 + 强制跟进，L1）**：列表展示请求 token/用户/提交时间/取消时间/原因/在线兽医数/跟进状态；`SYSTEM_FAILURE` 类**自动视觉预警**且**须标记跟进**（未跟进不可归档）；其它类型跟进可选。
4. **AC4（备注 + 标记跟进 + 归档可搜，L1）**：运营可加备注、标记「已跟进」；已跟进可移入归档（`archived_at`）；归档记录仍可搜索；活动/归档分区展示。
5. **AC5（写审计 + 门控，L1）**：标记跟进/归档/加备注同事务调 `AdminAuditService.record(actorAccountId, "FAILED_REQUEST_FOLLOWED_UP"/"FAILED_REQUEST_ARCHIVED", "FAILED_CONSULT_REQUEST", id, summary)`；页面/写端点 `@PreAuthorize("hasAuthority('vet.view')")`（兽医接诊能力监控归兽医管理权限组）。
6. **AC6（回归绿，L0/L1）**：既有 consult 发起/取消/排队流程不回归；`mvn -B compile` + consult/admin 单测全绿；本地 L1 全量回归绿（迁移 validate）。

## Dev Notes

### 架构约束（必须遵守）
- 对外暴露用**不可枚举 token**（`request_token`），不外露自增 id。[Source: CLAUDE.md 护栏；architecture.md#Naming Patterns target_id 不可枚举]
- 跨模块只经领域事件/service，**禁跨 repo**：consult 发 `ConsultRequestFailedEvent`，admin 订阅落自有表（admin 拥有 `failed_consult_requests`）。[Source: architecture.md#Module 边界 / Communication Patterns]
- 写操作强制审计；枚举 UPPER_SNAKE；时间 UTC；表名 snake_case 复数（业务实体表不加 admin_ 前缀）。[Source: architecture.md#Naming/Communication Patterns]

### 既有代码基线（READ 过，brownfield）
- **现状**：`ConsultSessionService`（`createWaiting` 建 WAITING + 入队 + 发 `ConsultRequestQueuedEvent`；`cancel` WAITING→CANCELLED + 出队；`WAITING_TIMEOUT_SECONDS=60`，**超时不迁状态**，仅供前端弹「继续等待/转 AI」，注释明确「用户直接退出 App 则请求自动从队列删除（前端行为）」）；`ConsultQueueService`（Redis ZSET `consult:waiting`，enqueue/dequeue）；`VetPresenceService.onlineVetIds()`→在线兽医 id 列表（`.size()` 即在线数）。
- **本故事改什么**：新建 `failed_consult_requests` 表/实体/repo（V35）；新建 `ConsultRequestFailedEvent` + 在 consult 失败点发布；admin 监听落库 + 列表/跟进/归档页。
- **不可破坏**：① `ConsultSessionService.cancel`/`createWaiting` 既有逻辑保留，仅**追加发事件**（不改状态机）；② `ConsultRequestQueuedEvent`/`ConsultAcceptedEvent` 等既有事件不动；③ 「同时仅 1 个」「继续等待」语义不变；④ Redis 队列收窄用途（只存 sessionId）不变。

### 关键边界 / 防坑
- **TIMEOUT 捕获是 brownfield 难点**：现有超时**不产生服务端状态迁移**（前端轮询 + 用户退出时前端删队列）。服务端没有可靠的「超时无人接单」落点。最小可行方案（dev 二选一并记录）：
  1. **在取消时分类**：用户在超时弹层选「不再等待/退出」触发的 cancel 携带 `reason=TIMEOUT`，其余主动撤回为 `USER_CANCEL`（需 App 侧在 cancel 请求带 reason——跨产品，标注；后端先接受 reason 参数，App 后续配合）；
  2. **轻量 stale 扫描**：新增 `@Scheduled` 扫描超时仍 WAITING 且队列中已无对应活跃轮询的请求（判定较脆，V1 可不做）。
  推荐**方案 1**：后端 `cancel` 接受可选 `reason`，无则默认 `USER_CANCEL`；这样 USER_CANCEL/TIMEOUT 都从 cancel 入口分流，SYSTEM_FAILURE 从异常路径入。**本故事后端先就绪 reason 入参与事件，App 侧传 reason 标为跨产品后续**。
- **online_vet_count 取值时机**：必须在**失败发生时刻**取（cancel/失败处），不是落库时（落库可能稍晚，在线数会变）。事件内携带该值，监听器直接用。
- **SYSTEM_FAILURE 强制跟进**：归档端点校验 `if (reason==SYSTEM_FAILURE && !followedUp) reject`；视觉预警在列表 badge（红）。
- **request_token 生成**：用既有不可枚举 token 生成方式（参考项目内 token 工具/UUID + 编码）；若以 sessionId 关联，token 是对外展示值，sessionId 仅内部（可不存或存内部列）。
- **与 Epic 5 异常工单区分**：本表=**从未建立会话**的请求（WAITING 阶段失败）；Epic 5 工单=**已建立会话**中兽医被封禁的异常。两者数据源、表、语义都不同，勿混（PRD 附录 A 明示）。
