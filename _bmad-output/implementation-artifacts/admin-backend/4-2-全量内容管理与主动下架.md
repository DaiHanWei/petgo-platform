# Story 4.2: 全量内容管理与主动下架

Status: ready-for-dev

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

> **范围域**：管理后台 V1.0.0 · Epic 4（内容审核）· Story 4.2（AB-3B）。**brownfield**：复用既有 `ContentService.softDelete`，新增「恢复」与「全量浏览/搜索」能力（content 模块拥有 `content_posts`，经 service 暴露给 admin）。
> **产物归属**：仅后端 `petgo-backend`（**无 Flutter 侧**）。SSR + HTMX，路由 `/admin/**`，**不返 JSON**。
> **无前向依赖**：4.1 → 4.2 → 4.3 互不依赖；可独立实现。

## Story

As a **持 `content.proactive_takedown`/`content.restore` 权限的运营成员**,
I want **浏览全平台已发布内容（三类）、按类型/作者/时间/状态筛选并正文全文搜索，对任意内容主动下架（必填原因）或恢复已下架内容**,
so that **无举报时也能主动处置违规内容，并可纠错恢复，且每次操作留审计**。

## Acceptance Criteria

> 验证层：**L0** 编译/单测/MockMvc 切片（无需 DB）· **L1** 集成（Docker postgres+redis，验筛选/搜索/下架/恢复/审计真跑）· **L2**（本故事无）。

1. **AC1（全量浏览，L1）**：`GET /admin/content` 列出**全平台**内容（三类 DAILY/KNOWLEDGE/GROWTH_MOMENT，跨所有作者），含 id/类型/作者/正文预览/创建时间/状态（上线中=`deleted_at IS NULL` / 已下架=`deleted_at IS NOT NULL`）；分页（keyset 或 offset，低频内部工具均可）。HTMX 请求返回 `admin/content :: rows`，整页请求返回完整视图。
2. **AC2（多维筛选，L1）**：支持按 `type`（三类）/ `authorId`（作者）/ 时间范围（created_at from~to）/ `status`（上线中 / 已下架）组合筛选，筛选走 query 参数（`/admin/content?type=DAILY&status=ONLINE&...`），HTMX 片段刷新。
3. **AC3（正文全文搜索，L1）**：支持对正文 `text` 关键词搜索（大小写不敏感子串，`ILIKE %term%`）；与 AC2 筛选可叠加；空词不过滤。
4. **AC4（主动下架，必填原因，L1）**：授权运营对一条上线中内容「主动下架」**必填原因**（服务端校验非空）→ 经既有 `ContentService.softDelete(postId, ADMIN_TAKEDOWN)` 从 Feed/我的发布/成长档案移除，并经既有 `ContentRemovedEvent → notify` 通知作者「因违反社区规范已被移除」；原因记入审计 summary。
5. **AC5（恢复已下架，L1）**：授权运营对一条已下架内容「恢复」→ 经新增 `ContentService.restore(postId)` 清除 `deleted_at`、内容重新进入公开口径（Feed/我的发布/成长档案）；恢复的级联边界见 Dev Notes（评论/点赞不随恢复，V1 可接受）。
6. **AC6（审计强制副作用，L0 切片 + L1）**：每次主动下架/恢复在同事务经 `AdminAuditService.record(...)` 写审计（`action_type`=`CONTENT_TAKEN_DOWN`/`CONTENT_RESTORED`，`target_type=CONTENT_POST`，summary 含 postId + 下架原因；**不含正文明文/PII**）。
7. **AC7（门控，L0/L1）**：浏览/筛选/搜索端点 `@PreAuthorize("hasAuthority('content.proactive_takedown')")`（模块访问）；下架端点 `content.proactive_takedown`；恢复端点 `content.restore`。无权限 → 403（`SUPER_ADMIN` 隐式全权）。
8. **AC8（回归绿，L0/L1）**：既有 Feed/我的发布/成长档案读取口径（`deleted_at IS NULL AND status=PUBLISHED`）与既有 `ContentService` 行为不回归；`mvn -B compile` + L0 全绿；本地 L1 全量回归绿。

## Dev Notes

### 架构约束（必须遵守）
- 后台 = SSR MVC + HTMX，`/admin/**`，**HTML 片段不返 JSON**；区分整页/片段用 `HX-Request`。[Source: architecture.md#Format Patterns]
- 包 `com.tailtopia.admin.moderation.{web,service,domain,dto}`；控制器 `XxxAdminController` 返视图/片段。[Source: architecture.md#Naming Patterns]
- **模块边界**：`content_posts` 由 content 模块拥有；admin **只经 `ContentService`** 读写（浏览/搜索/下架/恢复），**禁 admin 直读 content repo**。[Source: architecture.md#Architectural Boundaries]
- 下架经 `ContentService.softDelete`（既有，仅 `ADMIN_TAKEDOWN` 发 `ContentRemovedEvent` → notify 作者，复用 4.1 同一链路）。[Source: ContentService.java#doSoftDelete]
- 枚举 UPPER_SNAKE；时间 UTC 存储、运营时区展示；审计/日志 **严禁** 正文明文/PII/令牌。[Source: architecture.md#Format Patterns; CLAUDE.md]
- **禁 MQ/缓存/中间件**；本故事无异步。[Source: CLAUDE.md 护栏]

### 既有代码现状（READ 过，brownfield 基线）
- `content/service/ContentService`：有 `softDelete(postId, reason)`（软删 + 级联评论软删 + 点赞物删 + `ADMIN_TAKEDOWN` 发 `ContentRemovedEvent`）、`findSummary`、各类 growth 查询；**无全量浏览/搜索、无 restore → 本故事新增。**
- `content/repository/ContentPostRepository`：`findFeed`/`findMyPosts` 用「布尔可空参数 + (createdAt,id) keyset」式 `@Query`（**参照此式写 adminSearch 避免 PG 42P18 NULL 类型推断坑**，见仓库注释）；公开口径 `deleted_at IS NULL AND status=PUBLISHED`。
- `content/domain/ContentPost`：`id/authorId/type/petId/text(≤1000)/imageUrls/status(PUBLISHED)/deletedAt/createdAt`；**软删 = `deletedAt != null`，无 delete_reason 列**（DeleteReason 仅用于审计/事件，不入库）。
- `content/domain/DeleteReason`：`AUTHOR_DELETE | ADMIN_TAKEDOWN`（仅 2 值）。
- `notify/service/ContentNotifyListener.onContentRemoved`：消费 `ContentRemovedEvent` → 通知作者。**复用，零改动。**
- V4 `content_posts`：`text VARCHAR(1000)`，索引 author_id/pet_id/type/created_at；**无全文索引**（ILIKE 子串对 ≤500 DAU 低频后台可接受，见防坑）。
- 当前 Flyway 最大 = `V31`；本故事**无新迁移**。

### 关键边界 / 防坑
- **「已下架」无法区分作者自删 vs 运营下架**：`content_posts` 无 `delete_reason` 列（DeleteReason 不入库）。故 status「已下架」筛选 = `deleted_at IS NOT NULL`（含作者自删/注销级联/运营下架）。如产品要求只列「运营下架的」可恢复项，需另起 `ALTER` 加 `delete_reason` 列（决策 E2，**本故事不做**，按现状以 deletedAt 判定，恢复对任意已删项生效——dev 阶段如发现产品强约束需回 PO 确认）。
- **恢复的级联边界（V1 可接受，必须在 UI/PR 标注）**：`softDelete` 时评论被软删、点赞被**物理删除**。`restore` 仅清帖子 `deleted_at`，**评论保持软删、点赞不恢复**（点赞已物删无从还原）。V1.0.0 接受此不对称；如需完整恢复评论需扩 `restore` 级联清评论 deletedAt（可选增强，非本故事 AC）。
- **主动下架必填原因**：服务端权威校验 reason 非空（前端必填仅辅助）；原因进**审计 summary**，不进对作者的通知正文（作者通知文案固定「因违反社区规范已被移除」，复用既有）。
- **全文搜索**：V1 用 `ILIKE '%term%'`（无索引，低频后台够用）；**不**引入搜索引擎/中间件（护栏）。如后续量大可加 `pg_trgm` GIN 索引（ALTER，非本故事）。
- **作者展示**：AC1 展示 `authorId`（数字 id）即满足；富化显示名需用户查询 service（Epic 3 用户管理或既有 auth service），**经既有 service、禁直读 users 表**，非本故事必做。
- **content.view_reports 与 content.proactive_takedown**：本故事浏览/搜索用 `content.proactive_takedown` 作模块访问门（举报队列 4.1 用 `content.view_reports`，两模块权限独立）。

### 横切依赖（上游 Epic 1 故事，按执行顺序先行）
- **审计** `AdminAuditService.record(...)`（Story 1.3）：下架/恢复强制副作用。
- **门控** `@PreAuthorize("hasAuthority('content.proactive_takedown'|'content.restore')")`（Story 1.5）；`SUPER_ADMIN` 隐式全权。
- **i18n**（Story 1.6）：文案先中文，后续统一外化。

### Flyway
- **无新迁移**（不加表/列；以 `deleted_at` 判状态，ILIKE 搜索无需索引迁移）。若产品强约束需 `delete_reason` 列 → 从 **V32** 顺延（决策 E2，本故事不做）。

### 范围边界（不做）
- 不做举报队列（→ 4.1）、不做人工审核队列（→ 4.3）；不实现 AdminAuditService 本体/权限分配 UI/i18n 外化（各归 1.3/1.5/1.6）。
- 不加 `delete_reason` 列、不加全文索引、不做恢复评论级联（均为可选增强）。

### 测试标准
- L0：service（必填原因校验、下架/恢复委托 + 审计 mock 验证）、controller（MockMvc：视图/片段/门控/筛选绑定/必填原因回显）、`ContentService.restore`（清 deletedAt + 幂等）。沿用 `AdminWebControllerTest`/`AdminContentServiceTest` 的 MockMvc/Mockito 范式。
- L1（本地，需 Docker PG+Redis）：跨作者全量浏览、多维筛选、正文搜索、主动下架（作者通知 + 审计）、恢复 + 审计、既有 Feed 读取不回归。**云端 headless 只跑到 L0；L1 标注待本地验收。**
