# Story 4.1: 用户举报队列处理

Status: ready-for-dev

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

> **范围域**：管理后台 V1.0.0 · Epic 4（内容审核）· Story 4.1（AB-3A）。**brownfield 扩既有 `/admin/reports`**（Story 3.7 已落举报队列单条下架/驳回）。
> **产物归属**：仅后端 `petgo-backend`（**无 Flutter 侧**）。SSR + HTMX，路由 `/admin/**`，**不返 JSON**。
> **无前向依赖**：4.1 → 4.2 → 4.3 互不依赖；可独立实现。

## Story

As a **持 `content.view_reports`/`content.takedown` 权限的运营成员**,
I want **在举报队列里按时间查看举报、对单条或批量举报执行下架/驳回，并自动双向通知作者与举报人**,
so that **48h 内维护社区环境（AM-1），且每个处置可追溯、举报人有闭环反馈**。

## Acceptance Criteria

> 验证层：**L0** 编译/单测/MockMvc 切片（无需 DB）· **L1** 集成（Docker postgres+redis，验队列/下架/通知/审计真跑）· **L2**（本故事无）。

1. **AC1（队列列表，L1）**：`GET /admin/reports` 待处理队列按**举报时间倒序**展示，每行含：举报人（reporterId，无独立用户服务时展示 id；详情见 Dev Notes 边界）/内容预览（textPreview）/举报类型（reasonType）/举报时间/**作者**（content_posts.author_id）/该帖累计举报数/内容是否已下架。HTMX 请求（`HX-Request`）返回行片段，整页请求返回完整视图。
2. **AC2（单条下架，L1）**：对一条举报选择「下架」→ 经既有 `ContentService.softDelete(postId, ADMIN_TAKEDOWN)` 同步从 Feed/我的发布/成长档案移除（既有口径 `deleted_at`），并经既有 `ContentRemovedEvent → notify` 向**作者**发「你发布的内容因违反社区规范已被移除」（复用 Story 3.7 既有链路）；工单置 `RESOLVED`，处理人记 `adminAccountId` 对应的运营标识。
3. **AC3（单条驳回，L1）**：对一条举报选择「保留（驳回）」→ 工单置 `DISMISSED`、内容不动。
4. **AC4（举报人统一模糊通知，L1，新增行为）**：**无论下架还是驳回**，向**举报人**（reporterId）发一条**统一的模糊通知**（不透露处置结果、不透露内容/作者，如「感谢你的举报，我们已完成审核」）。〔注：这是对 Story 3.7「审核结果不回告」的**有意修订**——AB-3A 要求向举报人发模糊闭环通知；见 Dev Notes「关键行为变更」。〕
5. **AC5（批量下架/驳回，L1）**：队列支持多选 → 单次提交一组 reportId 批量「下架」或「驳回」→ service 内**逐条处理 + 逐条审计 + 逐条双向通知**，部分失败不回滚已成功项，**汇总结果回 HTMX 片段**（成功 N 条/失败 M 条及原因）。
6. **AC6（审计强制副作用，L0 切片 + L1）**：每条下架/驳回（含批量内每一条）在同事务经 `AdminAuditService.record(...)` 写一条审计（`action_type` = `CONTENT_TAKEN_DOWN` / `REPORT_DISMISSED`，`target_type=CONTENT_REPORT`/`CONTENT_POST`，summary 含工单与帖 id，**不含 PII/内容正文明文**）。
7. **AC7（已处理队列，L1）**：处理后工单移出待处理队列，可在「已处理」视图（filter `status=RESOLVED|DISMISSED`）按时间查看，含处理人与处理时间。
8. **AC8（门控，L0/L1）**：队列查看端点 `@PreAuthorize("hasAuthority('content.view_reports')")`；下架端点 `hasAuthority('content.takedown')`；驳回端点 `hasAuthority('content.view_reports')`（或 `content.takedown`，见 Dev Notes）。无权限访问 → 403（`SUPER_ADMIN` 隐式全权）。
9. **AC9（回归绿，L0/L1）**：既有 `/admin/reports` 单条下架/驳回行为与 `AdminWebControllerTest`/`ReportService`/`ContentService` 不回归；`mvn -B compile` + L0 全绿；本地 L1 全量回归绿。

## Dev Notes

### 架构约束（必须遵守）
- 后台 = SSR MVC + HTMX，路由 `/admin/**`，**HTMX 端点返 HTML 片段（`text/html`），绝不返 JSON / 绝不落 `/api/v1`**；区分整页/片段用 `HX-Request` 头。[Source: admin-backend/architecture.md#Format Patterns; #API & Communication Patterns]
- 包 `com.tailtopia.admin.moderation.{web,service,domain,dto}`；控制器后缀 `XxxAdminController`，返回视图名/片段。[Source: architecture.md#Naming Patterns; #Project Structure]
- **跨模块只经既有 service / 领域事件，禁跨 repo**：下架经 `ContentService.softDelete`，作者通知经既有 `ContentRemovedEvent`，举报人通知经新事件 → notify；**禁 admin 直读 content_posts / notifications 表**。[Source: architecture.md#Communication Patterns; #Architectural Boundaries]
- 枚举 UPPER_SNAKE；时间 `timestamptz` UTC（页面展示按运营时区格式化，存储仍 UTC）；日志/审计 summary **严禁** PII/内容正文明文/令牌。[Source: architecture.md#Format Patterns; CLAUDE.md 护栏]
- 异步只 `@TransactionalEventListener`（进程内事件）；**禁 MQ/缓存层/中间件**。[Source: CLAUDE.md 强制护栏]

### 既有代码现状（READ 过，brownfield 基线）
- `admin/service/AdminModerationService`：已有 `pendingQueue()`（PENDING 倒序、limit 50）、单条 `takedown(reportId, adminId)`（softDelete + mark RESOLVED）、`dismiss(reportId, adminId)`（mark DISMISSED）。**本故事在此加批量 + 审计 + 举报人通知；单条逻辑保留。** [现状：AdminModerationService.java]
- `admin/service/ReportQueueItem`：视图项含 reportId/postId/reasonType/status/createdAt/reportCount/postTextPreview/postDeleted。**缺 reporterId/authorId → 本故事补。**
- `admin/web/AdminWebController`：`GET /admin/reports`、`POST /admin/reports/{id}/takedown|dismiss` 三端点 + `templates/admin/reports.html`（含内联 `<style>`，无 HTMX、无批量、无筛选）。
- `moderation/service/ReportService`：`pendingQueue(limit)`、`find`、`countForPost`、`mark(reportId, adminId, decision)`。
- `moderation/domain/ContentReport`：含 `reporterId`、`reasonType`、`status(PENDING|RESOLVED|DISMISSED)`、`handledBy`、`handledAt`、`resolveBy(adminId, decision)`。
- `content/service/ContentService.softDelete(postId, DeleteReason.ADMIN_TAKEDOWN)`：软删 + 级联（评论软删、点赞物删）+ 仅 `ADMIN_TAKEDOWN` 发 `ContentRemovedEvent`；`findSummary(postId)→PostSummary(id,type,textPreview,deleted)`（**缺 authorId → 本故事经 content 模块补**）。
- `notify/service/ContentNotifyListener.onContentRemoved`：消费 `ContentRemovedEvent` → 通知作者「内容已被移除」。**作者通知本故事零改动，复用。**
- `notify/service/NotificationService.send(recipientUserId, type, title, body, deepLinkType, targetRef)`：统一推送出口（写 notifications + 角标 + 离线推送）。
- 当前 Flyway 最大 = `V31`；本故事**无新迁移**（NotificationType 加枚举值落 varchar 不需迁移）。

### 关键行为变更（务必显式处理，否则违反 AB-3A）
- **举报人通知是新增行为**：Story 3.7 明文「举报人侧仅收到提交反馈，审核结果不回告，V1 无举报进度查询/申诉入口」。AB-3A 要求**无论结果向举报人发统一模糊通知**。本故事**新增**该闭环通知，但保持「模糊」——
  - 通知文案对下架/驳回**完全相同**，**不透露**处置结果、**不透露**被举报内容/作者；
  - **不提供**申诉入口、不提供查询；仅一条「已处理」性质的告知。
  - 实现走**新事件 `ReportResolvedEvent` → notify**（与作者通知的事件式解耦一致），不在 moderation/admin 内直调 notify repo。

### 横切依赖（上游 Epic 1 故事，按执行顺序先于本故事完成）
- **审计**：`AdminAuditService.record(...)`（Story 1.3）——每个写操作强制副作用，哈希链 append-only。本故事下架/驳回（含批量逐条）必须调用。若 1.3 尚未落地，dev 阶段以接口占位、联调时接真实实现（按执行顺序 1.3 已先行）。
- **门控**：`@PreAuthorize("hasAuthority('<permission_code>')")`（Story 1.5）——`content.view_reports`/`content.takedown`；`SUPER_ADMIN` 隐式全权。模板 `sec:authorize` 隐藏入口仅体验、非安全边界。
- **i18n**：文案先中文，Story 1.6 统一外化 `#{admin.*}`；本故事不外化。

### 关键边界 / 防坑
- **批量不整体回滚**：逐条独立事务/独立 try-catch，部分失败回报每条结果（架构 §Process Patterns「批量操作」）；汇总片段明确成功/失败条数与原因。
- **举报人 = reporterId（users.id）**，作者 = content_posts.author_id；两者通知目标不同，勿混。展示「举报人/作者」昵称需用户服务——V1.0.0 用户详情服务（Epic 3）或不在范围；**本故事展示数字 id 即满足 AC1**，如已有可用的 `auth`/用户查询 service 可富化显示名（经既有 service，禁直读 users 表）。
- **作者通知零改动**：复用既有 `ContentRemovedEvent`（仅 `ADMIN_TAKEDOWN` 触发），不要重复发作者通知。
- **已删幂等**：`softDelete` 对已删内容幂等；批量中若某帖已被先前下架，视为成功（工单仍置 RESOLVED）。
- **审计 summary 脱敏**：只放工单 id / 帖 id / 处置类型，**不放内容正文/举报人邮箱**。

### Flyway
- **无新迁移**。`NotificationType` 追加 `REPORT_REVIEWED` 为枚举值（列已是 varchar，validate 不校验枚举集）。若误判需建表 → 从 **V32** 顺延（不预期）。

### 范围边界（不做）
- 不建人工审核队列（→ 4.3）；不做全量内容浏览/主动下架（→ 4.2）；不实现 AdminAuditService/权限分配 UI/i18n 外化本体（各归 1.3/1.5/1.6，本故事仅**调用/标注**）。
- 不为举报人提供申诉入口/进度查询（AB-3A 仅要求模糊通知）。

### 测试标准
- L0：`AdminModerationService`（批量逐条 + 部分失败 + 逐条审计/通知 mock 验证）、`AdminModerationController`（MockMvc：视图/片段/门控/批量绑定）、notify 监听器消费 `ReportResolvedEvent`（mock NotificationService）。沿用 `AdminWebControllerTest` 的 MockMvc/Mockito 范式。
- L1（本地，需 Docker PG+Redis）：队列列表、单条/批量下架→作者通知 + 举报人模糊通知 + 审计落库 + 工单 RESOLVED/DISMISSED 流转、已处理视图真跑。**云端 headless 只跑到 L0；L1 标注待本地验收**（CLAUDE.md 云端约定）。
