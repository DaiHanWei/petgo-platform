# Story 3.1: 用户搜索与详情查看

Status: review

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

> **范围域**：管理后台 V1.0.0 · Epic 3（用户账号治理）· Story 3.1（AB-UA-01）。**纯只读**：用户搜索 + 详情聚合，不写任何业务表、不可编辑用户资料。
> **产物归属**：仅后端 `petgo-backend`（**无 Flutter 侧**）。brownfield 扩 `com.tailtopia.admin`，新建 `admin.usermgmt` 子域。
> **前置**：建立在 Epic 1（admin_accounts + 服务端会话 + adminFilterChain + 权限门控）之上。本故事**只读**，**无前向依赖于 3.2/3.3**。

## Story

As a **客服/运营**,
I want **按用户 ID 或注册邮箱搜索用户，并打开一个只读的详情聚合页**,
so that **处理投诉与合规请求时能掌握账号全貌（档案/问诊/内容/状态），无需翻多个系统也不会误改用户数据**。

## Acceptance Criteria

> 验证层：**L0** 编译/单测/MockMvc 切片（mock 各 owning service，无需 DB）· **L1** 集成（Docker postgres+redis，验真实聚合读 + 门控 + 视图渲染）· **L2** 外部凭证（本故事无 L2）。

1. **AC1（搜索，L1）**
   **Given** 用户搜索页 `/admin/users`
   **When** 客服按**用户 ID** 或**注册邮箱**查询
   **Then** 返回匹配的普通用户（`users.role=USER`）；命中按 ID 唯一、按邮箱精确匹配；无结果显示空态文案；**已停用账号在结果中标注「已停用」**（停用语义由 Story 3.2 落地；本故事只读展示当前状态字段，3.2 未实现前一律显示「正常」）。（HTMX 片段刷新）

2. **AC2（详情聚合，L1）**
   **Given** 用户详情页 `/admin/users/{userId}`
   **When** 打开
   **Then** 聚合展示五块只读信息：①基本信息（昵称/注册邮箱/注册时间 `created_at`/最后活跃时间）②宠物档案列表（经 profile owning service 读）③历史问诊（**仅元数据**：会话 ID/兽医/起止时间/状态/评分若有，页面**显著标注「仅元数据，不含对话内容」**）④发布内容列表（经 content owning service 读）⑤账号当前状态。

3. **AC3（只读不可编辑，L0/L1）**
   **Given** 用户详情页
   **When** 渲染
   **Then** 页面**无任何编辑用户资料的表单/按钮**（停用/删除入口属 3.2/3.3，本故事不渲染）；本故事的所有端点均为 `GET`，**不产生任何写操作、不写审计**。

4. **AC4（数据边界，L0/L1）**
   **Given** 历史问诊聚合
   **When** 读取
   **Then** **绝不读取 IM 正文 / AI 分诊上下文 / 用户上传媒体**（储存于第三方腾讯 IM，V1.0.0 不跨系统拉取，与 AB-4B 一致）；仅展示 TailTopia 系统内元数据。

5. **AC5（模块边界，L0/L1）**
   **Given** 跨模块聚合
   **When** `AdminUserService` 读 profile/content/consult 数据
   **Then** **一律经各 owning service 读，禁止 admin 直接访问 profile/content/consult 的 repository、禁跨库 join**（沿用架构模块边界）。

6. **AC6（权限门控，L0）**
   **Given** 服务端逐端点授权（A5）
   **When** 无相应查看权限的后台账号请求 `/admin/users/**`
   **Then** 方法级 `@PreAuthorize` 拒绝 403；`SUPER_ADMIN` 隐式全权放行（门控基建复用 Story 1.5；权限码见 Dev Notes）。

7. **AC7（回归绿，L0/L1）**
   **Given** 新增子域
   **When** `mvn -B compile` + L0 单测 + 本地 L1 全量回归
   **Then** 全绿；既有端点（App `/api/v1` 与既有 `/admin/**`）无回归。

## Dev Notes

### 架构约束（必须遵守）
- 后台 = Thymeleaf SSR + HTMX 片段，路由 `/admin/**`，**绝不返回 JSON、绝不落 `/api/v1`**；HX-Request 返片段，整页返完整视图。[Source: admin-backend/architecture.md#API & Communication Patterns / Format Patterns]
- 包 `com.tailtopia.admin.usermgmt.{web,service,dto}`；控制器后缀 `XxxAdminController` 或 `AdminUserController`，返回视图名/片段。[Source: architecture.md#Naming Patterns / Project Structure → admin/usermgmt]
- **跨模块只经既有 owning service 或领域事件，禁跨 repo、禁跨库 join**（用户详情聚合经 owning service 读）。[Source: architecture.md#Architectural Boundaries 数据边界]
- **数据边界**：问诊消息正文 / AI 上下文 / 用户媒体储存于第三方（腾讯 IM），后台 V1.0.0 **不读取**，仅展示系统内元数据（与 AB-4B 一致，AG-6）。[Source: architecture.md#NFR5 / Gap AG-6]
- 时间：库内 `timestamptz` UTC，视图层按运营时区格式化并标注（沿用基线）。[Source: architecture.md#Format Patterns]
- 授权用 `permission_code` authority 方法级 `@PreAuthorize` 强制（A5），前端隐藏仅体验。[Source: architecture.md#Process Patterns 授权校验]

### 既有基线（READ — 现状 / 改什么 / 不可破坏）
- **auth**（`auth/service/AccountQueryService`、`auth/repository/UserRepository`、`auth/domain/User`）：
  - 现状：`UserRepository` 有 `findById`、`findByGoogleSub`、`findByEmailAndRole(email, Role)`；`User` 有 `id/email/displayName/nickname/role/createdAt/updatedAt/deletedAt`，**无 `lastActiveAt`、无独立 `status` 列**（停用列在 3.2 引入）。
  - 改：经 `AccountQueryService`（或在其上）暴露「按 id / 按 email 查 `USER`」的只读方法；搜索按邮箱用 `findByEmailAndRole(email, Role.USER)`。
  - 不可破坏：**不改 App 登录/注册路径**；本故事只读，不改 `User` 写语义。
- **profile**（`profile/service/ProfileService`、`PetProfileQueryService`、`PetProfileRepository.findByOwnerId(ownerId)`）：现状已有按 owner 读档案的能力 → 经 service 暴露给 admin 聚合。**不在 admin 直读 `PetProfileRepository`**。
- **content**（`content/service/ContentService`、`ContentPostRepository`）：现状有 `findMyPosts`/按作者类型分页等；运营视角需「列出该用户全部发布（含已下架）」——若现成 service 方法不足，在 content service **新增只读方法**返回内容摘要列表。`PostStatus` 当前仅 `PUBLISHED`，软删用 `deleted_at`。**不在 admin 直读 `ContentPostRepository`**。
- **consult**（`consult/service/ConsultHistoryService`、`ConsultSessionRepository.findByUserId(userId)`、`consult/domain/ConsultSession`）：`ConsultSession` 字段含 `userId/vetId/status(SessionStatus)/createdAt` 等可作元数据；`aiImageRefs`/症状文字属 PII，**严禁取用**。经 consult service 暴露「按 userId 列会话元数据 + 评分」的只读方法。
- **既有 admin slice**：`admin/web/AdminWebController` + `templates/admin/{layout,dashboard,...}.html` + `adminFilterChain`（Epic 1）。新页挂入既有导航/布局。

### 关键边界 / 防坑
- 「最后活跃时间」：`users` 表无现成 `last_active_at` 列。**不要为此新增列**（本故事零写、零迁移）；取可得的近似来源（如最近一次内容/问诊/会话活动时间，经 owning service）并在视图标注口径，或先标「暂无」。如确需精确字段，列为后续增量、不在本故事。
- 问诊聚合：DTO 只承载元数据字段，**从源头就不要把 `aiImageRefs`/症状文本投影进来**（避免误泄）。
- 搜索按邮箱：普通用户的 google 登录邮箱即注册邮箱；务必限定 `role=USER`，**不要把后台账号（admin_accounts）或兽医账号搜出来**。
- 只读纪律：本故事**不写审计**（审计是写操作的强制副作用，只读不触发）；不要顺手加停用/删除按钮（属 3.2/3.3）。

### Flyway
- **本故事无新迁移**（纯只读聚合）。若 3.2 已先合入其 `users` 状态列，本故事的「已停用」标注直接读之；3.2 未合入则一律显示「正常」，**不得**在本故事新增列。

### 范围边界（不做）
- 不做：停用/重新激活（→ 3.2）、删除 D1/D2（→ 3.3）、编辑用户资料（PRD §5 明确范围外，永不做）、双语外化（→ 1.6，文案先中文）、审计写入（只读无写）。
- 横切依赖：权限门控 `@PreAuthorize`（Story 1.5）；视觉令牌/i18n（Story 1.6）。若 1.5 门控基建尚未就绪，按其约定预留权限码 `user.view` 接线，先以 `ROLE_ADMIN` 兜底门控，待 1.5 收口。

### 测试标准
- L0：`AdminUserService`（mock 各 owning service）验搜索命中/未命中、详情五块装配、问诊只投影元数据；`AdminUserController` MockMvc 验页面含五块 + 「仅元数据」标注 + 无编辑控件 + HX 片段 + 无权限 403。沿用既有 `AdminWebControllerTest` 范式。
- L1（本地 docker PG+Redis）：真实聚合读跑通 + 全量回归绿。**云端 headless 只跑 L0；L1 标注待本地验收**（CLAUDE.md 云端约定）。
