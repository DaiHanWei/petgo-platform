# Story 5.2: 问诊会话元数据查询

Status: review

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

> **范围域**：管理后台 V1.0.0 · Epic 5（问诊异常与会话查询）· Story 5.2（AB-4B）。**建立在 Epic 1 地基（服务端会话 / permission authority 门控 / i18n）之上**；与 Story 5.1 同模块（`admin.anomaly`），但**独立、纯只读**，不依赖 5.1 的工单表。
> **产物归属**：仅后端 `petgo-backend`（**无 Flutter 侧**）。brownfield 扩 `com.tailtopia.admin.anomaly`。SSR（Thymeleaf）+ HTMX，走 `/admin/**`，**不返 JSON**。
> **数据边界（NFR5，硬约束）**：消息正文 / AI 分诊结果 / 用户上传媒体储存于第三方腾讯 IM，本故事**绝不读取**，仅展示 TailTopia 系统内的会话元数据 + 评分。**无新表**（只读既有 `consult` 数据）。

## Story

As a **被授权的运营人员**,
I want **按用户 ID / 兽医 ID / 日期范围查询问诊会话的元数据（含评分）**,
so that **能排查与审计问诊，而完全不触达第三方的消息正文/AI/媒体（NFR5）**。

## Acceptance Criteria

> 验证层：**L0** 编译/单测/MockMvc 切片（mock service，无需 DB）· **L1** 集成（Docker postgres+redis，验查询/门控/元数据边界）· **L2**（本故事无）。

1. **AC1（多维查询，L1 + L0 service 单测）**：会话查询页支持按**用户 ID**、**兽医 ID**、**日期范围**（起止，作用于会话创建时间，UTC）任意组合查询；HTMX 提交返回 `:: rows` 结果片段。
2. **AC2（元数据结果集，L1）**：每行展示**会话 ID / 用户账号（ID，匿名化后容空）/ 兽医账号（ID）/ 开始时间（createdAt）/ 结束时间（terminalAt：中断取 interruptedAt，否则 updatedAt）/ 会话状态 / 用户评分（若有，1-5 星 + 文字）**；无评分的会话评分列留空标注。
3. **AC3（绝不读第三方，L0/L1）**：DTO/模板**不含**消息正文、AI 分诊结果、用户上传媒体字段；查询路径**不调用**任何 IM / AI / 媒体拉取接口（NFR5）；页面明确标注「仅元数据」。
4. **AC4（跨模块经既有 service，禁跨 repo + 无新表，L0/L1）**：`admin.anomaly` **不直接 `@Autowire` `ConsultSessionRepository`**；改经 `consult` 模块新增的只读查询 service 方法（见 Dev Notes「跨模块读取约定」）取会话元数据 + 评分；本故事**不建任何表/不写迁移**。
5. **AC5（纯只读，无写无审计，L0）**：本故事**无任何写操作**，故**不**调 `AdminAuditService`、**无** `@PostMapping`；仅 `GET` 查询端点。
6. **AC6（权限门控，L0/L1）**：查询页/端点 `@PreAuthorize("hasAuthority('consult.view_sessions')")`；无该 authority 访问返回 403（A5）。
7. **AC7（回归绿，L0/L1）**：`mvn -B compile` + 既有 admin/consult 单测全绿；新增查询 service/controller 单测；本地 L1 全量回归绿。

## Dev Notes

- 后端 = Spring Boot 4 / Java 21 / Maven；控制器/模板归 `com.tailtopia.admin.anomaly.{web,dto}`（与 5.1 同模块）。[Source: admin-backend/architecture.md#Project Structure → `anomaly/`（line 250）；#结构映射「AB-4 问诊异常·会话查询 → admin/anomaly」（line 287）]
- 后台一律 `/admin/**` + HTML/HTMX 片段，**不返 JSON**；服务端会话 + CSRF（Epic 1 已立）。[Source: architecture.md#AI Agent Guidelines line 367]
- **跨模块只经既有 service / 领域事件，禁跨 repo**：admin 模块**不得**直接注入 `ConsultSessionRepository`/`ConsultRatingRepository`；会话元数据读取必须经 `consult` 模块暴露的只读查询 service。[Source: architecture.md line 188/294；CLAUDE.md 强制护栏]
- 权限 authority：`consult.view_sessions`；方法级 `@PreAuthorize`，前端隐藏仅体验（A5）。[Source: architecture.md line 182；Story 1.5]
- 时间一律 UTC；DB snake_case ↔ Java camelCase。[Source: CLAUDE.md 命名映射链]

### 跨模块读取约定（关键 —— 解「读既有 consult」与「禁跨 repo」的张力）
- 数据源是既有 `consult_sessions` / `consult_ratings`（**无新表**），但 admin **不直访** consult 的 repository（违反「禁跨 repo」护栏）。
- 解法：在 **consult 模块内**新增一个**只读**查询 service（如 `ConsultSessionAdminQueryService`），由它使用 `ConsultSessionRepository`（consult 自己的 repo —— 模块边界内合法）+ `ConsultRatingRepository`，返回**元数据 DTO**；`admin.anomaly` 控制器仅依赖该 service。这样既「只读既有 consult 数据」又「不跨 repo」。
- DTO 只含元数据 + 评分字段，**绝不**含 IM/AI/媒体 —— 即便 `ConsultSession` 实体持有 `aiSymptomText`/`aiImageRefs`/`imConversationId`/`vetDiagnosis`，查询 DTO **不映射**这些列（NFR5）。

### 既有代码基线（READ —— 现状 / 用什么 / 不可破坏）
- `consult/repository/ConsultSessionRepository`（**现状**）：已有 `findByUserIdAndStatusInOrderByCreatedAtDesc` 等多个 finder（模块注释明示「仅 consult.service 直接访问」）。**用什么**：增动态多条件查询（Specification / @Query），供新只读 service 调用。**不可破坏**：既有 finder 与用户/兽医历史查询语义。[Source: consult/repository/ConsultSessionRepository.java]
- `consult/domain/ConsultSession`（**现状**）：`getId/getUserId/getVetId/getStatus/getCreatedAt`、`terminalAt()`（终态时间单一口径，中断取 interruptedAt 否则 updatedAt 兜底 createdAt）。**用什么**：DTO 的 startedAt=createdAt、endedAt=terminalAt()、status=status。`userId` 可空（注销匿名化 `anonymize()`）。[Source: consult/domain/ConsultSession.java:276-281,167-171]
- `consult/domain/ConsultRating` + `ConsultRatingRepository`（**现状**）：`stars(1-5)` + `comment(≤100)`，「仅运营可见」，每会话至多一条；匿名化保留评分剥 userId。**用什么**：按 sessionId 取评分拼进 DTO（「评分若有」）。[Source: consult/domain/ConsultRating.java]
- `templates/admin/layout.html`（**现状**）：SSR 外壳侧栏。**改什么**：在「问诊异常（模块 4）」分组加「会话查询」入口。**不可破坏**：既有导航/渲染。[Source: templates/admin/layout.html:29-50]
- `@PreAuthorize` + permission authority（Story 1.5 产出）：authority 字符串即 `permission_code`。[Source: Story 1.5；architecture.md line 182]

### 关键边界 / 防坑
- **NFR5 红线**：查询 DTO/模板**零** IM 正文、AI 分诊结果、用户媒体；评审 grep 确认查询路径无 `imClient`/AI/`SignedUrlService` 拉媒体调用。
- **纯只读**：无 `@PostMapping`、无 `AdminAuditService`（审计只记**写**操作，查询不审计）；勿误加。
- **匿名化容空**：注销后会话 `user_id` 为空，结果集需容空展示（不报 NPE）。
- **日期范围语义**：作用于 `created_at`（UTC）；前端传日期需明确时区换算口径（建议按 UTC 日界，dev 在实现注释口径）。
- **不建表/不写迁移**：本故事 Flyway **无新增**（与 5.1 的 V38 无关）。若误判需新列，停下确认 —— 元数据均取自既有列，不应需要。

### 范围边界（避免前向依赖 / 过度实现）
- **本故事不做**：异常工单（→ Story 5.1）；评分总览/单兽医评分详情看板（→ Epic 6）；任何写操作/审计；任何 IM/AI/媒体读取（NFR5 永不做）。
