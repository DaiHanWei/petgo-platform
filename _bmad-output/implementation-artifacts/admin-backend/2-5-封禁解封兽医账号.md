# Story 2.5: 封禁/解封兽医账号

Status: ready-for-dev

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

> **范围域**：管理后台 V1.0.0 · Epic 2 · Story 2.5（AB-2E）。**安全攸关节点**（封禁副作用 + 跨产品契约）。
> **产物归属**：仅后端 `petgo-backend`（**无 Flutter 侧**）。SSR + HTMX，`/admin/vets/{id}/status`，不返 JSON。
> **brownfield 关键**：`POST /admin/vets/{id}/status` 封禁 + `AdminVetService.setBanned`（已含「清在线态 + `ConsultInterruptService.interruptByVetBan`」）**已存在**（Story 5.1/5.7）。本故事**补审计 + 门控 + 发 `ConsultAnomalyRaisedEvent` 供 Epic 5 + 对齐 2026-06-29 修订的用户重新匹配系统消息**。

## Story

As a **运营成员**,
I want **弹窗确认后封禁或解封兽医账号，且进行中问诊被妥善中断并通知用户**,
so that **及时处置违规兽医、用户能选择重新匹配或结束、异常进入 Epic 5 工单可追踪**。

## Acceptance Criteria

> 验证层：**L0** MockMvc 切片 + 单测 · **L1** 集成（Docker postgres+redis，验中断/事件/审计真跑）。

1. **AC1（弹窗确认 + 封禁即失登录权，L1）**：封禁须前端弹窗确认；封禁后兽医 `status=BANNED` 即时失去登录权（既有 `BannedVetFilter`/`authenticate` 已保障），列表显示「已封禁」。
2. **AC2（进行中问诊强制关闭 + 系统消息含重新匹配，L1，免费阶段 V1.0.0）**：封禁时兽医有进行中问诊（IN_PROGRESS/PENDING_CLOSE）→ 会话强制关闭（既有 `interruptByVetBan` 迁 INTERRUPTED）；向受影响用户发 IM 系统消息，告知**兽医已临时下线、本次中断**，并提供「重新匹配一位兽医继续」或「结束」选项（免费阶段**无退款**）。系统消息文案据此修订。
3. **AC3（发异常事件供 Epic 5，L1）**：每个被中断的会话发布 `ConsultAnomalyRaisedEvent`（会话/用户/兽医/起止时间/异常类型元数据），供 Epic 5 `ConsultAnomalyService` 订阅落工单（本故事只发事件，不建工单表）。
4. **AC4（解封恢复，L1）**：解封 → `status=ACTIVE`，兽医可重新登录；**不恢复已中断会话**（中断终态不可逆，用户须重新发起/已走重新匹配）。
5. **AC5（写审计含操作人+时间戳，L1）**：封禁/解封成功同事务调 `AdminAuditService.record(actorAccountId, "VET_BANNED"/"VET_UNBANNED", "VET_ACCOUNT", vetId, summary)`；审计含操作人（actor account）与时间戳（`created_at`）。
6. **AC6（门控，L0/L1）**：端点 `@PreAuthorize("hasAuthority('vet.ban')")`。
7. **AC7（回归绿，L0/L1）**：既有封禁副作用（清在线态 + 中断）不回归；既有 `ConsultInterruptedEvent`（推送/历史）仍发；`mvn -B compile` + vet/consult/admin 单测全绿；本地 L1 全量回归绿。

## Dev Notes

- 跨模块副作用走既有领域事件 + service，**禁跨 repo**：封禁→既有会话中断（`ConsultInterruptService`）+ 发 `ConsultAnomalyRaisedEvent`（admin/Epic5 订阅入工单）。[Source: architecture.md#Communication Patterns / Integration Points 数据流①]
- 红色态零变现 / 安全规则只升不降 / 封禁是安全攸关节点——勿埋绕过点。[Source: CLAUDE.md 强制护栏]
- 写操作强制审计（含操作人+时间戳）；`VET_BANNED`/`VET_UNBANNED` UPPER_SNAKE 过去式。授权 `vet.ban`。[Source: architecture.md#Communication Patterns；PRD 附录 B]
- 免费阶段 V1.0.0 无退款；付费退款分支属 📦1.1.0（不实现）。[Source: PRD#AB-2E]

### 既有代码基线（READ 过，brownfield）
- **现状**：`AdminVetService.setBanned(vetId, banned)`（`@Transactional`：`vetAccounts.setStatus(BANNED/ACTIVE)`；封禁时 `presence.goOffline(vetId)` + `interruptService.interruptByVetBan(vetId)`，**无审计、无门控**）；`ConsultInterruptService.interruptByVetBan`（查 IN_PROGRESS/PENDING_CLOSE → `s.interrupt(VET_BANNED)` → IM 系统消息「兽医已临时下线，本次问诊已中断，请重新发起咨询」→ 发 `ConsultInterruptedEvent`；返回中断数）；`ConsultSession.interrupt` 迁 INTERRUPTED 终态、不评分、不存档；`BannedVetFilter`/`VetAccountService.authenticate` 已挡 BANNED 登录；`AdminWebController#setVetStatus`（`POST /{id}/status`，`@RequestParam banned`，无门控/审计）；`vets.html` 封禁表单已有 confirm 弹窗。
- **`ConsultAnomalyRaisedEvent` 当前不存在**（grep 确认）——本故事新建。
- **本故事改什么**：新建 `ConsultAnomalyRaisedEvent` + 在中断循环发它；修订 IM 系统消息文案；`setVetStatus`/`setBanned` 加审计 + 门控。
- **不可破坏**：① 既有封禁副作用顺序（setStatus → goOffline → interrupt）保留；② `ConsultInterruptedEvent` 继续发（不替换，并存）；③ 解封不恢复会话（既有语义）；④ INTERRUPTED 终态/不存档语义不变；⑤ 既有 confirm 弹窗保留（仅更文案）。

### 关键边界 / 防坑
- **事件并存非替换**：`ConsultInterruptedEvent`（Epic 6 推送/历史）与新 `ConsultAnomalyRaisedEvent`（Epic 5 运营工单）是两个消费面，**都要发**；勿用一个替另一个。
- **事件字段取值**：`startedAt` 取会话 `createdAt`/接单时间，`endedAt` 取 `interruptedAt`（`s.terminalAt()` 可复用）；`anomalyType="VET_BANNED"`（UPPER_SNAKE）。
- **审计与中断同事务**：封禁是「setStatus + 中断 + 审计」原子；若拆在 controller 调两次 service 注意事务边界（推荐在 `AdminVetService.setBanned` 内注入 `AdminAuditService` 同事务记，或保证 controller 方法 `@Transactional`）。封禁数量/受影响会话数可入 summary（不含 PII）。
- **跨产品「重新匹配」**：后端只负责①强关会话②发系统消息告知选项③发 AnomalyRaisedEvent；用户点「重新匹配」走的是**既有问诊发起/排队路径**（用户中断后占用解除即可重新发起，见 `ConsultInterruptService` 注释）——**App 侧呈现选项是另一产品（petgo_app）的事，本故事不含 Flutter 改动**，仅在 Dev Notes/Completion 标注跨产品依赖。
- **解封幂等**：对已 ACTIVE 解封 / 已 BANNED 再封禁应幂等不报错（既有 setStatus 直接赋值，天然幂等；审计可仍记一条或判重——V1 记一条可接受）。

### Flyway
- 本故事**无新迁移**（封禁用既有 `vet_accounts.status` + `consult_sessions` 中断列）。`ConsultAnomalyRaisedEvent` 是内存事件，**不落表**（工单表在 Epic 5）。

### 范围边界（不做）
- 不做 Epic 5 异常工单表/订阅落库（→ Epic 5 Story 5.1，本故事只发事件）；不做付费退款分支（📦1.1.0）；不做 App 侧「重新匹配/结束」UI（跨产品，petgo_app）；不做双语外化（→ 1.6，新文案先中文）。
- 本故事**只**交付：封禁/解封审计 + 门控 + 发 `ConsultAnomalyRaisedEvent` + 系统消息文案修订（含重新匹配/结束告知）。

### 测试标准
- L0：`ConsultInterruptServiceTest`（mock sessions/im/events）：中断每会话发 `ConsultInterruptedEvent` **和** `ConsultAnomalyRaisedEvent`、系统消息文案含「重新匹配/结束」语义；`AdminWebControllerTest`：封禁/解封调 `AdminAuditService.record(VET_BANNED/VET_UNBANNED)`、无 `vet.ban` 403。
- L1（本地 docker）：封禁有 IN_PROGRESS 会话的兽医 → 会话 INTERRUPTED + 兽医被踢下线 + 不可登录 + 事件落工单（与 Epic 5 联调时验）；解封可重新登录。云端只 L0。
