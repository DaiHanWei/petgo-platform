# Story 2.3: 创建兽医账号

Status: review

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

> **范围域**：管理后台 V1.0.0 · Epic 2 · Story 2.3（AB-2B）。**依赖 Story 2.1**（建号后置资质 `PENDING_COMPLETION`）。
> **产物归属**：仅后端 `petgo-backend`（**无 Flutter 侧**）。SSR + HTMX，`/admin/vets`，不返 JSON。
> **brownfield 关键**：`POST /admin/vets` 开户 + `AdminVetService.create` + `CreateVetForm` + `vets.html` 开户卡 **已存在**（Story 5.1）。当前 `CreateVetForm` 只有 `displayName/username/password`，**本故事加联系手机号字段 + 写审计 + 建号后置待完善资质**。

## Story

As a **运营成员**,
I want **创建兽医账号（显示名/登录邮箱/联系手机号/初始密码）并一次性拿到初始凭证**,
so that **可人工交付给兽医登录工作台，且新号资质待完善暂不可接单**。

## Acceptance Criteria

> 验证层：**L0** MockMvc 切片 + 单测（无 DB）· **L1** 集成（Docker postgres+redis，验唯一校验/落库/资质行/审计）。

1. **AC1（必填字段，L0/L1）**：创建表单必填 **显示名 / 登录邮箱 / 联系手机号 / 初始密码（≥8 位）**；登录邮箱须邮箱格式校验；缺任一或格式错 → 表单内联回显错误，不落库。
2. **AC2（联系手机号入库，L0/L1）**：`vet_accounts` 增列 `contact_phone`（运营私下联系用，**非登录凭证**）；`VetAccount` 实体加字段；建号写入。
3. **AC3（建号可登录但不可接单，L1）**：新号立即可登录工作台（账号 `ACTIVE`）；建号后置资质 `PENDING_COMPLETION`（调 Story 2.1 `VetQualificationService.ensureForVet(newVetId)`），**不可接单**（2.1 门控）。
4. **AC4（凭证一次性显示，L1）**：初始密码界面**仅显示一次**（开户成功回显），**绝不写入审计 summary / 日志 / 列表**；运营手动复制交付。
5. **AC5（邮箱唯一，L0/L1）**：登录邮箱唯一；已被占用 → 校验拒绝并提示「该登录邮箱已被占用」（复用既有 `existsByUsername`，username 即登录邮箱）。
6. **AC6（写审计 + 门控，L1）**：开户成功在同事务调 `AdminAuditService.record(actorAccountId, "VET_CREATED", "VET_ACCOUNT", vetId, summary)`（summary 含显示名/邮箱，**不含密码/手机号明文**）；端点受 `@PreAuthorize("hasAuthority('vet.create')")` 门控。
7. **AC7（回归绿，L0/L1）**：既有列表/重置/封禁/IM 建号（5.1：`imClient.ensureAccount`）不回归；`mvn -B compile` + admin 单测全绿；本地 L1 全量回归绿（迁移 validate）。

## Dev Notes

### 架构约束（必须遵守）
- 写操作**强制审计**：同事务经 `AdminAuditService.record(...)`（Story 1.3 提供），`action_type` UPPER_SNAKE 过去式 `VET_CREATED`；**审计/日志绝不含一次性密码、手机号明文**。[Source: architecture.md#Communication Patterns / Enforcement；Format Patterns「一次性敏感值」]
- 授权 `@PreAuthorize("hasAuthority('vet.create')")`（Story 1.5 权限基础，附录 B）。[Source: PRD 附录 B；architecture.md#Process Patterns]
- 凭证人工交付，V1.0.0 **不做自动凭证发送**（PRD A-1 已确认）。[Source: PRD#AB-2B]

### 既有代码基线（READ 过，brownfield）
- **现状**：`CreateVetForm`(`displayName/username/password`，无手机号)；`AdminVetService.create(displayName, username, rawPassword)`→`VetAccountService.create(...)`（`existsByUsername` 唯一校验 + BCrypt encode + 落库）+ IM `ensureAccount(vetImId, displayName)`（失败不阻断）；`VetAccount`（无 `contact_phone`，无 email 列——**登录字段是 `username`**）；`AdminWebController#createVet`（`@Valid CreateVetForm` → create → 回显 `createdVetId`）；`vets.html` 开户卡（昵称/登录账号/初始密码）。
- **本故事改什么**：`CreateVetForm`(+contactPhone, username→@Email)、`vet_accounts`(+contact_phone，V38)、`VetAccount`(+contactPhone)、`VetAccountService.create`(+手机号)、`AdminVetService.create`(+手机号 + ensureForVet)、`AdminWebController#createVet`(+审计 +门控)、`vets.html`(+手机号字段)。
- **不可破坏**：① IM `ensureAccount` 兜底逻辑保留（建号不因 IM 失败回滚）；② 唯一校验复用 `existsByUsername`（不另建 email 列）；③ 既有 `createdVetId` 回显与表单清空保留；④ 重置/封禁/列表端点不动。

### 关键边界 / 防坑
- **「登录邮箱」沿用既有 `username` 列，不新增 email 列、不改列名**：Epic 5 兽医登录用 `vet_accounts.username`（`VetAccountService.authenticate` by username、`BannedVetFilter`、JWT subject 均依赖它）。改名/加列会牵连兽医登录与既有数据。本故事**仅在语义与 UI 上把 `username` 当作登录邮箱**（表单加 `@Email` 校验、标签「登录邮箱」），列保持 `username`。在 Dev Notes/Completion 记录此决策，2.4 编辑同此口径。
- **手机号非登录凭证**：`contact_phone` 仅运营联系用，**不参与认证、不唯一、不进 JWT/日志**；列 nullable（存量兽医无值）。
- **密码回显边界**：密码是运营在表单**自己输入**的明文（非系统生成），故「一次性显示」对建号而言=运营本就知道；**严禁**把密码写进 `createdVetId` 之外的任何回显/审计/日志。重置密码（2.4）才是系统生成需谨慎回显的场景。
- **资质门控依赖 2.1**：`ensureForVet` 必须在 2.1 合入后调用；若并行开发，注入 `VetQualificationService` 并标注依赖。
- `contact_phone` 长度别用 1；用 VARCHAR(32)（含国家码/分隔符余量）。

### 范围边界（不做）

- 本故事**只**交付：手机号字段/列 + 邮箱格式&唯一校验 + 建号置待完善资质 + 凭证一次性 + 审计 + vet.create 门控。
