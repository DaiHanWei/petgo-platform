# Story 4.3: 人工审核队列预建（未激活）

Status: ready-for-dev

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

> **范围域**：管理后台 V1.0.0 · Epic 4（内容审核）· Story 4.3（AB-3C）。**预建但 V1.0.0 默认未激活**：建表 + 开关 + 完整激活后逻辑，但开关默认关、行为与现网（FR-12 自动审核拦截直接发布失败）一致。
> **产物归属**：仅后端 `petgo-backend`（**无 Flutter 侧**）。SSR + HTMX，路由 `/admin/**`，**不返 JSON**。激活属跨 App+后台契约（AG-3），V1.0.0 不触发。
> **无前向依赖**：4.1 → 4.2 → 4.3 互不依赖；可独立实现。
> **AB-3D 已移除**：所有内容（含官方/种子账号发布）一律经三方自动审查，**无种子豁免**——本故事不引入 `users.is_seed_account`、不做标记 UI。

## Story

As a **超级管理员**,
I want **预建人工审核队列与一个超管可控的激活开关，并把激活后的完整流程接好但默认关闭**,
so that **将来招到审核人手后一键开启而无需改架构；V1.0.0 维持现网「自动审核拦截即发布失败」行为不变**。

## Acceptance Criteria

> 验证层：**L0** 编译/单测/MockMvc 切片（无需 DB）· **L1** 集成（Docker postgres+redis，验迁移 validate / 开关分支 / 超时定时 / 门控真跑）· **L2**（本故事无）。

1. **AC1（建表，L0 编译 + L1 迁移）**：新增 Flyway **`V36__init_manual_review_queue.sql`** 建 `manual_review_queue`（`content_id` / `submitted_at` / `status`(PENDING|APPROVED|REJECTED|TIMED_OUT) / `decided_by` / `decided_at` / 时间戳）与 **`V37__init_admin_settings.sql`** 建 `admin_settings`（**单行配置表**，含 `manual_review_enabled BOOLEAN NOT NULL DEFAULT false` + 时间戳）；JPA 实体与列**精确匹配**（`ddl-auto=validate` 启动不报错）；枚举落库 varchar UPPER_SNAKE。
2. **AC2（默认未激活＝现网行为，L1）**：`admin_settings.manual_review_enabled=false`（默认）时，内容发布走**现网 FR-12 路径**——三方自动审核 `TEXT_BLOCKED/IMAGE_BLOCKED` 即**直接发布失败、不落库、不进队列**（`ContentService.publish` 现有行为**不变**）；人工审核队列页**不在导航展示 / 不可达**。
3. **AC3（激活开关，仅超管，L1）**：超管可在设置页切换 `manual_review_enabled`；开关读写经 `AdminSettingsService`；切换写审计（`SETTING_CHANGED`）。开关 UI 与队列入口**仅超管可见/可达**（`@PreAuthorize` SUPER_ADMIN）；普通账号 403。
4. **AC4（激活后入队挂起，L1）**：`manual_review_enabled=true` 时，未过自动审核（verdict != PASS）的内容**改为入队挂起**——内容以「审核中」态落库（`PostStatus.UNDER_REVIEW`，不进 Feed/公开口径）+ 写一条 `manual_review_queue(content_id, PENDING)`；**用户侧显示「内容审核中，通常 3 天内完成」**（用户侧 App 呈现属跨产品配套，后端返回该挂起态语义；见 Dev Notes AG-3）。
5. **AC5（运营处置，L1）**：队列页（持权限运营）对一条 PENDING：**通过** → 内容转 `PUBLISHED` + 发既有 `ContentPublishedEvent`（进 Feed）+ 通知作者「已通过」+ 队列置 `APPROVED`；**拒绝** → 内容丢弃（软删/不发布）+ 通知作者「未通过」+ 队列置 `REJECTED`；均写审计 + 记 `decided_by`/`decided_at`。
6. **AC6（3 天超时自动丢弃，L1）**：`@Scheduled` 定时扫描（`@Async` + DB 去重，**禁中间件**）将 `submitted_at` 超 **3 天**仍 PENDING 的项置 `TIMED_OUT` + 丢弃内容 + 通知作者；仅在 `manual_review_enabled=true` 时实际执行处置（关闭时定时器空转/早返回）。
7. **AC7（超 24h 高亮，L1）**：队列列表对 `submitted_at` 超 **24h** 未处理的 PENDING 项视觉高亮（模板条件类）。
8. **AC8（门控，L0/L1）**：队列查看/处置端点 `@PreAuthorize("hasAuthority('content.takedown')")`（或运营审核权限，见 Dev Notes）；激活/停用开关 + 队列入口仅 `SUPER_ADMIN`。
9. **AC9（回归绿 + 现网零影响，L0/L1）**：开关默认关 → `ContentService.publish` 行为与所有既有内容测试**完全不回归**；`mvn -B compile` + L0 全绿；本地 L1 全量回归绿（含 V36/V37 迁移 validate）。

## Dev Notes

### 架构约束（必须遵守）
- 后台 = SSR MVC + HTMX，`/admin/**`，**HTML 片段不返 JSON**。[Source: architecture.md#Format Patterns]
- 包 `com.tailtopia.admin.moderation.{web,service,domain,repository,dto}`；控制器 `XxxAdminController`。[Source: architecture.md#Naming Patterns; #Project Structure（moderation: ManualReviewItem/ReviewStatus/AdminSettings/ManualReviewService/ReviewTimeoutScanner）]
- **新表 snake_case + Flyway 从 V30 顺延（决策 E2）**：本故事用 **V36 manual_review_queue + V37 admin_settings**（与 sprint-status 分配一致；实际号以落库顺序为准，若 V32~V35 已被 1-3/1-4/2-1/2-9 占用则本故事顺延后两号）。枚举 UPPER_SNAKE；时间 timestamptz UTC。[Source: architecture.md#Data Architecture; sprint-status.yaml Flyway 分配]
- **定时仅 `@Scheduled`+`@Async`+DB 去重，禁调度/消息中间件（F5）**。[Source: architecture.md#API & Communication Patterns 定时任务; CLAUDE.md 护栏]
- 审计强制副作用、日志脱敏；`ddl-auto=validate`（schema 归 Flyway）。[Source: architecture.md#Enforcement Guidelines]

### 既有代码现状（READ 过，brownfield 基线）
- `content/service/ContentService.publish`：发布写库**前**调 `ContentModerationService.moderate(text, imageUrls)`，verdict `TEXT_BLOCKED/IMAGE_BLOCKED` → **直接 throw `AppException`（不落库、不进队列）**，`PASS` → 落库 + 发 `ContentPublishedEvent`。**这是 FR-12 现网路径，AC2 默认必须保持。本故事在此分支接开关。**
- `content/service/ContentModerationService`：V1 stub（关键词 + 图像 marker），真实三方后接；**本故事不改审核判定，只改「未过之后怎么办」**。
- `content/domain/PostStatus`：当前**仅 `PUBLISHED`**；本故事加 `UNDER_REVIEW`（列 varchar(16)，无需迁移）。Feed/我的发布查询均 `status=PUBLISHED`，故 UNDER_REVIEW 天然不公开（确认所有公开口径不取它）。
- `content/event/ContentPublishedEvent`：发布领域事件（里程碑等订阅）；approve 时复用以进 Feed/触发既有副作用。
- `notify/service/NotificationService.send(...)` + `NotificationType`：通知作者出口；本故事加 `CONTENT_REVIEW_APPROVED`/`CONTENT_REVIEW_REJECTED`（varchar 枚举，无需迁移）。
- `notify/repository/ScheduledPushMarkRepository` + V22 `scheduled_push_dedup`：**既有 `@Scheduled`+DB 去重范式**，`ReviewTimeoutScanner` 参照（不引中间件）。
- 当前 Flyway 最大 = `V31`；本故事建 **V36/V37**（中间 V32~V35 归 1-3/1-4/2-1/2-9，按执行顺序先行占用）。

### 依赖方向（关键防坑）
- `ContentService`（content 模块）需要「查开关 + 入队挂起」，但 **content 不应反向依赖 admin 包**。处理：
  - 在 **content 模块定义一个出站端口接口**（如 `ManualReviewGate { boolean enabled(); void enqueue(long contentId); }`），由 **admin.moderation 提供实现 bean** 注入；content 只依赖接口。或
  - 用领域事件：content verdict != PASS 且需挂起时发事件，admin 订阅入队（但「内容是否落 UNDER_REVIEW」需同事务决定，接口式更直接）。
  - 推荐**接口式**（同步、同事务、可单测 mock）；保持「跨模块经 service/接口、禁跨 repo」。dev 可在 Completion Notes 记录最终选型。
- `admin.moderation` 调 content 的状态变更（approve/discard）经 `ContentService` 新方法，**禁直读 content_posts**。

### 关键边界 / 防坑
- **V1.0.0 默认未激活是硬要求**：`admin_settings` 种子行 `manual_review_enabled=false`；**所有「激活后」逻辑必须 dormant**——publish 默认 throw（现网）、队列页不可达、定时器空转。AC9「现网零回归」是本故事最重要验收点。
- **激活是跨产品契约（AG-3）**：开关打开会改变 App 用户侧发布体验（从「发布失败」变「审核中挂起」），需 App 侧配合呈现「内容审核中，通常 3 天内完成」。**V1.0.0 不激活、不阻塞**；真实激活时另起 App+后台协调故事（已记 Deferred）。本故事后端返回挂起态语义即可，不要求 App 改动。
- **超时定时去重**：多实例/重入用 DB 去重（参照 scheduled_push_dedup），不用内存锁、不引中间件。
- **AdminSettings 单行**：固定主键（id=1）或唯一约束保证单行；读时缺省按 false（防种子缺失）。
- **门控分层**：队列处置（通过/拒绝）= 运营权限（`content.takedown` 或新审核权限）；**激活/停用开关 + 队列入口可见性 = 仅 SUPER_ADMIN**（AB-3C「激活/停用仅超管可见」）。两者别混。
- **无种子豁免（AB-3D 移除）**：不写 `is_seed_account`、不做豁免分支；官方/种子内容同样走自动审核（与现网一致）。

- **门控** `@PreAuthorize`（Story 1.5）：`content.takedown` + `SUPER_ADMIN` 隐式全权；开关仅 SUPER_ADMIN。

- **V36 `manual_review_queue`** + **V37 `admin_settings`**（admin_settings 插入单行种子 `id=1, manual_review_enabled=false`）。实体 `@Column` 名/可空/长度与 DDL 精确一致（validate）。中间号若被上游故事占用则两表顺延为最新两号（决策 E2，单调分配）。

### 范围边界（不做）
