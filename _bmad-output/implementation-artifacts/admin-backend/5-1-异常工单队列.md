# Story 5.1: 异常工单队列

Status: ready-for-dev

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

> **范围域**：管理后台 V1.0.0 · Epic 5（问诊异常与会话查询）· Story 5.1（AB-4A）。**建立在 Epic 1 地基（admin_accounts / 服务端会话 / `AdminAuditService` 哈希链审计 / permission authority 门控 / i18n）与 Epic 2 封禁流（Story 2.5 发 `ConsultAnomalyRaisedEvent`）之上**。
> **产物归属**：仅后端 `petgo-backend`（**无 Flutter 侧**）。brownfield 扩 `com.tailtopia.admin`，**新建 `com.tailtopia.admin.anomaly` 包**。SSR（Thymeleaf）+ HTMX，走 `/admin/**`，**不返 JSON**。
> **数据边界（NFR5，硬约束）**：问诊消息正文 / AI 上下文 / 用户媒体储存于第三方腾讯 IM，本故事**绝不跨系统拉取**，工单只承载 TailTopia 系统内的会话元数据。

## Story

As a **运营成员**,
I want **处理因兽医被封禁而自动生成的问诊异常工单（仅元数据 + 内部备注 + 处理图 + 归档）**,
so that **能在 48h 内（AM-2）追溯并处置受影响的问诊，且全程不触达第三方消息正文**。

## Acceptance Criteria

> 验证层：**L0** 编译/单测/MockMvc 切片（mock service/repo，无需 DB）· **L1** 集成（Docker postgres+redis，验事件落工单/去重/门控/审计/迁移）· **L2** 外部凭证（处理图上传走 OSS 私密桶 + 签名 URL，留本地/真凭证手测）。

1. **AC1（建表，L0 编译 + L1 迁移）**：新增 Flyway `V38__init_consult_anomalies.sql` 建 `consult_anomalies`；对应 JPA 实体字段/列**精确匹配**（`ddl-auto=validate` 启动不报错）。枚举落库 `varchar` + UPPER_SNAKE；时间 `timestamptz` UTC；`session_id` 上唯一索引（一会话至多一工单，AC8 去重的库级兜底）。
2. **AC2（事件落工单，L1 + L0 listener 单测）**：`ConsultAnomalyService`（`admin.anomaly.service`）以 `@Async @EventListener` 订阅 Story 2.5 封禁流发出的 `ConsultAnomalyRaisedEvent`；每个事件落一条工单，**仅写元数据**（`session_id` / `user_id` / `vet_id` / `session_started_at` / `session_ended_at` / `session_status` / `anomaly_type` = `VET_BANNED`），`status=OPEN`。
3. **AC3（仅元数据，绝不读第三方，L0/L1）**：工单实体/DTO/模板**不含**任何 IM 正文、AI 分诊上下文、用户媒体字段；列表与详情仅展示会话元数据；不调用任何 IM / AI 拉取接口（NFR5）。
4. **AC4（内部备注，L1 + 审计）**：运营在工单详情可添加内部文字备注（用户不可见）；写操作经 `AdminAuditService.record(ANOMALY_NOTE_ADDED, anomalyId, summary)` 同事务落审计哈希链。
5. **AC5（标记已处理 + 处理图 + 归档，L1/L2）**：运营可「标记已处理」（选填上传处理过程图）→ 工单 `status=RESOLVED`、记 `resolved_by`/`resolved_at`、移入归档区；**已归档工单仍可被搜索/查看**；处理图存 OSS 私密桶（短签名 URL + EXIF 剥离，复用 `shared/media`），**库内只存对象 key，绝不存签名 URL**；写操作经 `AdminAuditService.record(ANOMALY_RESOLVED, anomalyId, summary)`（summary **不含**签名 URL/对象 key 之外的敏感内容）。
6. **AC6（工单不可删，L1）**：无任何删除工单的端点/UI；应用层不提供 DELETE 路径（与审计「只可追加」一致的姿态，AB-4A「工单记录不可删除」）。
7. **AC7（双入口可达 + 筛选，L0 视图 + L1）**：异常工单队列经**模块 4 导航**与**模块 1（兽医管理）**均可访问（两处侧栏入口指向同一 `/admin/anomalies`）；可按工单状态（待处理 OPEN / 已归档 RESOLVED）筛选，HTMX 片段返回匹配行；归档区可搜。
8. **AC8（去重幂等，L1）**：同一 `session_id` 重复事件（重放/并发）**不产生重复工单**（按 `session_id` upsert/存在即跳过 + 唯一索引兜底）；落工单失败有结构化日志（脱敏，不记 PII/健康数据）。
9. **AC9（权限门控，L0/L1）**：查看队列/详情 `@PreAuthorize("hasAuthority('consult.view_anomalies')")`；加备注/标记已处理 `@PreAuthorize("hasAuthority('consult.handle')")`；无对应 authority 访问返回 403（前端隐藏仅体验，A5）。
10. **AC10（回归绿，L0/L1）**：`mvn -B compile` + 既有 admin/consult 单测全绿；新增 listener/service/controller 单测；本地 L1 全量回归绿，新 bean 注入 `@SpringBootTest` 上下文无误。

## Dev Notes

### 架构约束（必须遵守）
- 后端 = Spring Boot 4 / Java 21 / Maven；新建包 `com.tailtopia.admin.anomaly.{web,service,domain,repository,dto}`。[Source: admin-backend/architecture.md#Project Structure → `anomaly/ {web,service,domain,repository,dto}`；line 250]
- 后台一律 `/admin/**` + HTML/HTMX 片段，**不返 JSON**；服务端会话 + CSRF（adminFilterChain，A1，Epic 1 已立）。[Source: architecture.md#AI Agent Guidelines line 367]
- **审计是强制副作用**：每个后台**写操作**同事务经 `AdminAuditService.record(...)` 写 `admin_audit_logs`（哈希链）；`action_type` UPPER_SNAKE 过去式。本故事新增动作码：`ANOMALY_NOTE_ADDED` / `ANOMALY_RESOLVED`。[Source: architecture.md#Implementation Patterns line 200；Story 1.3]
- **跨模块只经既有 service / 领域事件，禁跨 repo**：本故事消费 `consult` 模块发出的 `ConsultAnomalyRaisedEvent`，**不**直访 `consult` 任何 repository；兽医/用户展示名如需解析，经既有 `VetAccountService` / 用户侧 service（V1.0.0 工单展示以 ID 为主，名称解析可选）。[Source: architecture.md line 188/201/294；CLAUDE.md 强制护栏]
- 异步只用 `@Async`（+ `@EventListener`），**禁引入 MQ / 通用缓存 / 新中间件**。[Source: CLAUDE.md 强制护栏；architecture.md line 367]
- 权限 authority：`consult.view_anomalies`（查看）/ `consult.handle`（处理）；方法级 `@PreAuthorize`，前端隐藏仅体验（A5）。[Source: architecture.md line 182；Story 1.5]
- 敏感数据：处理图属高敏，私密桶 + 短签名 URL + EXIF 剥离 + 日志不落；审计/日志**绝不**含签名 URL / PII / 健康数据。[Source: architecture.md#NFR7 line 214；CLAUDE.md 命名映射链]

### 事件契约（与 Story 2.5 的反向依赖 —— 关键）
- `ConsultAnomalyRaisedEvent` **由 Story 2.5（封禁/解封兽医）产出**：封禁流复用既有 `ConsultInterruptService.interruptByVetBan(vetId)`（`IN_PROGRESS/PENDING_CLOSE → INTERRUPTED`），在**每个被中断会话**处，除既有 `ConsultInterruptedEvent` 外**再发一条** `ConsultAnomalyRaisedEvent`。建议事件载荷（record，置于 `consult.event`）：`sessionId / userId / vetId / sessionStartedAt(=createdAt) / sessionEndedAt(=interruptedAt/terminalAt) / sessionStatus(=INTERRUPTED) / anomalyType(=VET_BANNED)`。
- **执行顺序合规**：Epic 2 在 Epic 5 之前实现，故 5.1 编码时 2.5 已发布该事件（反向依赖在时间线上成立）。**若实现 5.1 时事件尚未就绪**，dev 须先在 `consult.event` 定义该事件并在 `ConsultInterruptService` 补发（属 2.5 的产出物，标注在 Completion Notes，勿改其它语义）。
- 本故事订阅用 `@Async @EventListener`（架构明示）；注意 `interruptByVetBan` 为 `@Transactional`，`@Async` 监听在独立线程消费 —— 工单落库与封禁事务**最终一致**即可（48h SLA，非强一致）；若需「封禁回滚则不建工单」更强保证，可改 `@TransactionalEventListener(phase=AFTER_COMMIT)` + `@Async`，dev 视实现权衡并在 Completion Notes 记录。[Source: ConsultInterruptService.java:36-51]

### 关键边界 / 防坑
- **NFR5 红线**：实体/DTO/模板**零** IM 正文、AI 上下文、用户媒体字段；评审时 grep 确认无任何 `imClient`/AI 拉取调用。
- **签名 URL 绝不入库/审计**：`consult_anomalies` 只存 `resolution_image_key`（对象 key）；展示时 `SignedUrlService` 现签；审计 summary 不含签名 URL。
- **去重**：`session_id` 唯一索引是兜底，service 层 `existsBySessionId` 是首道；`@Async` 并发可能撞唯一约束 → 捕获 `DataIntegrityViolationException` 当作「已存在」吞掉，不抛。
- **不可删**：不写任何 `@DeleteMapping`/删除 service 方法（AC6），归档=状态切 `RESOLVED` 而非删行。
- **双入口同页**：两个侧栏链接指向**同一** `/admin/anomalies`，避免重复实现/路由分叉。
- **用户匿名化兼容**：注销（D1）会把会话 `user_id` 置空（`ConsultSession.anonymize`），故工单 `user_id` 可空，展示需容空。[Source: consult/domain/ConsultSession.java:167-171]

### Flyway（V38 —— 顺延说明）
- 架构早期示意把 anomaly 表归在「V30+ 指示性序号」，**但序号按 sprint 实际执行顺序单调分配（决策 E2，勿照搬示例号）**：Story 1.1 已占用 **V30/V31**；后续 **V32/V33** 归 1.3/1.4（审计/会话日志），**V34** 归 2.1（兽医资质），**V35** 归 2.9（失败请求），**V36/V37** 归 4.3（人工审核队列 + admin_settings）。故本故事 anomaly 表实际顺延为 **V38**。实现时以 `db/migration/` 实测最大号 +1 为准（若前序 story 的实际占号有出入，按单调顺延取下一个空号，并在 Completion Notes 记录）。[Source: CLAUDE.md 实现纪律 §4；Memory: prd-r2-rework-delta-and-frozen-flyway]
- DDL 草案（列长度勿用 `VARCHAR(1)`，Hibernate6 会映 CHAR(1) 触发 validate 全红）：
```sql
-- V38__init_consult_anomalies.sql
CREATE TABLE consult_anomalies (
  id                   BIGINT GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
  session_id           BIGINT       NOT NULL,
  user_id              BIGINT,                          -- 注销匿名化后可空
  vet_id               BIGINT       NOT NULL,
  session_started_at   TIMESTAMPTZ,
  session_ended_at     TIMESTAMPTZ,
  session_status       VARCHAR(16)  NOT NULL,           -- 中断时会话状态快照（INTERRUPTED）
  anomaly_type         VARCHAR(32)  NOT NULL,           -- VET_BANNED
  status               VARCHAR(16)  NOT NULL,           -- OPEN | RESOLVED
  internal_note        TEXT,                            -- 内部备注（用户不可见）
  resolution_image_key VARCHAR(255),                    -- OSS 对象 key（绝不存签名 URL）
  resolved_by          BIGINT,                          -- admin_accounts.id
  resolved_at          TIMESTAMPTZ,
  created_at           TIMESTAMPTZ  NOT NULL DEFAULT now(),
  updated_at           TIMESTAMPTZ  NOT NULL DEFAULT now(),
  CONSTRAINT ck_consult_anomalies_status CHECK (status IN ('OPEN','RESOLVED')),
  CONSTRAINT ck_consult_anomalies_type   CHECK (anomaly_type IN ('VET_BANNED'))
);
CREATE UNIQUE INDEX uq_consult_anomalies_session ON consult_anomalies (session_id);
CREATE INDEX idx_consult_anomalies_status ON consult_anomalies (status);
```
- 实体 `@Column` 名/可空性/长度必须与 DDL 精确一致（`ddl-auto=validate`）；主键生成策略与既有 admin 迁移（V30）保持一致。
