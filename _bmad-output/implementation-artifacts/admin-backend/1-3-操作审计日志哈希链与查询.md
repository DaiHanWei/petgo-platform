# Story 1.3: 操作审计日志（哈希链 append-only）与查询

Status: ready-for-dev

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

> **范围域**：管理后台 V1.0.0 · Epic 1（后台地基）· Story 1.3。**建立在已完成的 Story 1.1/1.2 之上**（admin_accounts + 服务端会话 + 紧急账密 + Lark OAuth + adminFilterChain）。
> **产物归属**：仅后端 `petgo-backend`（**无 Flutter 侧**）。brownfield 扩 `com.tailtopia.admin`。
> **横切定位**：本故事交付的 `AdminAuditService.record(...)` 是 **Epic 2~6 所有后台写操作都要调用的横切件**（架构 §Communication Patterns「审计写入是强制副作用」），务必做成统一、易复用、难绕过的单一入口。

## Story

As a **后台账号**,
I want **所有后台写操作被不可篡改地记录，并能按维度筛查审计日志**,
so that **每个运营决策可追溯、且任何篡改可被检出（AM-C1 操作日志完整性零漏记录）**。

## Acceptance Criteria

> 验证层：**L0** 编译/单测/MockMvc 切片（mock repo，无需 DB）· **L1** 集成（Docker postgres+redis，验迁移/哈希链/append-only/门控真跑）· **L2** 外部凭证（本故事无 L2）。

1. **AC1（建表，L0 编译 + L1 迁移）**：新增 Flyway `V32__init_admin_audit_logs.sql` 建 `admin_audit_logs`（`id` / `actor_account_id` / `action_type`(varchar UPPER_SNAKE) / `target_type`(varchar) / `target_id`(varchar，外露用不可枚举值或业务 id 字符串) / `summary` / `created_at`(timestamptz UTC) / `prev_hash` / `row_hash`）；JPA 实体 `@Column` 与列精确匹配（`ddl-auto=validate` 启动不报错）。

2. **AC2（统一审计入口 + 哈希链，L0 + L1）**：提供统一 `AdminAuditService.record(actorAccountId, actionType, targetType, targetId, summary)`；每次写入：取链尾 `row_hash` 作为本行 `prev_hash`（首行 `prev_hash` 为约定创世值），`row_hash = SHA-256(canonical(prev_hash, actorAccountId, actionType, targetType, targetId, summary, createdAt))`；同一事务内随业务写操作落库。**各控制器/服务禁止自拼 SQL 或绕过此入口**（架构 Enforcement）。

3. **AC3（并发串行化，L1，AG-2）**：审计链写入串行化——取链尾时对链尾加锁（悲观行锁 / `SELECT ... FOR UPDATE` 链尾、或数据库 advisory lock 单写入路径），保证并发写入下 `prev_hash→row_hash` 链不断裂、无竞态空洞（后台写并发极低，正确性优先）。

4. **AC4（append-only 不可篡改，L1，A6）**：任何对 `admin_audit_logs` 的 UPDATE/DELETE 被拒——以 **DB 触发器**在 `BEFORE UPDATE OR DELETE` 抛异常阻断（迁移内建），且审计仓库接口**不暴露** `delete*`/改写方法（窄接口，仅 append + 查询）；链可被独立校验器逐行重算 `row_hash` 验证未被篡改。

5. **AC5（查询页 + 权限门控，L0 门控 + L1）**：`/admin/audit-logs` 审计查询页**仅 `SUPER_ADMIN` 或持 `admin.view_logs` 权限的账号可见**；无此权限的后台账号访问 → 403。〔需求修正 2026-06-29：AB-1B 由「全员可查」改为权限受限。STAFF 的 `permission_code` authority 由 Story 1.5 装载，本故事先以 `hasRole('SUPER_ADMIN') or hasAuthority('admin.view_logs')` 表达式门控——1.5 落地前实际仅超管可见，前向兼容、无前向依赖。〕

6. **AC6（筛选 + 永久保留，L1）**：查询页支持按**日期范围 / 操作人（actor account）/ 操作类型（action_type）**筛选；筛选提交走 HTMX 局部刷新结果区（`HX-Request` 返片段，整页请求返完整视图）；结果按 `created_at` 倒序分页；**永久保留不设年限**（无清理任务）。

7. **AC7（接入 1.1 紧急登录 hook，L1）**：Story 1.1 预留的「紧急账密成功登录 → 写审计 + 告警全体超管」在本故事接入——紧急账密（formLogin）登录成功时写一条 `EMERGENCY_LOGIN_SUCCEEDED` 审计，并向全体 `SUPER_ADMIN` 发后台告警（复用既有 notify/后台预警机制，不引入新中间件）；**审计/告警内容绝不含密码/令牌/签名 URL**（架构「一次性敏感值」护栏）。〔Lark 登录成功不属紧急入口，不在本 AC；会话登录/登出/过期审计归 Story 1.4。〕

8. **AC8（回归绿，L0/L1）**：`mvn -B compile` + L0 全绿；本地 L1 全量回归绿；既有端点（含 1.1/1.2 登录路径、种子发帖、举报处理）不回归。

## Dev Notes

### 架构约束（必须遵守）
- 后端 = Spring Boot 4 / Java 21 / Maven；包 `com.tailtopia.admin.audit.{web,service,domain,repository,dto}`。[Source: admin-backend/architecture.md#Project Structure → admin/audit]
- 后台 = SSR MVC + HTMX，路由 `/admin/**`，**返回视图/HTML 片段不返 JSON**；CSRF 开启；错误用友好页/内联条非 ProblemDetail。[Source: architecture.md#API & Communication Patterns / Format Patterns]
- **审计写入是强制副作用**：每个后台写操作必须在同一事务内经 `AdminAuditService.record(...)` 写一条审计（哈希链 `prev_hash→row_hash`）；`action_type` UPPER_SNAKE **过去式**；统一入口，**禁各控制器自拼**。[Source: architecture.md#Communication Patterns / Enforcement]
- **A6 不可篡改**：append-only + 应用无 UPDATE/DELETE 权（窄仓库接口）+ DB 触发器 + 行间哈希链（篡改可检）。[Source: architecture.md#Authentication & Security A6 / Data Architecture]
- 时间 `timestamptz` UTC；枚举/动作 varchar UPPER_SNAKE；新表 snake_case 复数 `admin_` 前缀。[Source: architecture.md#Naming Patterns]
- 日志/审计 **绝不含**一次性密码、令牌、签名 URL、健康/PII。[Source: architecture.md#Enforcement / Format Patterns「一次性敏感值」]

### 既有代码基线（READ，1.1/1.2 产出）
- `SecurityConfig.adminFilterChain`（shared/security）：session + form + CSRF + `hasRole("ADMIN")` + `AdminLoginThrottleFilter` + `AdminSessionGuardFilter` + 失败/成功处理器。**本故事在 formLogin successHandler 接入紧急登录审计（AC7）**；⚠️ 该文件含仓库既有「法律页」未提交 WIP，提交时**仅纳入 admin 审计增量**。[Source: SecurityConfig.java:57-99]
- `AdminUserDetails`（admin/service）：principal，携 `adminAccountId`/`operatorUserId`/`accountType`，authorities=`ROLE_ADMIN`(+`ROLE_SUPER_ADMIN`)。**细粒度 `permission_code` authority 尚未装载（Story 1.5 做）**——故本故事审计页门控用 `hasRole('SUPER_ADMIN') or hasAuthority('admin.view_logs')`，1.5 前实际仅超管命中。[Source: AdminUserDetails.java]
- `AdminAccountRepository`（admin/account）：`findByLarkEmail` / `findById` / `countByAccountType`。需「全体 SUPER_ADMIN」列表用于 AC7 告警 → 可加 `findByAccountType(SUPER_ADMIN)` 查询方法。[Source: AdminAccountRepository.java]
- 当前 Flyway 最大 = `V31`（admin_account_permissions）；本故事新表从 **V32** 起。[来源：`db/migration/` 实测]
- 既有后台告警/notify：复用 `notify` 模块或既有后台预警路径（READ `com.tailtopia.notify` 现状后择最小侵入方式）；**不引入 MQ/缓存层**。

### 关键边界 / 防坑
- **窄仓库接口防误删**：`JpaRepository` 默认暴露 `delete*`——审计仓库应改继承 `org.springframework.data.repository.Repository<AdminAuditLog, Long>` 并只声明 `save` + 查询，或显式不调用任何删除；DB 触发器是最后防线（即便代码误删也被 DB 拒）。
- **哈希链串行化（AG-2）**：取链尾必须加锁，否则并发两写可能读到同一 `prev_hash` → 分叉。后台写并发极低，用悲观锁/advisory lock 即可，勿引入队列中间件（F5）。
- **同事务**：`record(...)` 须随业务写操作在**同一事务**提交——业务回滚则审计回滚（避免悬挂审计）；本故事自身的紧急登录审计无业务事务，独立提交即可。
- **successHandler 取 bean**：Lambda successHandler 内需 `AdminAuditService`/`AdminAccountRepository`/告警服务——经方法参数注入或 `http.getSharedObject(ApplicationContext.class)` 取，避免循环依赖。
- **summary 脱敏**：紧急登录 summary 只记「邮箱 + 时间 + 来源」语义，**绝不**记密码/失败明文；其他动作的 summary 同样禁敏感值。
- **HTMX 片段 vs 整页**：依 `HX-Request` 头区分；筛选/分页返 `audit-logs :: rows`，直接访问返完整视图（`layout :: page`）。

### 范围边界（不做 / 避免前向依赖）
- **本故事不做**：会话登录/登出/过期审计落库（→ Story 1.4 `admin_session_logs`）；STAFF `permission_code` authority 装载与账号管理 UI、超管≤5、SUPER_ADMIN 隐式全权 PermissionEvaluator（→ Story 1.5——本故事门控表达式已前向兼容）；Epic 2~6 各业务写操作对 `AdminAuditService` 的实际调用（仅**定义** `record` 入口 + 预留 action_type 常量，调用在各业务故事）；双语外化（→ 1.6，新页文案先中文）。
- 本故事**只**交付：审计表/实体（V32）+ 哈希链 append-only 服务 + 查询页（权限门控 + 筛选 HTMX）+ 接入 1.1 紧急登录 hook。
