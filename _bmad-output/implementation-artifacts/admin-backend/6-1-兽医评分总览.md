# Story 6.1: 兽医评分总览

Status: ready-for-dev

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

> **范围域**：管理后台 V1.0.0 · Epic 6（兽医评分质量看板）· Story 6.1（AB-6A）。**纯只读看板**，与 6.2 互不依赖（无前向依赖）。
> **产物归属**：仅后端 `petgo-backend`（**无 Flutter 侧**）。brownfield 扩 `com.tailtopia.admin`。
> **brownfield 现状**：既有只有「单兽医评分」页（`/admin/vets/{id}/ratings`，Story 5.6）。本故事**新增**一张「全兽医评分总览」页（之前不存在），并复用既有 consult 评分读路径聚合。
> **隐私边界**：V1.0.0 评分**不对 App 用户/兽医公开**，仅运营可见（沿用 `ConsultRatingQueryService` 的「仅运营可见」语义）。

## Story

As a **运营成员**,
I want **在一张总览页看到所有兽医的评分均分、已评/未评问诊数，并能排序与按日期范围筛选**,
so that **我能横向监控整体服务质量、快速识别表现异常的兽医（FR-33 / AB-6A）**。

## Acceptance Criteria

> 验证层：**L0** 编译/单测/MockMvc 切片（mock service，无需 DB）· **L1** 集成（Docker postgres+redis，验聚合/筛选/排序真跑）· **L2**（本故事无）。

1. **AC1（总览页与字段，L0/L1）**：新增 `GET /admin/ratings`，渲染所有兽医一行一条：兽医显示名、均分（1-5，保留一位小数，无评分显示「—」或 0.0）、**已评问诊数**、**未评问诊数**。页面为只读，**无任何写操作/按钮**（除筛选与排序控件）。
   **Given** 多个兽医且各自有/无评分
   **When** 打开 `/admin/ratings`
   **Then** 每个兽医一行，均分/已评数/未评数与底层 consult 数据一致；零评分兽医也出现在列表（均分占位、计数为 0）
2. **AC2（已评/未评口径，L0/L1）**：**已评数 = 该兽医 `consult_ratings` 行数**（每会话至多一条，`uq_consult_ratings_session`）；**未评数 = 该兽医 `CLOSED` 且无对应评分的会话数**（即进入过评分门、最终无评分）。
   **Given** 某兽医有 N 条评分、M 个 CLOSED 但无评分的会话
   **When** 加载总览
   **Then** 已评=N、未评=M（口径见 Dev Notes「已评/未评定义」，含 `补评分` 与 `INTERRUPTED` 的处理约定）
3. **AC3（排序，L0/L1）**：支持按**均分升序/降序**与**问诊量**（已评+未评之和，或显式「总问诊量」列）排序，经 query 参数（如 `?sort=avg_desc|avg_asc|volume_desc`）切换；默认排序在 Dev Notes 约定。
   **Given** 总览列表
   **When** 选择某排序
   **Then** 行顺序按该维度正确重排（HTMX 片段刷新或整页重渲均可，与既有 admin 页风格一致）
4. **AC4（日期范围筛选，L1）**：支持 `from`/`to` 日期范围筛选；筛选后均分/已评/未评仅统计**会话发生时间落在区间内**的问诊；不传则统计全时段。
   **Given** 选定一个日期区间
   **When** 提交筛选
   **Then** 三项统计均只反映区间内的问诊（口径单一，见 Dev Notes「日期口径」）
5. **AC5（隐私与门控，L0/L1）**：端点经 `@PreAuthorize("hasAuthority('rating.view')")` 门控（横切依赖 Story 1.5）；评分数据**绝不**出现在任何 App 用户/兽医可达接口；模板入口按 `rating.view` 隐藏（仅体验）。
   **Given** 无 `rating.view` 权限的后台账号
   **When** 访问 `/admin/ratings`
   **Then** 方法级拒绝 403（前端隐藏非安全边界）
6. **AC6（跨模块边界，L0）**：聚合**只经既有/扩展的 consult `service` + `vet` service 接口**完成，`com.tailtopia.admin.rating` **绝不**直接注入 `consult.repository` / `vet.repository`（禁跨 repo，沿用基线边界）。
7. **AC7（回归绿，L0/L1）**：既有 `/admin/vets/{id}/ratings`（5.6）与全部既有端点不回归；`mvn -B compile` + L0 全绿；本地 L1 全量回归绿。

## Dev Notes

### 架构约束（必须遵守）
- 后端 = Spring Boot 4 / Java 21 / Maven；新增包 `com.tailtopia.admin.rating.{web,service,dto}`（架构已为 AB-6 预留此包 + `AdminRatingService`「读 consult 评分」）。[Source: admin-backend/architecture.md#Project Structure（admin/rating）；§Backend Module Map「AB-6 评分看板 → admin/rating（只读） → vet-ratings」]
- 路由一律 `/admin/**`，**绝不**落 `/api/v1`；总览页用名词复数 `/admin/ratings`；筛选/排序走 query 参数。[Source: architecture.md#Routing]
- 跨模块**只经既有业务 service 或领域事件，禁跨 repo**（沿用基线边界）。本故事 admin 侧不得注入 consult/vet 的 repository。[Source: architecture.md#Project Structure「跨模块只经既有各业务 service…禁跨 repo」]
- 授权 = 方法级 `@PreAuthorize("hasAuthority('rating.view')")`（A5，权威）；模板隐藏仅体验。`SUPER_ADMIN` 隐式全权（由 1.5 的 PermissionEvaluator 处理）。[Source: architecture.md#Authentication & Security A5；§Routing「permission_code」rating.view]
- **纯只读 → 本 Epic 不建表、不加 Flyway、不写操作审计**（无写操作即无 `AdminAuditService.record(...)` 落点）；仅靠 `rating.view` 门控。当前 Flyway 最大 = `V31`（Story 1.1）；本故事不应新增任何迁移。
- 异步/缓存/中间件：不涉及，禁引入。

### 既有代码现状（READ 过，brownfield 基线 — 现状 / 改什么 / 不可破坏）
- `admin/web/AdminWebController#vetRatings`（`GET /admin/vets/{id}/ratings`）→ `admin/vet-ratings.html`：**单兽医**评分页（Story 5.6）。**不可破坏**；本故事新增的是**总览**页，二者并存（总览每行可链到该单兽医页，作为通往 6.2 的入口）。
- `admin/service/AdminVetService#ratings(vetId)` → `ConsultRatingQueryService#forVet(vetId)` → `VetRatingsView(vetId, average, count, items[stars,comment,createdAt])`：既有单兽医读路径。本故事**复用其聚合思路**，但总览需「按兽医的计数 + 未评数 + 日期区间」，故在 consult 侧**新增只读读口**，不改既有 `forVet`。
- `consult/service/ConsultRatingQueryService`：仅注入 `ConsultRatingRepository`。**改什么**：扩展只读方法（或新增 `ConsultQualityQueryService`）以产出按兽医的 rated/unrated/avg 摘要；未评数需要 `ConsultSessionRepository`，故新服务可同时注入两个 consult repo（**同模块内**，不破边界）。
- `consult/repository/ConsultRatingRepository`：已有 `findByVetIdOrderByCreatedAtDesc`、`existsBySessionId`、`findBySessionId`、`findByUserId`。**可新增**只读聚合方法（count/按时间区间）。
- `consult/repository/ConsultSessionRepository`：已有 `findByVetIdAndStatusInOrderByCreatedAtDesc(vetId, [CLOSED,INTERRUPTED])`。**可新增** `count` 类只读方法用于未评数。模块边界注释写明「仅 consult.service 直接访问」——**保持**：admin 不得直接用它。
- `vet/service/VetAccountService#listAll()`：兽医全量清单（含 displayName）。总览的兽医行来源；经 `AdminVetService.list()`（→ `VetAdminView`）或直接 `VetAccountService.listAll()` 均可，**不直访 `VetAccountRepository`**。
- `consult/domain/ConsultSession`：终态相关 —— `closeRated()`→`CLOSED`+`closedReason=RATED`；`closeUnrated()`→`CLOSED`+`closedReason=UNRATED`（30min 评分门超时）；`interrupt(...)`→`INTERRUPTED`（封禁中断，**不评分**）。`terminalAt()` 是历史/排序的单一时间口径。
- `consult/domain/ClosedReason`：`RATED`（用户评分关闭）/ `UNRATED`（30min 超时未评自动关闭）。

### 🟡 已评/未评定义（关键，避免 agent 误统计）
- **已评数 = `consult_ratings` 行数**（`uq_consult_ratings_session` ⇒ 每会话至多一条）。注意：`UNRATED` 关闭后用户仍可能**补评分**（`rating_prompt_state=PENDING`→用户补评），此时该会话会**新增一条评分行**——这类会话应计入**已评**（以「是否存在评分行」为权威，**不**以 `closed_reason` 为权威）。
- **未评数 = `status=CLOSED` 且无对应评分行的会话数**（即进入过评分门、最终仍无评分）。实现上：CLOSED 会话总数 − 该兽医已评数，或 `count(CLOSED 且 NOT exists rating)`，二者一致即可。
- **`INTERRUPTED`（封禁中断）会话**：从不进入评分流程。**总览的「未评」默认不计入中断会话**（中断属问诊异常域，归 Epic 5 工单，不应污染质量分母）。若 PRD 评审坚持计入，需在 Completion Notes 记录并回填 PRD —— 本故事按「未评=CLOSED 无评分」实现，**不杜撰**。
- 均分口径沿用既有 `VetRatingsView`：评分行 stars 的算术平均，四舍五入一位小数；无评分行显示占位。

### 🟡 日期口径（单一，避免双口径漂移）
- 日期区间作用于**问诊会话的发生时间**，建议统一以**会话 `created_at`**（或 `terminalAt()`，二选一并在代码注释固定）为口径，让「已评/未评/均分」三项基于**同一会话集**统计，避免「评分按 rating.createdAt、会话按 session.createdAt」造成的口径漂移。
- 实现简化：若以会话集为口径，已评数可由「区间内 CLOSED 会话中存在评分的数量」得出；均分由这些会话对应的评分计算。V1 数据量小（≤500 DAU、兽医少），可接受 join/逐会话判断；但请在读口注释写清口径，便于后续核对。
- 默认（不传 `from`/`to`）= 全时段。

### 关键边界 / 防坑
- **N+1**：逐兽医调 `forVet` 再逐兽医数会话会放大查询；V1 兽医数少可接受，但请在 `AdminRatingService` 注释标注，必要时用一次性 `group by vet_id` 聚合查询（仍在 consult.service 内）。
- **零评分兽医必须出现在总览**（AC1）：兽医清单来自 vet 侧，评分/会话用 left-join 语义补 0，不能因无评分而漏行。
- **隐私**：评分仅运营可见——切勿在任何 `/api/v1` 路径或 App 可达响应里泄露均分/评价（沿用 `ConsultRatingQueryService` 注释的「仅运营可见」）。
- **门控就位**：`@PreAuthorize("hasAuthority('rating.view')")` 依赖 Story 1.5 的 `@EnableMethodSecurity` 与 permission authority 装配。若 1.5 尚未落地，注解先就位（待 1.5 生效），同时**确保至少不弱于既有 `/admin/**` 的 `ROLE_ADMIN` 链门控**（不得让评分页比既有页更开放）。在 Completion Notes 标注门控生效层级。

### 范围边界（不做 / 避免过度实现）
- **本故事不做**：单兽医评分详情的未评原因明细（→ Story 6.2）、任何评分写/编辑/删除/导出、向 App 公开评分、数据分析大盘（DAU/留存/漏斗，PRD §5 明确不做）。
- **无新表、无 Flyway、无操作审计**（纯只读，本 Epic 统一约定）。
- 双语外化（`#{admin.*}`）统一在 1.6 收口，本故事新文案先中文。
