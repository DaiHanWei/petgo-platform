# Story 3.2: 停用与重新激活用户账号

Status: ready-for-dev

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

> **范围域**：管理后台 V1.0.0 · Epic 3（用户账号治理）· Story 3.2（AB-UA-02）。可逆处置：停用使账号即时不可登录 + 进行中会话强关 + App 停用提示文案；可重新激活。
> **产物归属**：主后端 `petgo-backend`（**无 Flutter 侧实现**；App 仅需展示后端返回的停用文案，属跨产品呈现，见 Dev Notes）。brownfield 扩 `com.tailtopia.admin.usermgmt`。
> **前置**：Epic 1（admin_accounts + 会话 + 门控 + 审计 1.3）。**与 3.1/3.3 各自独立**（3.1 详情页会展示本故事落地的「已停用」状态，但本故事不依赖 3.1）。

## Story

As a **客服/运营**,
I want **停用一个违规或申诉中的用户账号（即时不可登录、进行中问诊强关、App 显示停用文案），并能在需要时重新激活**,
so that **能即时执行处置且可恢复，同时每步都留下可追溯的审计记录**。

## Acceptance Criteria

> 验证层：**L0** 编译/单测/MockMvc（mock service，无需 DB）· **L1** 集成（Docker postgres+redis，验状态列迁移/登录阻断/会话强关/审计）· **L2** 外部凭证（无）。

1. **AC1（停用即时不可登录，L1）**
   **Given** 停用弹窗（二次确认 + **必填原因**）
   **When** 运营确认停用一个 `status=ACTIVE` 的普通用户
   **Then** 账号状态切 `DEACTIVATED`；该用户**即时不可登录 App**——后续 Google 登录与 token 刷新均被拒（既有 refresh 句柄一并失效）；列表/详情显示「已停用」。

2. **AC2（进行中会话强关，L1）**
   **Given** 被停用用户存在进行中问诊（`IN_PROGRESS`/`PENDING_CLOSE`）
   **When** 停用执行
   **Then** 其进行中会话**强制关闭**（经 consult owning service / 领域事件，禁 admin 直写 consult 表）。

3. **AC3（App 停用提示文案，L1）**
   **Given** 已停用用户尝试访问 App
   **When** 被拒
   **Then** 后端返回明确的停用语义（RFC 9457 ProblemDetail，App 侧据此展示），文案：「账号已被停用，如有疑问请联系客服：WhatsApp 081290906953 / 邮箱 cs@tailtopia.id」（用户已无法进入 App 内工单，须走外部渠道）。

4. **AC4（必填原因 + 审计，L1）**
   **Given** 停用为写操作
   **When** 提交
   **Then** 原因**必填**（空则校验拒绝）；同事务经 `AdminAuditService.record(...)` 写一条审计（`action_type=USER_DEACTIVATED`、target=userId、summary 含操作人 + 时间戳 + 原因摘要）；审计 append-only（A6）。

5. **AC5（重新激活，L1）**
   **Given** 已停用账号
   **When** 运营重新激活
   **Then** 状态切回 `ACTIVE`、恢复登录权；写审计 `action_type=USER_REACTIVATED`（含操作人 + 时间戳）。

6. **AC6（权限门控，L0）**
   **Given** 逐端点授权（A5）
   **When** 无停用权限的后台账号请求停用/激活端点
   **Then** 方法级 `@PreAuthorize` 拒绝 403；`SUPER_ADMIN` 隐式全权（权限码见 Dev Notes）。

7. **AC7（回归绿，L0/L1）**
   **Given** 新增状态列与登录阻断
   **When** `mvn -B compile` + L0 + 本地 L1 全量回归
   **Then** 全绿；既有 App 登录/刷新/问诊路径与既有 `/admin/**` 无回归；`ddl-auto=validate` 通过。

## Dev Notes

### 架构约束（必须遵守）
- 后台 SSR + HTMX，路由 `/admin/**`，写操作走 POST + CSRF；错误用友好页/内联错误条，**不外泄堆栈**（ProblemDetail 仅 `/api/v1`，App 停用文案走 `/api/v1` 的 ProblemDetail）。[Source: architecture.md#Format Patterns]
- **每个写操作必须经 `AdminAuditService.record` 写审计（哈希链 append-only，A6），禁控制器自拼**；`action_type` UPPER_SNAKE 过去式（`USER_DEACTIVATED`/`USER_REACTIVATED`）。[Source: architecture.md#Communication Patterns 审计强制副作用]
- 跨模块副作用走既有 service/领域事件，**禁 admin 直接操作 auth/consult 的 repository**。[Source: architecture.md#Architectural Boundaries 模块边界]
- 授权方法级 `@PreAuthorize`（A5）；`SUPER_ADMIN` 隐式全权。[Source: architecture.md#Process Patterns]
- 枚举 `varchar`+UPPER_SNAKE；时间 `timestamptz` UTC；加列走 ALTER（决策 E2）。[Source: architecture.md#Implementation Patterns；CLAUDE.md]

### 既有基线（READ — 现状 / 改什么 / 不可破坏）
- **`auth/domain/User`**：现状有 `role/deletedAt/passwordHash`，**无独立 `status` 列**；`deletedAt` 列存在但**注销时走物理删除（`AuthAccountDeletionService.deleteByUserId` → `users.delete()`），`deletedAt` 对 `users` 实际为 vestigial、未参与登录判定**。→ 本故事**新增专用 `status` 列**承载「停用/激活」（理由见下「Flyway 与方案选型」）。
- **App 登录路径**（`auth/service/AuthService`、token 刷新）：现状登录/刷新**未校验任何停用状态**（grep 确认无 deletedAt/status 拦截）。→ 在登录与刷新入口**新增** `status==ACTIVE` 校验，拒绝时抛停用语义异常。**不可破坏**正常用户登录/刷新/onboarding。
- **`auth/repository/RefreshTokenRepository`**：已有 `deleteByUserIdAndSubjectType(userId, SubjectType.USER)`（注销级联用）→ 停用复用之撤销令牌。
- **`consult/service/ConsultInterruptService`**：现状 `interruptByVetBan(vetId)`（兽医封禁强关，Story 2.5/Epic 5 复用），内部 `session.interrupt(InterruptReason.VET_BANNED)` + IM 处理。→ **对称新增 user 侧强关**（新增 `InterruptReason.USER_DEACTIVATED` 或复用合适枚举；经 service，不让 admin 直写）。
- **`usermgmt`**（3.1 若先落地则已建）：在其上加停用/激活编排与 web 端点。
- **审计 `AdminAuditService`（Story 1.3）**：截至本故事撰写，audit 子域尚未在代码库落地（`com.tailtopia.admin.audit` 未建）。→ 横切依赖：按 1.3 约定的 `record(actionType, targetType, targetId, summary)` 接线，与写操作同事务；若 1.3 未就绪，先以接口占位 + log 占位并在 Completion Notes 标注「审计落库待 1.3」（**不得**跳过审计设计点）。

### Flyway 与方案选型（停用如何阻断登录 — 给方案）
两种承载停用语义的方案，**本故事推荐方案 A（新增 `status` 列）**：
- **方案 A（推荐）：`users.status VARCHAR(16) DEFAULT 'ACTIVE'`（`ACTIVE|DEACTIVATED`）**。语义清晰、与「删除/注销」（物理删 + `deletedAt`）正交；可逆（停用↔激活）天然贴合枚举；登录/刷新只判 `status==ACTIVE`，删除走既有物理删路径，互不耦合。**Flyway 从 V38 顺延**：`V38__add_users_status.sql`（ALTER ADD COLUMN，决策 E2；当前最大迁移在 Epic 2 占用 V32~V37 之后，实际号实现时按 `db/migration/` 实测顺延，**勿照搬**）。
- **方案 B（不推荐）：复用 `users.deletedAt`** 作停用标记。否决理由：`deletedAt` 语义是「已删除」，且注销走物理删除使该列已半废弃；用它表达「可恢复的停用」会与删除/注销/注销重扫语义混淆，且重新激活需「清 deletedAt」反直觉、易出 bug。仅在团队明确拒绝加列时退而求其次，并须全局审清 `deletedAt` 的所有读点。

### 关键边界 / 防坑
- **登录阻断必须覆盖登录 + 刷新两个入口**：只拦登录、不拦刷新，会让已持 token 的停用用户继续用到过期；故停用须同时撤 refresh 句柄 + 刷新入口判 status。
- 停用文案走 App 的 `/api/v1` ProblemDetail（App 据此渲染外部联系渠道），**后台页本身**不弹该文案；后台错误用友好内联条。
- 强关会话经 consult service，**不要**在 admin 直接 `session.interrupt()` 或写 consult 表。
- 「停用」≠「删除」：本故事**不**触发任何级联删除/匿名化（那是 3.3）；仅切状态 + 阻断 + 强关 + 审计，数据全保留、可逆。
- 仅作用于 `role=USER` 的普通用户；不得停用后台账号（admin_accounts，那是 1.5）或兽医账号（那是 2.5 封禁，独立机制）。

### 范围边界（不做）
- 不做：删除 D1/D2（→ 3.3）、详情聚合页（→ 3.1，本故事只在其上加按钮/状态展示）、双语外化（→ 1.6）、App 侧停用弹层 UI 实现（跨产品，App 据 ProblemDetail 自行呈现）。
- 横切依赖：审计 `AdminAuditService`（1.3）、权限门控 `@PreAuthorize`（1.5，建议权限码 `user.deactivate`）、视觉/i18n（1.6）。

### 测试标准
- L0：`AdminUserService` 停用/激活（mock：状态切换、refresh 撤销、会话强关、审计写入、原因必填）；controller MockMvc（必填原因校验、确认弹层、403、CSRF）。
- L1（本地 docker PG+Redis）：V38 validate 绿；停用→Google 登录/刷新被拒（停用文案）+ 进行中会话强关 + refresh 失效；重新激活→可再登录；审计 append-only；全量回归绿。**云端 headless 只跑 L0；L1/迁移待本地验收**（CLAUDE.md）。
