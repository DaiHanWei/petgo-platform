# Story 2.1: 兽医资质数据模型与接单门控

Status: review

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

> **范围域**：管理后台 V1.0.0 · Epic 2（兽医账号治理）· Story 2.1（AB-2H 后端模型部分）。**建立在 Epic 1（admin_accounts/认证/审计）之上**，并扩展既有 Epic 5 兽医账号体系（brownfield）。
> **产物归属**：仅后端 `petgo-backend`（**无 Flutter 侧**）。本故事是 **Epic 2 的数据地基**：2.2 列表的「资质状态」列、2.7 资质录入审核、2.8 到期阻断均依赖本故事建的 `vet_qualifications` 与状态机。
> **brownfield 关键**：Epic 5 已建 `vet_accounts` + `VetAccountService` + 接单流程（`ConsultAcceptService`）。本故事**新增 1:1 的资质表**并在**既有接单路径**插入资质门控，**不改 vet_accounts 结构、不破坏既有兽医登录/接单**。

## Story

As a **平台**,
I want **兽医资质以状态机建模，并据此门控接单**,
so that **仅完成 SIPDH 合法执业认证（已认证/即将到期）的兽医可进入接诊队列**。

## Acceptance Criteria

> 验证层：**L0** 编译/单测/MockMvc 切片（无需 DB）· **L1** 集成（Docker postgres+redis，验迁移/门控真跑）· **L2** 外部凭证（OSS 真传证件图属 2.7，本故事无 L2）。

1. **AC1（建表，L0 编译 + L1 迁移）**：新增 Flyway `V34__init_vet_qualifications.sql` 建 `vet_qualifications`（`id` / `vet_account_id`(**唯一**，1:1，FK→`vet_accounts(id)`) / `ktp_no` / `ktp_photo_key` / `sipdh_no` / `sipdh_issuer` / `sipdh_expiry`(date) / `sipdh_photo_key` / `degree_photo_key` / `profile_photo_key`(nullable) / `pdhi_photo_key`(nullable) / `specialties`(jsonb，nullable) / `status`(varchar，UPPER_SNAKE) / `reject_reason`(nullable) / `created_at` / `updated_at`(timestamptz UTC)）；JPA 实体 `@Column` 与列**精确匹配**（`spring.jpa.hibernate.ddl-auto=validate` 启动不报错）。
2. **AC2（状态机枚举，L0）**：`QualificationStatus` 落库 varchar + UPPER_SNAKE，**6 态**且与 PRD 中文一一对应：`PENDING_COMPLETION`(待完善) / `UNDER_REVIEW`(审核中) / `CERTIFIED`(已认证) / `REJECTED`(已驳回) / `EXPIRING_SOON`(证件即将到期) / `EXPIRED`(证件已过期)。证件图字段存 OSS **私密桶对象 key**（绝不存签名 URL）。
3. **AC3（默认待完善 + 不可接单，L1）**：兽医账号创建后其资质默认 `PENDING_COMPLETION`（由 `VetQualificationService.ensureForVet(vetId)` 幂等保证 1:1 行存在）；该状态不可接单。
4. **AC4（接单门控，L1）**：兽医资质状态**非** `CERTIFIED`/`EXPIRING_SOON` 时**不得接单**——在既有接单路径 `ConsultAcceptService.accept(vetId, sessionId)` 调用 `VetQualificationService.canTakeConsult(vetId)`，不满足即拒绝（友好错误，不进 IN_PROGRESS、不占用会话、不消队列）；满足则照旧。
5. **AC5（回归绿，L0/L1）**：不改 `vet_accounts` 表结构；既有兽医登录、`VetAccountService`、`AdminVetService`、`ConsultAcceptService` 其余逻辑不回归；`mvn -B compile` + 既有 vet/consult/admin 单测全绿；本地 L1 全量回归绿（迁移 validate）。

## Dev Notes

### 架构约束（必须遵守）
- 后端 = Spring Boot 4 / Java 21 / Maven；包 `com.tailtopia.admin.vetqual.{domain,repository,service}`（架构目录树明列）。[Source: architecture.md#Project Structure → admin/vetqual]
- 业务实体表**不加 `admin_` 前缀**（`vet_qualifications` 属业务实体表）；snake_case 复数；枚举 varchar + UPPER_SNAKE；时间 timestamptz UTC。[Source: architecture.md#Naming Patterns（增量）]
- 仅「已认证/即将到期」可接单，**接单门控复用既有 vet 队列/接单逻辑读此状态**；跨模块只经既有 service，**禁跨 repo**。[Source: architecture.md#Data Architecture vet_qualifications；epics.md#Story 2.1]
- 证件图复用 `shared/media` **私密桶（MediaScope.PRIVATE）+ 短签名 URL + EXIF 剥离**；本表只存对象 key，签名/上传/读取在 Story 2.7 落（本故事仅定义 key 列）。[Source: architecture.md#敏感证件媒体；NFR7]

### 既有代码基线（READ 过，brownfield）
- **现状**：`vet/domain/VetAccount`（`id/username/passwordHash/displayName/status(ACTIVE|BANNED)/createdAt/updatedAt`，**无资质字段**）；`vet/service/VetAccountService`（create/resetPassword/setStatus/authenticate/listAll）；接单在 `consult/service/ConsultAcceptService.accept(vetId, sessionId)`（状态=WAITING → `markInProgress` CAS → IM 建会话 → 出队 → `presence.goBusy`）。当前接单**只校验会话状态 WAITING**，**无资质门控**。
- **本故事改什么**：(1) 新增 `vet_qualifications` 表 + 实体 + repository + `VetQualificationService`；(2) 在 `ConsultAcceptService.accept` 注入并调用 `canTakeConsult` 门控。
- **不可破坏**：① 不改 `vet_accounts` 表与 `VetAccount` 实体（资质独立 1:1 表）；② `ConsultAcceptService` 既有 CAS 抢单/IM/出队/置忙逻辑保持原序，门控仅前置一个拒绝分支；③ 既有兽医登录（`VetAccountService.authenticate`）与 `BannedVetFilter` 不动——资质门控**只挡接单，不挡登录**（PRD AB-2B：新号可登录、不可接单）；④ 既有 `AdminVetService`/`AdminWebController` 不动（资质状态展示在 2.2 列表故事接入）。

### 关键边界 / 防坑
- **1:1 行的创建时机**：账号创建时建 `PENDING_COMPLETION` 行（2.3 调 `ensureForVet`）；但**存量兽医**（Epic 5 已建、无资质行）需兜底——`canTakeConsult`/`getStatus` 对**无行**视同 `PENDING_COMPLETION`（不可接单），避免存量兽医因缺行报 NPE。是否给存量兽医批量补行由 2.7 录入时 `ensureForVet` 惰性补，不做数据迁移脚本（决策 E2 冻结迁移，勿在 V34 写 INSERT）。
- **接单门控点唯一权威 = accept**：用户侧「是否有兽医在线」用 Redis presence（不读资质），故无资质兽医可能仍显示在线——这是可接受的（接单时被挡）；如要更干净可在 goOnline 处加门控（T3 可选），但**勿在两处写不一致的判定**，统一调 `canTakeConsult`。
- **EXPIRING_SOON 仍可接单**（PRD：≤30 天仅预警不停接），故 `canTakeConsult` = `{CERTIFIED, EXPIRING_SOON}`，**勿误把即将到期挡掉**。
- 列长度别用 `VARCHAR(1)`（Hibernate6 → CHAR(1) 触发 validate 全红，见 [[hibernate6-char1-and-cloud-l0-gap]]）；`sipdh_expiry` 用 `date`（无时分，到期按日判定）。
- 资质字段在 `PENDING_COMPLETION` 阶段多为空 → DDL 中除 `id/vet_account_id/status/created_at/updated_at` 外**均 nullable**（待完善时未填）。
