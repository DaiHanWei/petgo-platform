# Story 1.5: 后台账号管理 UI（两级权限与生命周期）

Status: review

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

> **范围域**：管理后台 V1.0.0 · Epic 1（后台地基）· Story 1.5。**关键故事**——交付两级权限模型的授权根（`permission_code` authority + SUPER_ADMIN 隐式全权），是 **Epic 2~6 所有 `@PreAuthorize` 方法级门控的基础**（架构 A5、AM-C2 零越权）。
> **建立在 1.1/1.2/1.3 之上**：admin_accounts(V30) + admin_account_permissions 表(V31，**本故事补实体/仓库**) + 服务端会话即时撤权(`AdminSessionGuardFilter`) + `AdminAuditService`(1.3)。
> **产物归属**：仅后端 `petgo-backend`（**无 Flutter 侧**）。brownfield 扩 `com.tailtopia.admin`。

## Story

As a **超级管理员**,
I want **创建普通后台账号、分配模块级权限、停用/重新激活账号**,
so that **按岗位最小授权、离职/调岗即时撤权，且账号永不硬删以保审计完整（AM-C2 权限越权零事故）**。

## Acceptance Criteria

> 验证层：**L0** 编译/单测/MockMvc 切片（mock repo，无需 DB）· **L1** 集成（Docker postgres+redis，验权限装载/门控/生命周期/审计真跑）· **L2** 外部凭证（本故事无 L2）。

1. **AC1（权限实体补全，L0 编译 + L1 validate）**：为已存在的 `admin_account_permissions` 表（1.1 的 V31）补 JPA 实体 `AdminAccountPermission`（复合主键 `account_id` + `permission_code`）+ 仓库；`ddl-auto=validate` 启动不报错（**本故事无新迁移**，表已建）。

2. **AC2（authorities 装载 permission_code，L0/L1）**：`AdminUserDetailsService.loadByEmail` 装配 `AdminUserDetails.authorities` 时——**STAFF** 从 `admin_account_permissions` 加载其 `permission_code` 列表注入为 `SimpleGrantedAuthority`（与既有 `ROLE_ADMIN` 并存）；**SUPER_ADMIN 隐式全权**（注入全部 `permission_code` 集合，或经自定义 `PermissionEvaluator`/表达式判定，两选一并文档化）。密码/Lark 两登录路径都生效（authorities 在 `loadByEmail` 统一装载）。

3. **AC3（创建普通账号 + 分配权限，L1）**：超管在 `/admin/accounts` 填写成员 **Lark 邮箱**（必填、唯一）+ 勾选模块 `permission_code` → 创建 `account_type=STAFF`、`status=ACTIVE` 账号并写 `admin_account_permissions`；该邮箱即入白名单（=`admin_accounts` 中一行 ACTIVE，语义见 1.2），成员**下次 Lark 登录**即按权限生效，**未授权模块不可见/不可达**；操作写审计 `ACCOUNT_CREATED`（含被授权限摘要，不含敏感值）。

4. **AC4（超管上限 5，L0/L1）**：创建/提升为 `SUPER_ADMIN` 时**应用层**校验现有 `SUPER_ADMIN`（status=ACTIVE 计数）< 5；第 6 个被拒并友好提示。（计数口径：ACTIVE 的 SUPER_ADMIN；DISABLED 不占名额——在 Dev Notes 固化口径。）

5. **AC5（停用即时撤权 + 重新激活，L1，A1）**：超管停用任一账号 → `status=DISABLED`；**即时撤销其所有会话**——复用 1.2 已实现的 `AdminSessionGuardFilter`（每请求复查 status），无需新撤权机制；可重新激活恢复 `ACTIVE`；**账号不可永久删除**（无 delete 端点/方法）；停用与激活均写审计 `ACCOUNT_DEACTIVATED` / `ACCOUNT_REACTIVATED`。

6. **AC6（方法级门控权威，L0 切片，A5）**：受保护后台操作以 `@PreAuthorize("hasAuthority('<permission_code>')")`（SUPER_ADMIN 隐式通过）方法级门控；普通账号直接请求未授权模块端点 → **403**（前端隐藏入口仅体验，非安全边界）；本故事至少对**账号管理自身**端点门控（`admin.create_account` / `admin.deactivate`，SUPER_ADMIN 隐式全权），并验证 1.3 审计页 `admin.view_logs` 现可对被授权 STAFF 放行（权限装载后闭环）。

7. **AC7（编辑权限，L1）**：超管可调整既有 STAFF 账号的 `permission_code`（增/删勾选）；保存后**下次请求**该账号 authorities 随之变化（authorities 每次 `loadByEmail` 重算，或会话内变更需重登/守卫复查——在 Dev Notes 固化生效时机）；变更写审计 `PERMISSION_GRANTED` / `PERMISSION_REVOKED`。

8. **AC8（回归绿，L0/L1）**：`mvn -B compile` + L0 全绿；本地 L1 全量回归绿；既有登录/会话/审计（1.1~1.3）不回归。

- 后端 Spring Boot 4 / Java 21 / Maven；包 `com.tailtopia.admin.account.{web,service,domain,repository,dto}`；控制器后缀 `XxxAdminController`，返视图/HTML 片段不返 JSON。[Source: architecture.md#Project Structure / Naming Patterns]

### 关键边界 / 防坑
- **SUPER_ADMIN 隐式全权的实现选型**：① 登录时把附录 B 全部 permission_code 注入 authorities（简单、直观，新增权限需同步全集常量）；② 自定义 `PermissionEvaluator`/表达式 `hasRole('SUPER_ADMIN') or hasAuthority(...)`（不必维护全集）。**推荐 ②** 更抗遗漏，但任选其一须全栈一致（含 1.3/1.4 门控表达式）；固化在代码注释。
- **authorities 生效时机（AC7）**：authorities 在 `loadByEmail` 计算并随会话 principal 固化——**会话存续期内改权限不会自动生效**，新权限于「下次重新登录」或可在改权限后由守卫/逻辑强制该账号会话失效（更强，但 V1 可只承诺「下次登录生效」，与 1.2 白名单语义一致）；**固化承诺并在 UI 文案/Dev Notes 说明**，勿留歧义。
- **超管上限口径**：计 ACTIVE 的 SUPER_ADMIN；DISABLED 不占名额（停用一个超管后可再建）。固化为 `countByAccountTypeAndStatus(SUPER_ADMIN, ACTIVE) < 5`。引导期注意架构 A3「≥2 超管避免找回死锁」。
- **不可删除**：服务/控制器**不得**提供账号 delete；权限表的 `deleteByAccountId` 仅用于「重置该账号权限再重建」，与「账号硬删」无关，勿混淆。
- **唯一性**：`lark_email` 唯一（V30 uniq 约束）——创建前校验 + 捕获 DB 唯一冲突友好回显。
- **审计脱敏**：ACCOUNT_CREATED/PERMISSION_GRANTED 的 summary 记邮箱 + 权限码摘要即可，**不含**任何密码（STAFF 本无密码，走 Lark）。
- **HTMX 片段 vs 整页**：依 `HX-Request` 区分；危险操作（停用）二次确认。

### 范围边界（不做 / 避免前向依赖）
- **本故事不做**：动态 RBAC 角色表/`admin.manage_roles`（后续版本）；Epic 2~6 各业务端点的实际 `@PreAuthorize` 标注（各业务故事按本故事建立的 permission_code 体系自行标注——本故事只**确立机制**并对账号管理自身端点示范）；双语外化（→ 1.6，新页文案先中文）。
- 本故事**只**交付：权限实体/仓库 + authorities 装载（含 SUPER_ADMIN 隐式全权）+ 账号管理 UI（创建/改权限/停用/激活，写审计）+ 超管上限 5 + 不可删除 + 方法级门控机制与示范。
