# Story 2.7: 兽医资质录入与审核

Status: review

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

> **范围域**：管理后台 V1.0.0 · Epic 2 · Story 2.7（AB-2H UI/流程部分）。**依赖 Story 2.1**（`vet_qualifications` 模型与 6 态）。
> **产物归属**：仅后端 `petgo-backend`（**无 Flutter 侧**；兽医自传入口属兽医工作台/App，本故事只做后台审核侧）。SSR + HTMX，`/admin/vets/{id}/qualification`，不返 JSON。
> **brownfield 关键**：资质表/状态机由 2.1 建。本故事新增**运营直录 / 审核（通过·驳回）/ 续期**三条后台流程 + 证件图 OSS 私密桶签名/上传，并接审计。

## Story

As a **运营成员**,
I want **直接录入兽医资质（直接已认证），或审核兽医自传的资质（通过/驳回），并能为已认证账号续期**,
so that **兽医通过资质后方可接单，且证件高敏 PII 安全留存**。

## Acceptance Criteria

> 验证层：**L0** MockMvc 切片 + 单测 · **L1** 集成（Docker postgres+redis，验状态机/审计）· **L2** OSS 真传证件图（私密桶 + 签名 URL + EXIF 剥离）。

1. **AC1（运营直录→已认证，L1/L2）**：运营在后台填写完整资质字段（KTP/SIPDH 编号·机构·有效期 + 证件图 key）提交 → 状态直接切 `CERTIFIED`（已认证），无需额外审核；证件图存 OSS **私密桶**（`MediaScope.PRIVATE`，短签名 URL 读、EXIF 剥离），表内只存对象 key。
2. **AC2（审核通过/驳回，L1）**：兽医工作台自传资质后状态为 `UNDER_REVIEW`（审核中）；运营审核 → 通过：`CERTIFIED`；驳回：`REJECTED` 且 **必填驳回原因**（写 `reject_reason`）；兽医下次登录工作台见状态与原因（V1.0.0 无外部通知）。
3. **AC3（续期保持已认证，L1）**：已认证账号更新资质（如 SIPDH 续期：新有效期 + 新证件图）→ 保持 `CERTIFIED`（若原为 `EXPIRING_SOON`/`EXPIRED`，续期后回到 `CERTIFIED`）。
4. **AC4（证件图读取签名 URL，L1/L2）**：后台查看资质时，证件图经 `SignedUrlService.sign(key)` 现签**短 TTL** URL 展示；**绝不存签名 URL、绝不把签名 URL/证件 key 落日志**。
5. **AC5（写审计 + 门控，L1）**：录入/审核通过/驳回/续期均同事务调 `AdminAuditService.record(actorAccountId, "VET_QUALIFICATION_RECORDED"/"VET_QUALIFICATION_APPROVED"/"VET_QUALIFICATION_REJECTED"/"VET_QUALIFICATION_RENEWED", "VET_QUALIFICATION", vetId, summary)`（summary **不含证件号明文/key/签名 URL**）；端点 `@PreAuthorize("hasAuthority('vet.qualify')")`。
6. **AC6（接单门控联动，L1）**：录入/审核通过后兽医资质 `CERTIFIED` → 可接单（2.1 `canTakeConsult` 放行）；驳回/待完善 → 不可接单。
7. **AC7（回归绿，L0/L1）**：既有资质 service（2.1）/列表（2.2）不回归；`mvn -B compile` + 单测全绿；本地 L1 全量回归绿。

## Dev Notes

### 架构约束（必须遵守）
- 资质证件（KTP/SIPDH/学位证）高敏 PII：**私密桶 + 短签名 URL + EXIF 剥离 + 日志不落**（NFR7）；表内存对象 key，**绝不存签名 URL**（参考 `ConsultSession.aiImageRefs` 注释「绝不存签名 URL」）。[Source: architecture.md#敏感证件媒体；NFR7]
- 写操作强制审计，summary **不含证件号/key/签名 URL**；授权 `vet.qualify`（扩展附录 B 兽医管理权限组，Story 1.5 基础；附录 B 未列资质专项，本故事新增此 code 并标注）。[Source: architecture.md#Communication Patterns；PRD 附录 B]
- 状态机合法迁移集中在实体方法（service 调用），非法迁移抛 `AppException`（沿用 `ConsultSession` 风格）。[Source: petgo-backend consult 实体状态机范式]

### 既有代码基线（READ 过，brownfield）
- **现状**：2.1 建 `vet_qualifications` + `QualificationStatus`(6 态) + `VetQualificationService`(ensureForVet/getStatus/canTakeConsult)。`shared/media`：`PresignedUploadService.issue(MediaScope, userId, contentType)`（PRIVATE 仅签 PUT、对象私有、publicUrl=null）；`SignedUrlService.sign(objectKey)`/`signAll(keys)`（短 TTL 签名读）；`MediaScope{PUBLIC,PRIVATE}`；`MediaController` 提供直传 URL 端点。
- **本故事改什么**：扩 `VetQualificationService`（录入/审核/续期）+ 实体状态机方法 + 后台资质控制器/模板 + 接审计 + 证件图签名/上传接线。
- **不可破坏**：① 2.1 的 `canTakeConsult`/`getStatus`/`ensureForVet` 与接单门控不变（本故事只改状态值，门控自动联动）；② `shared/media` 既有签名/上传逻辑不动（复用接口）；③ 列表（2.2）读资质状态口径不变。

### 关键边界 / 防坑
- **EXIF/日志脱敏**：证件 key、签名 URL、KTP/SIPDH 号**严禁进 SLF4J 日志/审计 summary/异常消息**（高敏 PII）。summary 仅记「录入资质」「审核通过」「驳回（原因长度 N）」之类。
- **签名 URL 现签现用**：每次渲染详情现签短 TTL URL，**不缓存、不入库、不回传给非授权页**；HTMX 片段刷新也现签。
- **状态机驳回必填原因**：`reject(vetId, reason)` reason 空/空白 → 校验拒绝（web 层 + 实体双保险）；通过/续期清空旧 `reject_reason`（避免历史驳回原因残留误导）。
- **续期起点态**：续期允许从 `CERTIFIED`/`EXPIRING_SOON`/`EXPIRED` 切回 `CERTIFIED`（运营补新证件）；从 `PENDING_COMPLETION`/`UNDER_REVIEW`/`REJECTED` 应走「直录」而非「续期」（区分两入口，勿混）。
- **兽医自传入口不在本故事**：`UNDER_REVIEW` 态由兽医工作台/App 提交产生（跨产品）；本故事**只做后台审核侧**——若兽医自传链路尚未实现，L1 可由测试直接置 `UNDER_REVIEW` 态再审核（标注自传入口为他处/后续）。

### Flyway
- 本故事**无新迁移**（资质字段/状态/reject_reason 均在 2.1 的 V34）。如确需新列从 **V38 之后**顺延并说明（不预期）。

### 范围边界（不做）
- 不做兽医工作台自传资质 UI（跨产品/他处）、不做到期定时扫描与 30 天预警（→ 2.8）、不做外部通知（PRD：V1.0.0 无邮件/推送）、不做双语外化（→ 1.6，新文案先中文）。
- 本故事**只**交付：后台直录/审核（通过·驳回必填原因）/续期 + 证件图私密桶签名/上传 + 审计 + vet.qualify 门控 + 接单门控联动。
