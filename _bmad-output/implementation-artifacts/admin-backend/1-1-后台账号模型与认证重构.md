# Story 1.1: 后台账号模型与认证重构（admin_accounts + 紧急账密）

Status: review

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

> **范围域**：管理后台 V1.0.0 · Epic 1（后台地基）· 本故事是**一切后台操作的前置**（AG-1）。
> **产物归属**：仅后端 `petgo-backend`（**无 Flutter 侧**）。
> **brownfield**：复用并重构既有 `com.tailtopia.admin` Thymeleaf slice（Story 3.1 建）。**不新建工程、不上 SPA、不引入中间件。**

## Story

As a **超级管理员**,
I want **后台账号迁移到独立的 `admin_accounts` 表，并保留用户名+密码紧急登录入口**,
so that **后台身份与 App 用户/兽医完全隔离（符合 PRD 术语表），且 Lark 服务异常时仍有可用的备用入口**。

## Acceptance Criteria

> 验证层：**L0** 编译/单测/MockMvc 切片（无需 DB）· **L1** 集成（需 Docker postgres+redis，验迁移/门控/登录真跑）· **L2** 外部凭证（本故事无 L2）。

1. **AC1（建表，L0 编译 + L1 迁移）**：新增 Flyway `V30__init_admin_accounts.sql` 建 `admin_accounts`、`V31__init_admin_account_permissions.sql` 建 `admin_account_permissions`；对应 JPA 实体字段与列**精确匹配**（`spring.jpa.hibernate.ddl-auto=validate` 启动不报错）。枚举落库 `varchar` + UPPER_SNAKE。
2. **AC2（开户迁移，L1）**：`AdminBootstrap` 改为从 env（`ADMIN_BOOTSTRAP_EMAIL` + `ADMIN_BOOTSTRAP_PASSWORD`）幂等 upsert 一个 **`SUPER_ADMIN`** 账号到 `admin_accounts`（`status=ACTIVE`、`password_hash` BCrypt）；env 缺任一则不开户（无默认管理员）；**明文密码绝不入库/日志**（仅记 email + created/updated 事件）。
3. **AC3（认证迁移，L1）**：`AdminUserDetailsService` 改为按 `lark_email` 从 `admin_accounts` 加载、且 `status=ACTIVE` 且 `password_hash != null`；超管用「邮箱 + 密码」经既有表单入口（`/admin/login`）登录成功进入后台；**App 用户 / 兽医 / 已停用 / 未设密码账号一律不可登录后台**。
4. **AC4（紧急入口硬化，L1）**：紧急账密登录失败统一提示「账号或密码错误」（不区分字段）；连续失败触发**限流 + 账号锁定**（如同邮箱 N 次失败后短时锁定）。〔注：「成功登录写审计 + 告警全体超管」依赖审计基建，**移至 Story 1.3 落地**（见 Dev Notes 范围边界），避免对未建的 `admin_audit_logs` 形成前向依赖。〕
5. **AC5（旧路径保留停用 + 不破坏种子发帖，L1）**：旧 `users.role=ADMIN` 行与 `V7` 的 `users.password_hash` 列**保留、不删**（决策 E2），但后台登录**不再依赖**它；**既有 Story 3.1 种子内容发布必须仍可用**——种子帖内容作者（`content_posts.author_id → users.id`）的来源在本故事中改为一个明确的「官方内容作者 users id」，不再取自后台登录主体（见 Dev Notes「关键集成点」）。
6. **AC6（授权主体形状，L0/L1）**：`AdminUserDetails` 承载 `adminAccountId` + `accountType`；authorities 含 `ROLE_ADMIN`，超管额外含 `ROLE_SUPER_ADMIN`；登录后既有后台页（`/admin/dashboard` 等）仍受 `ROLE_ADMIN` 门控且可正常访问。（细粒度 `permission_code` authority 与 SUPER_ADMIN 隐式全权由 Story 1.5 落地。）
7. **AC7（回归绿，L0）**：改动涉及的既有消费方（`AdminUserDetails` 构造、`AdminWebController` 取作者、`AdminWebControllerTest` 等）一并更新；`mvn -B compile` 与既有 admin 单测全绿，无残留对 `users` 表的后台登录依赖。

## Dev Notes

### 架构约束（必须遵守）
- 后端 = Spring Boot 4 / Java 21 / Maven；包 `com.tailtopia.admin.account.{web,service,domain,repository,dto}`（本故事主要 domain/service/repository + 改 shared/security 与既有 admin）。[Source: admin-backend/architecture.md#Project Structure]
- 后台认证 = **服务端 HttpSession + 表单登录 + CSRF**（与 App 的无状态 JWT 相反，A1）；独立 `adminFilterChain`（`securityMatcher("/admin/**")`，`@Order(1)`）。[Source: architecture.md#Authentication & Security; SecurityConfig.java:52-75]
- 后台账号与 App `users`/`vet_accounts` **完全隔离**（F-A）；后台账号走 `admin_accounts`，**禁止**复用 `users(role=ADMIN)`。[Source: architecture.md#Data Architecture 【F-A 定案】]
- 凭证 env 注入**绝不入库/日志**；枚举 `varchar`+UPPER_SNAKE；时间 `timestamptz` UTC；新表 snake_case 复数。[Source: architecture.md#Implementation Patterns]
- 异步/定时只 `@Async`/`@Scheduled`+DB；**禁中间件**。本故事不涉异步。

### 既有代码现状（READ 过，brownfield 基线）
- `SecurityConfig.java`（shared/security）：`adminFilterChain` 已是 session+form+CSRF+`hasRole("ADMIN")`，登录页 `/admin/login` permitAll，成功跳 `/admin/dashboard`。`PasswordEncoder` = BCrypt bean 已在此。**本故事认证范式不变，只换数据源。**
- `AdminUserDetailsService`：当前 `users.findByEmailAndRole(email, ADMIN)` + 过滤 `deletedAt==null` + `passwordHash!=null`。→ 改为查 `admin_accounts`。
- `AdminUserDetails`：当前承载 `userId/email/passwordHash`，authorities 固定 `ROLE_ADMIN`。→ 改字段为 `adminAccountId`，加 `accountType` 与 `ROLE_SUPER_ADMIN`。
- `AdminBootstrap`（ApplicationRunner）：当前 upsert 到 `users`（`User.newAdmin`）。→ 改为 upsert 到 `admin_accounts`。
- `V7__add_admin_password.sql`：`ALTER TABLE users ADD COLUMN password_hash VARCHAR(255)`。**保留不动**。
- `templates/admin/login.html`：表单 `username`+`password` POST `/admin/login`，`param.error` 文案「账号或密码不正确」。本故事可不改（双语外化与 Lark 入口在 1.6/1.2）。
- 当前 Flyway 最大 = `V29`；新表从 **V30** 起。[来源：`db/migration/` 实测]

### 🔴 关键集成点（不处理会破坏现网）
**种子内容发布的作者来源**：`AdminWebController` 种子发帖当前用 `AdminUserDetails.getUserId()` 作为 `content_posts.author_id`（指向 `users.id`）。admin 迁出 `users` 表后，登录主体不再有 `users.id` → 直接断掉 Story 3.1 种子发布。
- **要求**：本故事必须保持种子发布可用（AC5）。推荐做法二选一：
  1. 引入配置 `ADMIN_SEED_AUTHOR_USER_ID`（env），种子发帖固定归属该官方 users 行；或
  2. 保留旧 bootstrap 的 `users(role=ADMIN)` 行作为「官方内容作者 shim」，由配置指向其 id。
- 无论哪种，**`content_posts.author_id` 必须是有效 `users.id`**（外键/语义），不能塞 `admin_accounts.id`。
- 验收：改完跑 Story 3.1 的种子发布 L1（或其单测）仍绿。

### 范围边界（避免前向依赖 / 过度实现）
- **本故事不做**：Lark OAuth（→1.2）、白名单即时撤权每请求校验（→1.2）、操作/会话审计表与哈希链（→1.3/1.4）、紧急登录的审计+告警超管（→1.3，仅在审计就绪后接）、普通账号创建 UI + 模块权限分配 + 超管≤5 校验 + SUPER_ADMIN 隐式全权 PermissionEvaluator（→1.5）、双语外化（→1.6）。
- 本故事**只**交付：账号表/实体 + 开户迁移 + 紧急账密认证迁移 + 登录硬化（限流/锁定）+ 不破坏既有后台页与种子发帖。
- MC-2（就绪报告）：若实现偏大，可在 dev 内部拆「T1-T3 认证迁移」与「T4 紧急入口硬化」两提交，但同属本故事。

### Flyway DDL 指引（V30 / V31）
```sql
-- V30__init_admin_accounts.sql
CREATE TABLE admin_accounts (
  id            BIGINT GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
  lark_email    VARCHAR(255) NOT NULL,
  display_name  VARCHAR(100) NOT NULL,
  account_type  VARCHAR(20)  NOT NULL,   -- SUPER_ADMIN | STAFF
  status        VARCHAR(20)  NOT NULL,   -- ACTIVE | DISABLED
  password_hash VARCHAR(255),            -- 仅超管紧急入口；STAFF 为 NULL
  created_by    BIGINT,
  created_at    TIMESTAMPTZ  NOT NULL DEFAULT now(),
  updated_at    TIMESTAMPTZ  NOT NULL DEFAULT now()
);
CREATE UNIQUE INDEX uq_admin_accounts_lark_email ON admin_accounts (lark_email);

-- V31__init_admin_account_permissions.sql
CREATE TABLE admin_account_permissions (
  account_id      BIGINT      NOT NULL REFERENCES admin_accounts(id),
  permission_code VARCHAR(64) NOT NULL,  -- 如 vet.view / content.takedown
  PRIMARY KEY (account_id, permission_code)
);
```
- ⚠️ 与既有约定对齐：主键策略与既有迁移一致（参考既有 `V13__init_vet_accounts.sql` 等的 id 生成方式，保持全库一致）；列长度别用 `VARCHAR(1)`（Hibernate6 会映射 CHAR(1) 触发 validate 全红，见 [[hibernate6-char1-and-cloud-l0-gap]]）。
- 实体 `@Column` 名/可空性/长度必须与 DDL 精确一致（`ddl-auto=validate`）。
