# Story 5.6: 会话结束机制与用户评分门

Status: review

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

## Story

As a **用户**,
I want **兽医结束会话后我有保护窗口并通过评分确认关闭**,
so that **服务质量得到保障，我不被过早切断对话**。

> 本 Story 在 5.5（IN_PROGRESS 进行中会话 + IM 会话）之上，落地**会话状态机的收尾段**：兽医结束 → `PENDING_CLOSE`（非立即关闭）、30min 保护窗口内用户可继续发消息 / 评分立即关闭 / 超时自动关闭记「未评分」、关闭触发 FR-16 存档（IM→OSS 桥接，Epic 2 承接落地档案）+ FR-33 评分（1-5 星必填 + ≤100 字选填，仅运营可见，超时未评下次补弹一次），Admin 评分查看。
>
> 做：兽医「结束会话」二次确认 → PENDING_CLOSE、用户侧评分门（30min 窗口、继续发消息、评分立即 CLOSED、超时自动 CLOSED 记未评分、下次补弹一次）、关闭发 `ConsultClosedEvent`（触发存档桥接 + 评分记录）、IM→OSS 存档桥接（复制聊天媒体到私密桶档案路径）、Admin 评分查看页。
> 不做：封禁中断（5.7，那是 INTERRUPTED 旁路）、问诊历史列表渲染（5.8，本 Story 产出历史/存档数据，列表在 5.8）、档案落地展示（Epic 2 profile 承接 FR-16，本 Story 只发事件 + 复制媒体）、推送本身（Epic 6，本 Story 发事件/IM 系统消息）。
>
> 依赖前序：5.5（IN_PROGRESS + `im_conversation_id` + `ImCallbackController` 骨架）、5.4（AI 上下文私密图，存档一并纳入）、5.1（Admin slice 承评分查看）。本 Story 落 `ConsultClosedEvent`，profile 模块（Epic 2）订阅做 FR-16 存档落地。

## Acceptance Criteria

> **验证层标注**：**L0**（状态迁移 / 评分校验 / lint） · **L1**（需 Docker + pg + redis：PENDING_CLOSE 迁移、30min 定时关闭、评分落库、补弹标记、Admin 查看、存档媒体复制到私密桶） · **L2**（需真实腾讯 IM + 真机：PENDING_CLOSE 期间继续发消息、IM 系统消息、存档实际拉 IM 媒体）。
> 本 Story 状态机收尾与评分主体在 **L1**；30min 窗口内继续对话与 IM 系统消息在 **L2**。存档桥接（IM→OSS 复制）在 **L1/L2**（拉真实 IM 媒体需 L2）。

### AC1 — 兽医结束 → PENDING_CLOSE（非立即关闭）

**Given** 兽医点击「结束会话」并二次确认
**When** 触发结束
**Then** 会话进入 PENDING_CLOSE「待关闭」状态（非立即关闭），用户侧收到「兽医已完成本次问诊，请为本次服务评分」（FR-31）
> 验证层：状态迁移 `IN_PROGRESS→PENDING_CLOSE` + 启动 30min 计时 = **L1**；兽医二次确认 + 用户侧 IM 系统消息「请评分」= **L2**。**注意**：PENDING_CLOSE 非立即关闭——会话仍可发消息（见 AC2）；兽医结束后回 ONLINE（解除 5.5 BUSY）可接新单。

### AC2 — 30min 保护窗口三分支（继续/评分/超时）

**Given** 30 分钟保护窗口内
**When** 用户行为分支
**Then** 用户继续发消息→兽医收推送可继续回复；用户评分（1-5 星必填 + ≤100 字选填）→会话立即 CLOSED 并触发 FR-16 存档；超时未评分→自动 CLOSED 记「未评分」（FR-31、FR-33）
> 验证层：① 继续发消息（PENDING_CLOSE 仍可收发，兽医收推送）= **L2**（IM）；② 评分 1-5 星必填 + ≤100 字选填 → `PENDING_CLOSE→CLOSED` + `ConsultClosedEvent`(触发存档)= **L1**（评分落库/迁移）+ **L2**（存档拉 IM 媒体）；③ 30min 超时无评分 → 自动 `CLOSED` 记 `unrated` = **L1**（DB 状态机定时）。评分校验（星必填、字数 ≤100）服务端权威 = **L1**。

### AC3 — 超时未评分下次补弹一次

**Given** 用户超时未评分
**When** 下次进入问诊页面
**Then** 补弹评分弹窗一次；仍未评分则该次不再要求、记「未评分」（FR-33）
> 验证层：`unrated` 会话标记 `rating_prompt_pending` → 下次进问诊 Tab(5.8)弹一次 → 弹后置 `prompted` 不再弹 = **L1**（标记流转）+ **L2**（弹窗 UI）。**多态**：补弹后用户仍可评（→CLOSED 已存档，仅补评分记录）或仍跳过（永久 unrated，不再弹）。

### AC4 — Admin 评分查看（不对普通用户公开）

**Given** 运营在 Admin 后台
**When** 查看兽医评分
**Then** 可见各兽医历史评分与平均分（V1 评分不对普通用户公开）（FR-33、G-1）
> 验证层：**L1**（Admin `role=ADMIN` 评分查看端点 + 各兽医历史评分/平均分聚合）。评分**仅运营可见**——普通用户/其他兽医 API 不返回他人评分；前端用户侧不展示兽医评分（5.8 历史显示的是「用户自己给的评分」，非兽医公开均分）。

### AC5 — 对话中 kill 断线保护 + 补评分推迟 [R2·回改]

> **代码已实现，本轮回改补齐**（F12 退出态：对话进行中 kill 的断线保护窗口与恢复；并补「补评分推迟」语义）。

**Given** 用户在对话**进行中**（IN_PROGRESS / PENDING_CLOSE 等未终态会话）
**When** 用户强制退出 App / kill 进程，或带「进行中会话」进入问诊页
**Then** 分两条：
- **① kill 断线保护与恢复**：对话进行中强制退出 App（kill 进程）→ 会话视为**用户断线**，进入既有 **30 分钟保护窗口**（复用 5.6 已有的 PENDING_CLOSE/保护窗口定时机制，**不新增定时器、不引入 MQ**）；用户重开 App 可从「问诊」Tab「进行中」入口**恢复会话**（会话连续、IM 会话不重建）；30 分钟内未返回 → 复用既有超时自动关闭（窗口到期自动 CLOSED）。状态机沿用 `WAITING/IN_PROGRESS/PENDING_CLOSE/CLOSED/INTERRUPTED/CANCELLED`（决策 E1），断线属保护窗口内的「用户暂时不在」，**非 INTERRUPTED**（INTERRUPTED 是 5.7 封禁旁路，两者文案/状态不混淆，见 5.7）。
- **② 补评分推迟（FR-33）**：用户进入问诊页面时若**有进行中会话**，补评分弹窗（AC3 的 `pending-rating` 补弹）**推迟到该进行中会话结束后**再弹——不在用户正处理活跃会话时打断（避免补弹覆盖在恢复的对话上）。
> 验证层：**L1**（断线 → 30min 保护窗口 → 重开恢复 的状态流转复用既有迁移；补评分推迟逻辑：有进行中会话则不返回/不触发补弹，会话结束后才放行）+ **L0**（「问诊」Tab「进行中」恢复入口 widget；补评分推迟判定的前端门控 widget 测）+ **L2**（真机 kill→重开恢复对话、IM 会话连续，留本地/真机）。**禁 MQ / 分布式锁/新中间件**：保护窗口与超时关闭走既有 DB 状态机 + @Scheduled 扫描（5.6 已建），断线不新增机制只复用。任务：后端（断线视为保护窗口可恢复 + 恢复查询 + 补评分推迟判定）+ 前端（「进行中」恢复入口 + 进页有进行中会话时推迟补弹）。

## Dev Notes

### 关键架构约定（本 Story 必须落实）

- **会话状态机收尾在 Java 侧**：`IN_PROGRESS →(兽医结束) PENDING_CLOSE →(评分/30min超时) CLOSED`（架构 §Communication Patterns）。30min 评分门是**状态迁移触发器**（架构明确「评分门(30min)为状态迁移触发器」），用 DB 状态机定时扫描，**禁 MQ**。
- **IM→OSS 存档桥接（架构关键跨云点）**：`ConsultClosedEvent → profile 订阅做存档桥接`；媒体「问诊存档时把所需图从 IM 复制一份到私密桶，档案只引用应用自有 URL（不引用会过期的 IM URL）」（架构 §Data Architecture 媒体三层桥接规则 / §Decision Impact「存档桥接(IM→OSS)连接 consult→profile」）。本 Story 落 consult 侧事件 + `ImToOssArchiver` 媒体复制；profile/Epic 2 订阅落档案。
- **评分仅运营可见（FR-33 / G-1）**：Admin slice 查看，普通用户/兽医不可见他人/公开均分。用户侧 5.8 历史显示的是「用户本人给的评分」，非兽医均分。
- **跨模块经事件**：consult 发 `ConsultClosedEvent`，profile（存档）、notify（推送）订阅，不互相直调 repository（架构 §模块边界）。

### 会话状态机专项

> 全机收尾：`IN_PROGRESS →(本Story 兽医结束) PENDING_CLOSE`。PENDING_CLOSE 是「待关闭保护窗口」——**非立即关闭**，30min 内可继续对话。三出口：① 用户评分 → `CLOSED`(RATED) + 存档；② 30min 超时 → `CLOSED`(UNRATED) + 补弹标记；③ （封禁在 5.7 是另一条 `IN_PROGRESS→INTERRUPTED` 旁路，不经 PENDING_CLOSE）。
> CLOSED 触发 `ConsultClosedEvent` → 存档桥接（IM→OSS 私密桶复制）+ 评分记录 + 历史可见（5.8）。存档是「额外操作」，问诊历史无论是否存档都保留（5.8）——本 Story 产存档数据，5.8 区分「是否已存档」标记。

### 强制护栏（架构 §Enforcement —— 违反即返工）

- **禁 MQ / 通用缓存 / 新中间件**：30min 超时 + 存档重试用 DB 状态机 + @Async。
- **存档只引用应用自有 URL**：档案不引用会过期的 IM URL；私密图只走签名 URL；日志不落签名 URL/健康图/PII。
- **跨模块经事件**：`ConsultClosedEvent` 发布，profile/notify 订阅；consult 不直调 profile/notify repository。
- **角色门控**：结束 `role=vet`、评分 `role=user`、评分查看 `role=ADMIN`；评分不对普通用户/兽医公开。
- **Redis 收窄**：本 Story 不新增 Redis 用途（BUSY→ONLINE 复用 5.2 在线态）。

### 范围边界（防 scope creep —— 本 Story 明确不做）

- ❌ 不做封禁 → INTERRUPTED（Story 5.7，那是不经 PENDING_CLOSE 的旁路）。
- ❌ 不做问诊历史列表渲染 / 只读会话页（Story 5.8，本 Story 产历史/存档数据 + pending-rating 接口）。
