# Story 6.2: 兽医回复与兽医端新请求推送

Status: review

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

## Story

As a **用户与兽医**,
I want **兽医回复时提醒用户、新请求时提醒在线兽医**,
so that **双方都能及时响应问诊**。

> 本 Story 在 6.1 已建的统一推送出口 + 深链路由表上，接入**问诊域两个推送场景**：① 用户侧——兽医回复推送（FR-22A）；② 兽医侧——新问诊请求推送给所有在线兽医（FR-22E）。
> **依赖**：推送下发/写库/角标/深链均复用 **Story 6.1**；事件源来自 **Epic 5（consult/vet 会话状态机 + 兽医在线态 + 待接单队列）**——本 Story 通过 consult 模块的领域事件触发 notify，**不在 notify 里直访 consult/vet 的 repository**。兽医「在线」判定（FR-32）、待接单队列态读 Redis（Epic 5 已建）。

## Acceptance Criteria

> **验证层标注**：**L0** 静态/单测（无需 DB） · **L1** 集成/运行时（Docker + postgres + redis） · **L2** 端到端/外部（真实 APNs/FCM 经腾讯 IM、真机、冷启动/后台/前台三态）。
> 推送**送达与点击直达**为 L2（真机）；「正在查看不重复推送」的抑制判定、在线兽医筛选、深链组装可在 L0/L1 验证。

### AC1 — 兽医回复推送（用户不在对话页时，FR-22A）

**Given** 兽医在会话中发送回复
**When** 用户未在当前对话页查看
**Then** 推送「你的宠物问诊有新回复，点击查看」，点击直达对应会话；用户正在查看该对话时不重复推送（FR-22A）
> 验证层：**L1**（兽医回复事件 → notify 写库 + 角标 +1 + IM 离线投递触发，需 pg+redis；「正在查看抑制」依据客户端在场态/会话焦点判定）+ **L2**（真机收到推送、点击直达对应会话页为端到端）。**L0**：抑制逻辑（当前激活会话 == 推送目标会话 ⇒ 不推）+ 深链组装（type=VET_REPLY + 会话 token）单测。

### AC2 — 兽医端新请求推送（仅在线兽医，FR-22E）

**Given** 用户提交兽医咨询请求
**When** 请求进入待接单队列
**Then** 向所有在线（FR-32）兽医推送「有新的问诊请求，点击查看」，离线兽医不接收，点击跳工作台「待接单」（FR-22E）
> 验证层：**L1**（提交请求 → 读 Redis 兽医在线态筛选在线兽医 → 逐个 notify 投递；离线兽医不入投递名单，需 redis 造在线/离线态）+ **L2**（在线兽医真机收推送、点击跳工作台待接单 Tab 为端到端）。**L0**：在线筛选（离线/忙态过滤）+ 深链组装（type=NEW_CONSULT_REQUEST → 工作台待接单 location）单测。

## Tasks / Subtasks

> 三段组织。本 Story 形态 = **后端中（事件订阅 + 在线筛选 + 推送触发）+ 前端中（在场态抑制信号 + 工作台待接单深链落点）**。执行顺序：后端 → 前端 → 联调。

### 🟦 后端子任务（petgo-backend / Spring Boot）

- [ ] **B1. 订阅兽医回复事件触发用户侧推送** (AC: 1)
  - [ ] consult 模块在兽医发送回复时发布领域事件（如 `VetRepliedEvent{sessionToken, recipientUserId}`）；`notify` 订阅 → `NotificationService.send(type=VET_REPLY, title/body 双语 key, deepLink=会话 token)`。
  - [ ] **抑制约定**：服务端默认照发；「用户正在查看该对话不重复推送」由 IM 在场态 / 客户端前台焦点判定为主（见 F1）。若 Epic 5 已维护「用户当前激活会话」信号，则后端可据此跳过——以不漏推为底线、以不在看时不打扰为目标。
- [ ] **B2. 订阅新问诊请求事件触发兽医侧推送** (AC: 2)
  - [ ] 用户提交咨询请求进待接单队列时，consult 发布 `ConsultRequestQueuedEvent{requestToken}`；`notify` 订阅 → 读 Redis 兽医在线态（FR-32，Epic 5 已建在线/忙/离线）筛**在线**兽医集合 → 逐个 `send(type=NEW_CONSULT_REQUEST, deepLink=工作台待接单)`。
  - [ ] **离线兽医不入投递名单**；忙态按 Epic 5 在线态语义处理（在线含忙则推、纯离线不推）。
  - [ ] V1 逐条投递，**不引入 MQ**；批量在线兽医循环走 `@Async`。
- [ ] **B3. 推送文案双语 key** (AC: 1, 2)
  - [ ] 文案走 i18n（用户侧「你的宠物问诊有新回复，点击查看」；兽医侧「有新的问诊请求，点击查看」），下发时按收件人语言本地化或下发 key 由客户端本地化（与项目 i18n 约定一致）。

### 🟩 前端子任务（petgo_app / Flutter）

- [ ] **F1. 用户侧「正在查看抑制」信号** (AC: 1)
  - [ ] consult 会话页激活时上报/标记当前激活会话；前台收到指向**同一会话**的推送时不展示 in-app Banner、不重复打扰（与 6.1 前台 Banner 分发协同）。非当前会话或后台/冷启动正常按 6.1 三态直达。
- [ ] **F2. 兽医回复推送深链落点** (AC: 1)
  - [ ] 复用 6.1 `pushPayloadToLocation`：`VET_REPLY` → consult 会话页（token 定位）；目标会话已关闭/中断走 6.1 多态兜底。
- [ ] **F3. 兽医端待接单深链落点** (AC: 2)
  - [ ] `NEW_CONSULT_REQUEST` → 兽医工作台（`features/vet`）「待接单」Tab/列表 location；点击后聚焦待接单视图。兽医为独立角色门控，深链续跳遵守 role 门控。

### 🟨 联调验收子任务

- [ ] **J1. 兽医回复推送端到端** (AC: 1 / L2)
  - [ ] 兽医发回复 + 用户在后台 → 用户真机收「有新回复」推送 → 点击直达该会话；用户正在该会话页时 → 不弹重复推送。
- [ ] **J2. 新请求推送在线筛选** (AC: 2 / L1+L2)
  - [ ] 造 2 在线 + 1 离线兽医，用户提交请求 → 仅 2 在线兽医真机收推送、离线兽医无；点击跳工作台待接单。
- [ ] **J3. 抑制 + 在线筛选单测** (AC: 1, 2 / L0)
  - [ ] 抑制判定（激活会话 == 目标会话 ⇒ 不推/不弹）+ 在线兽医筛选（离线排除）单测覆盖。

## Dev Notes

### 关键架构约定（沿用全局）

- 后端 `com.tailtopia.notify.{web,service,domain,repository,dto,event}`；事件源 consult/vet 经领域事件通信，**禁 notify 直访 consult/vet repository**。
- 前端 `lib/features/notify` + `lib/features/consult`（会话页在场态）+ `lib/features/vet`（工作台待接单落点）。
- 深链复用 6.1 `core/router/deep_link_routes` 映射，token 定位、隐藏 Tab Bar。

### 推送专项约定

- 下发统一经 6.1 `NotificationService` → `shared/im` 离线推送（底层 APNs/FCM），**不开新通道、不引 TPNS**。
- 兽医在线态（FR-32）+ 待接单队列态读 **Redis**（Epic 5 已维护），notify 只读不写在线态。
- 角标计数复用 6.1 `notify:unread:{userId}`。

### 强制护栏（违反即返工）

- **禁 MQ / 独立推送服务**；批量在线兽医投递走 `@Async` 循环。
- **禁** notify 跨模块直访 consult/vet repository——必须经领域事件 / owning service。
- **禁** 深链带顺序 id（用会话 token / 请求 token）。
- 日志严禁记录健康数据/会话内容/PII。

### 范围边界（本 Story 明确不做）

- ❌ 不做内容点赞/评论推送（Story 6.3）。
- ❌ 不申请推送权限（Story 6.4）。
- ❌ 不建/不改深链路由表与三态分发器本体（Story 6.1 已建，本 Story 仅复用）。
- ❌ 不实现会话状态机/兽医在线态/待接单队列本体（Epic 5——本 Story 仅订阅其事件、只读其 Redis 态）。
- ✅ 只做：兽医回复推送（含正在查看抑制）+ 新请求推送（在线兽医筛选）两个场景的事件订阅、推送触发与深链落点。

### Project Structure Notes

- notify 订阅 consult 发布的 `VetRepliedEvent` / `ConsultRequestQueuedEvent`（事件名以 Epic 5 实际定义为准，过去式命名）。
- 兽医端深链落点在 `features/vet` 工作台待接单视图，遵守 role 门控。
