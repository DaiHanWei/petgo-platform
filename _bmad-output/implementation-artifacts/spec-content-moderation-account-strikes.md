# 账号违规计数 + 注销默认隐藏联动（内容审核 story 9）

> 内容审核补充规范落地的第 9 个（末位）story。**权威源 = 方案 v1.0.1**（§5.2 / §5.4 / §5.5）；代码契约遇冲突以 `CROSS-STORY-DECISIONS.md` 为准。
> 本 story 后端为主，前端无用户可见面（运营后台展示见 story 8）。

---

## 1. 背景与范围

方案 §5.4 引入**账号维度违规数据记录**：运营每次判定内容违规时，系统按账号、按内容类型（帖子/评论/昵称/头像）分别累加一个违规计数，供运营人工参考优先级、并为**下一版本（V1.2.0）账号级处置功能**（如多次违规自动限权）预留数据基础。

**本 story 的核心边界（务必贯穿实现）：**

- **仅记录，不处置。** 本版本**不**基于该计数触发任何自动动作 —— 不警告、不限制发帖、不限权、不封号。计数是**只写只读的旁路数据**，不进入任何审核/发布/风控判定链路。写代码时严禁埋任何"计数达阈值就……"的分支（那是 V1.2.0 的事，方案 §7 待确认 #10）。
- **为 V2 预留。** 表结构与累加口径要能支撑未来"按类型阈值触发处置"，但本 story 到此为止：建表 + 接线累加 + 注销联动，不实现消费端。
- **注销联动**（方案 §5.4 注销处置决策）：用户注销时其内容**默认对其他人隐藏**、在人工队列中的条目随注销**移出队列不再发布**、违规计数依 D1/D2 账号注销级联策略一并处置。

**明确不在本 story 范围：**
- 账号级自动处置 / 阈值告警 / 限权（V1.2.0）。
- 账号举报 FR-58（V1.2.0）。
- 违规内容/审核日志的留存与访问控制落地（方案 §5.5，属审计基建横切，已有 `admin_audit_logs` 哈希链承载处置记录，本 story 不重复造）。

## 3. 目标与非目标

**目标：**
1. 新建 `violation_counts` 表：按 `(account_id, violation_type)` 累计违规次数（帖子/评论/昵称/头像分类型）。
2. 提供单一累加入口 `ViolationCountService.record(accountId, violationType)`，与处置状态迁移同事务、幂等累加。
3. 接线**现有**帖子处置点（FR-12A 拒绝/超时、举报下架、后台巡查下架）累加计数；为评论/名称/头像处置点定义调用契约（由 story3/4/5 落地）。
4. 注销联动：注销时用户内容默认对其他人隐藏、队列中条目移除不再发布、违规计数依 D1/D2 处置。

**非目标（明确不做）：**
- ❌ 不做任何**自动限权 / 阈值告警 / 账号级处置**（V1.2.0，方案 §7 #10）。计数只写只读，不进判定链。
- ❌ 不把**自动同步拦截**（评论 FR-55 命中 L1 关键词库 / 风险评分 ≥0.8）计入 —— 那不是人工判定（见 §5.1 计入口径）。
- ❌ 不改 `account_deletions`（V20）表结构、不改 UGC 匿名化"已注销用户"文案（Story 7-3 的 D1，与本 story 的"隐藏"是两层，见 §5.3）。
- ❌ 不引入 MQ / 缓存层 / 新中间件（护栏）。

## 4. 数据与迁移

### 4.1 新表 `violation_counts`（占位 `V55__init_violation_counts.sql`，实际号顺延）

按账号、按类型的**聚合计数器**（每个 `(account_id, violation_type)` 一行）。选聚合计数器而非逐条事件流的理由：处置的逐条审计证据已由 `admin_audit_logs`（story 1.3 哈希链、append-only）承载；本表只需支撑运营快速查看"某账号各类违规多少次"与 V2 阈值判断，聚合表最轻（V1 姿态，不过度设计）。

```sql
-- V<n>__init_violation_counts.sql  （占位号见 frontmatter，CI 落地顺延，勿硬编码 55）
CREATE TABLE violation_counts (
    id                  BIGINT GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    account_id          BIGINT      NOT NULL,          -- 违规内容所属 App 用户 id（users.id）
    violation_type      VARCHAR(16) NOT NULL,          -- UPPER_SNAKE 枚举，见 §4.2
    violation_count     INTEGER     NOT NULL DEFAULT 0,-- 累计人工判定违规次数（仅记录，不驱动逻辑）
    first_violation_at  TIMESTAMPTZ,                   -- 首次计入时间（UTC）
    last_violation_at   TIMESTAMPTZ,                   -- 最近一次计入时间（UTC）
    created_at          TIMESTAMPTZ NOT NULL DEFAULT now(),
    updated_at          TIMESTAMPTZ NOT NULL DEFAULT now(),
    CONSTRAINT uq_violation_counts_account_type UNIQUE (account_id, violation_type),
    CONSTRAINT ck_violation_counts_type CHECK (
        violation_type IN ('POST','COMMENT','NAME','AVATAR')
    )
);
CREATE INDEX idx_violation_counts_account ON violation_counts (account_id);
```

**约定落地要点：**
- `account_id` **不加 FK**到 `users`：注销走 D1/D2 可能删除 user 行（本 story §5.3 对计数的处置在删除前完成），加 FK 会与级联删除次序打架；靠应用层保证一致性（V1 姿态）。
- `violation_type` 用 `VARCHAR(16)`（**非 CHAR/非 length=1**，规避 Hibernate6 CHAR(1) validate 坑）；UPPER_SNAKE + CHECK 约束落库。
- 时间戳一律 `TIMESTAMPTZ` UTC；命名 `snake_case` ↔ 实体 `camelCase`（`violationCount` / `firstViolationAt` …）。
- `ddl-auto=validate`：实体字段与列一一对应，勿让 Hibernate 造列。

### 4.2 枚举 `ViolationType`（Java，UPPER_SNAKE 落库 varchar）

| 枚举值 | 覆盖面板 | 触发处置点（人工判定） |
|---|---|---|
| `POST` | 帖子正文+图片 | FR-12A 队列拒绝/超时丢弃、举报下架、后台巡查下架 |
| `COMMENT` | 评论与回复 | FR-55A 巡查下架（story3） |
| `NAME` | 用户昵称 + 宠物名 | 名称人工判定违规 → 重置默认编码名（story4） |
| `AVATAR` | 用户头像 + 宠物头像 | 头像人工判定违规 → 重置默认头像（story5） |

> 方案 §5.4 列"帖子/评论/昵称/头像"4 类，本 story 按此 4 类落枚举：`NAME` 合并昵称与宠物名、`AVATAR` 合并用户与宠物头像。若 V2 运营要区分 user/pet，再新增枚举值 + 放宽 CHECK（增量 ALTER，勿改本迁移 —— 已提交即冻结，决策 E2）。

## 5. 后端设计

### 5.1 计入范围口径（方案 §5.4，务必逐字对齐）

**只计入「人工判定」违规**，即需要运营/后台人工动作或队列超时判定的场景：

| 计入 ✅ | 触发点 | story |
|---|---|---|
| 帖子 FR-12A 人工队列**判定拒绝** | `ManualReviewService.reject` | story2/本 |
| 帖子 FR-12A 队列**超时自动丢弃** | `ManualReviewService.scanTimeouts → TIMED_OUT` | story2/本 |
| 已发布帖子经**举报人工判定下架** | `AdminModerationService.takedown` | story6/本 |
| 已发布帖子/评论**后台巡查下架** | `AdminContentManageService.takedown`（帖子）、评论 FR-55A（story3） | story3/本 |
| 名称人工判定违规 → 重置默认编码名 | 名称处置点（story4） | story4 |
| 头像人工判定违规 → 重置默认头像 | 头像处置点（story5） | story5 |

**不计入 ❌（关键排除项）：** 评论 FR-55 **提交时的自动同步拦截** —— 命中 L1 强制拦截关键词库、或风险评分 ≥0.8 被同步拒绝。这类是**机器自动判定、内容从未发布**，不涉人工判定，**不累加计数**（方案 §5.4 明文）。同理帖子提交时命中 L1 硬拦截直接失败（未入队）也不计入。

> **口径本质：** "违规内容真实存在过并被人工确认"才计数。自动拦截 = 内容根本没落地，不计。

### 5.2 累加入口 `ViolationCountService.record(...)`

单一收口方法，所有处置点调用：

```
@Service
class ViolationCountService {
    // 与调用方处置的状态迁移【同事务】执行（不新起事务、不 @Async）：
    // upsert violation_counts：存在则 violation_count+1、刷新 last_violation_at；
    // 不存在则 INSERT count=1、first/last=now()。
    void record(long accountId, ViolationType type);
}
```

- **同事务累加**：`record(...)` 必须与处置动作（下架/拒绝/重置）在**同一数据库事务**内提交 —— 处置回滚则计数不落，处置成功则计数必落，二者原子一致。
- **UPSERT 实现**：`INSERT ... ON CONFLICT (account_id, violation_type) DO UPDATE SET violation_count = violation_counts.violation_count + 1, last_violation_at = now(), updated_at = now()`（PostgreSQL 原子 upsert，避免并发下的读改写竞态；配 UNIQUE 约束）。
- **不触发任何下游**：`record` 只写表，无事件、无通知、无风控回调（"仅记录不处置"的物理保证）。

### 5.3 累加接线点（本 story 落地现有帖子点；评论/名称/头像由对应 story 调用）

本 story **直接接线**已存在的帖子处置点：
1. `ManualReviewService.reject(...)` → 在 `contentService.discardReview` 后、状态置 `REJECTED` 的同事务内 `record(authorId, POST)`。
2. `ManualReviewService.scanTimeouts(...)` → 每条置 `TIMED_OUT` 的同事务内 `record(authorId, POST)`（authorId 从队列条目/内容取）。
3. `AdminModerationService.takedown(...)` → `softDelete` 成功后同事务 `record(postAuthorId, POST)`。
4. `AdminContentManageService.takedown(...)` → 同事务 `record(postAuthorId, POST)`。

评论/名称/头像点在 story3/4/5 落地时调用 `record(accountId, COMMENT|NAME|AVATAR)`（本 story 交付服务契约 + §8 记档，不阻塞那些 story）。

### 5.4 幂等（防重复计数）

处置点均为**幂等状态迁移**：队列条目 `status != PENDING` 即跳过（`ManualReviewService.java:122`）、帖子已 soft-deleted 再 takedown 无二次迁移。因累加挂在"发生状态迁移"的同事务分支内，**重复调用不会二次计数**，无需额外去重键。

> **已知边界（记入 §10）：** 若运营"下架 → 恢复 → 再下架"同一帖子，会计数两次。本 story 视其为两次独立人工违规判定，**接受**（符合"累计人工判定次数"语义）；不做跨恢复的抵扣。

### 5.5 注销联动（方案 §5.4 注销处置决策）

在 `AccountDeletionService.execute(...)`（`account/service/AccountDeletionService.java:87` 编排点）内，D1/D2 现有级联基础上**追加**三件事：

1. **内容默认对其他人隐藏**：注销用户的帖子/评论对**其他用户不可见**。
   > **与 Story 7-3 匿名化的关系（D-CM4 边界，务必分清）：**
   > - **匿名化（7-3，D1）= 显示层**：保留内容，作者名替换为"已注销用户"i18n 文案（`AccountDeletionService.java:95` "content 由 user 行删除自动匿名"）。这是**已实现**行为。
   > - **隐藏（本 story）= 可见性层**：内容对其他人**不再展示**。
   > - 二者**并存不冲突**：注销后内容既被匿名化（万一被查到不暴露身份）、又默认对他人隐藏（正常不展示）。实现上以 Feed/详情/我的发布查询的可见性过滤为准（复用/对齐 story2/story6 的"仅作者可见"可见性模型；作者已注销则等同对所有人隐藏）。
   > - ⚠️ 实现选择需与架构确认：优先复用现有可见性字段（story2 引入的挂起/隐藏态），避免为注销单造一套隐藏标志。若 story2 的可见性列可承载"因作者注销而隐藏"，本 story 只需在注销编排点把该用户内容置为隐藏态。
2. **队列条目随注销移除不再发布**：注销用户处于 `manual_review_queue`（V41）中的 `PENDING` 条目 → 标记移出队列（终态，如 `TIMED_OUT`/新终态或直接删条目按 story2 的队列语义），对应内容**不再发布**（保持仅作者可见 → 作者已注销即对所有人不可见）。
3. **违规计数依 D1/D2 级联处置**：
   - **D1（用户申请注销 `USER_REQUEST`）**：`violation_counts` 中该 `account_id` 的行按 PII/合规策略处理 —— 违规计数不含 PII（仅 account_id + 类型 + 次数），可**随 user 删除一并删除**（与 7-3 "档案删除"同档），或匿名化保留计数供 V2 统计。**本 story 默认：随注销删除该用户 violation_counts 行**（D1 语义 = 用户要求清除，计数属其账号数据）。
   - **D2（违规处置注销 `VIOLATION`）**：同 D1 处理计数行；D2 本就前置下架全部内容（`DeletionType.VIOLATION`），此处不重复累加（下架发生在注销前的处置流程，已各自计数）。

> **注销联动的护栏：** 上述都在 `execute` 现有同步事务/编排内完成，**不新起 @Async、不引 MQ**；删除计数行发生在 user 行删除**之前**（无 FK，但顺序保一致）。

## 7. 验收 AC（含验证层级与所需环境）

> 层级：**L0** 静态（`mvn -B compile|package`，无需 DB）｜**L1** 集成（Docker postgres+redis 真跑 + Flyway validate）｜**L2** 端到端（运营后台/真实注销流程）。

**A. 迁移与 schema**
- **AC-1（L1）** 新迁移创建 `violation_counts` 表，含 `UNIQUE(account_id, violation_type)` 与 `violation_type` 的 CHECK（`POST/COMMENT/NAME/AVATAR`）；应用启动 `ddl-auto=validate` **通过**（实体↔列一致，无 CHAR(1) 坑）。环境：Docker + Flyway。
- **AC-2（L0）** 迁移文件名占位 `V<n>__`，未硬编码具体号；`mvn -B package` 编译通过、实体与枚举 `ViolationType` 就位。

**B. 累加口径**
- **AC-3（L1）** 帖子 FR-12A 队列**判定拒绝** → 对应作者 `(account_id, POST)` 计数 +1、`last_violation_at` 刷新。
- **AC-4（L1）** 帖子 FR-12A 队列**超时丢弃**（`scanTimeouts`）→ `(account_id, POST)` +1。
- **AC-5（L1）** 举报下架 / 后台巡查下架 → `(account_id, POST)` +1。
- **AC-6（L1）** 计数累加与处置**同事务**：模拟处置事务回滚 → 计数**不落**；处置提交 → 计数落。
- **AC-7（L1）** **不计入自动拦截**：评论 FR-55 提交时命中 L1 / 评分 ≥0.8 被同步拒绝（story3 路径）**不产生**任何 `violation_counts` 行（本 story 以单测/集成断言"自动拒绝路径不调用 `record`"）。
- **AC-8（L1）** 幂等：对同一已拒绝队列条目重复调用 `reject`（`status != PENDING` 被守卫）→ 计数**不二次累加**。

**C. 仅记录不处置（负向断言）**
- **AC-9（L0/L1）** 全代码路径搜索确认：`ViolationCountService.record` **无任何下游副作用**（无事件发布、无通知、无限权、无发布/审核判定读取 `violation_counts`）。计数达任何值都不触发自动动作。（L0 静态审查 + L1 行为断言）

**D. 注销联动**
- **AC-10（L1）** 注销执行后，该用户帖子/评论对**其他用户不可见**（Feed/详情查询过滤），同时匿名化（"已注销用户"）保留不变 —— 二者并存。
- **AC-11（L1）** 注销时该用户在 `manual_review_queue` 的 `PENDING` 条目被移出队列、对应内容**不发布**。
- **AC-12（L1）** 注销（D1）后该用户 `violation_counts` 行按 §5.5 处置（默认删除）；D2 同处理；均在 user 删除前完成，无 FK 冲突、无残留。
- **AC-13（L2）** 运营后台走一次真实"违规判定 → 计数增长 → 该用户注销 → 计数与内容按策略处置"端到端（配合 story8 展示面）。回本地/预发验收。

- **D-CM4 区分（不可混用）**：
  - "违规名 → 重置默认编码名 `user_8f3a2b`/`Pet_8f3a2b`"（story4 违规处置）

## 10. 风险与待确认

1. **注销"隐藏"的承载字段依赖 story2**：若 story2 的可见性模型未先落地，"内容因作者注销而对他人隐藏"缺现成字段。**待确认**：是否等 story2 合并后再接线，或本 story 临时用 story2 约定的可见性列。建议与架构对齐可见性列复用方案，避免双套隐藏标志。
2. **D1 计数行处置策略（删除 vs 匿名保留）**：§5.5 默认"D1 随注销删除 violation_counts 行"。但方案 §5.5 又要求违规/处置数据**无限期留存**供 V2 追溯 —— 计数行删除是否与留存诉求冲突？**待确认**：计数行是否应改为"保留但脱离 account_id 关联"（匿名保留），还是"处置记录留在 admin_audit_logs 即足够、计数聚合表可随账号删除"。倾向后者（审计证据在 audit_logs，聚合计数属账号数据可删），但需 PM/合规拍板。
3. **"下架→恢复→再下架"重复计数**（§5.4 边界）：接受两次计数。若运营认为应抵扣，需引入逐条事件流去重键（本 story 不做，记为 V2 可选）。
4. **计数类型粒度**：本 story 按方案 4 类（POST/COMMENT/NAME/AVATAR），未区分 user/pet 名称与头像。若 V2 账号处置需更细粒度，须增枚举 + 放宽 CHECK（增量 ALTER）。
5. **account_id 无 FK**：靠应用层保证注销时"先删计数、再删 user"的次序。**待确认**：是否需在 `execute` 编排里显式保证顺序（本 story 已按此设计），或接受最终一致（V1 姿态）。
6. **方案 §7 #10 未闭环**：违规计数的"具体字段与统计口径（是否含误报次数等）"标待确认。本 story 按"仅计入人工判定违规、误报恢复不计/不抵扣"落地；若运营口径变动需回改累加点。
