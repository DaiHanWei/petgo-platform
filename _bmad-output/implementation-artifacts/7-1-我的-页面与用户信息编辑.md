# Story 7.1: 「我的」页面与用户信息编辑

Status: review

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

## Story

As a **用户**,
I want **一个集中管理我的信息与发布的个人中心**,
so that **我能编辑资料、回看我发过的内容、并找到设置（语言/退出/注销/帮助）入口**。

> 本 Story 是 Epic 7 的入口聚合页（FR-20「我的」页面结构），**混合前后端**：后端补齐用户信息读写（头像/昵称编辑）+「我的发布」时间线聚合接口；前端搭「我的」Tab 五大区块壳与编辑交互。语言设置详细收口在 **7.2**、退出登录/账号注销在 **7.3**、无障碍达标在 **7.4**——本 Story 只放它们的**入口**，不实现其内部逻辑。宠物状态修改与档案编辑入口**复用 Epic 2 的 FR-39 编辑页 / FR-0F 状态选择界面**，本 Story 不重写编辑页，只接入口并保证双向同步（FR-21）。

## Acceptance Criteria

> **验证层标注**：每条 AC 末尾标注验证层级与所需本地环境——
> **L0 静态**（编译/lint，无需 DB） · **L1 集成/运行时**（需 Docker daemon + postgres + redis） · **L2 端到端/外部凭证**（需真实第三方或真机模拟器）。
> 本 Story 主要落 **L0 / L1**；头像替换走 STS 直传 OSS 的端到端验证需真机 + 真实 OSS = **L2**。

### AC1 — 「我的」页面信息架构（顶栏双图标 + 设置项收二级页 + 主体干净）

**Given** 登录用户进入「我的」Tab
**When** 查看页面
**Then** **顶部导航栏**：左为「我的」固定标题，右上角并列「帮助反馈」图标 +「设置」图标；**主页面主体**自上而下展示用户信息（头像 + 昵称 + 编辑按钮）、宠物卡片/引导卡（见 AC5）、宠物状态（展示 A/B/C + 修改入口）、我的发布——主页面保持干净，**不再平铺语言设置 / 退出登录 / 账号注销三项**；语言设置 / 退出登录 / 账号注销**收进二级「设置页」**（点右上角设置图标进入）（FR-20）
> 🔄 **PRD V1.0.0 修订（F8 · 2026-06-08）：** 个人中心信息架构重组——「帮助反馈」「设置」改为右上角双图标；语言/退出/注销三项从主页区块挪入二级设置页（点设置图标进入），主页面只承载「人 + 宠物」内容，UX 套 mint 原型。
> 验证层：**L0**（顶栏双图标 + 主体区块组件树 widget test；右上角设置图标 → 设置页含语言/退出/注销三项入口的路由 widget test）+ **L1**（页面数据来自 `GET /api/v1/me` 聚合，需后端起栈拉真实用户行）。设置页内退出登录/注销入口仅占位跳转，其逻辑在 7.3。

### AC2 — 用户信息编辑即时生效（头像 + 昵称）

**Given** 用户编辑资料
**When** 修改头像（可替换 Google 默认头像）或昵称（≤20 字）
**Then** 即时生效
> 验证层：**L1**（昵称改 `PATCH /api/v1/me`，Bean Validation `≤20` 字、超长返回 422 ProblemDetail，需后端起栈）+ **L2**（头像替换走客户端压缩→STS 直传①公开桶→回填 URL 的真机/真实 OSS 全链路；昵称即时生效可在 L0 widget 层验状态回填）。

### AC3 — 「我的发布」三类混合时间线倒序

**Given** 我的发布区块
**When** 展示
**Then** 以时间线倒序展示用户全部已发布内容（三类混合），单条目可进入详情页或删除（FR-36）
> 验证层：**L1**（`GET /api/v1/me/posts?cursor=...` 游标分页 20/批、仅返回当前用户 `author_id` 且 `deleted_at IS NULL` 的内容，倒序按 `created_at`；删除复用 FR-36 二次确认 + 级联软删，需后端起栈 + 种子数据）。空态/网络错误/已删除态走 UX-DR18。

### AC4 — 状态 A 用户的档案入口与跨页同步

**Given** 状态 A 用户
**When** 查看「我的」
**Then** 显示「编辑宠物档案」入口（复用 FR-39 编辑页）；宠物状态修改与成长档案 Tab 同步（FR-21）；**B/C → A 改状态后立即触发 FR-0G 宠物档案创建引导（与注册时体验一致，含「跳过」选项），用户跳过后触发 FR-0H 首页提示条**（与 1-6 状态修改 / 1-7 注册引导联动）
> 🔄 **PRD V1.0.0 修订（F8 · 2026-06-08）：** 补 B/C → A 改状态后触发 FR-0G 创建引导 / FR-0H 首页提示条的衔接断言（PRD FR-21 后果 + FR-0G/FR-0H），与 1-6/1-7 既有引导流复用、不另写逻辑。
> 验证层：**L1**（状态 A=有宠物档案 → 显示「编辑宠物档案」入口跳 FR-39 编辑页；状态 B/C 不显示该入口而显示状态选择入口；修改宠物状态后「我的」与成长档案 Tab 双向即时同步、首页 Feed 按新状态刷新——需后端 `pet_profiles` 真实行 + Riverpod 共享状态）+ **L0**（B/C→A 触发 FR-0G 引导跳转、跳过后置 FR-0H 提示条标志位的衔接 widget test，引导/提示条本体逻辑复用 1-6/1-7）。

### AC5 — 宠物卡片 / 引导卡的状态分支渲染（人 60% / 宠物 40%）

**Given** 「我的」主页面用户信息区下方的宠物区位
**When** 按当前宠物状态与建档情况渲染
**Then** 分三支：**状态 A 且已建档** → 渲染**宠物卡片**（宠物头像 + 名字 + 最近一条快乐时刻首图），点击跳转成长档案 Tab；**状态 A 未建档** → 渲染「**给你的宠物创建专属档案**」**引导卡**（替代宠物卡片位置，点击进入 FR-11 创建流程）；**状态 B / C** → 宠物卡片与引导卡**均不显示**。视觉权重「**人占 60% / 宠物占 40%**」——用户信息区为主视觉，宠物区位权重低于用户信息区，呈现宠物陪伴主人的关系
> 🔄 **PRD V1.0.0 修订（F8 · 2026-06-08）：** 新增宠物卡片/引导卡三态分支渲染 AC（PRD FR-20 区块表 + 后果），落实「人 60%/宠物 40%」视觉定位；UX 套 mint 原型。
> 验证层：**L0**（三态分支组件树 widget test：A+已建档=宠物卡片含最近快乐时刻首图、A 未建档=引导卡、B/C=两者均不渲染；视觉权重断言宠物区位低于用户信息区）+ **L1**（卡片数据与「最近一条快乐时刻首图」来自 `GET /api/v1/me` 聚合，需后端起栈拉真实用户行 + content 真实快乐时刻）。

### AC6 — 「我的发布」排序口径 = 发布时间（created_at）倒序（F9）[R2·回改]

> **[R2·回改]** 代码已实现，本轮回改补齐。明确「我的发布」排序口径为**发布时间 `created_at`**，与成长档案时间线/日历的**事件日期 `event_date`** 口径区分（决策 F9），杜绝两处当成同一副本。

**Given** 「我的发布」三类混合时间线
**When** 排序展示
**Then** 严格按**发布时间 `content_posts.created_at` 倒序**排列（最新发布在前）——**非**事件日期排序
**And** 明确区别于成长档案时间线 / 日历视图——后者按**事件日期 `content_posts.event_date`** 排序（同一条成长日历类内容在「我的发布」与「成长档案」**两处排序口径不同，是两套口径而非同一副本**：一条 `event_date` 在过去、`created_at` 是今天补录的快乐时刻，在「我的发布」按今天的 `created_at` 靠前、在成长档案按过去的 `event_date` 归位）
> 验证层：**L0**（widget test：「我的发布」列表项按 `created_at` 倒序断言；口径文案/注释标注 created_at vs event_date 不混用）+ **L1**（`GET /api/v1/me/posts` 返回顺序按 `created_at desc`，与成长档案 `event_date` 排序端点口径分离，需后端起栈 + 含 `event_date≠created_at` 的种子数据验两口径不同序）。

## Dev Notes

### 关键架构约定（本 Story 必须落实）

- **目录**：后端 `com.tailtopia.profile/{web,service,domain,repository,dto,event}` + `com.tailtopia.auth/`（用户身份）；「我的发布」经 content service 接口拿，**禁跨模块 join**。前端 `lib/features/me/{data,domain,presentation}`，复用 content 的 Feed 分页组件与 profile 的编辑页。
- **命名映射链**：DB `snake_case`（`users.avatar_url`、`users.nickname`、`content_posts.author_id`）↔ JSON camelCase（`avatarUrl`/`nickname`/`authorId`）↔ Dart/Java camelCase。时间 UTC ISO-8601。
- **聚合读**：`GET /api/v1/me` 走 BFF 式一次返回，减少印尼↔德国串行往返。
- **状态**：Riverpod `AsyncValue<MeState>`，不可变 copyWith；宠物状态用跨页共享 provider 保证三处同步。
- **错误**：统一 ProblemDetail；dio 拦截器映射本地化文案；昵称 422 含 `errors[].field`。

### ⚖️ 合规攸关约束

- **PDP 数据主体权利可达（NFR-8）**：本 Story 的「我的」页面是 PDP「数据主体权利可达路径」的**承载入口**——账号注销入口（→7.3）、帮助与反馈、隐私政策链接必须从此页**可达**（注销逻辑在 7.3，但入口缺失即合规缺口）。本 Story 须确保入口存在且可点击进入对应流程占位。
- **头像/昵称即 PII**：`avatar_url`/`nickname` 属个人身份信息——**日志严禁记录昵称、头像签名/URL、令牌**（架构 §日志，SLF4J JSON 不落 PII）。头像走①公开桶（Feed 可见），但上传链路仍须 EXIF GPS 剥离（G-4）防定位泄漏。
- **越权隔离**：`GET/PATCH /api/v1/me` 只可操作 JWT `sub` 本人数据；任何以他人 id 越权读写 → 403 ProblemDetail，不得返回他人 PII。

### 无障碍（NFR-13 / UX-DR15）—— 本页须满足，正式横切验收在 7.4

- 编辑按钮、状态修改入口、列表项、退出/注销入口等可交互元素触摸目标 **≥44×44pt**。
- 文本/图标在 #FAF8F5 底色对比度 **≥4.5:1**；三色态宠物状态徽标用 icon+text+color **不依赖颜色单一**。
- 昵称编辑输入框、各入口须有可访问性语义标签（screen reader 可读）。

### 强制护栏

- **禁跨模块直接访问对方 repository**：「我的发布」必须经 content service 接口；宠物档案/状态经 profile/FR-21 既有端点——本 Story **不重写**编辑页与删除逻辑（复用 FR-39/FR-0F/FR-36）。
- 头像 OSS URL 只存应用自有 URL，不存第三方临时 URL。
- 异步只用 `@Async`+事件，**禁引入 MQ/通用缓存层**。

### 范围边界（本 Story 明确不做）

- ❌ 不实现语言设置内部逻辑与双语文案收口（7.2）——只放语言设置入口。
- ❌ 不实现退出登录与账号注销级联删除逻辑（7.3）——只放入口。
