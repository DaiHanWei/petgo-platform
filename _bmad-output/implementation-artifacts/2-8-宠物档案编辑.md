# Story 2.8: 宠物档案编辑

Status: review

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

## Story

As a **宠物主**,
I want **随时修改宠物档案信息**,
so that **档案与名片始终保持准确最新**。

> 本 Story 收口 Epic 2：编辑头像/名字/品种/生日/介绍，两入口（档案 Tab 信息卡 + 「我的」Tab）复用同一编辑页，修改即时生效并联动名片 H5（触发 2.6 OG 图重渲染）。**前后端均衡**。
>
> **依赖前序**：2.1（头像上传）、2.2（pet_profiles + 创建端点/DTO 复用）、2.4（档案 Tab 信息卡编辑入口）、2.6（名片重渲染 service，编辑后调用）。「我的」Tab 入口由 Epic 7 提供，本 Story 提供编辑页 + 入口对接 contract。

## Acceptance Criteria

> **验证层标注**：**L0 静态** · **L1 集成/运行时**（Docker+postgres+redis）· **L2 端到端/外部凭证**（真实 OSS 头像、名片 H5 联动）。核心落 **L0/L1**；名片即时同步实时读库已在 2.6 覆盖（L1），头像直传真实 OSS 在 2.1 覆盖（L2）。

### AC1 — 两入口复用同一编辑页，改五字段

**Given** 已创建档案用户
**When** 从成长档案 Tab 信息卡右上角「编辑」或「我的」Tab「编辑宠物档案」入口进入
**Then** 两处入口复用同一编辑页，可改头像/名字（≤20字）/品种/生日/一句话介绍（≤30字）（FR-39）
**And** 修改即时生效，宠物名片 H5（FR-14）同步更新，V1 不限编辑次数
**And** **宠物类型 `pet_type` 硬禁修改**——服务端拒绝任何 pet_type 变更（请求带该字段即忽略或拒，落库值不变）、前端编辑页该字段**置灰只读**展示
> 🔄 **PRD V1.0.0 修订（F6 · 2026-06-08）：** 编辑页硬禁改 `pet_type`（创建后不可改，避免已完成里程碑数据错乱）；其余字段（头像/名字/品种/生日/介绍）编辑不变。
> 验证层：**L1**（`PATCH /pet-profiles/me` 更新写库 + 编辑后访问 `/p/{cardToken}` 即最新 + OG 图重渲染触发 + **pet_type 变更被拒/落库不变**，需 postgres）+ **L0**（同一 `pet_profile_edit_page` 被两入口路由复用、名字 ≤20/介绍 ≤30 校验、表单预填既有值、**pet_type 字段置灰只读** — widget/MockMvc 测试）。

### 🟦 后端子任务（petgo-backend / `com.petgo.profile`）

- [ ] **B1. 编辑端点** (AC: 1)
  - [ ] `ProfileApiController` `PATCH /api/v1/pet-profiles/me`（JWT），body `PetProfileUpdateRequest{avatarUrl?, name?(≤20), breed?, birthday?, intro?(≤30)}`（部分更新，record + 校验）。owner 取自 JWT，**仅能改自己档案**。
    > 🔄 **PRD V1.0.0 修订（F6 · 2026-06-08）：** `PetProfileUpdateRequest` **不含 `pet_type` 可变字段**；即便请求体携带 pet_type，服务端硬拒/忽略，绝不更新该列。

## Dev Notes

### 架构约定

- **命名/复用**：复用 2.2 的 `PetProfile`/Repository；DTO `PetProfileUpdateRequest`(record，可空部分更新)；`PATCH` 语义（200 读/改成功）。owner 自 JWT。
- **名片联动**：FR-39 编辑联动——名片正文 2.6 实时读库自动最新；OG 预渲染图需主动重渲染（异步 `@Async`，无 MQ）。
- **token 不变**：编辑不重生成 cardToken（已分享链接保持有效）。
- **头像**：复用 2.1 公开桶 + EXIF 剥离。

### 强制护栏

- **禁 MQ/缓存层/新中间件**；重渲染用 `@Async`。
- `ddl-auto=validate`（本 Story 无新表，复用 2.2/2.6 表/列）。
- owner 自 JWT，仅改自己档案（越权 403/404）。
- 校验 422、ProblemDetail 统一。
- 日志记编辑事件 id 可，不落用户正文做检索；无健康数据。
- 删除级联/旧头像清理策略记录（与注销 FR-20 一致）。

### 范围边界（不做）

- ❌ 不做创建（2.2）、时间线（2.4）、名片 H5 渲染本体（2.6，仅触发其重渲染）、分享 FAB（2.7）。
- ❌ 不做「我的」Tab 本体（Epic 7，仅对接编辑入口 route）。
- ❌ 不做多宠物/删除档案（V1 范围；删除走注销级联 Epic 7）。
- ✅ 只做：`PATCH /pet-profiles/me` 编辑端点 + 名片 OG 重渲染联动 + 复用创建表单的编辑页 + 两入口同页复用 + 校验。

### Project Structure Notes

- 后端 `com.petgo.profile.{web/ProfileApiController(加 PATCH), service/ProfileService(update + 触发 2.6 重渲染), dto/PetProfileUpdateRequest}`。
- 前端 `lib/features/profile/presentation/pet_profile_edit_page.dart`（复用 2.2 创建表单组件）；go_router 两入口指同一页。
