# Story 4.4: 客服判定与订单解锁/回落

Status: ready-for-dev

> V1.1 **Epic 4 第 4 story**，**纯后端**（客服判定行为 + 订单状态机联动 + 通知，L1）。消费 4-3 的 `refund_requests`：把 4-3 只「记 need_decision」的 `submitNeed` 升级为带**后果**的判定——**批准**→订单 `COMPLETED→REFUNDING` + 解锁用户选方式入口（**不发通知**，AB-5B）；**驳回**→发「退款申请未通过」`REFUND_REJECTED` 通知 + **订单回落 `COMPLETED` 置 `refund_rejected=true`**（A-2，订单状态不再假装在退款）。
> 源：`epics-v1.1.md` Story 4.4（批准→订单 REFUNDING + App 解锁选方式 + 不发通知 L1；驳回→发「退款申请未通过」+ 订单回落 COMPLETED 置 refund_rejected L1）· AB-5B · A-2 · 架构 `architecture-v1.1-delta.md` §3.4/§5.2 · 通知扩枚举 `extend_notification_types_v11`（line 151）。
> **⚠️ 承接 4-3（已就位，勿重建）**：`pay.refund` 模块 + `RefundService`（`submitNeed(refundToken, submitterAdminId, decision)` 已置 need_decision + 审计）+ `refund_requests` 表 + 职责分离守卫 + payout 加密。本 story **在 4-3 基础上加行为后果 + 通知 + admin 控制器**，不碰 4-3 的职责分离/加密/净额。
> **范围边界**：本 story = 客服判定行为（批准订单 REFUNDING+解锁 / 驳回订单回落+通知）+ `NotificationType` 扩枚举（V72）+ 订单状态机 CAS（COMPLETED→REFUNDING / 置 refund_rejected）+ admin 客服判定端点。**不做**：用户端选退款方式/填收款（4-5）、主管审批 + 财务实际 Midtrans 打款（4-6）、CSAT 问卷/工单结案通知（4-7，`CSAT_SURVEY`/`TICKET_RESOLVED` 枚举本 story 一并加进 CHECK 但不发）、前端（用户端投诉在 4-2 已建；退款方式选择 UI 4-5）。

## Story

As a 客服,
I want 批准退款需求即解锁用户选方式、驳回即通知并回落订单,
so that 两段审核可区分、驳回后订单状态诚实（AB-5B/A-2）。

2. **后端：批准退款需求（订单 REFUNDING + 不发通知）（L1）**
   **Given** `refund_request.need_decision=PENDING`、订单 `COMPLETED`
   **When** 客服批准（`refund.submit` 权限）
   **Then** `refund_request.need_decision=APPROVED` + `submitter_admin_id` + 订单 **CAS `COMPLETED→REFUNDING`**（`markRefundingFromCompleted`，幂等：仅 COMPLETED 可转）+ **不发任何通知**（AB-5B，解锁选方式经 need_decision=APPROVED，App 4-5 据此暴露入口）+ 审计（**L1**）
   **And** 订单已非 COMPLETED（如已 REFUNDING/REFUNDED）→ 幂等不重复置（CAS 返回 0 时不报错或按已处理，dev 定）（**L1**）

3. **后端：驳回退款需求（订单回落 + 通知）（A-2）（L1）**
   **Given** `refund_request.need_decision=PENDING`
   **When** 客服驳回（`refund.submit` 权限）
   **Then** `refund_request.need_decision=REJECTED` + `submitter_admin_id` + 订单 **置 `refund_rejected=true`（状态回落/保持 `COMPLETED`，不再假装在退款，A-2）** + 发 `REFUND_REJECTED` 通知给 `refund_request.user_id`（`NotificationService.send`，REQUIRES_NEW）+ 审计（**L1**）
   **And** 通知**不含**PII/退款金额细节（护栏）；`deepLinkType`/`targetRef` 指向工单或退款（照既有 send 范式，非随机 token——见 [[notify-after-commit-tx-swallow-bug]]）（**L1**）

4. **后端：admin 客服判定端点（L1）**
   **Given** 客服在后台判定
   **When** `POST /admin/refunds/{refundToken}/approve` / `POST /admin/refunds/{refundToken}/reject`（Thymeleaf admin，`@PreAuthorize` `hasRole('SUPER_ADMIN') or hasAuthority('refund.submit')`）
   **Then** `@AuthenticationPrincipal AdminUserDetails admin` → `admin.getAdminAccountId()` 作 submitter；调 `RefundService` 批准/驳回；redirect + flash（照 `AdminContentManageController` 范式）（**L1**）
   **And** 无 `refund.submit` 权限 → 403；已判定（非 PENDING）幂等/拒绝重复判定（dev 定，建议拒绝 + flash）（**L1**）

5. **后端：契约 + 静态验收（L0/L1）**
   **Given** 全部实现
   **When** 验证
   **Then** `DB_NAME=petgo_l1 mvn -B test` 绿：L0（V72 建约束、NotificationType 枚举）+ L1（批准→订单 REFUNDING + 无通知 + need APPROVED；驳回→订单 COMPLETED+refund_rejected + REFUND_REJECTED 通知落库 + need REJECTED；权限门控 403；幂等）；schema `validate` 一致（**L0/L1**）
   **And** 不破坏 3-8 既有 REFUNDING 路径（封禁挂起 IN_PROGRESS→REFUNDING 不受影响）（**L1**）

## Dev Notes

> 纯后端，在 4-3 `RefundService` 上加**判定后果**（订单联动 + 通知）+ admin 端点 + 通知扩枚举。**勿破坏** 3-8 的封禁挂起 REFUNDING 路径（那是 IN_PROGRESS→REFUNDING，本 story 是 COMPLETED→REFUNDING，两条 CAS 独立）。

### 承接现状（已核实，照此复用/勿改）

- **4-3 `RefundService`**：`submitNeed(refundToken, submitterAdminId, NeedDecision)` 仅 `r.markNeedDecision` + 审计。本 story 拆/扩为 `approveNeed`/`rejectNeed`（带订单联动 + 通知）。职责分离守卫在 approve(主管)/payout(财务) 那两段（4-3），本 story 的客服判定是 **submitter 单角色**，无职责分离冲突。
- **订单状态机**（`ConsultOrder`/`ConsultOrderRepository`）：`markRefunding` = CAS `IN_PROGRESS→REFUNDING`（@Modifying @Query，返回受影响行数，3-8 封禁挂起用）；`markRefunded` = `REFUNDING→REFUNDED`；`markCompleted` = `IN_PROGRESS→COMPLETED`。本 story **新增** `markRefundingFromCompleted`（`COMPLETED→REFUNDING`）+ `markRefundRejected`（`status=COMPLETED` 置 `refund_rejected=true`）。CAS 幂等：返回 0 = 已非期望态，不报错（按已处理）。`refund_rejected` 列已存在（V66）。
- **通知**（`notify`）：`NotificationType` 枚举（落库 UPPER_SNAKE + `ck_notifications_type` CHECK）；`NotificationService.send(long recipientUserId, NotificationType, String title, String body, String deepLinkType...)` **`@Transactional(REQUIRES_NEW)`**（必需——同步 `@TransactionalEventListener` AFTER_COMMIT 无环境事务坑，[[notify-after-commit-tx-swallow-bug]]）。扩 CHECK 照 `V40__extend_notification_types_report_reviewed.sql`（DROP+ADD constraint 全集）。**批准不发通知（AB-5B）**；仅驳回发 `REFUND_REJECTED`。通知文案本地化 + `targetRef` 非随机 token（既有范式）。
- **admin 端点**：Thymeleaf admin slice（`@Controller` + redirect string + `RedirectAttributes` flash）；`@PreAuthorize("hasRole('SUPER_ADMIN') or hasAuthority('refund.submit')")`；`@AuthenticationPrincipal AdminUserDetails admin` → `admin.getAdminAccountId()`。照 `AdminContentManageController.takedown/restore`。权限点 `refund.submit`（4-3 已加 `AdminPermissions`）。
- **审计**：`AdminAuditService.record`（4-3 已接）。批准/驳回记 `REFUND_NEED_SUBMITTED`（含 decision）或新增 `REFUND_NEED_APPROVED`/`REFUND_NEED_REJECTED`（dev 择；建议复用 REFUND_NEED_SUBMITTED + summary 含 decision，或加两常量到 `AuditActions`）。summary 无 PII。

### 关键设计决策（dev 照此）

- **D-1 批准 = 订单 COMPLETED→REFUNDING（新 CAS）**：退款工单针对已完成订单（服务已交付），批准后订单进 REFUNDING 等用户填收款（4-5）→ 主管审批 + 财务打款（4-6）→ REFUNDED。**解锁「选方式」不需额外状态**，App 据 `need_decision=APPROVED` 暴露入口（4-5）。**不发通知**（AB-5B）。
- **D-2 驳回 = 订单回落 COMPLETED + refund_rejected（A-2，UX 不撒谎）**：订单一直是 COMPLETED（未曾 REFUNDING），置 `refund_rejected=true` 标记「申请过但未通过」；发 `REFUND_REJECTED` 通知。**不新增订单终态枚举**（架构 A-2）。
- **D-3 通知 REQUIRES_NEW + 无 PII**：`REFUND_REJECTED` 经 `NotificationService.send`（自带 REQUIRES_NEW）；title/body 本地化占位「退款申请未通过」，**不含金额/账号**。
- **D-4 扩枚举一次加全**：V72 CHECK 含 `REFUND_REJECTED`/`TICKET_RESOLVED`/`CSAT_SURVEY`/`IDENTITY_REQUIRE_MODIFY`（`extend_notification_types_v11`），避 4-7/9-x 二次迁移；`NotificationType` 枚举同步加全 4 值。本 story 只**发** REFUND_REJECTED。
- **D-5 勿破坏 3-8**：3-8 封禁挂起走 `markRefunding`(IN_PROGRESS→REFUNDING)；本 story 加的是 `markRefundingFromCompleted`(COMPLETED→REFUNDING) —— 两条独立 CAS，互不影响。回归须含 `ConsultSuspensionIntegrationTest` 绿。
- **Flyway V72**：当前 max V71（4-3）。移动靶顺延。CHECK 约束改动是 ALTER（DROP+ADD），非新表。
