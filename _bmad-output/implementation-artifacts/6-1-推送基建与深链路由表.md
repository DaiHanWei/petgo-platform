# Story 6.1: 推送基建与深链路由表

Status: review

<!-- Note: Validation is optional. Run validate-create-story for quality check before dev-story. -->

## Story

As a **用户**,
I want **点击任意推送都能直达对应内容**,
so that **我不必从首页层层点回，体验连贯**。

> 这是 **Epic 6 的第一个 Story（基建型）**，为后续 6.2~6.6 全部推送场景铺地基。本 Story **不实现任何具体业务推送内容**（兽医回复推送在 6.2、互动推送在 6.3、版本提醒在 6.5），只交付：① 后端复用腾讯 IM 离线推送的**统一推送下发能力 + `notifications` 表**；② 前端 go_router **深链接路由表（FR-38）+ 推送点击三态（冷启动/后台/前台）落地分发** + 深链目标多态兜底。
> **依赖**：底层离线推送复用 Epic 5 已接入的 `shared/im`（腾讯 IM）；本 Story 不新建 IM 连接，只在其离线推送通道上叠加推送 payload 约定与客户端路由分发。具体业务事件源（兽医回复=Epic 5、内容互动=Epic 3）在后续 Story 接入，本 Story 用一条可触发的测试推送验证全链路。

## Acceptance Criteria

> **验证层标注**：每条 AC 末尾标注验证层级与所需本地环境——
> **L0 静态**（编译/lint，无需 DB） · **L1 集成/运行时**（需 Docker daemon + postgres + redis） · **L2 端到端/外部凭证**（需真实 APNs/FCM 经腾讯 IM、真机推送权限、冷启动/后台/前台三态真机验证）。
> 推送**送达**与深链**点击直达**本质是 L2（真机 + 真实 APNs/FCM 通道），无法在纯模拟器/CI 完全验证；路由表映射逻辑、payload 解析、目标多态兜底可在 L0/L1 用单测与本地 deep link 模拟（`flutter run` + 手动 URI/notification payload 注入）验证。

### AC1 — 复用腾讯 IM 离线推送，不引入独立推送服务

**Given** 后端需推送
**When** 通过腾讯 IM 离线推送能力（底层 APNs/FCM）下发
**Then** 复用 IM 推送，不引入独立 TPNS（架构决策）
**And** 后端统一推送出口落在 `notify/service` + `shared/im`，每条推送写一行 `notifications`（供 6.6 通知中心读取），并携带标准深链 payload（type + token，**绝不带顺序 id**）
> 验证层：**L0**（推送出口/payload 组装/`notifications` 写入可用 MockMvc + 单测验结构）+ **L1**（实际写 `notifications` 表、Redis 角标计数自增需 pg+redis 在跑）+ **L2**（真实经腾讯 IM 离线通道把通知投递到真机 APNs/FCM，需 IM 凭证 + 真机）。

### AC2 — 深链路由表按 FR-38 映射，三态均直达

**Given** go_router 深链路由表
**When** 用户点击推送
**Then** 按 FR-38 映射直达目标页（兽医回复→问诊会话 / 问诊结束→评分 / 被赞→详情页 / 被评→详情页定位评论区 / **宠物生日→「+发布」预选成长日历 / 陪伴纪念日→成长档案 Tab / L级里程碑节点→成长档案 Tab→里程碑列表页（壳）**）
**And** App 未启动冷启动后直达目标页（不经首页）、后台切前台直达、前台时以 Banner 展示点击直达
> 验证层：**L2**（冷启动/后台/前台三态 + 真机推送点击是真机端到端，必须真机验证）。**L0**：路由表 `deep_link_routes` 的「推送 payload → go_router location」纯映射函数可单测全覆盖（含**七类目标** + 评论区锚点参数 + 两个新增 location 目标）。
> 🔄 **PRD V1.0.0 修订（Fx · 2026-06-08）：** 深链路由表新增 3 条映射（生日 FR-40→「+发布预选成长日历」、纪念日 FR-41→「成长档案 Tab」、L级里程碑节点 FR-42→「成长档案 Tab→里程碑列表页（壳）」）+ 2 个新目标 location；对外深链仍只用不可枚举 token（决策 F2/F5）。

### AC3 — 深链目标资源失效时的多态兜底

**Given** 推送深链目标资源已删除/会话已关闭或中断
**When** 用户点击推送落地
**Then** 直达对应页并展示其 not-found / 已关闭 / 已中断态（UX-DR18 ④⑥），不停留首页也不白屏
> 验证层：**L1**（目标页拉取目标资源命中后端 404/已关闭状态需 pg 在跑造数据）+ **L0**（目标页对 not-found/已关闭/已中断三态的渲染分支可 widget test 验证，不依赖真机推送）。

## Dev Notes

### 关键架构约定（沿用全局）

- **目录/分层**：后端 `com.petgo.notify.{web,service,domain,repository,dto,event}` + 复用 `com.petgo.shared.im`（离线推送底层）；前端 `lib/features/notify/{data,domain,presentation}` + `lib/core/router/{app_router,deep_link_routes}`。
- **命名映射链**：DB `snake_case`(`notifications`) ↔ Java/Dart `camelCase` ↔ JSON `camelCase`；枚举落库 UPPER_SNAKE；Flyway `V<序号>__<描述>.sql`。
- **错误**：目标资源失效经后端返 404/409 语义 → 客户端转多态兜底（不裸抛、不白屏）。

### 推送专项约定（本 Story 地基，6.2~6.6 沿用）

- **复用腾讯 IM 离线推送**（底层 APNs/FCM），**砍独立 TPNS**——[architecture.md §技术取舍 / §运行时设计]。推送下发不开新通道，统一经 `shared/im`。
- **深链路由表（FR-38）** 在前端 go_router 落地（`core/router/deep_link_routes`），后端只下发 `type + token`，**映射规则在客户端**——[architecture.md §结构映射 FR-22*,34,38]。
- **未读角标 = Redis 计数**（`notify:unread:{userId}`），Redis 仅角标/限流/幂等/在线态用途，**不当通用缓存/队列**。
- **对外标识一律不可枚举 token**：推送深链用 `deepLinkToken`，**绝不暴露顺序 id**——[architecture.md §主键与对外标识：推送深链亦用 token]。

### 强制护栏（违反即返工）

- **禁** 为推送引入 MQ / 独立推送服务（TPNS）/ 新中间件；离线投递走 `@Async`。
- **禁** 推送 payload / 深链携带顺序 id 或健康数据明文。
- **禁** 跨模块直接访问对方 repository；业务事件源（6.2/6.3）经领域事件 → notify 订阅触发推送（如 `ContentLikedEvent → notify`）——本 Story 先建 NotificationService 入口，事件订阅在后续 Story 接。
- 日志严禁记录 PII/健康数据/令牌。

### 范围边界（本 Story 明确不做）

- ❌ 不实现兽医回复推送内容 / 兽医端新请求推送（Story 6.2）。
- ❌ 不实现内容互动推送（Story 6.3）。
- ❌ 不申请推送运行时权限（Story 6.4——权限时机=完成首次问诊后）。
- ❌ 不做 App 版本更新提醒（Story 6.5，App 内提示不走推送）。
- ❌ 不做通知中心铃铛 UI/列表（Story 6.6——本 Story 只建 `notifications` 表与角标计数键供其消费）。
- ✅ 只做：统一推送出口 + `notifications` 表 + Redis 角标计数 + 深链路由表 + 三态分发 + 目标多态兜底 + 平台通知通道接线。
