## Story

As a 用户,
I want 每月有免费的 AI 问诊额度,
so that 我能低门槛体验、超额再付费（FR-43B）。

## Acceptance Criteria

1. **后端：额度表与实体（L0/L1）**
   **Given** 尚无免费额度存储
   **When** 引入每月免费额度
   **Then** 新建 Flyway **`V63__init_user_monthly_free_quota.sql`**：表 `user_monthly_free_quota`（`id` bigint PK；`user_id` bigint NOT NULL；`period` `VARCHAR(7)` NOT NULL = `YYYY-MM`（**WIB**）；`used_count` int NOT NULL DEFAULT 0 **`CHECK(used_count>=0)`**；`created_at`/`updated_at` `timestamptz` DEFAULT now()）+ **唯一索引 `uq_user_monthly_free_quota (user_id, period)`**（决策 E2：新表新迁移，接 1-6 的 V62 顺延 V63，原架构示例号 V48 作废）；`UserMonthlyFreeQuota` JPA 实体只读映射，`period` 长度 7 用 `VARCHAR(7)` **非 CHAR**（避 Hibernate CHAR 坑）（**L0/L1**）

2. **后端：并发加锁读写、原子递增、不超发（L1）**
   **Given** 同账号本月并发多次发起消耗
   **When** 调 `FreeQuotaService.tryConsume(userId)`
   **Then** 走 `insertIfAbsent(userId, period)`（`INSERT ... ON CONFLICT (user_id, period) DO NOTHING`，靠唯一约束兜并发建行）+ **单行原子条件 UPDATE** `used_count = used_count + 1 WHERE user_id=? AND period=? AND used_count < :limit`（返回受影响行数：1=扣成功 / 0=已达上限）；**禁应用层「读-改-写」**（并发丢更新）；库级 `CHECK(used_count>=0)` 兜底。并发 N 次只成功 min(N, limit) 次、`used_count` 不超 `limit`（**L1**，等价 PawCoin `balance+delta>=0` 守卫范式）

3. **后端：period 按 WIB 每月 1 日切换（惰性重置）（L0/L1）**
   **Given** 免费额度「每月 1 号刷新」是面向印尼用户的产品语义
   **When** 计算当前 period
   **Then** `period = YearMonth.now(ZoneId.of("Asia/Jakarta")).toString()`（`YYYY-MM`，**刻意偏离项目全局 UTC 惯例**，架构 §调度④，代码注释标「有意为之，勿按 UTC 订正」）；**换月自然产生新 `period` 行 = 惰性重置，不需 `@Scheduled`**；跨 WIB 月初边界判定归属正确月份（**L0** period 纯函数可单测 / **L1** 实扣落库）

4. **后端：默认额度可后台调（本 story 走 env）（L0/L1）**
   **Given** FR-43B「默认 1 次、后台可调 0-35」
   **When** 读本月上限 `limit()`
   **Then** 默认值经 `TriageProperties`（`@ConfigurationProperties("petgo.triage")`，`defaultFreeQuota` env `TRIAGE_DEFAULT_FREE_QUOTA` 默认 1）读取，**clamp 到 `[0,35]`**（越界夹取不抛异常；`0`=全付费无免费额度、合法）；**`@ConfigurationProperties` 已注册生效**（`@EnableConfigurationProperties` 或 `@ConfigurationPropertiesScan`）+ `application.yml` 有 `petgo.triage.default-free-quota: ${TRIAGE_DEFAULT_FREE_QUOTA:1}`；9-2 后台 `pricing_config.monthly_free_quota` 落地后换 DB 读（照 1-3 `TopupTierProvider` / 1-5 `PAWCOIN_TOPUP_PAUSED` 渐进模式）（**L0/L1**）

5. **后端：本月额度状态查询端点（L0/L1）**
   **Given** 前端 paywall（2-4）与展示需读剩余额度
   **When** `GET /api/v1/me/free-quota`（JWT 门控，`userId` 一律取自 JWT sub、不信任客户端）
   **Then** 返回 `FreeQuotaView {period, limit, used, remaining}`（`remaining = max(0, limit - used)`；无当月行时 `used=0`）；**只读、不建行、不扣减**；无/非法 JWT → 401 ProblemDetail（照 `PawCoinController` 范式）（**L0/L1**）

6. **红色评级策略边界（本 story 显式不实现，防埋违反点）（L0 文档级）**
   **Given** 安全护栏「红色评级零变现，即使额度耗尽也永不拦红」（FR-43C，安全规则层只升不降）
   **When** 设计本 service
   **Then** `FreeQuotaService` **不感知评级颜色**——红色是否计入消耗、耗尽是否拦截，全在**调用侧策略**（2-2 分字段下发 / 2-3 解锁流决定「红色 unlock_source 恒 FREE、不调 tryConsume 或调了也不拦」）；本 story 只提供「有没有额度 / 扣一次 / 查状态」三个无颜色语义的原子能力，**不在此埋红色特判**（**L0**，AC 与代码注释显式声明，避免后续 story 误以为额度层已处理红色）

## Dev Notes

> Epic 2 开篇、资金之上第一个收费场景地基。核心难点只有一个：**并发不超发**（用单行原子条件 UPDATE，别用应用层读改写）。其余是标准 CRUD + 配置。**先读 scratchpad/stash_draft.diff 的草稿**——实现思路正确、可直接采用，但**测试和配置注册是必须补的缺口**。

### 承接（已 done / 既有可复用范式）
- **1.2 并发范式（直接照抄思路）**：`PawCoinWalletRepository.applyDelta(userId, delta)` = 单行原子条件 UPDATE（`WHERE balance+delta>=0`，返回行数）+ `insertIfAbsent`（`ON CONFLICT DO NOTHING`）。本 story 的 `tryConsume`（`WHERE used_count < limit`）与 `insertIfAbsent`（`ON CONFLICT (user_id,period)`）是同一模式的移植。`PawCoinWalletService` 的 `@Transactional` 用法同样可参照。
- **1.3/1.5 配置渐进范式**：`TopupTierProvider`（充值档位）、`PAWCOIN_TOPUP_PAUSED`（暂停开关）都走「本 story 先 env / `@ConfigurationProperties`，Epic 9 后台落地后换 DB 读」。本 story `defaultFreeQuota` 照此——9-2 `pricing_config.monthly_free_quota` 是最终归宿，现在先 env。
- **JWT 取用户范式**：`PawCoinController` / 既有 `TriageController` 都从 `@AuthenticationPrincipal Jwt` 取 `sub` 作 userId，`AppException.unauthorized` 兜 null/非法。`/api/v1/me` 主体（决策 C1，不用 `/users/me`）。
- **triage 模块已存在**：`com.tailtopia.triage.*`（V1.0 分诊）已有 domain/web/service 分层，本 story 往里加 `domain.UserMonthlyFreeQuota` / `repository.UserMonthlyFreeQuotaRepository` / `service.FreeQuotaService` / `web.FreeQuotaController` / `dto.FreeQuotaView`；配置类放 `shared.triage.TriageProperties`（草稿如此，与既有 shared 惯例一致）。

### 后端要点（核心）
- **并发不超发（AC2，最重要）**：唯一正确解是**数据库单行原子操作**——`INSERT ... ON CONFLICT DO NOTHING` 保证并发建行只成一行，`UPDATE ... WHERE used_count < :limit` 的条件在行锁下串行化，返回 0 即已满。**绝不能**「先 `findByUserIdAndPeriod` 读 used、Java 里 +1、再 save」——两个并发事务都读到 used=0 会各自写 used=1，超发。`used_count>=0` CHECK 是兜底不是主防线。
- **period WIB（AC3）**：`YearMonth.now(ZoneId.of("Asia/Jakarta"))`。**这是全项目唯一有意用非 UTC 的地方**（项目铁律是时间戳一律 UTC 存储）——但 period 是「业务归属月份」标签不是时间戳，用户在雅加达时间 7 月 1 日 06:00 应算 7 月（UTC 还是 6 月 30 日 23:00）。代码注释务必写清「有意为之，勿订正为 UTC」，否则后人 code-review 会当 bug 改。存库的 `created_at/updated_at` 仍是 UTC timestamptz，只有 `period` 字符串用 WIB 算。
- **clamp [0,35]（AC4）**：`Math.max(0, Math.min(props.getDefaultFreeQuota(), 35))`。`0` 合法（全付费）。`limit<=0` 时 `tryConsume` 直接 false 短路（不必碰库）。
- **不感知红色（AC6）**：本 service 三个方法都无「评级颜色」参数。红色永不锁的护栏在 2-3 解锁流：红色 `unlock_source` 恒 `FREE`/`RED_FREE`，要么不调 `tryConsume`、要么调了额度满也照样放行（策略在调用侧）。**本 story 别加 `if (isRed)` 特判**——那会把安全策略散落到额度层，违反「安全规则层只升不降、单点」。AC 里显式声明是为了让 2-2/2-3 的实现者知道额度层是「哑」的。
- **@Transactional 边界**：`tryConsume` 用默认 REQUIRED——独立调用自开事务；2-3「生成结果+扣额度+写解锁来源」同一事务调用时合并进外层（架构 NFR-3「同一事务」）。`hasFreeQuota`/`status` 用 `readOnly=true`。
- **不接调用点**：本 story triage 生成/解锁链路**一行不改**。`tryConsume`/`hasFreeQuota` 现在只有测试和未来 2-3 调。这是有意的「先建能力、后接线」，避免本 story 触碰 2-2/2-3 的复杂事务。
