# Story 3.2: 免费入队与广播

Status: review

> V1.1 **Epic 3 第二个** story，**纯后端**。承接 3-1（`consult_requests` 付费前 CAS 两表已建）。用户免费发起咨询 → 建 `consult_requests`(QUEUEING)**不扣费、不建订单** + 广播给所有在线兽医（FR-22E）；入队 1 分钟无兽医接单 → 请求**静默物理删除、不生成订单、无痕**（A-5）。`@Scheduled` 扫入队超时，**无 MQ / 无延迟队列**（enforcement 护栏）。**无迁移**（3-1 已建表）。
> 源：`epics-v1.1.md` Story 3.2（`POST /consultations` 不扣费不建订单；1min 无接单静默消失无痕；`@Scheduled` 扫超时 + 广播在线兽医 FR-22E，无 MQ）· 架构 `architecture-v1.1-delta.md` §3.2/§5.1（免费入队→限时支付流、服务端权威计时）· 决策 A-5（未扣费不建单）。
> **⚠️ brownfield 关系（务必理解）**：V1.0 已有免费直连流 `POST /consult-sessions`→`consult_sessions`(WAITING)+`ConsultQueueService`(Redis ZSET)+`ConsultRequestQueuedEvent`→`ConsultNotifyListener` 广播在线兽医。**V1.1 计费流用新 `POST /consultations`→`consult_requests`(QUEUEING)**，与 V1.0 会话流并存（本 story 不删旧端点；旧「接单即免费会话」被计费流取代由后续 story/前端切换，3-2 只加新入队+广播+超时删）。**广播/在线兽医基建直接复用**（`VetPresenceService.onlineVetIds()` + `NotificationType.NEW_CONSULT_REQUEST` + notify listener 范式）。
> **范围边界**：`POST /consultations`（建 QUEUEING 请求 + 广播）+ 入队超时 `@Scheduled` 删行。**不做**：兽医接单（3-3 CAS `tryAccept`）、限时支付/建会话（3-4）、前端下单 UI（3-5）、兽医端队列 UI（3-6）、既有 consult_sessions/V1.0 流改动、扣费/建订单（本 story 明令不建单）。

## Story

As a 用户,
I want 免费发起咨询进入排队并广播给在线兽医,
so that 我不用先付费就能求诊（FR-43D）。

## Acceptance Criteria

1. **后端：免费发起入队（不扣费、不建订单）（L1）**
   **Given** 3-1 的 `consult_requests`(QUEUEING) + `ConsultRequestRepository`
   **When** `POST /api/v1/consultations`（JWT USER，body `{petProfileId, symptomText?, imageObjectKeys?}`）
   **Then** 建 `consult_requests`：`state=QUEUEING`、`request_token`（`CardTokenGenerator` 不可枚举）、`user_id`（JWT）、`pet_profile_id`（校验属本人）、`queue_deadline_at = now + 1min`、`rebroadcast_count=0`；**绝不扣费、绝不建 `consult_orders`**（A-5「未扣费不建单」）；返回 `{requestToken, state, queueDeadlineAt}`（202/200）（**L1**）
   **And** **占用校验（FR-4B 同时仅 1 个）**：该用户已有 live `consult_requests`（QUEUEING/ACCEPTED_AWAIT_PAY——即表内任一行，取消/超时已删故「存在即 live」）时**不新建**，返回现有请求（`alreadyActive` 语义，200，前端跳「进行中」）（**L1**）

2. **后端：广播给所有在线兽医（FR-22E，复用现有基建）（L1）**
   **Given** `VetPresenceService.onlineVetIds()`（Redis `vet:online` 显式在线态）+ `NotificationType.NEW_CONSULT_REQUEST` + `notificationService.sendToVet`（既有 `ConsultNotifyListener` 范式）
   **When** 入队成功后
   **Then** 发领域事件（如 `ConsultRequestQueuedForBillingEvent(requestId)`，跨模块经事件、**不直访 vet repo**）→ notify 侧监听 → 遍历 `onlineVetIds()` 向**在线**兽医推 `NEW_CONSULT_REQUEST`（**离线不推**，FR-22E）；事件 `AFTER_COMMIT` 确保请求已落库（**L1**）
   **And** 广播**不透传在线兽医人数/身份**给用户（隐私，照既有 `ConsultAvailabilityController` 惯例）（**L1**）

3. **后端：入队超时静默删除（无痕，无 MQ）（L1）**
   **Given** `queue_deadline_at`（入队 1min，服务端权威计时）
   **When** 到期仍无兽医接单（仍 `QUEUEING`）
   **Then** `@Scheduled`（`fixedDelayString`，照 `ConsultCloseScanner` 范式，**禁 MQ/延迟队列**）周期扫 → **物理删** `state=QUEUEING AND queue_deadline_at < now` 的行（`ConsultRequestRepository.deleteExpiredQueueing(now)` 批量 `@Modifying DELETE`）；**不生成订单、不留痕、不迁移状态**（A-5）；已被接单（`ACCEPTED_AWAIT_PAY`）的行**不删**（state 谓词保护）（**L1**）
   **And** 删除幂等、可重扫（已删则命中 0 行）；扫描异常不 crash 调度（try/catch + 日志，照 `ConsultCloseScanner`）（**L1**）

4. **后端：端到端无痕 + 契约（L0/L1）**
   **Given** 完整免费入队生命周期
   **When** 发起 → 超时无接单 → 扫描
   **Then** 请求最终**从 `consult_requests` 消失**、`consult_orders` **零新增**、无「已取消」态残留（订单中心干净，A-5）；`POST /consultations` 缺 JWT 401、非法 body 422、他人 pet_profile 403/校验失败；DTO 契约（`requestToken`/`state`/`queueDeadlineAt`）（**L0** DTO/校验 / **L1** 全链路）

## Dev Notes

> 纯后端。**90% 复用既有基建**（在线兽医查询、广播 notify、@Scheduled 范式、token 生成、占用校验思路）。核心新增：新入队端点写 `consult_requests`(QUEUEING) + 超时**物理删**（无痕，区别于 V1.0「WAITING 保留态」）。**不建订单、不扣费**是本 story 的红线（A-5）。

### 承接现状（brownfield，已读，照此复用/勿改旧流）
- **3-1（已 review）**：`ConsultRequest`（`state` QUEUEING/ACCEPTED_AWAIT_PAY，`queue`工厂建 QUEUEING）；`ConsultRequestRepository`（`tryAccept` CAS、`deleteIfQueueing`、`findByRequestToken`）。**取消/超时=物理删行、无 CANCELLED**——故 `consult_requests` 内「存在即 live」，占用校验只需 `existsByUserId`。
- **V1.0 免费直连流（不动）**：`ConsultSessionService.createWaiting`（占用校验 FR-4B「同时仅 1 个」→ 命中现有返回 alreadyActive；否则建 `consult_sessions`(WAITING) + `queue.enqueue`(Redis) + 发 `ConsultRequestQueuedEvent`）；`WAITING_TIMEOUT_SECONDS=60`（V1.0 **超时不删、保留 WAITING** 供前端弹继续等待——V1.1 计费流不同，**超时物理删无痕**）。**本 story 新建平行入队（consult_requests），不改 ConsultSessionService/consult_sessions。**
- **广播基建（直接复用）**：`VetPresenceService.onlineVetIds()`（Redis `vet:online` ZSET，显式在线态，无 TTL 心跳兜底——见记忆库 vet-presence-explicit-only）；`ConsultNotifyListener.onConsultRequestQueued`（`@TransactionalEventListener` → 遍历 `onlineVetIds()` → `notificationService.sendToVet(vetId, NotificationType.NEW_CONSULT_REQUEST, ...)`，**离线不推** FR-22E）。本 story 发**新事件** `ConsultRequestQueuedForBillingEvent`（携 requestId，与 V1.0 `ConsultRequestQueuedEvent` 携 sessionId 区分，避免两流事件混淆）+ notify 侧加一个监听（或扩 `ConsultNotifyListener`），广播逻辑照抄。
- **@Scheduled 范式（直接复用）**：`ConsultCloseScanner`（`@Scheduled(fixedDelayString="${...ms:60000}")` → service 方法 → try/catch 日志不 crash，**禁 MQ**）。本 story `ConsultRequestTimeoutScanner` 同构。1min 队列超时用 60s 扫足够（或更密 30s，dev 定，配 property）。
- **token/校验**：`CardTokenGenerator.generate()`→`request_token`；pet_profile 属本人校验复用既有（profile 侧或 consult 既有直连病例校验，见 `ConsultSessionService.createWaiting` 的 `bindDirectCase`）。

### 关键设计（dev 照此）
- **不建订单红线（A-5）**：本 story `POST /consultations` 只写 `consult_requests`，**绝不碰 `consult_orders`**。订单在支付成功（3-4）才建。L1 断言 `consult_orders` 零新增是硬指标。
- **超时物理删无痕**：`deleteExpiredQueueing(now)` = `@Modifying DELETE WHERE state=QUEUEING AND queue_deadline_at < :now`。**state 谓词保护**：已被接单（ACCEPTED_AWAIT_PAY，pay_deadline 另算）的行不被队列扫描删（其超时由 3-4 支付窗扫描管）。物理删=无痕（不留「已取消」，A-5「订单中心无已取消态」的入队侧对应）。
- **服务端权威计时**：`queue_deadline_at` 是 DB timestamptz（`now + 1min`），扫描以 DB 时间为准，不信客户端。前端（3-5）展示同一倒计时。
- **占用校验（FR-4B）**：`existsByUserId(userId)` on `consult_requests`（存在即 live）。**跨流占用**（用户已有 V1.0 active `consult_session` 或 IN_PROGRESS `consult_order` 时能否再发起）——本 story 只管 `consult_requests` 单流占用；跨流统一留 [OPEN]（两流并存过渡期，前端引导单一入口）。
- **广播用 AFTER_COMMIT**：请求落库后再推（notify 既有范式），避免推了但事务回滚。**注意**：与资金入账的「禁 AFTER_COMMIT」不同——那是**入账写**必须同事务；**通知推送**用 AFTER_COMMIT 是对的（已落库才推，不吞资金写）。两者场景不同，勿混。
