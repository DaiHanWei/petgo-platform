# Story 5.1: 订单聚合接口

Status: review

> V1.1 **Epic 5 第 1 story**（订单中心开篇），**纯后端**（泛化订单聚合读接口 `GET /orders`，L1）。一个接口聚合多类支付订单为统一卡片契约，供 5-2 列表页 / 5-3 详情渲染。**无 Flyway 迁移**（3 类订单源表 `consult_orders`/`ai_consult_orders`/`payment_intents` 均已建，本 story 只读聚合 + 加 getter/repo 游标查询）。
> 源：`epics-v1.1.md` Story 5.1（`GET /orders` 泛化聚合 + 游标分页 + 类型筛选 + PawCoin 汇总 + 待接单 amount null + 退款处理中语义支持前端 info 映射）· FR-54 · UX-DR7 · 架构 `architecture-v1.1-delta.md` §6（`/orders` line 229）· §订单中心泛化契约（line 241）· A-5（两表拆分：`consult_requests` 不进订单中心）· P0（订单模型泛化 4 类）。
> **⚠️ 承接现状（已核实）**：3 类订单源已建——① **兽医** `consult_orders`（ConsultOrder，3.x：order_token/amount/status[IN_PROGRESS/COMPLETED/REFUNDING/REFUNDED]/refund_rejected/pay_channel/paid_at/created_at）② **AI 解锁** `ai_consult_orders`（V65，`triage.domain.AiConsultOrder`：order_token/amount/status[PENDING_PAYMENT/COMPLETED/ABNORMAL]/pay_channel/paid_at/created_at/triage_task_id）③ **PawCoin 充值** `payment_intents`（V60，purpose=`PAWCOIN_TOPUP`：public_token/amount/status[PENDING/PAID/FAILED/EXPIRED]/channel/created_at）。`PawCoinWalletService.balanceOf(userId)` 已建（余额汇总）。`PawCoinQueryService` 游标分页范式（`findByUserIdAndCreatedAtLessThanOrderByCreatedAtDesc`）。**无既有 `/orders` 控制器**——全新建。
> **决策（本轮已与用户对齐）**：① **本 story 聚合 3 类**（兽医/AI/充值）；**HD 下载无领域订单表**（`PaymentPurpose.ID_HD` 枚举存在但 HD 流程属 Epic 6 未建）→ `orderType` 泛化契约**预留 ID_HD 但本 story 不查**，Epic 6 建 HD 流时零改接口接入第 4 类。② **仅已付/持久订单入订单中心（遵 A-5）**：`consult_requests`（待接单 QUEUEING / 待支付 ACCEPTED_AWAIT_PAY，取消即删不留痕）**不进订单中心**；订单中心=持久已付订单。兽医订单实际 4 态（非 UX 描述的 6 态——那 6 态含下单流的待接单/待支付，属 3-5/3-6 下单链路非订单中心）；amount 恒非 null（`amount 待接单可为 null` 是泛化契约的防御位，本 story 兽医订单不产 null）。
> **范围边界**：本 story = 泛化聚合读接口（3 类 + 游标 + 类型筛选 + PawCoin 汇总 + 统一卡片契约 + 退款态 statusColor=info 语义）。**不做**：前端卡片/列表（5-2）；订单详情各态/失效态（5-3）；HD 订单（Epic 6）；待接单/待支付（下单流 3-5/3-6，A-5 不进订单中心）。**无迁移**（可选加 (user_id, created_at) 索引提游标性能，V1 低量不做，记 OPEN）。

## Story

As a 平台/用户,
I want `GET /orders` 泛化聚合兽医/AI/充值多类订单为统一卡片契约,
so that 前端一个接口渲染全部订单、退款中显 info 蓝（FR-54，UX-DR7）。

## Acceptance Criteria

1. **后端：`GET /orders` 泛化聚合 + 游标分页（L1）**
   **Given** 当前用户有多类订单（兽医 `consult_orders` / AI `ai_consult_orders` / 充值 `payment_intents.PAWCOIN_TOPUP`）
   **When** `GET /api/v1/orders?cursor=&limit=`（JWT role=USER，仅本人——照 `PawCoinController` 取 jwt.sub，绝不接受任意 userId，C1）
   **Then** 返回统一卡片视图列表（`OrderSummaryView`：`orderType`/`orderToken`/`statusCode`/`statusColor`/`amount`(可空 Long)/`payChannel`/`createdAt`），**跨 3 源按 `created_at` 倒序合并** + 游标分页（`nextCursor`=末条 createdAt epochMillis，`hasMore`）+ **PawCoin 余额汇总** `pawcoinBalance`（`balanceOf`）（**L1**）
   **And** 仅本人订单；`limit` 默认 20 上限 50（**L1**）

2. **后端：类型筛选（L1）**
   **Given** `?type=VET_CONSULT|AI_UNLOCK|PAWCOIN_TOPUP`
   **When** 带 type 请求
   **Then** 仅聚合该类订单源（无 type=全 3 类）；非法 type→400 或忽略（dev 定，建议 400 Bean 校验）（**L1**）
   **And** `ID_HD` 类型泛化预留但本 story 无源→空（Epic 6 接入）（**L1**）

3. **后端：退款处理中 statusColor=info（非 error）语义（L1）**
   **Given** 兽医订单 `status=REFUNDING`（退款处理中）
   **When** 映射卡片
   **Then** `statusColor=INFO`（前端映射蓝，**非 error 红**，UX-DR2/DESIGN.delta 组件②）；后端权威返 statusColor 语义（WARN/INFO/SUCCESS）供前端着色（**L1**）
   **And** 各态映射：兽医 IN_PROGRESS→INFO / COMPLETED→SUCCESS / REFUNDING→INFO / REFUNDED→SUCCESS；AI COMPLETED→SUCCESS；充值 PAID→SUCCESS（**L1**）

4. **后端：仅已付/持久订单（遵 A-5，L1）**
   **Given** 用户有进行中 `consult_requests`（待接单/待支付）
   **When** 聚合订单中心
   **Then** `consult_requests` **不入订单中心**（A-5）；聚合仅：`consult_orders`（全 4 态）+ `ai_consult_orders`（`COMPLETED`）+ `payment_intents.PAWCOIN_TOPUP`（`PAID`）（**L1**）
   **And** 兽医订单 `amount` 恒非 null（本 story 不产待接单 null 卡；`amount` 契约可空为 HD/未来预留）（**L1**）

5. **契约 + 验收（L0/L1）**
   **Given** 全部实现
   **When** 验证
   **Then** `DB_NAME=petgo_l1 mvn -B test` 绿：L0（DTO 契约、statusColor 映射纯函数）+ L1（3 类聚合倒序合并 + 游标翻页 + 类型筛选 + PawCoin 汇总 + REFUNDING→INFO + 仅本人 + consult_requests 不入）；schema validate 一致（**L0/L1**）
   **And** 不破坏 consult/ai/pay 既有查询（回归绿）（**L1**）

### 承接现状（已核实，照此复用/勿改）

- **3 类订单源**：① `consult_orders`（ConsultOrder）4 态 + refund_rejected；② `ai_consult_orders`（`triage.domain.AiConsultOrder`）——聚合仅 `COMPLETED`（`PENDING_PAYMENT`=现金到账前锚态、`ABNORMAL`=对账态，**非用户凭证不入**，OPEN-2）；③ `payment_intents`（PaymentIntent）purpose=`PAWCOIN_TOPUP`——聚合仅 `PAID`（PENDING/FAILED/EXPIRED 为过程态不入，OPEN-2）。
- **游标分页范式**：`PawCoinQueryService.view` + `findByUserIdAndCreatedAtLessThanOrderByCreatedAtDesc(userId, cursor, PageRequest.of(0, limit))`。cursor=createdAt epochMillis 字符串（首页传 now/MAX）。本 story 三源各按此查，service 层合并。
- **PawCoin 汇总**：`PawCoinWalletService.balanceOf(userId)`（无钱包→0）。
- **用户端点**：`/api/v1/orders`——JWT sub→userId（照 `PawCoinController.currentUserId`，绝不接受 body/path userId，C1）；limit 钳制（默认 20 上限 50）。SecurityConfig：`/api/v1/orders` 未在既有 matcher → 加 `hasRole('USER')`（照 support-tickets/refund matcher，4-5/4-7 已加范式）。
- **A-5 两表拆分**：`consult_requests`（QUEUEING/ACCEPTED_AWAIT_PAY）取消/超时即删、**不进订单中心**；订单中心只聚合 `consult_orders`（付费成功才建）。

### 关键设计决策（dev 照此）

- **D-1 聚合 3 类，HD 预留（用户拍板）**：`OrderType` 含 ID_HD 但本 story 无源不查；Epic 6 建 HD 订单流后加一源查询即接入，`OrderSummaryView` 契约零改。
- **D-2 仅已付/持久订单（遵 A-5，用户拍板）**：consult_requests 不入；兽医订单 amount 恒非 null（`amount` 契约留 nullable 仅为泛化/HD 预留防御）。
- **D-3 statusColor 后端权威（WARN/INFO/SUCCESS）**：后端按 orderType+statusCode 计算 statusColor 语义返前端着色。**退款处理中 REFUNDING→INFO(蓝)非 error(红)**（UX-DR2/DESIGN.delta 组件②，安全攸关体验：退款不制造焦虑红）。映射纯函数 L0 可断言。**title/subtitle 不由后端下发**（App 按 orderType+statusCode 本地化，[[petgo-i18n-model-and-debt]]；后端只回结构化 code + amount + token + 时间）。
- **D-4 跨源 in-memory 合并游标**：各源查 `createdAt < cursor` 取 `limit+1` → 映射 → 归并排序（createdAt desc）→ 取 `limit`，`nextCursor`=末条 createdAt.millis，`hasMore`=合并集 > limit。cursor 单调（createdAt）跨源全局有序。V1 低量可接受；高量优化（keyset union / 物化视图）留后续（OPEN-3）。
- **D-5 类型筛选**：`type` 指定则只查该源；无则 3 源。非法 type→400（Bean/枚举校验）。
- **D-6 无迁移**：3 源表已建；仅加 getter + 派生 repo 查询方法（无 schema 改）。可选 (user_id, created_at DESC) 索引提游标性能，V1 低量不做（OPEN-4）。
