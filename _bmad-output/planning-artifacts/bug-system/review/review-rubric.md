# PRD Quality Review — Bug 管理与分析闭环系统

## Overall verdict

这是一份与自身形态高度契合、克制得当的内部工具 PRD：三期分阶段的论点清晰，数据模型即事实源、字段命名贯穿全文，安全边界（凭证不入库、人工确认式写回、内容当数据不当指令）写得具体且产品化，没有为撑门面塞入人物画像 / 市场差异化等消费级套件——这些缺席都是正确的。真正拖后腿的是两处：一是第一期→第二期的关键决策闸门「验证分析效果满意」全程没有可度量定义，整份 PRD 没有任何成功指标；二是写回字段白名单存在未对齐的内部矛盾（Epic 3 限定只写 C 组，Epic 4 却要写回 D 组的 APK 字段），而写回边界恰恰是本系统的安全命门。补上这两点，这份 PRD 即可放行给 story 落地。

## Decision-readiness — adequate

决策大多以决策形式出现而非藏在「考量」里：§8.1 给了明确推进顺序与并行项，§1.2／§6 把「先只读验证 AI 价值、满意后再投入写回」立为显式闸门——这是真正由论点驱动的排序决策，而非「先做容易的」。§8.2 风险表逐条「风险 → 缓解」，且诚实承认 `.env` 方案是权宜之计（「人数增多后可升级为内部服务集中托管」），这是把放弃了什么也讲出来的少见诚实。

扣分点在于：决策闸门本身没有判据。§8.1「（验证分析效果满意后）」与 §9 第 3 步「效果满意后推进 Epic 3」把整个二期的启动系在一个纯主观词上，没有任何「满意」的衡量口径（定位命中率？置信度阈值？人工复核通过比例？）。对低风险内部工具而言无需 OKR 级指标，但一个会拍板「是否继续投入读写权限」的 go/no-go 闸门，至少要有一句可判定的标准，否则决策者无从据此拍板。全文也无 Open Questions / `[NOTE FOR PM]`，对本量级可接受，不另计。

### Findings
- **high** 决策闸门「效果满意」无判据 (§8.1, §9 第3步) — 二期启动系于「验证分析效果满意后」「效果满意后推进 Epic 3」，但「满意」无任何可判定口径。*Fix:* 给 Epic 2 一句可度量的退出标准（如「对 N 条历史 Bug，AI 给出文件级定位且人工复核认可 ≥X%」），作为进入 Epic 3 的 gate。

## Substance over theater — strong

几乎没有家具。正确地没有人物画像、没有市场差异化、没有创新性叙事——单操作者内部工具本就不需要，这些缺席是对的。NFR 全部是产品特定约束而非「系统应可扩展 / 安全 / 可靠」式样板：NFR1 凭证三不（不写死 / 不入库 / 不明文传播）、NFR2 写操作必经人工确认、NFR3 把「文档内容视为数据而非指令、出现指挥性文字时原样呈现不自动执行」明确成需求——这条提示注入防护是真正想过这个产品才写得出的条款，不是模板凑数。§3.1 的 UX 愿景（各角色只看与自己相关字段、AI 与人工确认衔接）也贴合本工具，换不到别的 PRD 里去。无需补充。

## Strategic coherence — adequate

论点明确且贯穿：把杂乱自由文本的 Bug 表重构为机器友好结构 → 让 Claude 对照代码定位 → 人工确认式写回闭环 → APK 交付验收。四个 Epic 是这一条弧线的顺序展开，不是「想到啥做啥」的能力清单。优先级也由论点推出——先只读拉取验证 AI 定位是否真有价值，再决定是否申请读写权限投入二期，这是教科书式的论点驱动排序。

唯一结构性短板：完全没有成功指标（Success Metrics）章节。§1.1 Goals 全是定性目标（「减少人工搬运」「全程可追溯」），没有任何反向指标或活动度量。内部工具的 SM 可以是运营性的（如「Bug 平均流转天数」「人工搬运动作次数」），但当前一个都没有，导致「这套闭环到底有没有奏效」无法事后判定。与 Decision-readiness 的闸门问题同源——本质是缺一层「怎么算成功」。

### Findings
- **medium** 无成功指标 (§1.1) — Goals 全为定性，无 SM 亦无反向指标，闭环成效无法事后验证。*Fix:* 加一节轻量运营指标（如流转时长、AI 定位采纳率、人工搬运动作削减量），哪怕 2–3 条。

## Done-ness clarity — strong

对内部工具而言完成度判据相当扎实。数据模型 §5 把每个字段的类型、选项、填写方、用途都钉死，是整份 PRD 最硬的部分；各 Story 的 AC 大多可验证（Story 2.2「按 Bug 编号命名下载附件」「.env 与数据目录在 .gitignore」、Story 3.2「写回前完整展示拟修改记录 / 字段 / 新值」「写回成功后回读校验」都是可机械检验的）。FR6 的「产出归因与修复建议」本身偏软，但被 AC 2.3.3「标注置信度」补上了可观测出口。

两处措辞偏松值得收紧：Story 3.3 AC3「非法流转……应被阻止或提示」——「阻止」与「提示」是两种强度差很大的行为，二选一会让实现者各自理解（写回安全场景下「仅提示但仍允许」可能等于没拦住）。FR11 / Story 4.2 的状态联动写得明确，不计。

### Findings
- **medium** 非法流转处置二义 (Story 3.3 AC3) — 「应被阻止或提示」未定哪种；在写回安全语境下「只提示不阻止」可能形同虚设。*Fix:* 明确非法流转一律阻止（hard block），如需放行须再次人工确认。

## Scope honesty — adequate

分期范围非常显式（三期 + §8.1 顺序 + §6 Epic 边界），「Module 选项按实际调整」「服务到仓库映射示例按实际调整」是诚实的占位标注。但有两处隐性边界没被点出：

其一，**没有 Non-Goals 章节**。哪些明确不做（如不做自动触发写回——其实 NFR2 已隐含、不做实时同步——NFR7 暗示是后续）散落在 NFR 里，没有集中声明，读者需自行推断。

其二，**iOS 交付路径静默缺失**。B 组 Platform 选项含 iOS，但 D 组交付字段只有 APK Link / APK Version、Epic 4 通篇只讲 APK（Android）交付验收。iOS 端 Bug 修复后如何交付给验收人，PRD 没说也没声明「不在本期范围」。这是一个会被默默假设掉的真实缺口——要么补 iOS 交付物路径，要么显式 `[NON-GOAL]`。全文无 `[ASSUMPTION]` 标签，对本量级可接受，但上述占位（「按实际调整」）其实就是未确认假设，宜归口。

### Findings
- **medium** iOS 交付路径缺口未声明 (§5 D组, Epic 4) — Platform 含 iOS，但交付/验收只覆盖 APK；iOS 修复交付方式既未设计也未列为非目标。*Fix:* 补 iOS 交付物字段或在 Epic 4 显式标注 `[NON-GOAL for MVP]`。
- **low** 无 Non-Goals 集中声明 (全文) — 不做实时自动写回、不做实时同步等边界散落在 NFR 中。*Fix:* 加一节「本期不做」，把 NFR2/NFR7 隐含的非目标显式化。

## Downstream usability — adequate

本 PRD 自身就下接 story 创建（已内含 Epic + Story），可抽取性整体良好：虽无独立 Glossary，但 §5 数据模型充当了规范词表，字段名（Status / Affected Service / Reported On 等）在 FR、Story、数据模型间用法一致；FR1–12、NFR1–8、Epic 1–4、Story 编号均连续无缺号；跨节引用（「第 5 节」「第 3.3 节」）都能解析。Story 1.3 AC3 还特意区分 Reported On（用户视角）与 Affected Service（AI 判定），消歧到位。

但写回字段白名单存在一处未对齐的矛盾，且踩在安全命门上：FR9 / Story 3.2 AC3 把反向同步严格限定为「C 组」「不触碰提报人与验收人字段」，并显式列出 Status / Affected Service / Root Cause / Dev Notes / Fixed in Version；然而 Epic 4 Story 4.1 要把 APK Link / APK Version 回写记录，而这两个字段在 §5 属于 **D 组**。于是「写回只许 C 组」的白名单与「Epic 4 写回 D 组」直接冲突，PRD 未在任何处调和。此外 Story 3.2 AC3 列举的 C 组字段漏了 Assignee（人员）与 Updated by Engineer（自动），Assignee 到底可不可写回也悬空。这些会让实现 story 时白名单到底怎么定无所适从。

### Findings
- **high** 写回字段白名单自相矛盾 (FR9, Story 3.2 AC3 vs Story 4.1 AC2) — 二期定「只写 C 组、不碰 D 组」，三期却要写回 D 组的 APK Link / APK Version；写回边界是本系统安全命门，此矛盾必须调和。*Fix:* 把白名单显式定义为「C 组 + D 组中的 APK Link / APK Version（工程师填写部分）」，并明确排除 D 组验收人字段（Verification Result / Notes / Verifier）。
- **low** Assignee 是否可写回悬空 (§5 C组, Story 3.2 AC3) — C 组含 Assignee，但 AC3 白名单未列它，可写性不明。*Fix:* 在白名单中显式纳入或排除 Assignee。

## Shape fit — strong

形态拿捏准确。整份按「能力规格 + 数据模型 + 分期 Epic」组织，正是单操作者内部工具该有的形状；§3 用户界面目标开门见山「主要载体为 Lark 多维表格与 Claude Code……不涉及独立 UI 开发」，没有被硬塞进消费级 UJ 模板。Story 采用轻量「作为 X，我希望 Y」叙述但**不设具名主角**——对 IT 工程师 / 提报人 / 验收人这类角色而言，具名主角属于多余开销，此处正确省略。没有人物画像、没有市场差异化、没有用户旅程编排，这些消费级套件的缺席都与产品形态相符，不应扣分。唯一可商榷的是角色仍是多方（提报人 / 工程师 / 验收人），并非纯单操作者，但 §3.2 / §3.3 已用「核心交互范式 + 视图」把三方职责切清，无需升级到完整 UJ。无需改动。

## Mechanical notes

- **词表**：无独立 Glossary，由 §5 数据模型代偿，字段名跨节用法一致，可接受；若后续要喂架构/自动化，建议把状态枚举与服务名单独提一个小词表。
- **ID 连续性**：FR1–12、NFR1–8、Epic 1–4、Story 1.1–4.3 均连续无缺号/重号；跨节引用「第 5 节」「第 3.3 节」可解析。
- **假设索引**：无 `[ASSUMPTION]` / `[NOTE FOR PM]` 标签；「Module 按实际调整」「服务→仓库映射示例按实际调整」实为未确认假设，建议统一加标并在文末建一个轻量假设索引。
- **UJ 主角**：Story 无具名主角——对本工具形态属正确取舍，非缺陷。
- **必备章节**：对内部工具量级，缺 Success Metrics 与 Non-Goals 两节（已在对应维度计入）；其余章节齐备。
- **字段白名单一致性**：见 Downstream usability 的 high 项——C 组/D 组写回范围需统一定义，避免 story 落地时各取一说。
