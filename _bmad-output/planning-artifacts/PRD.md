# PRD：PetGo 宠物 App V1.0

## 0. 文档说明

本 PRD 面向产品、设计、开发及下游工作流负责人。功能需求以全局编号 FR-N 标注，用于跨文档稳定引用。假设项以 `[ASSUMPTION]` 标注并汇总于 §9。V1.0 阶段核心目标为用户引入与功能验证，全功能免费，不涉及支付体系。

输入来源：团队初版头脑风暴文档（Hex / Rose / Dai / Aria / Shawn / Albern）及头脑风暴提炼产出文档（2026-05-29）。

## 1. 产品愿景

PetGo 是印尼第一款以「宠物主人的全生命周期陪伴」为核心的宠物社区 App。它为雅加达及印尼各地的宠物主解决三个长期无解的痛点：宠物突发健康问题时找不到低价可信的专业指导、线下宠物社群活动散落在 Instagram 和 WhatsApp 里无处聚合、以及宠物成长的珍贵记忆没有一个专属的家。

V1.0 以两个核心功能构建最小完整体验：**Konsultasi Kilat**（专业问诊）提供 AI 驱动的初步分诊和免费在线兽医咨询，让深夜的焦虑宠物主通过手机获得专业指引；**Paspor Tumbuh Kembang**（成长档案）让每一位炫娃型宠物主拥有一张可分享的「宠物名片」，把生命中的美好瞬间永久留存。

两个功能相互连接——问诊记录沉淀为宠物健康历史，数据积累成为用户不愿离开的核心资产。宠物聚会活动（Gabung Gath）规划于 V2 版本。

## 2. 目标用户

### 2.1 用户核心诉求（Jobs To Be Done）

**功能性**
- 宠物出现症状时，快速获得「危险还是不危险」的初步判断，决定是否需要立刻就医
- 获得可执行的用药参考建议，不必为了小问题跑一趟诊所

**情感性**
- 深夜宠物生病时，不再孤立无援、极度焦虑
- 把宠物当孩子一样，留下一本完整、永久可看的「成长相册」
- 在社群里炫耀自己的宠物，获得同类人群的认同和互动

**社交性**
- 结识同品种/同爱好的宠物主，扩展线下社交圈
- 把宠物名片分享给朋友，让不用 App 的人也能看到自己的宠物故事

### 2.2 V1 明确不服务的用户

- 有宠物寄养需求的用户（V2+ 规划）
- 宠物商品买家（不做电商）
- 寻找宠物美容、洗澡等线下到店服务的用户（V2+ 规划）

### 2.3 核心用户旅程

**UJ-1. Putri——深夜发现猫咪异常，用 PetGo 完成分诊和问诊决策**

- **Persona + 场景：** Putri，大二学生，住 Depok，养猫，深夜 11:30 发现猫咪 Oyen 呕吐白色泡沫
- **入口状态：** 已注册登录；极度焦虑；不知道是否需要立刻去急诊
- **路径：**
  1. 打开 App → 进入问诊模块 → 选择 AI 问诊入口
  2. 上传猫咪呕吐物的照片 + 文字描述症状
  3. AI 返回黄色评级，包含三项输出：危险等级（黄色）/ 观察建议（「接下来2小时：若再次呕吐或精神萎靡加重→立即就医；若未再出现→明早预约即可」）/ 用药建议（居家处理方案）
  4. 结果页底部出现「想要更个性化的建议？」软性引导，Putri 进入兽医咨询入口
  5. 与合作机构派驻兽医实时图文沟通，进一步确认处理方案
  6. 问诊结束，App 询问「是否将本次咨询存入 Oyen 的档案？」Putri 确认存入
- **价值到达时刻：** Putri 看到具体的观察倒计时和用药建议，焦虑消解，不需要深夜出门
- **结束状态：** 回到首页；问诊记录存入成长档案
- **边缘场景：** 若 AI 给出红色评级 → 界面触发半屏红色遮罩 + 强文案「请立即带宠物就医」，询问是否需要打开地图导航；不出现兽医咨询升级入口

---

**UJ-2. Aurel——用 PetGo 记录宠物日常，并把宠物名片分享给闺蜜**

- **Persona + 场景：** Aurel，25岁，雅加达白领，养了一只橘猫 Mochi，习惯在 Instagram 上晒猫，想要一个专属于 Mochi 的「主页」
- **入口状态：** 首次注册，为 Mochi 创建档案（名字、品种、生日、一句话介绍「Mochi，3岁橘猫，Jakarta 最懒的男孩」）
- **路径：**
  1. 点击「+」→ 选择「成长日历」→ 选择「快乐时刻」→ 上传 Mochi 晒太阳的照片 + 一句话
  2. 发布成功，Mochi 的档案时间线新增一条记录
  3. 档案页右下角动效浮动按钮引导 Aurel 分享 → 生成精美 H5 宠物名片链接
  4. Aurel 把链接发到闺蜜 WhatsApp 群：「来看我家 Mochi！」
  5. 闺蜜点开链接，查看精美宠物名片预览（最近5条记录），想看更多 → 页面提示「下载 App 查看 Mochi 的完整成长故事」
- **价值到达时刻：** 闺蜜回复「好可爱！这个 App 是什么？」→ Aurel 成为自发传播者
- **结束状态：** Mochi 的档案持续积累，问诊记录存入健康历史
- **边缘场景：** Aurel 在问诊结束后确认存档 → 咨询记录以「健康事件」标记出现在 Mochi 时间线

## 3. 术语表（Glossary）

下游文档和团队沟通须使用以下术语，不得使用同义词替换。

| 术语 | 定义 |
|------|------|
| **宠物主（Pet Owner）** | App 的主要 C 端用户，拥有至少一只宠物 |
| **兽医（Vet）** | 通过合作机构接入的在线执业兽医或兽医学生，提供图文问诊服务 |
| **AI 分诊** | 用户上传照片（图文）后，系统自动返回绿/黄/红危险等级 + 观察建议 + 用药建议的能力 |
| **绿/黄/红评级** | 绿=无需就医；黄=观察期（附条件倒计时协议）；红=立即就医 |
| **条件倒计时协议** | 黄色评级下的结构化观察指引：明确时间窗口 + 升级触发条件 + 可执行处理建议 |

| **宠物聚会活动（Gath）** | 由用户发起的宠物主题线下聚会，V2 功能 |
| **临时群聊** | 绑定单次 Gath 的群聊频道，活动结束后7天自动关闭，V2 功能 |
| **GPS 签到** | 参与者到达活动现场后触发的位置验证动作，V2 功能 |
| **成长档案** | 单只宠物的完整生命记录，包含快乐时刻、问诊记录等内容 |
| **宠物名片** | 成长档案的对外 H5 分享页，展示宠物头像/品种/介绍和精选照片流 |
| **照片认领** | 群聊中他人拍摄的宠物照片，由宠物主一键归入自己宠物的成长档案 |
| **场地方（Venue）** | 提供宠物活动场地的商业机构（V1 不接入平台预订，V2 规划）|

## 4. 功能与需求

### 4.0 用户登录与引导（User Auth & Onboarding）

**功能描述：** V1 采用 Google 登录作为唯一登录方式。未登录用户可浏览首页内容，浏览至第3页时触发软性登录推荐，触发核心功能时跳出强引导弹窗。新用户完成 Google 授权后，依次完成昵称确认和宠物状态选择（三类，必选），进入定制化首页体验。选择「有宠物」的用户进入档案创建引导（可跳过），跳过后前3次重启 App 显示提示条。手机号+OTP 和 Apple 登录规划于 V2。

**功能需求：**

#### FR-0A：未登录状态首页浏览
未登录用户打开 App 后可直接浏览首页内容流，无强制登录拦截。

**后果：**
- 首页内容流对未登录用户可见
- 问诊入口、成长档案入口、发布「+」按钮均可见，点击后触发登录引导（FR-0C）
- 未登录状态下无法提交任何数据

#### FR-0B：滚动触发软性登录推荐
用户在未登录状态下浏览至首页第3页内容时，触发软性登录推荐浮层。

**后果：**
- 浮层样式：底部弹起半屏，主 CTA「Google 登录」样式显著
- 关闭按钮：小号、浅色，视觉权重明显低于主 CTA
- 用户关闭后：当次 session 内不再弹出此浮层
- 触发条件：仅未登录状态，每次 session 最多触发一次

#### FR-0C：核心功能触发登录引导
未登录用户点击需要登录的功能（AI 问诊、兽医咨询、发布内容、创建宠物档案）时，触发登录引导弹窗。

**后果：**
- 弹窗视觉强度高于 FR-0B（用户有明确操作意图）
- 弹窗包含：说明文案「登录后继续使用该功能」+ Google 登录按钮 + 关闭
- 用户登录成功后，自动回到触发点继续操作

#### FR-0D：Google 登录（V1 唯一方式）
用户通过 Google OAuth 完成登录/注册。首次授权自动创建账户，无需区分登录/注册入口。

**后果：**
- 调起系统 Google 账号选择器
- 授权成功后返回 App，获取：Google ID、Email、Display Name、头像
- 老用户：直接进入 App，回到触发点继续操作
- 新用户：进入注册引导流程（FR-0E → FR-0F）

`[ASSUMPTION: Google OAuth SDK 在印尼 Android/iOS 环境下稳定可用]`

**服务条款与隐私政策：**
- Google 登录按钮下方展示文字链接：「登录即表示同意《服务条款》和《隐私政策》」
- 《服务条款》和《隐私政策》为可点击链接，跳转对应 H5 页面
- 无需用户单独勾选确认（Text Link 模式，符合 App Store 审核要求）

#### FR-0E：新用户昵称确认
新用户首次 Google 授权后，进入昵称确认页。

**后果：**
- 昵称默认读取 Google 账户 Display Name，用户可修改
- 字数限制 ≤20 字
- 确认后进入宠物状态选择（FR-0F）

#### FR-0F：宠物状态选择
新用户完成昵称确认后，必须选择一项宠物状态。该选择影响首页内容推荐方向，后续可在个人设置中修改。

| 选项 | 标签 | 首页内容方向 |
|------|------|------------|
| A | 我有宠物，上传宠物档案 | 健康/问诊/档案为主 |
| B | 暂无宠物，但计划养 | 养宠知识/选宠指南为主 |
| C | 宠物爱好者，无养宠计划 | 宠物内容/社区为主 |

**后果：**
- 宠物状态为必选，不可跳过
- 选择 A → 进入宠物档案创建引导（FR-0G）
- 选择 B 或 C → 直接进入首页，不显示档案提示条
- 用户可在「个人设置 → 宠物状态」中随时修改

#### FR-0G：宠物档案创建引导（仅状态 A）
选择「我有宠物」的新用户完成宠物状态选择后，进入档案创建引导，复用 FR-11 创建表单。

**后果：**
- 页面底部提供「跳过，稍后创建」入口
- 完成创建 → 进入首页，不显示档案提示条
- 跳过 → 进入首页，触发提示条逻辑（FR-0H）

#### FR-0H：首页宠物档案提示条
仅针对「选择状态 A 但尚未完成宠物档案上传」的用户，首页顶部显示提示条。

**后果：**
- 提示条样式：首页顶部固定区域，含「立即创建」CTA 和关闭（X）按钮
- 用户关闭后：当次 session 不再显示，下次重启 App 继续显示
- 显示频次上限：前3次重启 App 均显示，第3次关闭后永不再显示
- 停止显示条件（满足任意一项）：①用户完成宠物档案上传；②超过3次重启关闭记录
- 选择状态 B 或 C 的用户：不触发此提示条

### 4.1 专业问诊（Konsultasi Kilat）

**功能描述：** 宠物主在宠物出现异常症状时，通过两个独立入口获得专业支持：AI 问诊（即时分诊，仅图文输入）或兽医咨询（实时聊天，支持文字/图片/视频）。AI 分诊返回危险等级、观察建议和用药参考三项内容；黄色区间附带条件倒计时协议；红色评级触发半屏强提醒并询问是否导航就医。平台定位为工具提供方，不承担医疗建议的法律责任；所有输出附有免责声明，最终决策权明确归属用户。V1 通过机构合作接入兽医资源，不做个人兽医注册认证体系。

**功能需求：**

#### FR-1：AI 图文分诊
用户可上传宠物照片（最多3张）并用文字描述症状，系统在提交后返回分诊结果。实现 UJ-1。

**媒体规格限制（AI 问诊上传）：**
- 图片：JPG / PNG / HEIC，单张 ≤ 10MB
- 最多上传 3 张图片
- `[NOTE: AI 分诊 V1 仅支持图文输入（图片 + 文字）；视频分诊移至 V2。技术决策 2026-06-01：采用 Gemini 单模型图文分诊]`

**后果（可测试）：**
- 提交后 ≤15 秒内返回分诊结果
- 结果同屏呈现三项内容：危险等级（绿/黄/红）、观察建议、居家用药参考
- 结果页底部显示免责声明，字体小号、次要色，不干扰主内容视觉层级

**AI 超时 / 服务异常处理：**
- 提交后超过 15 秒未返回结果 → 显示加载超时提示：「分析时间较长，请稍后重试」+ 「重新提交」按钮
- AI 服务异常返回错误 → 显示：「AI 服务暂时不可用，请稍后再试」+ 「重新提交」按钮 + 软引导「或直接联系兽医」跳转 FR-4B
- 重新提交不需要用户重新上传文件，复用上次提交内容

**绿色评级结果页（补充定义）：**
- 危险等级展示：绿色图标 + 「暂无紧急风险」文案
- 内容：居家观察建议（如正常饮食、休息等具体建议）
- 软性引导（非强制）：「想更放心？可以咨询兽医确认」→ 跳转 FR-4B
- 底部：「将本次结果存入档案」引导（触发 FR-16）

#### FR-2：黄色区间条件倒计时协议
当危险等级为黄色时，观察建议必须包含：具体观察指标、时间窗口、升级触发条件。

**后果：**
- 黄色结果页不得出现「不严重」「可以放心」等终结性表述
- 时间窗口和升级触发条件以视觉上可区分的方式展示（如独立卡片或加粗文字）

#### FR-3：红色半屏强提醒
当危险等级为红色时，界面触发半屏红色遮罩 + 强提醒文案（「⚠️ 请立即带宠物就医」）。不展示兽医咨询入口。

**后果：**
- 红色遮罩不可在5秒内关闭，防止用户误划过
- 遮罩内展示「是否需要打开地图导航前往附近宠物医院？」确认按钮，用户选择后方调用系统地图
- 系统地图传入搜索词「宠物医院/Klinik Hewan」

#### FR-4A：AI 问诊入口
用户可从问诊模块直接进入 AI 分诊流程。V1 免费，无支付步骤。实现 UJ-1。

**后果：**
- 入口与兽医咨询入口在问诊模块内平级展示
- AI 问诊结果页可展示「想咨询兽医？」软性引导，但兽医咨询入口独立可达

#### FR-4B：兽医咨询入口
用户可从问诊模块直接进入兽医图文咨询界面，与合作机构派驻兽医实时沟通。V1 免费，机构合作接入。`[ASSUMPTION: 合作机构已确定接入方式]`

**V1 限制：用户同时只能有 1 个进行中的兽医咨询。**

**后果：**
- 入口与 AI 问诊入口平级展示
- 在线状态展示为**概率性**（非实时人数）：显示常规可用时段，如「工作日 8:00–23:00 通常有兽医在线」
- 不展示实时在线人数，降低运营维护复杂度
- **已有进行中咨询时**：兽医咨询入口显示「查看进行中的咨询 →」，点击直接跳转至该会话，不允许发起新的咨询，直到当前会话结束

**无人接单超时处理（等待 1 分钟）：**
- 用户提交兽医咨询后，进入等待界面，显示「正在为你匹配兽医…」
- 等待超过 1 分钟仍无兽医接单 → 触发超时提示
- 超时页面展示两个选项：
  - 「继续等待」：重置等待计时，继续寻找可用兽医
  - 「先用 AI 分诊」：跳转至 FR-4A（AI 问诊），原兽医咨询请求保留在队列中
- 用户选择继续等待后，若后续有兽医接单，推送通知提醒用户（FR-22A）

**主动取消等待：**
- 等待期间显示「取消」按钮
- 用户点击「取消」→ 弹窗确认「确认取消本次匹配？」
- 确认后请求从兽医待接单队列中删除，返回问诊模块主页

**AI 升级至兽医时的数据传递：**
- 用户从 AI 分诊结果页点击「咨询兽医」升级时，将以下数据完整传递给接单兽医：
  - AI 危险评级（绿/黄/红）
  - 用户的症状文字描述
  - 用户上传的全部图片
- 兽医在「待接单」列表可预览上述信息，接单后在对话界面顶部展示完整 AI 上下文
- 用户无需在对话中重新描述或重新上传

**兽医无人在线时的状态（FR-4B 离线态）：**
- 进入兽医咨询入口后检测到无兽医可接单
- 页面显示：「当前暂无兽医在线」+ 预期恢复时段说明
- 软性引导文案：「先用 AI 分诊？AI 可立刻给出初步评估，等兽医上线后随时咨询」
- 引导按钮跳转至 FR-4A（AI 问诊入口）
- 不强制跳转，用户可选择留在页面等待

#### FR-5：兽医接单辅助
兽医接单后，系统提供辅助工具：同类症状历史判断摘要（若有）、无历史数据时展示 AI 参考回复、用户侧自动显示免责提示「本次建议为参考，最终决策权在您」。

**后果：**
- 历史判断库按症状关键词自动匹配，无需兽医手动搜索
- 免责提示在用户对话界面自动展示，无需兽医手动添加
- **数据库建立方式**：完全依赖真实问诊数据自动累积，平台上线初期数据库为空
- **冷启动阶段**：数据库为空时，仅展示 AI 参考回复，不展示历史判断摘要
- 随平台问诊数量增长，历史数据库逐步丰富，历史参考价值逐步提升

### 4.4 首页 Feed（Homepage Content Feed）

**功能描述：** 首页 Feed 汇聚全平台用户公开发布的三类内容（日常分享、专业知识科普、成长日历快乐时刻），按发布时间倒序排列。已登录用户根据宠物状态进行**硬过滤**，显示对应内容类型；未登录用户显示全部内容，不过滤。V1 不做关注/好友关系，所有内容均来自全平台公共内容流。平台上线前由运营团队提前准备种子内容填充 Feed，Feed 为空时提供空状态引导用户发布。

**功能需求：**

#### FR-17：首页 Feed 内容模型
首页展示全平台用户公开发布的内容，按发布时间倒序排列，根据用户宠物状态进行硬过滤。

**宠物状态过滤规则：**

| 用户状态 | 显示内容类型 |
|---------|------------|
| A（有宠物）| 三类全显示（日常分享 + 专业知识科普 + 成长日历快乐时刻）|
| B（计划养宠）| 专业知识科普 + 日常分享（**不显示**成长日历快乐时刻）|
| C（宠物爱好者）| 三类全显示 |
| 未登录 | 三类全显示，无过滤 |

**后果：**
- 已登录用户 Feed 仅显示与宠物状态匹配的内容类型（硬过滤，非权重调整）
- V1 不做关注/好友关系，Feed 为全平台公共内容流
- 排序为纯时间倒序，V1 不做算法推荐
- 用户修改宠物状态后，Feed 即时按新状态刷新

**Feed 卡片结构：**

| 元素 | 说明 |
|------|------|
| 作者头像 + 昵称 | 左上角；点击触发 FR-26 迷你主页预览卡 |
| 内容预览 | 正文前2行文字（超出省略）|
| 第一张图片 | 若内容含图则展示第一张，无图则纯文字卡片 |
| 点赞/评论数 | **不在卡片上展示**（点击卡片进入详情页后可见）|

- 点击卡片任意区域跳转至内容详情页（FR-28）
- Feed 采用**无限滚动**（Infinite Scroll）：用户滚动至距底部约5条时自动触发加载下一批内容
- 每次加载：20条内容（首次加载 20 条，后续每批 20 条）
- 支持下拉刷新（Pull to Refresh）获取最新内容

#### FR-18：冷启动内容与 Feed 空状态
平台上线前运营团队提前发布种子内容；Feed 内容为空时显示引导发布的空状态页面。

**后果：**
- 种子内容与用户内容在 Feed 中混合显示，不做特殊标记区分
- Feed 无内容时显示引导文案（「成为第一个分享的人！」）+ 发布「+」入口
- `[ASSUMPTION: 运营团队有管理员账号可提前在平台内发布种子内容]`

**上传失败处理（通用规则，适用于 FR-1 和 FR-12）：**
- 上传过程中网络中断或服务器错误 → 显示失败提示：「图片上传失败，请检查网络后重试」+ 「重试」按钮
- 点击「重试」仅重新上传失败的文件，用户已填写的文字内容**在当前 session 内存中保留**，无需重新填写
- **V1 不做持久化草稿箱**：用户关闭编辑页面或退出 App，内容直接清空，不存服务器
- 如需重新发布，用户需重新操作

### 4.5 App 导航结构（Navigation Architecture）

**功能描述：** App 采用5位置底部 Tab Bar 作为主导航。中间位置为凸起样式的「+」发布按钮，全局可达。首页为唯一无需登录可访问的 Tab；其余四个 Tab 均需登录，未登录点击时触发 FR-0C 登录引导弹窗。

**功能需求：**

#### FR-19：底部 Tab Bar 导航

**Tab Bar 结构（从左至右）：**

```
首页  |  成长档案  |  [+]  |  问诊  |  我的
```

| Tab | 图标位置 | 未登录 | 说明 |
|-----|---------|--------|------|
| 首页 | 1（最左）| ✅ 可访问 | 显示公共 Feed（FR-17）|
| 成长档案 | 2 | ❌ 触发 FR-0C | 默认展示当前用户自己的宠物档案 |
| [+] 发布 | 3（中间）| ❌ 触发 FR-0C | 凸起样式，全局可见；点击进入 FR-12 发布入口 |
| 问诊 | 4 | ❌ 触发 FR-0C | 进入专业问诊模块（FR-1～FR-5）|
| 我的 | 5（最右）| ❌ 触发 FR-0C | 个人信息、宠物状态设置、账号管理 |

**后果：**
- [+] 按钮视觉上凸起于 Tab Bar，区别于其他4个平级 Tab
- 未登录用户点击成长档案 / [+] / 问诊 / 我的，触发登录引导弹窗（FR-0C），登录完成后自动跳转至目标 Tab
- Tab Bar 在所有一级页面持续显示，进入二级页面（如问诊详情、档案编辑）时隐藏
- **角标规则**：「问诊」Tab 上显示红点，当有兽医回复消息未读时；铃铛角标显示所有未读通知总数（兽医回复 + 互动通知）；两者同时显示，不互斥

**问诊进行中切换 Tab 的行为：**
- 用户在兽医咨询对话中点击其他 Tab，会话在后台保持连接，**不中断、不关闭**
- 「问诊」Tab 上的红点持续显示，提示用户有未读消息
- 用户切回「问诊」Tab，直接恢复至对话界面，消息连续无中断

**兽医账号被运营封禁时的处理：**
- 运营后台封禁某兽医账号后，该账号立即无法登录
- 若封禁时有进行中的会话：用户侧收到系统消息「兽医已临时下线，本次问诊已中断，请重新发起咨询」
- 中断的会话进入用户问诊历史，标记为「已中断」
- 用户可立即重新发起新的问诊请求

### 4.6 个人中心（我的）

**功能描述：** 「我的」Tab 是用户账号信息与设置的统一入口，包含用户基本信息展示与编辑、宠物状态修改、我的发布历史、账号管理（退出登录 / 账号注销）和帮助反馈。宠物状态修改同时在「成长档案」Tab 设有快捷编辑入口，两处均可修改，效果同步。

**功能需求：**

#### FR-20：「我的」页面结构与内容

**页面区块：**

| 区块 | 内容 | 说明 |
|------|------|------|
| 用户信息 | 头像 + 昵称 + 编辑按钮 | 头像默认读取 Google 账户头像，可替换；昵称可修改（≤20字）|
| 宠物状态 | 展示当前状态（A/B/C）+ 修改入口 | 修改后 Feed 即时按新状态刷新（FR-17）；同步入口也在成长档案 Tab（FR-21）|

- 不做独立弹窗，直接复用 FR-0F 的状态选择界面

### 4.7 推送通知（Push Notifications）

**功能描述：** V1 支持三类通知：兽医回复推送、内容互动推送（点赞/评论）、App 版本更新提醒。前两类为系统推送（依赖用户授权），版本更新为 App 内提示（不需要推送权限）。推送权限申请在用户完成第一次问诊后触发，避免冷启动即要求权限导致用户直接拒绝。

**功能需求：**

#### FR-22A：兽医回复推送通知
兽医在问诊会话中发送回复时，向用户发送推送通知。

**后果：**
- 用户未在当前页面查看对话时，触发推送
- 通知内容：「你的宠物问诊有新回复，点击查看」
- 点击通知直接跳转至对应问诊会话
- 用户正在查看该对话时，不重复推送

#### FR-22B：内容互动推送通知
其他用户对当前用户发布的内容点赞或评论时，发送推送通知。

**后果：**
- 被点赞触发通知：「有人赞了你的 [内容标题/宠物名]」
- 被评论触发通知：「有人评论了你的内容，点击查看」
- 点击通知跳转至对应内容详情页
- V1 不做通知合并（单条逐条推送），V2 可考虑批量合并

#### FR-22C：App 版本更新提醒（App 内提示）
App 有新版本时，在 App 内弹出版本更新提示。**不使用系统推送，无需推送权限。**

**后果：**
- 每次冷启动时检测是否有新版本可用
- **推荐更新**（常规版本）：弹窗提示「发现新版本，建议更新」，可点击「稍后」跳过，下次启动继续提示
- **强制更新**（重大版本）：弹窗不可跳过，必须前往应用商店更新后才能继续使用
- 跳转目标：iOS → App Store，Android → Google Play 或应用内更新

#### FR-22E：兽医端新请求推送通知
有新问诊请求进入待接单队列时，向所有在线兽医推送通知。

**后果：**
- 触发条件：用户提交兽医咨询请求（AI 问诊不触发）
- 通知内容：「有新的问诊请求，点击查看」
- 推送对象：当前在线状态为「在线」（FR-32）的所有兽医账号
- 离线兽医不接收推送
- 点击通知 → 跳转兽医工作台「待接单」Tab

---

#### FR-22D：应用权限申请与拒绝处理

V1 需要申请的权限：相机、相册（照片库）、推送通知。三类权限的申请时机和拒绝处理规则如下。

**相机 / 相册权限：**

| 触发场景 | 申请时机 |
|---------|---------|
| AI 问诊上传照片（FR-1）| 用户点击「上传」时触发 |
| 兽医咨询发送图片/视频（FR-30）| 用户在聊天内点击「+」上传时触发 |
| 宠物档案头像上传（FR-11）| 用户点击头像编辑时触发 |
| 成长日历快乐时刻上传（FR-12）| 用户点击「+」发布时触发 |

- 用户**授权**：正常进入相机/相册选择
- 用户**拒绝**：弹出引导弹窗「需要相册权限才能上传，请前往设置开启」，附「去设置」跳转按钮和「取消」按钮
- 「去设置」跳转至系统 App 权限设置页（iOS: Settings > App Name；Android: 应用权限管理）

**推送通知权限：**
- **不在 App 首次启动时请求权限**（授权率极低）
- 在用户**完成第一次问诊**后请求：「开启通知，兽医回复时第一时间提醒你」
- 用户拒绝后不再主动弹起系统权限弹窗
- 可在「我的」页面引导用户手动前往设置开启

### 4.8 内容互动（点赞与评论）

**功能描述：** V1 支持对三类内容（日常分享、专业知识科普、成长日历快乐时刻）进行点赞和评论。评论支持两级结构：一级评论（直接回复内容）和二级评论（回复评论）。所有互动均需登录，未登录触发 FR-0C。V1 不做收藏、@提及功能。

**功能需求：**

#### FR-23：内容点赞
用户可对首页 Feed 中的任意内容进行点赞，点赞为开关状态（可取消）。

**后果：**
- 三类内容（日常分享、专业知识科普、成长日历快乐时刻）均支持点赞
- 点赞/取消点赞即时更新点赞数，显示在内容卡片上
- 点赞需登录，未登录触发 FR-0C 登录引导
- 点赞触发 FR-22B 推送通知给内容发布者（用户对自己内容点赞不通知）

#### FR-24：内容评论（两级结构）
用户可对内容发表评论，支持一级评论和二级评论（回复评论）。

**一级评论（直接评论内容）：**
- 任何登录用户均可发表
- 字数限制 ≤200 字
- 发布后即时显示，按时间正序排列
- 评论发布触发 FR-22B 推送通知给内容发布者

**二级评论（回复评论）：**
- 对一级评论进行回复，显示在该一级评论下方
- 字数限制 ≤200 字
- 二级评论下不再支持继续嵌套（最多两级）
- 回复触发 FR-22B 推送通知给被回复的评论者

**删除权限：**
- 用户可删除自己发表的评论（任意级别）
- 内容发布者可删除自己内容下的任意评论
- 删除一级评论时，该评论下所有二级评论一并删除

**后果（共同）：**
- 评论数量显示在内容卡片上
- 评论需登录，未登录触发 FR-0C 登录引导
- V1 不做对评论点赞功能

#### FR-36：内容删除（无编辑功能）
用户可删除自己发布的内容，V1 不支持编辑已发布内容。

**后果：**
- 删除入口：内容详情页「···」菜单 → 「删除」（仅内容作者可见此选项）
- 删除前显示确认弹窗：「删除后内容将永久移除，无法恢复」
- 删除后内容从 Feed 和成长档案时间线同步移除
- 其他用户对该内容的点赞和评论数据一并删除
- V1 不做内容编辑；需修改只能删除后重新发布

### 4.9 内容详情页（Content Detail Page）

**功能描述：** 用户在首页 Feed 点击内容卡片后，全屏跳转至内容详情页，展示完整内容、作者信息、互动数据及评论列表。返回按钮（左上角）回到 Feed，保持 Feed 滚动位置。

**功能需求：**

#### FR-28：内容详情页结构

**页面区块（从上至下）：**

| 区块 | 内容 |
|------|------|
| 顶部导航栏 | 左：返回按钮；右：「···」菜单（含举报入口 FR-25）|
| 作者信息 | 头像 + 昵称，点击触发 FR-26 迷你主页预览卡 |
| 内容正文 | 完整文字内容 |
| 媒体内容 | 图片（多张可左右滑动查看）；V1 不含社区视频展示 |
| 互动栏 | 点赞按钮 + 点赞数 / 评论数（点击评论数跳转至评论区）|
| 评论区 | 一级评论列表（时间正序）+ 各级回复（FR-24）|
| 评论输入框 | 固定在页面底部，点击展开键盘，支持发布一级评论 |

**后果：**
- Feed 中点击内容卡片全屏跳转，左上角返回箭头回到 Feed，**保持 Feed 原滚动位置**（不回顶）
- 图片多于1张时，支持左右滑动，角标显示「当前页/总页数」（如「2/5」）
- V1 详情页无社区视频展示（内容发布不支持视频，V2 规划）
- 评论输入框固定底部，未登录时点击触发 FR-0C 登录引导
- **评论区加载规则**：首次加载10条一级评论；超出10条时显示「查看更多评论」按钮，点击每次加载10条；二级评论默认展示前3条，超过3条显示「查看全部 X 条回复」

### 4.10 内容举报与审核（Content Reporting）

**功能描述：** V1 支持对 Feed 中的内容进行举报。举报进入运营人工审核队列，由运营团队处理。V1 不做评论举报和用户举报，不做关键词自动过滤。

**功能需求：**

#### FR-25：内容举报
用户可对首页 Feed 中的内容发起举报。

**后果：**
- 举报入口：内容卡片右上角「···」菜单内（不占主界面空间）
- 举报类型选项（单选）：违法违规 / 虚假信息 / 不当内容 / 骚扰 / 其他
- 提交后：用户侧显示「已收到你的举报，我们会尽快处理」
- 举报数据进入运营审核队列，由运营团队人工处理
- V1 不做自动下架，所有处理决定由运营人工做出
- V1 不做对评论的举报、不做对用户账号的举报

`[ASSUMPTION: 运营团队有内容审核后台可查看和处理举报工单]`

### 4.11 其他用户头像点击（迷你主页预览卡）

**功能描述：** V1 不做其他用户的完整公开主页。用户点击 Feed 内容中的其他人头像/昵称时，弹出迷你主页预览卡，展示有限信息。设计目标：让用户感受到「主页正在精心筹备」，而非「功能不存在」。

**功能需求：**

#### FR-26：其他用户迷你主页预览卡
用户在 Feed 中点击他人头像或昵称时，从底部弹出迷你主页卡片。

**卡片内容：**
- 用户头像 + 昵称
- 已发布内容数量（「TA 发布了 X 条内容」）
- 文案：「TA 的专属主页正在精心筹备中，更多精彩即将呈现 ✨」
- 关闭按钮（点击卡片外部或右上角「×」关闭）

**后果：**
- 不展示「关注」按钮（V2 功能），不展示「查看主页」按钮
- 文案措辞不得出现「功能开发中」「敬请期待」「暂不支持」等技术性表达
- 卡片为底部半屏弹出样式（bottom sheet），不跳转新页面
- V2 升级路径：将此卡片直接升级为完整个人主页跳转入口，无需改动 Feed 层触发逻辑

### 4.12 兽医端（Vet Workbench）

**功能描述：** App 内存在两种账号类型：普通用户（Google 登录）和兽医（账号密码登录）。兽医账号由运营后台统一创建并发放凭证。兽医登录后进入独立工作台界面，拥有与普通用户完全不同的 Tab Bar，不可访问用户侧功能。

---

#### FR-29：兽医账号登录入口
兽医通过独立入口使用账号密码登录，与用户侧 Google 登录流程完全隔离。

**后果：**
- 登录页 Google 按钮下方显示小字链接：「兽医登录」
- 点击跳转至账号密码登录页（独立页面）
- 输入框：账号（邮箱/用户名）+ 密码；无「忘记密码」V1 暂不做（由运营重置）
- 登录成功后，系统识别账号类型为兽医，跳转至兽医工作台（FR-30）
- 兽医账号不可访问用户侧首页、成长档案等功能

---

#### FR-30：兽医工作台（独立 Tab Bar）
兽医登录后进入专属工作台，Tab Bar 与用户侧完全不同。

**兽医 Tab Bar（从左至右）：**

```
待接单  |  进行中  |  历史记录  |  我的
```

| Tab | 内容 |
|-----|------|
| 待接单 | 用户提交的待接问诊请求列表，兽医可接受或忽略 |
| 进行中 | 当前进行中的会话列表，点击进入对话界面 |
| 历史记录 | 已结束的问诊会话记录 |
| 我的 | 兽医个人信息 + **在线/离线状态切换**（影响 FR-4B 用户侧显示）|

**接单流程：**
1. 用户提交问诊请求（FR-4A / FR-4B）→ 请求进入待接单队列
2. 兽医收到推送通知「有新问诊请求」（复用 FR-22 推送框架）
3. 兽医在「待接单」Tab 查看请求摘要（用户描述 + 上传的照片）
4. 兽医点击「接受」→ 会话创建，自动移入「进行中」Tab
5. 用户侧收到通知「兽医已接受你的问诊，点击开始对话」

**对话界面（进行中）：**
- 图文聊天，支持发送文字、图片、视频（经腾讯云 IM 传输，视频 ≤ 60 秒）
- 兽医侧展示 FR-5 辅助工具（症状历史数据库、AI 参考回复、免责提示）
- 对话框顶部显示用户上传的原始描述和媒体文件

---

#### FR-31：问诊会话结束机制（含用户评分确认门）
兽医主动关闭会话，但会话不立即终止——需要用户评分确认，或等待30分钟保护窗口结束后自动关闭。此设计防止兽医过早结束对话、服务质量不达标。

**结束流程：**

```
兽医点击「结束会话」
        ↓
兽医侧弹窗确认「确认结束？」
        ↓
用户侧收到通知「兽医已完成本次问诊，请为本次服务评分」
        ↓
    ┌──────────────────────────────┐
    │ 用户立即评分（30分钟内）        │  → 会话立即关闭，进入档案存储（FR-16）
    │ 用户未评分（超过30分钟）        │  → 会话自动关闭，记录为「未评分」
    │ 用户在30分钟内继续发消息        │  → 兽医收到通知，可继续回复
    └──────────────────────────────┘
```

**后果：**
- 兽医触发结束后，会话进入「待关闭」状态（非立即关闭）
- 用户在30分钟保护窗口内**仍可发送消息**，兽医收到推送可选择继续回复
- 30分钟内用户完成评分 → 会话正式关闭
- 30分钟内用户未评分 → 会话自动关闭，评分标记为「未评分」
- 会话关闭后触发 FR-16（档案存储弹窗）和 FR-33（若未评分则再弹评分请求一次）

---

#### FR-32：兽医在线状态管理
兽医可在「我的」Tab 手动切换在线/离线状态，影响用户侧对兽医可用性的感知（FR-4B）。

**后果：**
- 在线状态时：可接收新问诊请求，出现在可接单队列
- 离线状态时：不接收新请求，用户侧问诊入口显示「当前暂无兽医在线」
- 状态切换即时生效

### 4.13 兽医评分（Vet Rating）

#### FR-33：问诊结束后用户评分
用户在问诊结束后对兽医服务进行评分，作为会话正式关闭的确认门。

**评分界面：**
- 1-5 星评分（必填）
- 可选文字评价（≤100字，选填）
- 确认按钮：「提交评分」

**后果：**
- 评分数据记录至兽医账号，运营后台可查看各兽医历史评分和平均分
- V1 评分结果不对普通用户公开展示（仅运营可见）
- 若用户超时未评分（30分钟），系统关闭会话后，用户**下次进入问诊页面**时弹出补评分弹窗一次；若仍未评分，该次问诊不再要求评分，记录为「未评分」
- 兽医端「历史记录」Tab 可查看每次问诊的用户评分

---

### 4.14 通知中心与问诊历史

#### FR-38：推送通知深链接规则
用户点击推送通知后，App 必须直接跳转至对应内容页面，不得仅跳转首页。

| 推送类型 | 点击后跳转目标 |
|---------|-------------|
| 兽医已回复（FR-22A）| 问诊 Tab → 对应进行中会话 |
| 问诊已结束（FR-31）| 问诊 Tab → 评分弹窗（FR-33）|
| 被点赞（FR-22B）| 内容详情页（FR-28）|
| 被评论（FR-22B）| 内容详情页（FR-28）→ 定位至评论区 |

**后果：**
- App 未启动时点击通知：冷启动后直接跳转目标页（不经过首页）
- App 在后台时点击通知：切到前台后直接跳转目标页
- App 已在前台时：通知以 Banner 形式展示，点击 Banner 跳转目标页

---

#### FR-34：全局通知中心（铃铛图标）
首页顶部导航栏右侧设置铃铛图标，汇聚所有系统通知。

**后果：**
- 铃铛图标位于首页顶部导航栏右侧
- 有未读通知时显示红色数字角标
- 点击进入通知列表页，按时间倒序展示所有通知：
  - 兽医已回复（跳转至对应问诊会话）
  - 问诊已结束（跳转至评分界面）
  - 被点赞（跳转至对应内容详情页）
  - 被评论（跳转至对应内容详情页）
- 通知条目点击后标记为已读

#### FR-35：问诊 Tab 结构（含历史记录）
「问诊」Tab 内不仅有功能入口，还整合用户的问诊历史记录。

**问诊 Tab 页面结构（从上至下）：**

| 区块 | 内容 |
|------|------|
| 功能入口区 | AI 问诊入口（FR-4A）+ 兽医咨询入口（FR-4B），平级展示 |
| 进行中 | 若有进行中的问诊会话，显示会话卡片（点击进入对话）|
| 我的问诊历史 | 所有历史问诊记录列表，含 AI 问诊结果和兽医咨询记录 |

**历史记录条目包含：**
- 日期 + 问诊类型（AI / 兽医）
- AI 问诊：危险等级（绿/黄/红）+ 症状摘要
- 兽医问诊：兽医昵称 + 会话摘要 + 用户评分
- 是否已存入档案的标记

**后果：**
- 用户无论是否将记录存入档案（FR-16），问诊历史均保留在此列表
- 关闭 FR-16 不影响此处的历史可见性（档案存入是额外操作，不是唯一留存路径）

## 7. 成功指标

V1 核心目标：用户规模积累 + 功能使用验证。无收入目标。

**Primary（主要指标）**

- **SM-1：新增用户数（New Users）**——V1 上线后3个月内新增达到 [待定] 用户。验证 FR-1、FR-17、FR-11。
- **SM-2：宠物名片分享→下载转化率**——分享链接点击后下载 App 的转化率 ≥ [待定]%。验证 FR-14、FR-15（核心病毒拉新飞轮）。
- **SM-3.1：AI 问诊点击转化率**——进入问诊模块后选择 AI 问诊的用户比例。验证 FR-4A。
- **SM-3.2：兽医咨询点击转化率**——进入问诊模块后选择兽医咨询的用户比例。验证 FR-4B。

**Secondary（次要指标）**

- **SM-4：成长档案内容发布频率**——有档案用户的平均月发布次数 > 2次。验证留存锚点设计（FR-12）。
- **SM-5：问诊记录存档率**——问诊结束后用户选择存入档案的比例。验证跨功能连接价值（FR-16）。

**Counter-metrics（不可优化的反向指标）**

- **SM-C1：兽医咨询响应质量**——V1 阶段观察实际数据，不设硬性目标；不可为拉高 SM-3.2 而牺牲响应质量。
- **SM-C2：月卸载率**——不可以牺牲留存为代价换取拉新，卸载率须控制在 [待定]% 以下。

## 附录 C：约束与护栏（Constraints & Guardrails）

**医疗责任边界：**
平台定位为工具提供方，不承担医疗建议的法律责任。所有 AI 分诊结果和兽医回复须附有免责声明（小号字体，不干扰主内容），明确告知用户「本建议为参考，最终决策权归属您」。红色评级不展示兽医咨询入口，避免平台在紧急场景下被视为替代急诊的工具。

**数据隐私：**
用户宠物健康数据和问诊记录不对第三方出售。⏭️ **PDP 合规 V1 暂不处理**（见 §8.8）；TECH_FRAMEWORK 已标注核心数据落德国的跨境出境风险，留待合规专项。

**内容审核：**
V1 UGC 内容（成长档案发布、评论）采用**人工审核**——运营人工处理用户举报队列（FR-25）。V1 不做机器/关键词自动过滤。（攒局群聊已移 V2，本期不涉及。）
