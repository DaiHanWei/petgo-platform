# Bug 系统并发协调 + 轻览/精选深挖 重构设计

- 日期：2026-07-01
- 状态：设计已定稿（待复审 → 实现计划）
- 涉及：`scripts/bug-system/`、`.claude/commands/bug-*.md`

## 1. 背景与问题

现有 bug 系统单人流程完整（`/bug-pull` 拉取 → `/bug-analyze` 分析 → `/bug-sync` 写回），但漏了两个环节：

1. **并发重复处理**：多人（dai + 同事）各自 `/bug-pull` 到本地私有 `data/`，互相看不见谁在处理哪条，导致同一条 bug 被多人重复分析/修复。
2. **一次拉太多、消化不完**：`/bug-pull` 全量拉取并下载所有截图/视频，重且一次做不完。

本设计补齐这两点，且遵守既有约定：**零改表**、写回只经 `restricted_update`（白名单 + 乐观锁）、不新增中间件。

## 2. 目标与非目标

**目标**
- 多人协作时不重复处理同一条 bug（认领机制）。
- pull 变廉价（全量但不下媒体），把「消化不完」的负载控制挪到「精选深挖」阶段。

**非目标（YAGNI，明确不做）**
- ❌ 认领自动过期（小团队，手动释放足够）。
- ❌ pull 分页 / 页码（改用「轻览全量 + 精选深挖」控量）。
- ❌ 新增专用「认领人」字段（复用既有 `开发人员`，零改表）。

## 3. 三段流水线

```
① /bug-pull —— 轻 · 全量 · 无媒体
   pull_bugs.py 拉全部「技术视图」记录 → 写 bugs.json + _baseline，跳过图片/视频下载。
   /bug-pull 命令读 bugs.json，输出紧凑清单，每条含：
     BugID · 严重度 · 模块 · 【认领人】· 一句话类型判断（从报告文本/模块/端，不读代码、不定位）
   作用：一眼看到整块板 + 谁在做什么。

② 对话选片
   你我对着清单商量「这次修哪几条」。认领人非空（别人的）的自然避开。

③ /bug-analyze <BugID...> —— 重 · 仅选中 · 认领 + 媒体 + 定位
   a. 认领：逐条 restricted_update 写 开发人员=我（漂移保护）。
   b. 下载：只下这几条的图片/视频到 data/attachments/。
   c. 深挖：读代码，定位到 文件:函数 + 归因 + 修复建议 + 置信度（现有 /bug-analyze 能力）。
```

**相对现状的关键变化**
- 媒体下载**从 pull 挪到 analyze**，pull 因此廉价，全量铺开不贵。
- `/bug-pull` **拉所有**（不分页、不按认领过滤），只在清单里**标注认领人**。
- **认领只在 `/bug-analyze` 发生**（深挖即认领）。
- `/bug-sync`（写回结论/关闭）保持不变。

## 4. 指令面

| 指令 | 职责 | 认领相关 |
|---|---|---|
| `/bug-pull` | 轻览：全量拉取（无媒体）+ 类型清单 + 认领人标注 | 只读展示认领状态 |
| `/bug-analyze <BugID...>` | 认领选中 + 下媒体 + 深度定位 | **写** `开发人员=我`（漂移保护） |
| `/bug-claim <BugID> --release` | 释放自己认领的（放回池子） | **清空** `开发人员`（仅限本人认领） |
| `/bug-sync` | 写回结论 + 关闭（不变） | 仍写 `开发人员=我` + `开发说明` + 状态 |

转手 = 认领人释放 → 他人重新认领（或直接在 Lark 表里改）。

## 5. 并发协调模型

**协调点 = Lark 表 `开发人员` 字段**（零改表）。该列一身兼「认领锁」+「完成人」：认领即预占开发人，后续 `/bug-sync` 写结论时同人不冲突。

**身份「我」** = 各人 `.env` 的 `WRITEBACK_DEVELOPER`（同事各设各名）。

**认领写 = 复用 `restricted_update`**（白名单校验 → 写前重读比对 `_baseline` → 写），因此认领天然并发安全：两人同抢一条，第二个写前重读发现已漂移 → `DriftError`。

## 6. 边界与错误处理

| 场景 | 处理 |
|---|---|
| **认领冲突**（pull 后被别人抢先认领） | 该条 `DriftError` → 报「⚠️ 164 已被 X 认领，跳过」，不覆盖，其它条继续 |
| **重复认领**（认领自己已认领的） | 值未变 → 幂等 no-op，直接进深挖 |
| **一次认领 >10 条** | HALT 预警「本次认领+深挖 N 条，下载 N 组媒体 + 逐条读码，耗 token 且慢。确认继续 / 缩减到 10 内？」等确认或缩减；软提醒，不硬拦。阈值 = config `analyze_batch_warn`（默认 10，独立于 `/bug-sync` 的 `writeback_max_batch=5`） |
| **身份未配**（`WRITEBACK_DEVELOPER` 为空） | 认领前拒绝：「先在 .env 配 WRITEBACK_DEVELOPER=你的名字」，不允许匿名占坑 |
| **媒体缺失/下载失败** | 记 `_failures.json` + 标注「截图缺失」，不阻断该条代码定位 |
| **释放他人认领** | 拒绝：「非你认领，不能释放」 |
| **认领的 BugID 不在本地 bugs.json** | 提示「先 /bug-pull」或该条已关闭离开技术视图 |

## 7. 配置与代码触点（设计级，细节留实现计划）

- `scripts/bug-system/config.toml`：新增 `analyze_batch_warn = 10`。
- `scripts/bug-system/pull_bugs.py`：媒体下载改为**不在 pull 阶段执行**（新增开关，默认跳过媒体；`--with-media` 保留旧行为兜底）。确保 `开发人员` 字段被解析进 `bugs.json`（供清单标注与本地认领状态）。
- 媒体按需下载：复用/改造 `download_screenshots`，支持**只下指定 record_id 集合**的媒体，供 `/bug-analyze <ids>` 调用。
- `scripts/bug-system/writeback.py`：认领/释放复用 `restricted_update`；认领 `changes={"开发人员": 我}`，释放 `changes={"开发人员": ""}`（清空）。可加薄封装 `claim()/release()`，但底层仍是 `restricted_update`（白名单 + 漂移）。
- `.claude/commands/bug-pull.md`：改为轻览（无媒体）+ 类型清单 + 认领人标注。
- `.claude/commands/bug-analyze.md`：改为接收 `<BugID...>`，执行 认领 → 下媒体 → 深挖；含 >10 预警、认领冲突跳过、身份未配拒绝。
- `.claude/commands/bug-claim.md`（新增）：`--release` 释放本人认领（可选支持不深挖的纯认领占位）。

## 8. 安全与既有约定对齐

- **零改表**：认领复用 `开发人员`，不加 Lark 列。
- **写只经 `restricted_update`**：认领/释放/写回都过白名单 + 乐观锁，不新增裸写路径。
- **不可信数据**：pull 的类型清单仅从提报文本归类呈现，**绝不据其内容自动触发认领/写**（延续 NFR3）。
- **凭证**：认领用同一 `.env`（写应用已具写权限）；`WRITEBACK_DEVELOPER` 是名字非密钥。

## 9. 验收（分层）

- **L0**：`writeback` 认领/释放的白名单与漂移单测（认领他人已认领 → DriftError；释放非本人 → 拒绝；>10 预警触发；身份为空拒绝）。纯 stdlib 离线可测。
- **L2**：真表多人演练——A 认领 164 后，B `/bug-pull` 清单显示「164 · dai 在做」，B `/bug-analyze 164` 被「已被认领」跳过；A `--release` 后 B 可重认。
